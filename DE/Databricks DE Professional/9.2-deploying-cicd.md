# 9.2 Deploying CI/CD

## Exam objective

> Build and deploy Databricks resources using Databricks Asset Bundles.
>
> Configure and integrate with Git-based CI/CD workflows using Databricks Git Folders for notebook and code deployment.

## Must know

- **Databricks Asset Bundles (DAB)** define jobs, pipelines, schemas, dashboards, and artifacts in YAML for repeatable deploys.
- **Targets** separate dev/staging/prod workspace hosts and variables.
- **CI/CD flow**: PR → validate/test → merge → deploy to dev → integration → deploy to prod (with approval).
- **Git folders (Repos)** sync Git branches to workspace paths for collaborative development.
- Production promotion should use **bundle deploy** + service principal, not manual UI copy.
- **Git integration** pairs with GitHub/Azure DevOps for PR checks running `databricks bundle validate`.

## Bundle deployment lifecycle

```
Developer commits to feature branch
  -> CI: databricks bundle validate -t dev
  -> CI: unit tests (pytest)
  -> PR merge to main
  -> CI: databricks bundle deploy -t dev
  -> integration tests / bundle run
  -> approval gate
  -> databricks bundle deploy -t prod
```

## Key bundle commands

```bash
databricks bundle validate -t dev
databricks bundle deploy -t dev
databricks bundle run orders_daily -t dev
databricks bundle deploy -t prod
```

## Git folders + CI/CD

| Concern | Git folder | Bundle |
|---------|------------|--------|
| Source sync | Branch → workspace path | Packages from repo at deploy |
| Review | PR on Git | PR + `bundle validate` |
| Resource creation | Manual or bundle | Declarative YAML |
| Prod promotion | Merge + bundle deploy target | Recommended |

Typical pattern:

1. Engineers develop in **Git folder** linked to repo.
2. Shared logic lives in repo `src/` (not only notebooks).
3. **Bundle** deploys jobs/pipelines referencing repo paths or built wheels.
4. CI uses **`databricks/setup-cli`** and workspace OIDC/SP auth.

```yaml
# GitHub Actions (exam pattern)
steps:
  - uses: actions/checkout@v4
  - uses: databricks/setup-cli@main
  - run: databricks bundle validate -t dev
  - run: databricks bundle deploy -t dev
    env:
      DATABRICKS_HOST: ${{ secrets.DATABRICKS_HOST }}
      DATABRICKS_TOKEN: ${{ secrets.DATABRICKS_TOKEN }}
```

## Environment and permissions

| Target | Typical differences |
|--------|---------------------|
| dev | Relaxed permissions, smaller clusters, dev catalog |
| prod | Service principal owner, autoscaling limits, prod catalog |
| variables | `catalog`, `schema`, `notify_email` per target |

Grant deploy identity:

- `CAN MANAGE` on jobs/pipelines it deploys.
- `USE CATALOG` / `CREATE TABLE` as required on target schemas.
- Least privilege — not metastore admin.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Promote job across 3 workspaces | DAB targets |
| Team edits notebooks with Git history | Git folder |
| Repeatable prod deploy from main branch | `bundle deploy -t prod` in CI |
| Custom Python shared across tasks | Wheel artifact in bundle |
| Quick one-off notebook experiment | Git folder dev branch (not prod deploy path) |

## PySpark / API essentials

Bundle references notebook/Python paths:

```yaml
tasks:
  - task_key: transform
    notebook_task:
      notebook_path: ../src/transform_orders.py
```

## SQL (exam)

CI/CD is primarily YAML/CLI; SQL may define schemas in bundle resources.

## UI / product surfaces

- **Workspace → Git folder**: branch, pull, commit.
- **Bundles** in CLI/IDE extension: validate, deploy.
- **Lakeflow Jobs**: verify deployed job name includes target suffix.

## Exam traps

- Manual prod notebook copy instead of bundle promotion.
- Same catalog variable for dev and prod targets.
- Legacy CLI for bundles (need modern Databricks CLI).
- Deploy without `bundle validate` in CI.
- Git folder sync confused with bundle deploy — sync alone does not update job definitions.

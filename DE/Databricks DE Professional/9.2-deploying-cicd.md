# 9.2 Deploying CI/CD

## Exam objective

> Build and deploy Databricks resources using Databricks Asset Bundles.
>
> Configure and integrate with Git-based CI/CD workflows using Databricks Git Folders for notebook and code deployment.

## Naming note

- The exam guide says **Databricks Asset Bundles (DABs)**. Current docs increasingly say **Declarative Automation Bundles**, formerly Databricks Asset Bundles. Treat them as the same bundle concept for the exam.
- **Git folders** are the current name for Databricks workspace Git integration. Older docs and UI references may say Repos.
- Current Databricks docs use **Lakeflow Jobs** and **Lakeflow Spark Declarative Pipelines (SDP)**.

## Must know

- Bundles define Databricks resources as code in `databricks.yml` and included YAML files.
- A bundle can deploy Lakeflow Jobs, Lakeflow SDP pipelines, schemas, volumes, dashboards, alerts, permissions, variables, and artifacts.
- **Targets** such as `dev`, `test`, `staging`, and `prod` let the same reviewed commit deploy with environment-specific workspace hosts, variables, permissions, and identities.
- CI/CD should run tests and `databricks bundle validate` before deployment.
- Production deployments should use a service principal or workload identity, not a personal user's token.
- Production jobs should run as a controlled identity through `run_as`, usually a service principal.
- Git folders support collaborative development and Git-backed workspace editing, but syncing a Git folder is not the same as deploying bundle-managed resources.
- Bundle-managed resources should be changed through Git and redeployed. Manual UI edits can be overwritten by the next deploy.
- Use approval gates, environment protection, and target-specific variables to promote safely to production.
- `bundle deploy` creates or updates resources. `bundle run` executes a deployed workflow. `bundle destroy` removes deployed resources and should be used carefully.

## Professional focus vs Associate

Associate notes cover Git folders, variables, CLI commands, and bundle deployment basics. For Professional, focus on production deployment design:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know `databricks bundle deploy` | Build a CI/CD promotion flow with validation, tests, approval, and rollback plan |
| Know bundle targets exist | Separate workspaces, catalogs, schemas, compute, schedules, permissions, and run identities by target |
| Know Git folders sync code | Distinguish collaborative development from production deployment |
| Know modern CLI is required | Install/authenticate CLI safely in CI/CD and avoid long-lived personal tokens |
| Know variables override values | Prevent dev/prod cross-contamination through target-specific variables |
| Know resources are YAML | Manage drift, permissions, artifacts, and ownership as code |

## CI/CD architecture

```text
Feature branch
  -> developer tests locally / in dev Git folder
  -> pull request
  -> CI: lint + unit tests
  -> CI: databricks bundle validate -t dev
  -> merge to main
  -> CI/CD: databricks bundle deploy -t dev
  -> CI/CD: databricks bundle run -t dev validation_job
  -> integration tests / smoke checks
  -> approval gate
  -> CI/CD: databricks bundle deploy -t prod
  -> optional prod smoke check
  -> monitoring + alerts
```

Deployment principles:

| Principle | Correct production shape |
|-----------|--------------------------|
| Source control is the source of truth | Resource changes happen through PRs |
| Same commit promotes | Dev/test/prod differ by target config, not code edits |
| Validate before deploy | CI runs `bundle validate` and tests |
| Deploy with automation identity | CI authenticates as service principal/workload identity |
| Run with controlled identity | Jobs/pipelines use `run_as` where supported |
| Keep secrets out of Git | Use secret scopes, CI secrets, OIDC/workload identity |
| Avoid manual prod edits | Bundle deploy restores declared state |
| Make deployment observable | CI logs, bundle summary, job run result, alerts |

## Bundle project layout

```text
orders_bundle/
  databricks.yml
  resources/
    jobs.yml
    pipelines.yml
    permissions.yml
  src/
    orders_etl/
      transforms.py
      __init__.py
  tests/
    test_transforms.py
  pyproject.toml
```

Bundle root:

```yaml
bundle:
  name: orders_etl

include:
  - resources/*.yml

variables:
  catalog:
    description: Target Unity Catalog catalog
  schema:
    description: Target schema
  notify_email:
    description: Notification recipient

artifacts:
  default:
    type: whl
    path: .
    build: python -m build

targets:
  dev:
    mode: development
    default: true
    workspace:
      host: https://adb-dev.example.com
    variables:
      catalog: dev_main
      schema: orders_dev
      notify_email: data-eng-dev@example.com

  prod:
    mode: production
    workspace:
      host: https://adb-prod.example.com
    variables:
      catalog: prod_main
      schema: orders
      notify_email: data-eng-oncall@example.com
    run_as:
      service_principal_name: sp-orders-prod
```

Resource example:

```yaml
resources:
  jobs:
    orders_daily:
      name: orders_daily_${bundle.target}
      email_notifications:
        on_failure:
          - ${var.notify_email}
      tasks:
        - task_key: transform_orders
          python_wheel_task:
            package_name: orders_etl
            entry_point: transform_orders
            parameters:
              - --catalog
              - ${var.catalog}
              - --schema
              - ${var.schema}
          libraries:
            - whl: ../dist/*.whl
```

## Key bundle commands

| Command | Purpose |
|---------|---------|
| `databricks bundle validate -t dev` | Validate config, variables, and resource definitions |
| `databricks bundle deploy -t dev` | Create/update resources in target workspace |
| `databricks bundle run -t dev orders_daily` | Run a deployed workflow |
| `databricks bundle summary -t dev` | Inspect deployed resources and URLs |
| `databricks bundle destroy -t dev` | Remove deployed resources for a target |

Command sequence:

```bash
databricks bundle validate -t dev
databricks bundle deploy -t dev
databricks bundle run -t dev orders_daily
databricks bundle summary -t dev
databricks bundle deploy -t prod
```

Exam shortcuts:

```text
validate -> checks config
deploy   -> creates/updates resources
run      -> executes a workflow
summary  -> shows deployed resources
destroy  -> removes resources
```

## Git folders vs bundles

| Concern | Git folders | Bundles |
|---------|-------------|---------|
| Primary purpose | Workspace Git-backed development | Declarative deployment and promotion |
| Best for | Editing notebooks/files, branch workflow, collaborative dev | Jobs, pipelines, artifacts, permissions, environment config |
| Production resource updates | Not enough by itself | `bundle deploy` |
| Source of truth | Remote Git repo | Remote Git repo + bundle YAML |
| Common user | Developer | CI/CD pipeline and platform deployment |
| Risk | Syncing code but not resource definitions | Deploying wrong target or identity if misconfigured |

Professional rule: use Git folders for development workflow and bundles for repeatable environment promotion. A Git folder branch update does not automatically update a job definition unless the job is intentionally configured to use that code path or a bundle deploy updates the resource.

## Git-based workflow

| Stage | Recommended pattern |
|-------|---------------------|
| Individual development | User Git folder under workspace user path, feature branch |
| Shared integration | PR into main or release branch, CI validation |
| Dev deployment | Bundle deploy from main to dev target |
| Staging deployment | Same commit deployed to staging with production-like variables |
| Production deployment | Protected branch/environment approval, service principal deploy |
| Emergency fix | Hotfix branch, tests/validate, targeted deploy, post-incident PR hygiene |

Git folder patterns:

- Use remote Git as source of truth, not uncommitted workspace notebook changes.
- Avoid multiple users editing the same Git folder/branch at once.
- Keep production Git folders read-only or admin-managed if used for production notebook paths.
- Prefer bundle-deployed wheels or files for production logic rather than relying on mutable interactive state.

## CI/CD authentication and identities

| Identity | Use |
|----------|-----|
| Developer user | Local development and PR work |
| CI/CD service principal | Deploy bundles to workspaces |
| Job run-as service principal | Run production workflows |
| Workspace admin | Configure workspace-level settings, not routine deployment |
| Account admin | Manage account-level identity/governance, not routine deployment |

Authentication guidance:

- Prefer service principals, workload identity federation, or OIDC where supported.
- Avoid long-lived personal access tokens for production CI/CD.
- Store secrets in the CI secret manager or Databricks secrets, not in Git.
- For GitHub Actions workload identity federation, the workflow needs `id-token: write`, `DATABRICKS_AUTH_TYPE: github-oidc`, `DATABRICKS_HOST`, and the service principal `DATABRICKS_CLIENT_ID`.
- Grant deploy identity only the permissions needed to create/update its target resources.
- Grant run-as identity the data permissions needed to execute the workflow.
- Separate deploy permissions from runtime data permissions when possible.

Permission split:

| Need | Deploy identity | Run identity |
|------|-----------------|--------------|
| Create/update job | Can manage deployed job/resource | Not necessarily |
| Run production ETL | Not necessarily | UC read/write privileges |
| Deploy pipeline | Can manage pipeline resource | Pipeline run-as/owner has data privileges |
| Write target tables | Usually no unless deployment validates data | Yes, scoped to target schemas/tables |
| Manage prod catalog | Avoid broad admin | Avoid broad admin |

## Target configuration

Use targets to isolate environments.

| Setting | Dev | Prod |
|---------|-----|------|
| Workspace host | Dev workspace | Prod workspace |
| Catalog/schema | Dev catalog/schema | Prod catalog/schema |
| Compute | Smaller/cheaper, schedules paused | Production policies, schedules enabled |
| Permissions | Broader developer access | Least privilege, service principals |
| Notifications | Dev/team channel | On-call/prod channel |
| Mode | `development` | `production` |
| Run identity | Developer or dev SP | Prod service principal |

Target safeguards:

- Use different catalogs/schemas for dev and prod.
- Use prod-specific `run_as`.
- Use prod-specific notification destinations.
- Disable or pause schedules in dev if needed.
- Do not let dev deploys write to prod catalogs.
- Use target names in resource names only where helpful; avoid accidentally creating duplicate prod resources.

## Testing and validation in CI/CD

| Stage | What it catches |
|-------|-----------------|
| Lint/type checks | Syntax, style, import-level issues |
| Unit tests | Transform logic before deployment |
| `bundle validate` | Bundle config, variables, syntax, resource references |
| Artifact build | Wheel/package failures |
| Deploy to dev | Workspace permissions, resource creation, config resolution |
| Integration run | Runtime dependencies, UC grants, compute, source/target paths |
| Smoke query | Tables exist, row counts, quality expectations |
| Approval gate | Human control before prod deployment |

Example GitHub Actions shape:

```yaml
name: deploy-orders

on:
  pull_request:
  push:
    branches: [main]

permissions:
  id-token: write
  contents: read

jobs:
  validate:
    runs-on: ubuntu-latest
    environment: Dev
    env:
      DATABRICKS_AUTH_TYPE: github-oidc
      DATABRICKS_HOST: ${{ vars.DATABRICKS_DEV_HOST }}
      DATABRICKS_CLIENT_ID: ${{ secrets.DATABRICKS_DEV_CLIENT_ID }}
    steps:
      - uses: actions/checkout@v4
      - uses: databricks/setup-cli@main
      - run: python -m pip install -e .[test]
      - run: pytest
      - run: databricks bundle validate -t dev

  deploy-dev:
    if: github.ref == 'refs/heads/main'
    needs: validate
    runs-on: ubuntu-latest
    environment: Dev
    env:
      DATABRICKS_AUTH_TYPE: github-oidc
      DATABRICKS_HOST: ${{ vars.DATABRICKS_DEV_HOST }}
      DATABRICKS_CLIENT_ID: ${{ secrets.DATABRICKS_DEV_CLIENT_ID }}
    steps:
      - uses: actions/checkout@v4
      - uses: databricks/setup-cli@main
      - run: databricks bundle deploy -t dev
      - run: databricks bundle run -t dev orders_daily
```

For production, use a separate protected environment, approval gate, workspace host, and service principal client ID. A token-based example may appear in older docs, but the correct production approach is to prefer short-lived federated authentication when it is available.

## Promotion and release strategy

| Strategy | Use when |
|----------|----------|
| Trunk-based deploy | Frequent reviewed deployments from main |
| Release branch/tag | Regulated environments needing explicit release artifact |
| Dev -> staging -> prod | Need integration validation before prod |
| Blue/green style resources | Need safer cutover for critical workflows |
| Roll forward | Most code/config fixes |
| Roll back redeploy | Previous known-good commit is safer |

Promotion rule:

```text
Same commit + different bundle target = normal promotion.
Different code per environment = drift risk.
```

Rollback patterns:

| Problem | Response |
|---------|----------|
| Bad job config deployed | Revert YAML, redeploy previous commit |
| Bad code in wheel | Redeploy previous package/commit |
| Bad data written | Use table history/restore or corrective job if appropriate |
| Job failed after transient issue | Repair run after confirming upstream outputs are valid |
| Prod deploy partially completed | Inspect bundle summary/resources, rerun deploy after fix |

## Drift and state management

Bundle deployment treats bundle YAML as desired state for managed resources.

| Drift source | Impact | Correct response |
|--------------|--------|------------------|
| Manual UI edit to bundle-managed job | May be overwritten on next deploy | Change YAML and redeploy |
| Changing bundle name/target/root path | Can create separate resources or affect state | Treat identity fields carefully |
| Deploying wrong target | Writes resources to wrong workspace/catalog | Validate target and host before deploy |
| Manual permission changes | CI/CD may restore declared permissions | Manage permissions as code |
| Deleting a managed resource manually | Next deploy may recreate or fail depending on state | Use bundle deploy/summary to reconcile |

Use:

```bash
databricks bundle summary -t prod
```

to inspect deployed resources and URLs before or after deployment.

## Deployment failures and troubleshooting

| Symptom | Likely cause | Correct response |
|---------|--------------|------------------|
| `bundle validate` fails | YAML syntax, missing variable, bad reference | Fix config before deploy |
| Deploy auth fails | Missing/expired token, wrong host, wrong profile | Fix CI credentials/OIDC/profile |
| Resource create/update denied | Deploy identity lacks workspace/resource permission | Grant least privilege to deploy identity |
| Job runs but table access denied | Run-as identity lacks UC privileges | Grant data privileges to run-as SP |
| Wheel task import fails | Artifact not built/attached, package mismatch | Fix `artifacts` and task libraries |
| Dev deploy writes prod tables | Wrong target variables | Separate catalogs and validate variables |
| Manual UI changes disappear | Bundle is source of truth | Change bundle YAML |
| Production job runs as deployer | Missing `run_as` | Configure prod run identity |
| Git folder sync does not change job | Job definition not redeployed | Run `bundle deploy` |

## Git folders in CI/CD

Git folders can be part of CI/CD, but they solve a different problem than bundles.

Use Git folders for:

- Notebook/file development in workspace.
- Reviewing and committing notebook changes.
- Keeping shared workspace code synced to a branch.
- Development jobs that intentionally run code from a Git-backed path.

Use bundles for:

- Creating/updating jobs and pipelines.
- Target-specific deployment.
- Permissions, variables, schedules, notifications.
- Artifact builds and wheel-backed production code.
- Repeatable promotion and rollback.

Git folder automation often updates a workspace Git folder to a branch. Bundle automation deploys resources. The exam may test that these are related but not interchangeable.

## Production readiness checklist

| Area | Check |
|------|-------|
| Source control | All resource/code changes reviewed in PR |
| CLI | Modern Databricks CLI installed in CI |
| Validation | `bundle validate` runs for target |
| Tests | Unit tests and integration smoke tests pass |
| Identity | CI deploys as service principal/workload identity |
| Runtime identity | Production workflows use `run_as` service principal |
| Variables | Dev/prod catalogs, schemas, hosts, notifications differ |
| Permissions | Least privilege declared or reviewed |
| Artifacts | Wheel/package builds and attaches correctly |
| Schedules | Production schedules enabled; dev schedules controlled |
| Monitoring | Notifications and alerts configured |
| Rollback | Previous known-good commit can be redeployed |

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Promote same job through dev/test/prod | Bundle targets |
| Create/update jobs and pipelines as code | Bundle resources |
| Team edits notebooks with Git history | Git folder |
| Production deploy from main after approval | CI/CD `bundle deploy -t prod` |
| Shared Python logic across tasks | Wheel artifact in bundle |
| Validate config before PR merge | `databricks bundle validate` |
| Run deployed validation workflow | `databricks bundle run -t dev <job>` |
| Prod workflow must not run as human | `run_as` service principal |
| UI job change was overwritten | Update bundle YAML and redeploy |
| Remove dev resources after experiment | `bundle destroy -t dev`, carefully |

| Surface | Use it for |
|---------|------------|
| Git folders | Branch, pull, commit, review workspace code |
| Databricks CLI | Validate, deploy, run, summarize bundles |
| Lakeflow Jobs UI | Inspect deployed jobs, runs, permissions, schedules |
| Lakeflow Pipelines UI | Inspect deployed pipelines and update history |

| Workspace permissions | Git folder and job ACLs |

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Manually copying notebooks to production | Deploy reviewed code/resources with bundles |
| Treating Git folder sync as production resource deployment | Use `bundle deploy` to update jobs, pipelines, permissions, and artifacts |
| Deploying dev and prod from different code | Promote the same reviewed commit with target variables |
| Using the same catalog/schema variables for dev and prod | Separate target-specific catalogs/schemas |
| Running production jobs as the developer who deployed them | Configure `run_as` service principal |
| Storing tokens or secrets in bundle YAML | Use CI secrets, workload identity, or Databricks secrets |
| Skipping `bundle validate` in CI | Validate before deploying |
| Manually editing bundle-managed jobs in the UI | Change YAML in Git and redeploy |
| Assuming `bundle deploy` runs the workflow | Use `bundle run` or Jobs API to execute it |
| Using `bundle destroy` casually in production | Reserve destroy for intentional resource removal with approval |

# 10.1 Data modelling

## Exam objective

> Design and implement scalable data models using Delta Lake to manage large datasets.
>
> Simplify data layout decisions and optimize query performance using Liquid Clustering.
>
> Identify the benefits of using liquid Clustering over Partitioning and ZOrder.
>
> Design Dimensional Models for analytical workloads, ensuring efficient querying and aggregation.

## Must know

- **Delta Lake** is the physical layer for curated models: ACID, `MERGE`, time travel, CDF, layout optimization.
- **Medallion** (bronze/silver/gold) organizes refinement; **dimensional models** organize analytics (facts/dimensions).
- **Grain** = what one row represents; every fact table must have explicit grain.
- **Star schema**: fact table + denormalized dimensions — best default for lakehouse BI.
- **Snowflake**: normalized dimensions — fewer duplicates, more joins.
- **Liquid clustering** replaces many partition/Z-order decisions with `CLUSTER BY` on filter columns.
- **Partitioning**: low-cardinality columns only; high cardinality causes small files.
- **Z-order**: legacy; do not combine with liquid clustering on same table.

## Dimensional model components

| Object | Role | Example |
|--------|------|---------|
| Fact | Measurable events at a grain | `fact_orders` (one row per order line) |
| Dimension | Descriptive context | `dim_customer`, `dim_product` |
| Degenerate dimension | Fact grain identifier without own table | `order_number` on fact |
| Surrogate key | Stable warehouse key | `customer_sk` |
| SCD Type 2 | Track history with `valid_from`/`valid_to` | Customer address changes |

## Scalable Delta patterns

| Pattern | Use |
|---------|-----|
| Incremental fact load | `MERGE` or `APPLY CHANGES` on `order_id` |
| Append-only events | Streaming table or append to partitioned/clustered fact |
| Wide gold marts | Pre-join for BI or use materialized views |
| SCD2 dimensions | `MERGE` with `valid_to` close + new row insert |

```sql
-- Fact table with liquid clustering on common filters
CREATE TABLE main.gold.fact_orders (
  order_line_sk BIGINT,
  order_id STRING,
  customer_sk BIGINT,
  product_sk BIGINT,
  order_date DATE,
  amount DECIMAL(12,2)
)
USING DELTA
CLUSTER BY (order_date, customer_sk);
```

## Liquid clustering vs partitioning vs Z-order

| Feature | Benefit | Drawback |
|---------|---------|----------|
| Partitioning | Prunes directories on low-cardinality keys | Too many partitions → small files |
| Z-order | Multi-column colocation (legacy) | Maintenance overhead; incompatible with liquid |
| Liquid clustering | Adaptive layout, up to 4 keys, incremental OPTIMIZE | Exam: preferred for new large tables |

Exam answer for "benefits of liquid over partition/Z-order":

- No rigid directory-per-value explosion on high-cardinality columns.
- Keys can be changed with less pain than repartitioning.
- Incremental clustering on write/optimize.
- Works with predictive optimization on UC managed tables.

## Efficient querying and aggregation

- Match **cluster keys** to `WHERE` and `JOIN` columns (`order_date`, `region`).
- Pre-aggregate to **daily grain** in gold when dashboards never need line-level.
- Use **materialized views** for repeated heavy aggregates (see Domain 1.2).
- Broadcast **small dimensions** in PySpark joins.

```sql
-- Star schema join for BI
SELECT
  d.calendar_year,
  c.region,
  SUM(f.amount) AS revenue
FROM main.gold.fact_orders f
JOIN main.gold.dim_date d ON f.order_date = d.date_key
JOIN main.gold.dim_customer c ON f.customer_sk = c.customer_sk
WHERE f.order_date >= DATE '2026-01-01'
GROUP BY d.calendar_year, c.region;
```

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Executive dashboards on daily revenue | Fact + dimensions or gold aggregate table |
| Customer history with versions | SCD2 dimension + `MERGE` |
| Filter large fact on date + region | `CLUSTER BY (order_date, region)` |
| High-cardinality user_id partition | Liquid clustering, not partition |
| Repeated expensive aggregate | Materialized view |

## Exam traps

- Fact table grain unclear (duplicate metrics when joining dimensions).
- Partitioning on timestamp at second-level cardinality.
- Snowflake every dimension when star schema would meet BI needs with fewer joins.
- Z-order and liquid clustering on same table.
- Gold wide table with raw PII when dimensional model should use masked dimensions.

# 10.1 Data modelling

## Exam objective

> Design and implement scalable data models using Delta Lake to manage large datasets.
>
> Simplify data layout decisions and optimize query performance using Liquid Clustering.
>
> Identify the benefits of using liquid Clustering over Partitioning and ZOrder.
>
> Design Dimensional Models for analytical workloads, ensuring efficient querying and aggregation.

## Naming note

- The exam says **Liquid Clustering** and **ZOrder**. Current Databricks docs usually write **liquid clustering** and `ZORDER`.
- Current docs increasingly recommend liquid clustering for new tables instead of legacy partitioning and `ZORDER`.
- Current Lakeflow docs use **AUTO CDC** for change data capture. Older material and some exam-adjacent examples may still mention `APPLY CHANGES`.
- This note uses **data modelling** in the file title to match the existing note name. Databricks docs often use **data modeling**.

## Must know

- Delta Lake is the physical storage layer for scalable models: ACID transactions, schema enforcement/evolution, `MERGE`, time travel, Change Data Feed, deletion vectors, and file layout optimization.
- Medallion architecture organizes refinement: bronze captures source reality, silver standardizes and validates, and gold serves business-friendly facts, dimensions, aggregates, views, and materialized views.
- Dimensional modeling organizes analytical consumption. The most important design choice is the **grain**: what one row represents.
- A fact table stores measurements or events at a declared grain. A dimension table stores descriptive context.
- Star schema is the default BI-friendly pattern: a central fact table joined to denormalized dimensions.
- Snowflake schema normalizes dimensions into more tables. Use it when shared hierarchies, storage reduction, or governance justify extra joins.
- Surrogate keys make dimensions stable even when source natural keys change or are reused.
- SCD Type 1 overwrites history. SCD Type 2 preserves history with effective dates, current-row indicators, and versioned rows.
- Liquid clustering uses `CLUSTER BY` to organize data for skipping and query pruning. It replaces many partitioning and `ZORDER` decisions.
- Partitioning should be rare and deliberate. It is only a good fit for low-cardinality, frequently filtered columns with enough data per partition.
- Do not use liquid clustering and `ZORDER` on the same Delta table. They are alternative layout strategies.

## Professional focus vs Associate

Associate notes cover medallion basics, gold objects, joins, aggregates, and liquid clustering fundamentals. For Professional, focus on design choices that affect correctness, scalability, cost, and BI usability:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know bronze/silver/gold | Decide which business entities become facts, dimensions, marts, materialized views, or semantic outputs |
| Know joins and aggregates | Preserve grain so joins do not duplicate measures |
| Know gold tables and views | Choose table, view, materialized view, or aggregate table based on freshness, cost, and latency |
| Know liquid clustering exists | Select clustering keys from workload patterns and avoid legacy over-partitioning |
| Know `MERGE` supports upserts | Design idempotent incremental loads and SCD processing |
| Know Unity Catalog governs tables | Model access boundaries, PII exposure, lineage, and ownership into the gold design |

## Modeling workflow

A production model starts with business questions, not with table syntax.

```text
Business process
  -> metric definitions
  -> required grain
  -> facts and dimensions
  -> source-to-target mapping
  -> incremental load strategy
  -> physical layout
  -> governance and access pattern
  -> BI validation
```

Design checks:

| Question | Why it matters |
|----------|----------------|
| What business process is represented? | Sales, shipments, subscriptions, payments, support cases, and web events have different grains |
| What is one row? | Prevents duplicate metrics and ambiguous joins |
| Which measures are additive? | Determines whether dashboards can safely sum across dimensions |
| Which dimensions change over time? | Determines SCD Type 1 vs Type 2 |
| Which filters are common? | Drives clustering, aggregate tables, and materialized views |
| How fresh must the data be? | Drives batch table, streaming table, or materialized view refresh strategy |
| Who can see which fields? | Drives masking, row filters, separate marts, and grants |

## Databricks-specific modeling caveats

| Caveat | Correct design response |
|--------|-------------------------|
| Primary and foreign key constraints are informational, not enforced | Use them to document relationships and help optimization, but enforce uniqueness and referential quality in ETL/tests |
| Delta transactions are table-level | Design multi-table fact/dimension updates to tolerate partial progress, retries, and repair |
| Joins can dominate cost | Prefer star schemas and conformed dimensions over heavily normalized models for common BI paths |
| Statistics drive skipping and planning | Keep important filter/join columns eligible for statistics, especially clustering columns |
| Nested/semi-structured fields are flexible | Extract fields used for filtering, joining, grouping, governance, and BI into typed columns |

Primary and foreign key example:

```sql
CREATE TABLE main.gold.dim_product (
  product_sk BIGINT NOT NULL PRIMARY KEY,
  product_id STRING,
  product_name STRING,
  category STRING
);

CREATE TABLE main.gold.fact_order_line (
  order_id STRING,
  order_line_id STRING,
  product_sk BIGINT,
  net_amount DECIMAL(18,2),
  CONSTRAINT product_fk FOREIGN KEY (product_sk)
    REFERENCES main.gold.dim_product(product_sk)
);
```

The constraint documents the relationship. It does not replace checks that reject or quarantine fact rows whose `product_sk` is missing from the dimension.

## Grain

Grain is the contract for a table.

| Table | Grain example | Measures |
|-------|---------------|----------|
| `fact_order_line` | One row per order line | Quantity, gross amount, discount amount |
| `fact_payment` | One row per payment transaction | Payment amount, fee amount |
| `fact_daily_inventory` | One row per product, store, and date | Ending quantity, on-hand value |
| `agg_daily_revenue` | One row per date, region, and channel | Revenue, order count |
| `dim_customer` | One row per customer version for SCD2 | Usually no additive measures |

Fact grain rules:

- Declare the grain in the table comment or design document.
- Store facts at the lowest useful grain when downstream questions are not yet stable.
- Build aggregate tables or materialized views for repeated higher-grain queries.
- Do not mix grains in one fact table, such as order header measures and order line measures, unless the design is explicit and tested.
- Store degenerate dimensions, such as `order_number`, directly on the fact when they identify the transaction but do not require a separate dimension table.

## Facts and dimensions

| Object | Role | Common columns |
|--------|------|----------------|
| Fact table | Stores events or measurements | Surrogate foreign keys, event dates, numeric measures, source identifiers |
| Dimension table | Stores descriptive context | Surrogate key, natural key, attributes, effective dates, current flag |
| Conformed dimension | Shared dimension used by multiple facts | `dim_date`, `dim_customer`, `dim_product` |
| Degenerate dimension | Business identifier stored on the fact | `order_number`, `invoice_number` |
| Bridge table | Handles many-to-many relationships | Fact key, dimension key, allocation percentage |
| Aggregate table | Stores precomputed metrics | Grouping keys and summarized measures |

Simple star schema:

```text
              dim_date
                 |
dim_customer -- fact_order_line -- dim_product
                 |
              dim_store
```

Fact DDL example:

```sql
CREATE TABLE main.gold.fact_order_line (
  order_line_sk BIGINT GENERATED ALWAYS AS IDENTITY,
  order_id STRING COMMENT 'Degenerate dimension from source order header',
  order_line_id STRING,
  customer_sk BIGINT,
  product_sk BIGINT,
  store_sk BIGINT,
  order_date DATE,
  quantity INT,
  gross_amount DECIMAL(18,2),
  discount_amount DECIMAL(18,2),
  net_amount DECIMAL(18,2),
  source_system STRING,
  loaded_at TIMESTAMP
)
USING DELTA
CLUSTER BY (order_date, customer_sk)
COMMENT 'Grain: one row per source order line.';
```

Dimension DDL example:

```sql
CREATE TABLE main.gold.dim_customer (
  customer_sk BIGINT GENERATED ALWAYS AS IDENTITY,
  customer_id STRING,
  customer_name STRING,
  segment STRING,
  region STRING,
  effective_from TIMESTAMP,
  effective_to TIMESTAMP,
  is_current BOOLEAN,
  source_updated_at TIMESTAMP
)
USING DELTA
CLUSTER BY (customer_id, is_current)
COMMENT 'SCD2 customer dimension. Grain: one row per customer version.';
```

## Star vs snowflake vs wide marts

| Model | Best fit | Tradeoff |
|-------|----------|----------|
| Star schema | General BI, ad hoc slicing, shared metrics | Dimensions may repeat descriptive attributes |
| Snowflake schema | Complex hierarchies or shared normalized reference data | More joins and harder BI navigation |
| One Big Table / wide mart | High-volume dashboards with stable fields | Duplication, larger refresh cost, harder governance |
| Aggregate table | Repeated metrics at stable grain | Less flexible than atomic fact |
| Materialized view | Repeated expensive SQL where managed refresh is acceptable | Refresh behavior and limitations must fit workload |
| Semantic/metric layer | Governed metric definitions for BI users | Does not replace physical grain and layout design |

Professional default:

```text
Use star schema for flexible BI.
Add aggregate tables or materialized views for hot dashboard paths.
Use wide marts only when the access pattern is stable and the duplication is worth the speed.
```

## SCD design

Slowly changing dimensions preserve either current state or historical versions.

| Type | Behavior | Use when |
|------|----------|----------|
| SCD Type 0 | Never change after first load | Immutable attributes like original signup date |
| SCD Type 1 | Overwrite current row | Only current state matters |
| SCD Type 2 | Close old version and insert new version | Historical reporting needs point-in-time attributes |

SCD2 columns commonly include:

- Surrogate key, such as `customer_sk`.
- Natural key, such as `customer_id`.
- `effective_from` and `effective_to`.
- `is_current`.
- Source sequence or update timestamp.
- Optional hash of tracked attributes to detect changes.

Manual SCD2 merge shape:

```sql
MERGE INTO main.gold.dim_customer AS target
USING main.silver.customer_changes AS source
ON target.customer_id = source.customer_id
AND target.is_current = true
WHEN MATCHED AND (
  target.customer_name <> source.customer_name OR
  target.segment <> source.segment OR
  target.region <> source.region
) THEN UPDATE SET
  effective_to = source.source_updated_at,
  is_current = false;
```

The closing update is only half of the SCD2 process. A production flow must also insert the new current row after closing the previous version. In Lakeflow Spark Declarative Pipelines, prefer `AUTO CDC` when the source provides CDC or snapshot data and the requirement matches supported SCD behavior.

Lakeflow SQL shape:

```sql
CREATE OR REFRESH STREAMING TABLE dim_customer;

CREATE FLOW customer_cdc AS AUTO CDC INTO dim_customer
FROM stream(customer_changes)
KEYS (customer_id)
SEQUENCE BY source_updated_at
COLUMNS * EXCEPT (_change_type)
STORED AS SCD TYPE 2;
```

## Incremental fact loading

Fact tables usually need idempotent incremental loads.

| Source behavior | Correct loading pattern |
|-----------------|-------------------------|
| Append-only events | Append new rows, deduplicate by event key if needed |
| Updates to existing facts | `MERGE` by durable business key |
| Late-arriving facts | Merge into existing partition/cluster range and update affected aggregates |
| Deletes or cancellations | Represent as status/reversal facts, or process CDC deletes intentionally |
| Reprocessing/backfill | Parameterize date range and make writes idempotent |

Fact merge example:

```sql
MERGE INTO main.gold.fact_order_line AS target
USING main.silver.order_line_updates AS source
ON target.order_id = source.order_id
AND target.order_line_id = source.order_line_id
WHEN MATCHED THEN UPDATE SET
  customer_sk = source.customer_sk,
  product_sk = source.product_sk,
  store_sk = source.store_sk,
  order_date = source.order_date,
  quantity = source.quantity,
  gross_amount = source.gross_amount,
  discount_amount = source.discount_amount,
  net_amount = source.net_amount,
  loaded_at = current_timestamp()
WHEN NOT MATCHED THEN INSERT (
  order_id,
  order_line_id,
  customer_sk,
  product_sk,
  store_sk,
  order_date,
  quantity,
  gross_amount,
  discount_amount,
  net_amount,
  source_system,
  loaded_at
) VALUES (
  source.order_id,
  source.order_line_id,
  source.customer_sk,
  source.product_sk,
  source.store_sk,
  source.order_date,
  source.quantity,
  source.gross_amount,
  source.discount_amount,
  source.net_amount,
  source.source_system,
  current_timestamp()
);
```

Late-arriving dimension handling:

| Situation | Correct approach |
|-----------|------------------|
| Fact arrives before dimension row | Use an unknown surrogate key and repair later, or delay fact publish until dimension is available |
| Dimension attribute changes after fact date | Join to SCD2 version where fact event time falls between effective dates |
| Source reuses natural key | Use surrogate keys and source-system identifiers |
| Dimension correction affects history | Define whether correction rewrites history or creates a new SCD2 version |

Point-in-time dimension join:

```sql
SELECT
  f.order_id,
  f.order_date,
  c.customer_sk,
  c.segment,
  f.net_amount
FROM main.silver.orders f
JOIN main.gold.dim_customer c
  ON f.customer_id = c.customer_id
 AND f.order_ts >= c.effective_from
 AND f.order_ts < COALESCE(c.effective_to, TIMESTAMP '9999-12-31 00:00:00');
```

## Liquid clustering

Liquid clustering is the preferred layout strategy for most new large Delta tables on Databricks.

| Capability | Why it matters |
|------------|----------------|
| `CLUSTER BY` at table creation | Ties physical layout to common query predicates |
| Change clustering keys later | Allows layout to evolve as workloads change |
| Incremental `OPTIMIZE` | Reduces maintenance cost compared with full rewrites |
| Works for high-cardinality filters | Avoids directory explosion from partitioning |
| Applies to streaming tables and materialized views where supported | Keeps modern Lakeflow outputs optimizable |
| `CLUSTER BY AUTO` on supported UC managed Delta tables | Lets Databricks select keys based on workload patterns |

Create with manual keys:

```sql
CREATE TABLE main.gold.fact_clickstream (
  event_id STRING,
  user_id STRING,
  event_date DATE,
  event_ts TIMESTAMP,
  page STRING,
  campaign_id STRING
)
USING DELTA
CLUSTER BY (event_date, user_id);
```

Create with automatic liquid clustering:

```sql
CREATE OR REPLACE TABLE main.gold.fact_clickstream_auto (
  event_id STRING,
  user_id STRING,
  event_date DATE,
  event_ts TIMESTAMP,
  page STRING,
  campaign_id STRING
)
USING DELTA
CLUSTER BY AUTO;
```

Change keys:

```sql
ALTER TABLE main.gold.fact_clickstream
CLUSTER BY (event_date, campaign_id);
```

Run layout maintenance:

```sql
OPTIMIZE main.gold.fact_clickstream;
```

Force reclustering after a major key change when required:

```sql
OPTIMIZE main.gold.fact_clickstream FULL;
```

Key selection guidance:

| Query pattern | Candidate cluster key |
|---------------|-----------------------|
| Most queries filter by date range | Date or timestamp column |
| BI filters by customer/account | Customer or account key |
| Queries filter by region and date | Region and date if both are selective |
| Queries filter by high-cardinality ID | High-cardinality ID can fit liquid clustering better than partitioning |
| Columns are highly correlated | Usually choose one, not both |
| Workload changes often | Consider `CLUSTER BY AUTO` where supported |

## Liquid clustering vs partitioning vs ZORDER

| Layout choice | Best fit | What is wrong when misused |
|---------------|----------|----------------------------|
| Liquid clustering | New large tables, high-cardinality filters, skewed data, changing workloads | Treating it as magic without matching keys to query predicates |
| Partitioning | Low-cardinality columns with large data per partition and frequent pruning | Partitioning by user ID, timestamp, or other high-cardinality values creates many small files |
| `ZORDER` | Legacy Delta layout on non-liquid tables | Combining it with liquid clustering or using it for new tables when liquid clustering is available |
| No explicit layout | Small tables or tables with natural ingestion-time locality | Leaving a large hot fact table unoptimized when query profile shows heavy file scanning |

Exam answer for benefits of liquid clustering:

- It simplifies layout tuning compared with partitioning and `ZORDER`.
- It supports high-cardinality clustering keys without creating one directory per value.
- It lets clustering keys change without rewriting the table into a new partition layout.
- It works with incremental `OPTIMIZE`.
- It can use automatic key selection on supported Unity Catalog managed Delta tables.
- It avoids the operational burden of maintaining separate partition and `ZORDER` strategies.

## Gold serving choices

| Requirement | Good choice |
|-------------|-------------|
| BI users need flexible slicing by many dimensions | Star schema with fact and dimension tables |
| Dashboard repeatedly runs the same expensive aggregate | Materialized view or physical aggregate table |
| Analysts need a lightweight governed projection | Unity Catalog view |
| Metrics must refresh with streaming source | Streaming table or Lakeflow pipeline output |
| Data must be shared across teams | Governed gold tables with clear grants and lineage |
| PII should not be broadly exposed | Masked dimension columns, row filters, or separate restricted marts |

Gold aggregate example:

```sql
CREATE OR REPLACE TABLE main.gold.agg_daily_revenue
USING DELTA
CLUSTER BY (order_date, region)
AS
SELECT
  f.order_date,
  c.region,
  p.category,
  SUM(f.net_amount) AS revenue,
  COUNT(DISTINCT f.order_id) AS order_count
FROM main.gold.fact_order_line f
JOIN main.gold.dim_customer c ON f.customer_sk = c.customer_sk
JOIN main.gold.dim_product p ON f.product_sk = p.product_sk
GROUP BY
  f.order_date,
  c.region,
  p.category;
```

Materialized view example:

```sql
CREATE MATERIALIZED VIEW main.gold.mv_daily_revenue AS
SELECT
  order_date,
  SUM(net_amount) AS revenue,
  COUNT(DISTINCT order_id) AS order_count
FROM main.gold.fact_order_line
GROUP BY order_date;
```

Use physical tables when you need explicit job control, complex incremental logic, or custom validation. Use materialized views when the SQL definition is supported and managed refresh behavior fits the workload.

## Query performance design

| Design area | Correct production shape |
|-------------|--------------------------|
| Fact size | Keep atomic fact tables narrow enough for common scans; move descriptive text into dimensions |
| Join keys | Use stable surrogate keys for dimension joins and durable natural keys for source matching |
| Statistics | Ensure filter/join columns have statistics collected, especially clustering keys |
| File layout | Use liquid clustering and `OPTIMIZE`/predictive optimization for hot large tables |
| Aggregates | Precompute expensive, repeated dashboard grains |
| Small dimensions | Let Spark broadcast automatically or use hints when stats are missing and table is safely small |
| Skew | Salt, pre-aggregate, or split skewed keys when one value dominates |
| BI SQL | Avoid requiring every dashboard to repeat complex source cleansing joins |

Star query:

```sql
SELECT
  d.calendar_year,
  d.calendar_month,
  c.region,
  p.category,
  SUM(f.net_amount) AS revenue
FROM main.gold.fact_order_line f
JOIN main.gold.dim_date d ON f.order_date = d.date_key
JOIN main.gold.dim_customer c ON f.customer_sk = c.customer_sk
JOIN main.gold.dim_product p ON f.product_sk = p.product_sk
WHERE f.order_date >= DATE '2026-01-01'
GROUP BY
  d.calendar_year,
  d.calendar_month,
  c.region,
  p.category;
```

Validation query for grain:

```sql
SELECT
  order_id,
  order_line_id,
  COUNT(*) AS row_count
FROM main.gold.fact_order_line
GROUP BY order_id, order_line_id
HAVING COUNT(*) > 1;
```

## Governance in the model

Modeling is also a governance decision.

| Concern | Modeling response |
|---------|-------------------|
| PII in customer dimension | Mask sensitive columns or split restricted attributes into a separate dimension |
| Regional access rules | Row filters, separate regional marts, or group-specific views |
| Metric ownership | Table comments, column comments, tags, and documented definitions |
| Cross-team reuse | Conformed dimensions and stable table contracts |
| Lineage requirements | Build through Unity Catalog tables/views/materialized views, not unmanaged paths |
| Retention requirements | Align table retention, deletion handling, and audit needs |

Do not expose bronze/raw tables directly to BI users when curated silver/gold models are expected. Bronze is for source fidelity and replay; gold is for business consumption.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Flexible BI over sales, customers, products, stores | Star schema |
| Need customer address as of order date | SCD Type 2 dimension |
| Only latest customer segment matters | SCD Type 1 dimension |
| Repeated daily dashboard aggregate | Materialized view or aggregate gold table |
| Large fact filtered by date and customer | Liquid clustering on date/customer |
| High-cardinality user ID filter | Liquid clustering, not partitioning |
| Low-cardinality date/month partition with huge data per partition | Partitioning can be considered, but liquid clustering is usually preferred for new tables |
| Existing table used `ZORDER BY (customer_id)` | Migrate to liquid clustering with `customer_id` as a clustering key |
| Many-to-many customer-to-campaign attribution | Bridge table |
| Dashboard double-counts revenue after dimension join | Recheck fact grain and dimension relationship |

## SQL essentials

Create clustered table:

```sql
CREATE TABLE main.gold.fact_orders (
  order_id STRING,
  order_date DATE,
  customer_sk BIGINT,
  amount DECIMAL(18,2)
)
USING DELTA
CLUSTER BY (order_date, customer_sk);
```

Enable clustering on an existing unpartitioned table:

```sql
ALTER TABLE main.gold.fact_orders
CLUSTER BY (order_date, customer_sk);
```

Use automatic liquid clustering:

```sql
ALTER TABLE main.gold.fact_orders
CLUSTER BY AUTO;
```

Remove clustering keys:

```sql
ALTER TABLE main.gold.fact_orders
CLUSTER BY NONE;
```

Inspect table metadata:

```sql
DESCRIBE DETAIL main.gold.fact_orders;
DESCRIBE EXTENDED main.gold.fact_orders;
SHOW TBLPROPERTIES main.gold.fact_orders;
```

## PySpark / API essentials

Fact merge:

```python
from delta.tables import DeltaTable

target = DeltaTable.forName(spark, "main.gold.fact_order_line")

(
    target.alias("t")
    .merge(
        updates.alias("s"),
        "t.order_id = s.order_id AND t.order_line_id = s.order_line_id",
    )
    .whenMatchedUpdateAll()
    .whenNotMatchedInsertAll()
    .execute()
)
```

Create clustered table from DataFrame:

```python
(
    fact_df.write
    .format("delta")
    .clusterBy("order_date", "customer_sk")
    .mode("overwrite")
    .saveAsTable("main.gold.fact_order_line")
)
```

Broadcast small dimension when appropriate:

```python
from pyspark.sql.functions import broadcast

result = fact_df.join(
    broadcast(dim_product_df),
    on="product_sk",
    how="left",
)
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Building gold tables without declaring grain | Define one-row meaning before writing DDL or ETL |
| Joining facts to non-unique dimensions and accepting duplicated measures | Enforce dimension uniqueness or join to the correct SCD2 version |
| Partitioning a fact table by high-cardinality `user_id` | Use liquid clustering on common filter keys |
| Combining `ZORDER` with liquid clustering | Choose liquid clustering for new clustered tables |
| Creating a wide gold table for every possible question | Keep canonical facts/dimensions and add aggregates for hot paths |
| Overwriting historical customer attributes when reports need point-in-time history | Use SCD Type 2 |
| Using SCD Type 2 for every attribute by default | Track history only where the business needs it |
| Letting dashboards define core revenue logic independently | Put governed metric logic in gold tables, materialized views, or a semantic layer |
| Exposing raw PII in a shared gold dimension | Mask, filter, split, or restrict sensitive attributes |
| Assuming `OPTIMIZE` fixes a bad model grain | Fix the model; layout tuning only improves access to correctly modeled data |

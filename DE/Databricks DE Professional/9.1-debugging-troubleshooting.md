# 9.1 Debugging and troubleshooting

## Exam objective

> Identify pertinent diagnostic information using Spark UI, cluster logs, system tables, and query profiles to troubleshoot errors.
>
> Analyze the errors and remediate the failed job runs with job repairs and parameter overrides.
>
> Use Lakeflow Spark Declarative Pipelines event logs and the Spark UI to debug Lakeflow Spark Declarative Pipelines and Spark pipelines.

## Must know

- Start from **failure surface**: job run page → failed task → driver/executor logs → Spark UI → query profile.
- **Driver log**: import errors, config, auth, `collect()` OOM, orchestration exceptions.
- **Executor log**: task failures, shuffle errors, data skew, Python UDF errors.
- **Spark UI**: stages, tasks, shuffle read/write, spill, stragglers.
- **Query profile**: SQL operator tree, bytes scanned, join type.
- **System tables**: historical runs, audit, billing correlation.
- **SDP event log**: flow-level errors, expectation failures, upstream dependency issues.
- **Job repair**: rerun failed tasks and downstream without re-running successful upstream tasks.
- **Parameter overrides**: fix bad `catalog`, date, or env for a single rerun.

## Diagnostic workflow

```
Job/pipeline failed
  -> Read error summary + failed task_key
  -> Driver stderr (first root cause)
  -> Spark UI: which stage/task failed?
  -> Query profile (if SQL)
  -> SDP event log (if pipeline)
  -> Fix code/config -> repair run or full redeploy
```

| Symptom | First place to look |
|---------|-------------------|
| `AnalysisException` / SQL syntax | Driver log, notebook cell |
| `OutOfMemoryError` driver | `collect()`, `toPandas()`, huge broadcast |
| `OutOfMemoryError` executor | skew, large partitions, UDF memory |
| Task failed repeatedly | Executor stack trace, shuffle spill |
| Permission denied | Driver log + UC grants |
| Pipeline DQ failure | SDP event log expectations |
| Sudden full table scan | Query profile scan size |

## Job repair and parameter overrides

**Repair run** (`Repair run` in UI / `repair-run` API):

- Reruns **failed** tasks and **dependent** downstream tasks.
- Skips **successful** upstream tasks — saves time and cost.
- Use after fixing transient infra or corrected parameters.

**Parameter overrides** for one-off backfill:

```json
{
  "job_parameters": [
    {"name": "run_date", "value": "2026-05-20"},
    {"name": "catalog", "value": "main"}
  ]
}
```

```bash
databricks jobs run-now --job-id 12345 \
  --json '{"job_parameters": [{"name": "run_date", "value": "2026-05-20"}]}'

databricks jobs repair-run --run-id 67890 --rerun-tasks transform_orders
```

| Situation | Action |
|-----------|--------|
| Transient cloud blip | Repair run |
| Wrong date parameter caused bad partition | Override parameter + repair |
| Code bug in transform | Fix Git + redeploy bundle + new run |
| Upstream succeeded, downstream failed | Repair downstream only |

## SDP debugging

```sql
SELECT timestamp, level, message, details
FROM event_log(TABLE main.silver.orders_pipeline)
WHERE level = 'ERROR'
ORDER BY timestamp DESC
LIMIT 50;
```

Check:

- Upstream flow failed before current dataset.
- Expectation `failed_records` > 0.
- Autoloader schema mismatch in `details`.

Link to **Spark UI** from pipeline cluster for transform performance issues (not expectation logic errors).

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| SQL warehouse query slow | Query profile |
| PySpark shuffle OOM | Spark UI stages + spill metrics |
| Who deleted production table | `system.access.audit` |
| Pipeline expectation dropped all rows | SDP event log DQ section |
| Rerun only failed branch after fix | Job repair |
| One-time backfill date | Parameter override |

## PySpark / API essentials

```python
# Reproduce failure locally with same parameters
catalog = "main"
run_date = "2026-05-20"
df = spark.table(f"{catalog}.bronze.orders_raw").filter(f"order_date = '{run_date}'")
```

## SQL (exam)

`event_log()`, system table queries, `DESCRIBE HISTORY` for data issues vs code issues.

## UI / product surfaces

- **Lakeflow Jobs → Run → Task run → Logs / Spark UI**
- **Lakeflow Pipelines → Event log / Update details**
- **Compute → Cluster event log**
- **SQL → Query history → Query profile**

## Exam traps

- Re-running entire job when repair would suffice.
- Repair without fixing root cause — infinite failure loop.
- Using Spark UI alone for SQL warehouse-only workload.
- Ignoring expectation metrics and only reading final "pipeline failed".
- Scaling cluster before checking skew/spill in Spark UI.

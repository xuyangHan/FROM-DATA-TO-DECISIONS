# 9.1 Debugging and troubleshooting

## Exam objective

> Identify pertinent diagnostic information using Spark UI, cluster logs, system tables, and query profiles to troubleshoot errors.
>
> Analyze the errors and remediate the failed job runs with job repairs and parameter overrides.
>
> Use Lakeflow Spark Declarative Pipelines event logs and the Spark UI to debug Lakeflow Spark Declarative Pipelines and Spark pipelines.

## Naming note

- Current docs use **Lakeflow Jobs** and **Lakeflow Spark Declarative Pipelines (SDP)**. Older material may say Workflows and Delta Live Tables.
- The exam says **Query Profiler UI** in some places; Databricks docs usually call it **Query Profile**.
- The pipeline event log is the main observability primitive for Lakeflow SDP.

## Must know

- Start from the highest-level failure surface, then drill down: Jobs run, task run, logs, Spark UI/Query Profile, system tables, pipeline event logs.
- **Driver logs** are best for import errors, Python exceptions, orchestration failures, authentication/config errors, and driver OOM.
- **Executor logs** are best for repeated task failures, executor OOM, Python worker errors, JVM errors, shuffle failures, and data-dependent errors.
- **Compute event logs** are best for cluster startup, termination, resize, policy, init script, and library installation issues.
- **Spark UI** is best for Spark stages, tasks, shuffle, spill, skew, executors, and streaming tab behavior.
- **Query Profile** is best for SQL warehouse / SQL query bottlenecks: scans, joins, shuffles, operator time, and spills.
- **System tables** provide historical/account-wide context for job runs, task timings, audit events, costs, and compute usage.
- **Lakeflow SDP event logs** show update events, flow progress, data quality expectations, lineage, user actions, and errors.
- **Repair runs** rerun selected failed/skipped tasks and dependent tasks as part of the original job run.
- **Parameter overrides** are useful for one repaired run or backfill, but code/config bugs should be fixed in source and redeployed.

## Professional focus vs Associate

Associate notes cover run history, Jobs UI health, Spark UI bottlenecks, and cluster/OOM diagnosis. For Professional, focus on choosing the right diagnostic surface and remediation action:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Open the failed job run | Identify first failed task, upstream blockers, retry/repair behavior, and parameter state |
| Read Spark UI metrics | Connect Spark UI symptoms to join, skew, spill, code, or data-layout fixes |
| Read cluster logs | Distinguish driver, executor, library, init script, and compute startup failures |
| Know system tables exist | Use them for historical trends and cross-workspace failure investigation |
| Know pipeline event logs exist | Query event log JSON for flow errors and expectation metrics |
| Know repair run exists | Decide repair vs full rerun vs redeploy vs parameter override |

## Diagnostic decision map

| Symptom | First place to look | Next place |
|---------|---------------------|------------|
| Multi-task job failed | Jobs run graph/list | First failed task logs |
| Downstream task skipped | Jobs run graph and `run_if` | Upstream failed task |
| SQL warehouse query slow | Query Profile | Query history/system tables |
| PySpark job slow or OOM | Spark UI stages/executors | Driver/executor logs |
| Job failed before user code | Task run details and compute event log | Library/init script logs |
| Permission denied | Driver/task log | UC grants and `system.access.audit` |
| Library import error | Driver log and library status | Bundle/job library config |
| Pipeline expectation failure | Lakeflow SDP event log | Quality rule and source data |
| Auto Loader schema issue | Pipeline event log or driver log | Schema location / rescued data / source files |
| Cluster did not start | Compute event log | Policy, quota, init scripts, libraries, cloud permissions |
| Need historical failure trend | System tables | Jobs UI for one run |

## Root-cause workflow

```text
1. Identify the failed workflow or query.
2. Find the first failed task/query/update, not only the final failure.
3. Read the error summary, task output, and driver stderr.
4. Decide category: code, data, permission, dependency, compute, performance, pipeline quality, or external system.
5. Open the correct deep-dive tool:
   - Query Profile for SQL.
   - Spark UI for Spark stages/tasks.
   - Compute event log for startup/lifecycle.
   - Pipeline event log for Lakeflow SDP.
   - System tables for trend/history.
6. Fix the root cause.
7. Choose remediation: repair run, parameter override, full rerun, or redeploy.
8. Verify the fixed run and add monitoring/tests if the issue can recur.
```

Important exam habit: if many downstream tasks are skipped, the skipped task is rarely the root cause. Find the earliest failed upstream task.

## Error categories

| Error category | Typical evidence | Correct response |
|----------------|------------------|------------------|
| Code bug | Stack trace in driver/task logs | Fix code, test, redeploy |
| Bad parameter | Wrong catalog/date/path in task output | Override parameter if one-off, fix default if persistent |
| Permission issue | `PERMISSION_DENIED`, `UNAUTHORIZED`, UC error | Grant least privilege to run-as identity |
| Dependency issue | `ModuleNotFoundError`, library install failure | Declare/pin dependency in job/bundle/cluster scope |
| Data quality issue | Expectation failure, invalid rows, schema mismatch | Fix source, adjust rule, quarantine, or repair |
| Data volume/performance | Query slow, shuffle/spill/skew | Tune query/layout/compute after diagnosis |
| Compute startup | Cluster error before code runs | Event log, policy, quota, init script, libraries |
| External system | Timeout, HTTP 429, JDBC failure | Retries/backoff, credentials, source availability |
| Governance change | Previously working query denied | Audit logs and grant history |

## Logs and diagnostics

| Surface | Best for | Look for |
|---------|----------|----------|
| Jobs run page | DAG status, failed/skipped tasks, retry/repair | First failed task, dependencies, parameters |
| Task output | Notebook/script stdout and high-level exception | Business error, parameter value, stack trace |
| Driver stdout/stderr | Driver-side exceptions | Imports, auth, config, driver OOM, Python exception |
| Executor logs | Distributed task failures | Python worker crash, executor OOM, data-dependent error |
| Compute event log | Lifecycle and startup failures | Policy, init script, library install, termination code |
| Libraries tab | Package installation status | Failed/pending libraries, version conflicts |
| Spark UI | Spark stage/task behavior | Shuffle, spill, skew, failed tasks |
| Query Profile | SQL physical plan and bottlenecks | Scan bytes, join type, operator time |
| System tables | Historical/account-level evidence | Repeated failures, slow tasks, audit events |
| Pipeline event log | Lakeflow SDP flow/pipeline state | Expectations, flow progress, errors |

## Spark UI troubleshooting

Use Spark UI when the task is a Spark job, PySpark job, Python wheel using Spark, or pipeline transformation.

| Spark UI observation | Likely cause | Correct response |
|----------------------|--------------|------------------|
| One task much slower than the rest | Skew | Inspect key distribution, AQE/skew handling, salting, split hot keys |
| Huge shuffle read/write | Join, aggregation, distinct, window, repartition | Filter/project early, pre-aggregate, broadcast small side |
| High memory/disk spill | Large partitions, memory pressure, skew | Reduce data, adjust partitions, fix skew, increase memory if justified |
| Executor lost | OOM, spot loss, infra issue | Executor logs, compute events, retry pattern |
| Many tiny tasks | Small files or over-partitioning | Compact files, reduce shuffle partitions |
| One/few huge tasks | Under-partitioning or single large file | Repartition or fix source layout |
| Streaming tab shows input but no progress | State/store/backpressure issue | Check processing rate, state size, checkpoint, source backlog |

Spark UI is not the first tool for SQL warehouse-only queries; use Query Profile there.

## Query Profile troubleshooting

Use Query Profile when the workload is SQL warehouse, DBSQL, SQL task, or a SQL query with a profile.

| Profile symptom | Likely cause | Correct response |
|-----------------|--------------|------------------|
| Scan bytes much larger than output | Poor pruning/skipping or broad query | Add predicates, clustering/stats, optimize layout |
| Slow join operator | Bad join order/type, missing stats, large shuffle | Analyze stats, broadcast safe small side, rewrite join |
| Sort-merge join where dimension is small | Missing stats or broadcast threshold issue | `ANALYZE`, broadcast hint if safe |
| Large shuffle | GroupBy, distinct, join, window | Reduce rows/columns before wide operation |
| Spill | Memory pressure or too much shuffled data | Reduce data, pre-aggregate, warehouse/compute sizing |
| Repeated expensive subquery | Query shape issue | Materialize intermediate result or use materialized view |
| UDF dominates | Python/SQL function cost | Prefer built-in functions or simplify UDF |

Use Query Profile to decide whether the fix is query rewrite, table layout/statistics, or compute sizing. Do not scale first when the profile shows a preventable full scan or many-to-many join.

## System tables for troubleshooting

Use system tables when the question is historical, cross-workspace, or audit-oriented.

Recent failed jobs:

```sql
SELECT
  workspace_id,
  job_id,
  run_id,
  period_start_time,
  period_end_time,
  result_state,
  termination_code
FROM system.lakeflow.job_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 1 DAY
  AND result_state = 'FAILED'
ORDER BY period_end_time DESC;
```

Slow task trend:

```sql
SELECT
  workspace_id,
  job_id,
  task_key,
  COUNT(*) AS task_runs,
  AVG(execution_duration_seconds) AS avg_execution_seconds,
  PERCENTILE(execution_duration_seconds, 0.95) AS p95_execution_seconds
FROM system.lakeflow.job_task_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 14 DAYS
GROUP BY workspace_id, job_id, task_key
ORDER BY p95_execution_seconds DESC;
```

Audit a permission/data change:

```sql
SELECT
  event_time,
  action_name,
  user_identity.email AS user_email,
  request_params
FROM system.access.audit
WHERE event_time >= current_timestamp() - INTERVAL 7 DAYS
  AND (
    action_name ILIKE '%grant%'
    OR action_name ILIKE '%revoke%'
    OR action_name ILIKE '%delete%'
    OR action_name ILIKE '%drop%'
  )
ORDER BY event_time DESC;
```

System table use cases:

| Question | System table |
|----------|--------------|
| Is this job failing more often this week? | `system.lakeflow.job_run_timeline` |
| Which task is the recurring bottleneck? | `system.lakeflow.job_task_run_timeline` |
| Who changed permissions or dropped a table? | `system.access.audit` |
| Did cost spike with failures/retries? | `system.billing.usage` joined to job metadata |
| Is compute startup/setup time high? | Job timeline tables and compute system tables |

## Job repair and rerun decisions

Repair runs are useful after you identify and fix the problem. They are not a substitute for root-cause analysis.

| Situation | Correct action |
|-----------|----------------|
| Transient cloud/source failure and upstream successful tasks are valid | Repair failed task and dependents |
| Wrong parameter for one run | Repair with parameter override |
| Code bug | Fix code, redeploy bundle/job, then rerun or repair if safe |
| Bad data corrected in source | Repair affected task and downstream tasks |
| Successful upstream task produced wrong output | Full rerun from the bad task or upstream, not downstream-only repair |
| Pipeline code/config changed | Redeploy or update pipeline, then rerun |
| Non-idempotent side effect task failed | Repair carefully; verify idempotency or compensate manually |

Repair behavior:

- Repair reruns selected tasks as part of the original job run.
- It can rerun failed tasks, skipped tasks, and dependent tasks based on selection.
- Successful upstream tasks can be skipped to save time and cost.
- Repaired tasks use current job/task settings.
- Parameters entered in the repair dialog/API override values for the repair.
- For subsequent repairs of the same run, the API may require the latest repair ID.

## Parameter overrides

Use parameter overrides for controlled reruns, backfills, and one-off corrections.

Examples:

| Override | Use case |
|----------|----------|
| `run_date` | Backfill or rerun one partition/day |
| `catalog` / `schema` | Dev/prod target correction |
| `source_path` | Reprocess specific landing path |
| `full_refresh` | Pipeline-specific refresh behavior |
| task parameters | Fix one task's bad argument |

CLI/API pattern:

```bash
databricks jobs run-now --job-id 12345 --json '{"job_parameters": {"run_date": "2026-05-20"}}'
```

```bash
databricks jobs repair-run --run-id 67890 --rerun-tasks transform_orders
```

Conceptual repair API fields:

```json
{
  "run_id": 67890,
  "rerun_tasks": ["transform_orders"],
  "rerun_dependent_tasks": true,
  "job_parameters": {
    "run_date": "2026-05-20"
  }
}
```

Parameter discipline:

- Log effective parameter values at task start.
- Validate catalog/schema/env values before destructive writes.
- Keep defaults in bundle/job config, not hidden notebook state.
- Use one-off overrides for one-off reruns; fix bad defaults in source config.

## Lakeflow SDP debugging

Use the pipeline UI and event log first, then Spark UI for Spark execution bottlenecks.

Event log query:

```sql
SELECT
  timestamp,
  event_type,
  message,
  details
FROM event_log('<pipeline-id>')
ORDER BY timestamp DESC
LIMIT 100;
```

Flow progress and expectations:

```sql
SELECT
  timestamp,
  details:flow_progress.status AS flow_status,
  details:flow_progress.metrics AS flow_metrics,
  details:flow_progress.data_quality.expectations AS expectations
FROM event_log('<pipeline-id>')
WHERE event_type = 'flow_progress'
ORDER BY timestamp DESC;
```

Errors:

```sql
SELECT
  timestamp,
  event_type,
  message,
  details
FROM event_log('<pipeline-id>')
WHERE event_type = 'error'
   OR message ILIKE '%fail%'
ORDER BY timestamp DESC;
```

Pipeline symptoms:

| Symptom | First check | Likely issue |
|---------|-------------|--------------|
| Update failed quickly | Event log error events | Syntax/config/source permission |
| Flow failed | `flow_progress` details | Dataset logic or upstream dependency |
| Expectation failed | Data quality section | Bad source data or too-strict rule |
| Auto Loader schema failure | Event details and driver logs | Schema evolution/rescued data/source drift |
| Pipeline slow | Pipeline UI + Spark UI | Large scan, shuffle, skew, state growth |
| Pipeline cannot update after event log deletion | Event log/catalog/schema status | Recreate/fix pipeline configuration; do not delete event logs |

Do not delete Lakeflow SDP event logs or the parent catalog/schema where they are published; this can break future updates.

## Data issues vs code issues

| Clue | Data issue | Code/config issue |
|------|------------|-------------------|
| Same code fails only for one date/partition | Likely | Possible |
| Stack trace points to missing column after source change | Likely schema drift | Also code robustness issue |
| All runs fail after deployment | Possible | Likely |
| Permission error after grants changed | No | Config/governance issue |
| Expectation failed with many invalid rows | Likely | Rule may also be too strict |
| Query returns wrong counts but no exception | Possible duplicates/join grain | Possible logic bug |

Useful checks:

```sql
DESCRIBE HISTORY main.silver.orders;
DESCRIBE DETAIL main.silver.orders;
```

```sql
SELECT _metadata.file_path, COUNT(*) AS rows
FROM main.bronze.orders_raw
WHERE order_date = DATE '2026-05-20'
GROUP BY _metadata.file_path
ORDER BY rows DESC;
```

Use table history, source file metadata, quality metrics, and row counts to avoid blaming code when the source data changed.

## Common remediation patterns

| Root cause | Remediation |
|------------|-------------|
| Missing UC privilege | Grant least privilege to run-as service principal |
| Wrong environment parameter | Override for repair; fix bundle/job default |
| Schema drift | Update schema handling, rescued data, or source contract |
| Bad source file | Quarantine/reprocess source file; repair downstream |
| Library conflict | Pin compatible dependency and redeploy/restart compute |
| Driver OOM | Remove `collect()`/`toPandas()` or increase driver only if justified |
| Executor OOM | Fix skew/partitions/cache/join, then consider memory |
| Full table scan | Improve predicates, stats, clustering, data skipping |
| Non-idempotent external call failed halfway | Use idempotency keys/compensation before repair |
| Pipeline expectation failure | Fix bad records, adjust expectation action, quarantine, or fail intentionally |

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| SQL query slow | Query Profile |
| PySpark shuffle OOM | Spark UI stages/executors and executor logs |
| Driver OOM from collected result | Driver log and code review |
| Cluster startup failed | Compute event log |
| Library install failed | Libraries tab and task/cluster logs |
| Who changed a production table or grant | `system.access.audit` |
| Lakeflow SDP expectation dropped all rows | Pipeline event log quality details |
| Rerun only failed branch after fix | Repair run |
| One-time backfill date | Parameter override |
| Persistent code bug | Fix source, redeploy bundle, rerun |

## PySpark / API essentials

Log effective parameters:

```python
catalog = dbutils.widgets.get("catalog")
schema = dbutils.widgets.get("schema")
run_date = dbutils.widgets.get("run_date")

print(f"catalog={catalog}, schema={schema}, run_date={run_date}")
```

Reproduce one failing partition:

```python
orders = (
    spark.table(f"{catalog}.bronze.orders_raw")
    .filter(f"order_date = DATE '{run_date}'")
)
```

Avoid driver OOM:

```python
# Bad for large data
# rows = orders.collect()

# Better
orders.groupBy("status").count().show()
```

Inspect plan:

```python
orders.explain("formatted")
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Rerunning the entire job when only one downstream branch failed | Use repair run for failed/skipped tasks and dependents when upstream outputs are valid |
| Repairing repeatedly without fixing root cause | Diagnose and fix code/data/config first, then repair |
| Debugging a skipped task as if it failed | Find the upstream failed or unmet condition that caused the skip |
| Using Spark UI for a SQL warehouse-only query | Use Query Profile and query history |
| Scaling compute before checking shuffle, skew, spill, or scan size | Diagnose with Spark UI/Query Profile first |
| Treating job-level failure as enough context | Open the failed task, logs, parameters, and compute details |
| Ignoring parameter values during rerun | Log and validate effective parameters; override only intentionally |
| Repairing non-idempotent side effects blindly | Add idempotency/compensation or handle manually |
| Reading only final pipeline failed message | Query Lakeflow event log for flow and expectation details |
| Deleting pipeline event logs | Keep event logs; they are required for observability and can affect updates |

# 8.1 Data governance

## Exam objective

> Create and add descriptions/metadata about enterprise data to make it more discoverable.
>
> Demonstrate understanding of Unity Catalog permission inheritance model.

## Must know

- **Discoverability** = comments, tags, owners, lineage, business glossary links, and search in Catalog Explorer.
- **Unity Catalog** is the system of record for data + AI asset metadata and permissions.
- **Permission inheritance** flows down the hierarchy: metastore → catalog → schema → table/view/volume/function.
- Grants at **higher level** can simplify administration; **DENY** at lower level can block inherited access.
- **Ownership** implies broad control on that object (`MANAGE` capabilities).
- **Service principals** and **groups** are preferred grantees for production automation.

## Metadata for discoverability

| Metadata | Purpose |
|----------|---------|
| `COMMENT ON TABLE/COLUMN` | Business definitions |
| Tags (`pii`, `domain:sales`) | Search, ABAC, policy |
| Owner | Accountability |
| Lineage | Upstream/downstream impact analysis |
| Table properties | SLA, quality tier, source system |

```sql
COMMENT ON TABLE main.gold.daily_revenue IS 'Daily revenue by region for exec dashboards';
COMMENT ON COLUMN main.gold.daily_revenue.revenue IS 'Sum of order amount after returns';

ALTER TABLE main.gold.daily_revenue SET TAGS ('domain' = 'finance', 'tier' = 'gold');
ALTER TABLE main.gold.daily_revenue SET OWNER TO `data-platform@example.com`;
```

## Permission inheritance model

```
Metastore (account/regional)
  └── Catalog
        └── Schema
              └── Table / View / Volume / Function
```

| Grant at | Effect on children |
|----------|-------------------|
| `USE CATALOG` on catalog | Required to access schemas inside |
| `USE SCHEMA` on schema | Required to access objects inside |
| `SELECT` on schema | All current/future tables? — exam: schema-level grants apply to tables in schema per UC rules |
| `SELECT` on table | That object only |
| `MANAGE` | Administrative control on securable |

Inheritance concepts for exam:

- Privileges granted on a **schema** apply to existing and often **future** tables in that schema (know schema-level `SELECT` vs table-level).
- **`DENY`** overrides inherited `ALLOW` for same privilege.
- **`OWNERSHIP`** is not inherited the same way as `SELECT` — each object has an owner.
- External locations, storage credentials, connections are separate securables with their own grants.

```sql
-- Pattern: least privilege via schema for read-only analysts
GRANT USE CATALOG ON CATALOG main TO `analysts`;
GRANT USE SCHEMA ON SCHEMA main.gold TO `analysts`;
GRANT SELECT ON SCHEMA main.gold TO `analysts`;

-- Deny sensitive table inside otherwise readable schema
DENY SELECT ON TABLE main.gold.salary_detail TO `analysts`;
```

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Users cannot find datasets | Comments, tags, owner, glossary |
| Onboard 50 analysts read-only to all gold | Schema-level `SELECT` on `main.gold` |
| One sensitive table in shared schema | `DENY` or no grant + table-level only |
| ETL service needs write to silver only | `MODIFY` on specific tables, not catalog admin |
| Impact analysis before column rename | Lineage in Catalog Explorer |

`COMMENT`, `SET TAGS`, `SET OWNER`, `GRANT`, `REVOKE`, `DENY`, `SHOW GRANTS`.

## UI / product surfaces

- **Catalog Explorer**: search, tags, lineage graph, permissions tab.
- **Data lineage**: pipeline → table dependencies.
- **Admin → Metastore**: catalog attachment to workspaces.

## Exam traps

- Granting `ALL PRIVILEGES` on catalog for a read-only team.
- Assuming cloud IAM replaces UC grants for table access.
- Forgetting `USE CATALOG` / `USE SCHEMA` prerequisites.
- No metadata on bronze — still document source system and PII tags.
- Confusing **owner** with **superuser on all child objects**.

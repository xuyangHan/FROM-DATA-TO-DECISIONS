# 8.1 Data governance

## Exam objective

> Create and add descriptions/metadata about enterprise data to make it more discoverable.
>
> Demonstrate understanding of Unity Catalog permission inheritance model.

## Naming note

- **Unity Catalog (UC)** is the governance layer for Databricks data and AI assets.
- Current Unity Catalog docs do **not** support `DENY` for UC objects. `DENY` applies to legacy `hive_metastore` objects.
- Databricks now has both ordinary tags and **governed tags**. Governed tags are the controlled-tag mechanism used by ABAC policies.

## Must know

- Data governance is not only access control. It includes discoverability, ownership, metadata, lineage, classifications, quality signals, and permission design.
- **Catalog Explorer** is the primary UI for discovering governed assets, viewing metadata, applying tags/comments, inspecting lineage, and managing permissions.
- Enterprise data becomes discoverable through table/column comments, owners, tags, lineage, schemas, domains, quality tier, source system, and business definitions.
- Unity Catalog securables are hierarchical: metastore, catalog, schema, then tables/views/volumes/functions/models.
- Most object access requires parent privileges such as `USE CATALOG` and `USE SCHEMA` plus object privileges such as `SELECT`.
- Privileges are additive through direct grants, group membership, inheritance, and ownership. `REVOKE` removes a grant but does not override access inherited from another group.
- Grants on catalogs and schemas can simplify administration and apply to current and future child objects where the privilege is inheritable.
- Ownership does not behave like normal inherited read access. Each object has an owner, and owners/container owners can manage child objects according to UC rules.
- `MANAGE` can allow permission management without automatically granting data access.
- Grant to groups for users and service principals for production automation.

## Professional focus vs Associate

Associate notes cover table types, basic grants, and governance objects. For Professional, focus on scalable governance design:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know `catalog.schema.table` | Design catalog/schema boundaries around domains, environments, and ownership |
| Know `GRANT SELECT` syntax | Choose table-level vs schema-level grants and understand inherited access |
| Know comments/tags exist | Use metadata for search, ownership, ABAC, PII classification, and operations |
| Know Unity Catalog governs data | Govern tables, volumes, functions, models, shares, connections, and external locations |
| Know lineage exists | Use lineage for impact analysis, compliance, and safe schema changes |
| Know groups are preferred | Build group/service-principal access patterns rather than one-off user grants |

## Governance model

```text
Metastore
  -> Catalog       domain/environment boundary
      -> Schema    subject area / layer / team boundary
          -> Tables, views, materialized views, streaming tables
          -> Volumes
          -> Functions
          -> Models
```

Design principles:

| Principle | Correct production shape |
|-----------|--------------------------|
| Domain-oriented catalogs/schemas | Names reflect business domains, environments, or layers |
| Clear ownership | Every important object has an accountable owner group |
| Metadata by default | Tables and columns have comments, tags, quality tier, and source info |
| Least privilege | Grant only what is needed, preferably to groups/service principals |
| Inherited grants with care | Use schema-level grants for broad read/write patterns; table grants for exceptions |
| Governed tags for policy | Use controlled tags for ABAC, PII, sensitivity, and classification |
| Lineage-aware changes | Check upstream/downstream impact before renames, drops, or schema changes |
| Auditability | Use audit logs, table history, owners, comments, and tags as governance evidence |

## Metadata for discoverability

| Metadata | Purpose | Example |
|----------|---------|---------|
| Table comment | Business definition and usage context | "Daily revenue by region for executive dashboards" |
| Column comment | Meaning, units, derivation, allowed values | "Net revenue after returns, USD" |
| Owner | Accountability and approval path | `finance_data_owners` |
| Tags | Classification, domain, lifecycle, SLA, PII | `domain = finance`, `pii = email` |
| Governed tags | Controlled values for policy and ABAC | `sensitivity = restricted` |
| Table properties | Technical metadata and operational hints | quality tier, source system, retention |
| Lineage | Upstream/downstream dependency graph | Job writes table, dashboard reads table |
| Certification/status | Consumer trust signal where available | certified, deprecated, experimental |
| README / notebook / dashboard link | Human-facing docs and examples | owner docs, SLA page, runbook |

Recommended minimum metadata:

| Object | Minimum metadata |
|--------|------------------|
| Bronze table | Source system, landing pattern, raw/PII tags, owner |
| Silver table | Business key, cleansing rules, quality expectations, owner |
| Gold table | Business definition, SLA, consumers, metric definitions |
| Sensitive column | PII/sensitivity tag and column comment |
| Volume | File purpose, retention, owner, sensitivity |
| Function | Purpose, arguments, owner, security impact |

## SQL metadata patterns

Comments:

```sql
COMMENT ON TABLE main.gold.daily_revenue
IS 'Daily revenue by region for executive dashboards. One row per region per order_date.';

COMMENT ON COLUMN main.gold.daily_revenue.revenue
IS 'Net revenue after returns, in USD.';

COMMENT ON COLUMN main.gold.daily_revenue.order_date
IS 'Business date of the order, not ingestion date.';
```

Tags:

```sql
ALTER TABLE main.gold.daily_revenue
SET TAGS (
  'domain' = 'finance',
  'quality' = 'gold',
  'source_system' = 'orders',
  'sla' = 'daily_08_00'
);

ALTER TABLE main.gold.customers
ALTER COLUMN email
SET TAGS ('pii' = 'email');
```

Owner:

```sql
ALTER TABLE main.gold.daily_revenue
SET OWNER TO `finance_data_owners`;
```

Inspection:

```sql
DESCRIBE EXTENDED main.gold.daily_revenue;
SHOW GRANTS ON TABLE main.gold.daily_revenue;
```

Do not store sensitive values in comments or tag values. Tags and comments are metadata and may be visible to users who can browse metadata.

## Tags, governed tags, and ABAC

| Tag type | Use |
|----------|-----|
| Ordinary tags | Search, classification, documentation, lightweight organization |
| Governed tags | Controlled taxonomy and policy targeting |
| Column tags | Classify sensitive columns such as PII or PHI |
| Catalog/schema/table tags | Classify domain, lifecycle, quality tier, retention, sensitivity |

Governance taxonomy examples:

| Tag key | Example values | Use |
|---------|----------------|-----|
| `domain` | finance, sales, hr | Search and ownership |
| `quality` | bronze, silver, gold | Layer and trust |
| `pii` | email, phone, ssn | Masking/ABAC and compliance |
| `sensitivity` | public, internal, restricted | Access review and ABAC |
| `source_system` | salesforce, sap, orders | Lineage/discovery |
| `retention` | 90_days, 7_years | Purge workflows |
| `owner_team` | data_platform, finance | Accountability |

ABAC fit:

| Requirement | Correct metadata/policy approach |
|-------------|----------------------------------|
| All columns tagged `pii = email` must be masked | Governed tag + ABAC column mask policy |
| High-sensitivity tables need regional row filter | Governed tag + ABAC row filter policy |
| Users need to search for finance gold tables | Tags/comments/owners |
| New tagged assets should be protected automatically | Governed tags + policy at catalog/schema scope |

## Lineage and impact analysis

Lineage helps answer "what depends on this?" and "where did this value come from?"

| Governance question | Lineage use |
|---------------------|-------------|
| Can I rename this column? | Inspect downstream tables, dashboards, queries, and jobs |
| Where did this sensitive field flow? | Trace upstream/downstream sensitive columns |
| Which job produced this table? | Find write operations and pipeline/job ownership |
| Why did a dashboard metric change? | Inspect upstream source and transformation path |
| Which objects are affected by a source outage? | Trace downstream dependencies |

Lineage is useful, but not magic:

- Lineage quality depends on supported workloads and access paths.
- Path-based reads/writes and external tools may not produce complete lineage.
- Use lineage with table history, job definitions, audit logs, and code search for high-risk changes.

## Permission inheritance model

Unity Catalog privileges can be granted at different levels.

```text
Metastore
  -> Catalog
      -> Schema
          -> Table / View / Materialized View / Streaming Table
          -> Volume
          -> Function
          -> Model
```

Core rules:

| Rule | Meaning |
|------|---------|
| Parent usage is required | A user generally needs `USE CATALOG` and `USE SCHEMA` before accessing child objects |
| Privileges can inherit | Grants on catalogs/schemas can apply to current and future child objects |
| Privileges are additive | Direct grants, groups, inherited grants, and ownership add together |
| `REVOKE` is not `DENY` | Removing one grant does not block access from another grant/group |
| UC has no `DENY` | Use least privilege, group design, masks/filters, ABAC, and narrower grants |
| Ownership is special | Owners and container owners can manage objects, but ownership is not just inherited `SELECT` |
| Some securables are separate | External locations, storage credentials, connections, shares, and recipients have their own privileges |

## Common privileges and use cases

| Privilege | Use |
|-----------|-----|
| `USE CATALOG` | Resolve/access objects in catalog |
| `USE SCHEMA` | Resolve/access objects in schema |
| `SELECT` | Read tables/views |
| `MODIFY` | Write/modify table data |
| `CREATE TABLE` | Create tables in schema |
| `CREATE VOLUME` | Create volumes |
| `CREATE FUNCTION` | Create functions |
| `EXECUTE` | Execute functions/models |
| `READ VOLUME` | Read files in a volume |
| `WRITE VOLUME` | Write files in a volume |
| `BROWSE` | Discover metadata without data access |
| `APPLY TAG` | Assign governed tags where allowed |
| `MANAGE` | Manage privileges/ownership without automatically granting data access |
| `ALL PRIVILEGES` | Broad object privileges, but not every special privilege such as `MANAGE` |

Access patterns:

| Requirement | Better grant |
|-------------|--------------|
| Analysts read one table | Parent `USE` + table `SELECT` |
| Analysts read all gold tables in schema | Parent `USE` + schema `SELECT` |
| BI users discover but cannot read | `BROWSE` |
| Job writes one table | Parent `USE` + table `MODIFY` |
| Job creates many silver tables | Parent `USE` + schema `CREATE TABLE` and scoped `MODIFY` |
| Data steward applies governed tags | `APPLY TAG` plus object permissions |
| Team manages permissions but should not read data | `MANAGE`, not `SELECT` |

## Least-privilege examples

Read-only analysts for all gold tables:

```sql
GRANT USE CATALOG ON CATALOG main TO `finance_analysts`;
GRANT USE SCHEMA ON SCHEMA main.gold TO `finance_analysts`;
GRANT SELECT ON SCHEMA main.gold TO `finance_analysts`;
```

Read one sensitive table through explicit access:

```sql
GRANT USE CATALOG ON CATALOG main TO `hr_analysts`;
GRANT USE SCHEMA ON SCHEMA main.gold TO `hr_analysts`;
GRANT SELECT ON TABLE main.gold.salary_detail TO `hr_analysts`;
```

Remove an inherited access problem correctly:

```text
If analysts inherit SELECT on main.gold from a group,
revoking a direct table grant from one user does not block access.

Correct options:
1. Remove the user from the broad group.
2. Split sensitive tables into a restricted schema.
3. Stop granting SELECT broadly at schema level.
4. Apply row filters/column masks if users may read but not see all data.
```

Service principal for production ETL:

```sql
GRANT USE CATALOG ON CATALOG main TO `sp_lakeflow_orders`;
GRANT USE SCHEMA ON SCHEMA main.silver TO `sp_lakeflow_orders`;
GRANT CREATE TABLE ON SCHEMA main.silver TO `sp_lakeflow_orders`;
GRANT MODIFY ON SCHEMA main.silver TO `sp_lakeflow_orders`;
```

## Ownership, MANAGE, and accountability

| Concept | Key point |
|---------|-----------|
| Owner | Full administrative control of the object |
| Container owner | Can manage child objects according to UC rules |
| `MANAGE` | Can manage privileges/ownership behavior without automatically granting data access |
| Group ownership | Preferred for enterprise continuity |
| Personal ownership | Risky when people change roles or leave |

Governance practice:

- Use group owners such as `finance_data_owners`, not individual users.
- Assign service principals as runtime identities, not broad human groups.
- Review owners during offboarding and team changes.
- Use comments/tags to point users to owner and support channels.

## Catalog and schema design

| Design choice | Use when |
|---------------|----------|
| Catalog per environment | `dev`, `test`, `prod` separation is a primary control |
| Catalog per domain | Finance, sales, HR need separate ownership/governance |
| Schema per medallion layer | `bronze`, `silver`, `gold` within a domain |
| Schema per sensitivity | Restricted data needs narrower grants |
| Schema per source system | Useful for ingestion/raw organization |

Example:

```text
prod_finance
  bronze
  silver
  gold
  restricted

dev_finance
  bronze
  silver
  gold
```

Schema boundaries are governance boundaries. If a table needs fundamentally different access rules, placing it in a separate restricted schema is often cleaner than broad grants plus many exceptions.

## Metadata quality checks

Governance should be monitored like data quality.

| Check | Why it matters |
|-------|----------------|
| Tables without comments | Hard to discover and trust |
| Columns tagged `pii` but no mask/policy | Compliance gap |
| Sensitive columns with no tags | Policy cannot target them |
| Tables owned by individual users | Offboarding and accountability risk |
| Gold tables without SLA/owner tags | Consumers lack support path |
| External tables without external location review | Storage lifecycle/governance risk |
| Broad grants on restricted schemas | Least-privilege risk |

Example metadata audit query shape:

```sql
SELECT
  table_catalog,
  table_schema,
  table_name,
  comment
FROM main.information_schema.tables
WHERE table_catalog = 'main'
  AND table_schema IN ('silver', 'gold')
  AND (comment IS NULL OR comment = '');
```

Exact information schema/system table availability and column names can vary by workspace and rollout. The exam expects the concept: use metadata queries and Catalog Explorer to find governance gaps.

## Governance with sharing and federation

| Scenario | Governance point |
|----------|------------------|
| Delta Sharing | Provider controls shared assets and recipients; recipient controls local user grants |
| Lakehouse Federation | UC connection and foreign catalog govern access to remote source |
| External table | UC governs table access; external cloud access still needs storage governance |
| Volume | Use UC grants such as `READ VOLUME` and `WRITE VOLUME` |
| Connection | Grant use/manage only to users or jobs that need remote source access |

Do not confuse cloud IAM with Unity Catalog grants. Cloud credentials can control storage access, but normal governed users should access tables by UC names and privileges.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Users cannot find or understand datasets | Comments, tags, owners, lineage, Catalog Explorer search |
| Need enterprise PII classification | Governed tags on sensitive columns |
| Same policy must apply to many tagged columns | ABAC policy |
| Onboard many analysts to all gold tables | Schema-level `SELECT` plus parent `USE` |
| One table has stricter access than sibling tables | Move to restricted schema or table-level grants |
| User still has access after revoke | Check group membership and inherited grants |
| Need impact analysis before change | Catalog Explorer lineage and table history |
| Production job needs write access | Service principal with scoped schema/table privileges |
| Team should manage permissions but not read data | `MANAGE` without `SELECT` |
| Consumers should discover data but not read it | `BROWSE` |

## SQL essentials

Add comments:

```sql
COMMENT ON TABLE main.silver.orders
IS 'Cleaned orders at one row per order_id.';

COMMENT ON COLUMN main.silver.orders.customer_id
IS 'Business customer identifier from CRM.';
```

Add tags:

```sql
ALTER TABLE main.silver.orders
SET TAGS ('domain' = 'sales', 'quality' = 'silver');

ALTER TABLE main.silver.customers
ALTER COLUMN email
SET TAGS ('pii' = 'email');
```

Grant inherited schema read:

```sql
GRANT USE CATALOG ON CATALOG main TO `analysts`;
GRANT USE SCHEMA ON SCHEMA main.gold TO `analysts`;
GRANT SELECT ON SCHEMA main.gold TO `analysts`;
```

Inspect grants:

```sql
SHOW GRANTS ON SCHEMA main.gold;
SHOW GRANTS ON TABLE main.gold.daily_revenue;
```

Transfer ownership to a group:

```sql
ALTER TABLE main.gold.daily_revenue
SET OWNER TO `finance_data_owners`;
```

## PySpark / API essentials

Metadata operations are SQL operations; they can be issued from notebooks or jobs when the principal has permission:

```python
spark.sql("""
COMMENT ON TABLE main.silver.orders
IS 'Cleaned order records, one row per order_id'
""")

spark.sql("""
ALTER TABLE main.silver.orders
SET TAGS ('domain' = 'sales', 'quality' = 'silver')
""")
```

Read governed data by UC name:

```python
orders = spark.table("main.silver.orders")
```

Avoid path-based access for governed tables unless the scenario explicitly requires it and the path is governed through UC external locations/volumes.

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Using `DENY` for Unity Catalog access control | Use least privilege, `REVOKE`, group design, restricted schemas, row filters, masks, or ABAC |
| Granting `ALL PRIVILEGES` on a catalog to a read-only team | Grant parent `USE` plus `SELECT` at the narrowest useful scope |
| Granting table `SELECT` but forgetting parent privileges | Also grant `USE CATALOG` and `USE SCHEMA` |
| Revoking a direct user grant while the user still belongs to a reader group | Remove inherited access through group/schema grants or adjust group design |
| Leaving important tables without comments or owners | Add business comments, owner group, tags, and source metadata |
| Putting sensitive values in tag names/comments | Use classification tags only; do not expose secrets or PII in metadata |
| Making individuals owners of production data objects | Use owner groups for continuity |
| Relying on cloud IAM instead of UC grants for table access | Govern table access through Unity Catalog privileges |
| Placing one restricted table in a broadly readable schema | Move it to a restricted schema or use explicit table-level grants/policies |
| Ignoring lineage before schema changes | Check downstream tables, jobs, dashboards, and queries first |

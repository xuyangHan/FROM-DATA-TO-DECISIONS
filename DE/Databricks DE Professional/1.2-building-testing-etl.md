# 1.2 Building and testing ETL with Lakeflow SDP, SQL, and Spark

## Exam objective

> Build and manage reliable, production-ready data pipelines for batch and streaming data using Lakeflow Spark Declarative Pipelines and Autoloader.
>
> Create and Automate ETL workloads using Jobs via UI/APIs/CLI.
>
> Explain the advantages and disadvantages of streaming tables compared to materialized views.
>
> Use APPLY CHANGES APIs to simplify CDC in Lakeflow Spark Declarative Pipelines.
>
> Compare Spark Structured Streaming and Lakeflow Spark Declarative Pipelines to determine the optimal approach for building scalable ETL pipelines.
>
> Create a pipeline component that uses control flow operators (e.g., if/else, for/each, etc.).
>
> Choose the appropriate configs for environments and dependencies, high memory for notebook tasks, and auto-optimization to disallow retries.
>
> Develop unit and integration tests using assertDataFrameEqual, assertSchemaEqual, DataFrame.transform, and testing frameworks, to ensure code correctness, including a built-in debugger.

## Naming note

- **Lakeflow Spark Declarative Pipelines (SDP)** is the current name for Databricks' declarative pipeline framework. In older material, you may see **Delta Live Tables (DLT)**.
- The exam guide says **APPLY CHANGES**. Current Databricks docs say **AUTO CDC** replaces `APPLY CHANGES` and has the same syntax. For the exam, recognize both as the declarative CDC feature in Lakeflow SDP.
- The exam says **Autoloader**; current docs usually write **Auto Loader** and use the `cloudFiles` source.

## Must know

- **Lakeflow SDP** builds batch and streaming pipelines in SQL or Python using declarative datasets, flows, streaming tables, materialized views, expectations, and event logs.
- **Streaming tables** are for ingestion, append-heavy workloads, CDC targets, and low-latency incremental processing.
- **Materialized views** are for complex joins, aggregations, and query acceleration where cached results can be refreshed.
- **Temporary views** organize intermediate pipeline logic without publishing or materializing a table.
- **Auto Loader** incrementally discovers and processes new files from cloud storage or Unity Catalog volumes through `cloudFiles`.
- **AUTO CDC / APPLY CHANGES** handles CDC ordering, deduplication, deletes, out-of-order events, and SCD Type 1/2 more cleanly than hand-written `MERGE`.
- **Lakeflow Jobs** orchestrate notebooks, Python scripts, Python wheels, SQL tasks, dbt tasks, pipelines, condition tasks, for-each tasks, schedules, notifications, and repairs.
- **Structured Streaming** is still the right tool when you need lower-level stream control, custom sinks, `foreachBatch`, or logic not supported in SDP.
- **Testing** should isolate DataFrame transforms, validate schema and results, and run integration checks against dev targets before promotion.

## Professional focus vs Associate

Associate notes cover basic Auto Loader, jobs, task dependencies, gold objects, and data quality. For Professional, expect design decisions:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know what Auto Loader does | Decide when to use Auto Loader in SDP, Jobs, or Structured Streaming |
| Know jobs have tasks | Choose task types, dependencies, retries, compute, parameters, and repair strategy |
| Know streaming tables and materialized views exist | Pick the correct dataset type for cost, latency, and correctness |
| Know data quality expectations | Decide whether to drop, fail, quarantine, or monitor bad records |
| Know basic unit tests | Build testable transforms with `DataFrame.transform`, `assertDataFrameEqual`, and `assertSchemaEqual` |
| Know bundles deploy workflows | Use bundle targets and parameters to test and promote pipelines safely |

## Production ETL architecture

```text
Cloud storage / message bus / source tables
  -> Auto Loader or streaming read
  -> bronze streaming table
  -> silver streaming table or materialized view with expectations
  -> AUTO CDC / APPLY CHANGES for CDC targets
  -> gold materialized views or Delta tables
  -> Lakeflow Jobs schedule, trigger, monitor, and notify
  -> bundle CI/CD promotes dev -> staging -> prod
```

Design principles:

| Principle | Correct production shape |
|-----------|--------------------------|
| Declarative pipeline logic | Define what each dataset should be, let SDP manage execution and maintenance |
| Environment isolation | Use bundle targets, pipeline parameters, and separate dev/prod catalogs |
| Incremental ingestion | Use Auto Loader for file arrivals rather than repeatedly listing and rereading everything |
| CDC simplicity | Use AUTO CDC / APPLY CHANGES instead of custom merge code when the question gives keys and sequence columns |
| Cost-aware scheduling | Use `Trigger.AvailableNow` or scheduled jobs when continuous low latency is not required |
| Testable code | Put transform logic in functions and test with small fixtures before deploying |
| Observable operation | Use pipeline event logs, expectation metrics, Jobs run history, and repair runs |

## Dataset types in Lakeflow SDP

| Dataset type | How it processes data | Best for | Avoid when |
|--------------|-----------------------|----------|------------|
| Streaming table | Processes each incoming record once, assuming append-oriented input | Ingestion, high-volume files, CDC targets, low-latency transformations | Source is frequently overwritten or requires full recomputation |
| Materialized view | Stores query results and refreshes them, often incrementally | Aggregations, joins, BI acceleration, expensive repeated computations | You need every event processed exactly once as it arrives |
| Temporary view | Recomputed inside the pipeline and not published | Intermediate transformations and reusable logic | Downstream users need to query it directly |

### Streaming tables vs materialized views

| Dimension | Streaming table | Materialized view |
|-----------|-----------------|-------------------|
| Primary use | Incremental ingest or append processing | Precomputed transformations and query acceleration |
| Latency | Good for low-latency/event-driven data | Depends on refresh trigger or schedule |
| Input assumption | Append-oriented data works best | Can reason over current state of upstream data |
| Output | A table maintained by streaming/incremental updates | Cached result of a query |
| Cost shape | Ongoing stream or triggered incremental cost | Refresh cost plus result storage |
| Best exam clue | "New records arrive continuously" | "Dashboard repeatedly runs the same expensive aggregation" |
| Key limitation | Not ideal for arbitrary rewrites of source files | Not always up to date between refreshes |

## Lakeflow SDP vs Structured Streaming

| Factor | Lakeflow SDP | Structured Streaming |
|--------|--------------|----------------------|
| Programming model | Declarative SQL/Python datasets and flows | Imperative Spark streaming query code |
| Managed tables | Yes, pipeline manages Delta tables | You manage sinks and checkpoints |
| Expectations / DQ | Built in | You implement manually |
| Event logs and pipeline graph | Built in | Jobs/Spark UI/logs primarily |
| CDC simplification | AUTO CDC / APPLY CHANGES | Manual `MERGE` or `foreachBatch` logic |
| Custom sinks and side effects | Limited or specific APIs | More flexible with `foreachBatch` |
| Default professional choice | Managed medallion ETL | Custom stream behavior or unsupported pattern |

Decision rule:

| Requirement clue | Better choice |
|------------------|---------------|
| Managed medallion pipeline with quality rules and lineage | Lakeflow SDP |
| Low-code incremental file ingestion into bronze | Lakeflow SDP + Auto Loader |
| CDC with primary key, sequence column, deletes, SCD Type 1/2 | AUTO CDC / APPLY CHANGES in SDP |
| Complex external API writes per microbatch | Structured Streaming with `foreachBatch` |
| Custom checkpoint/sink behavior is central to the solution | Structured Streaming |
| SQL-first BI acceleration | Materialized view or standalone materialized view |

## Auto Loader in production

Auto Loader is the standard pattern for incrementally loading files because it tracks discovered files, supports schema inference/evolution, and can scale to very large file counts.

Python in Lakeflow SDP:

```python
from pyspark import pipelines as dp


@dp.table(
    name="bronze_orders",
    comment="Raw orders ingested from landing volume",
)
def bronze_orders():
    return (
        spark.readStream.format("cloudFiles")
        .option("cloudFiles.format", "json")
        .load("/Volumes/main/landing/orders/")
    )
```

Structured Streaming outside SDP:

```python
query = (
    spark.readStream.format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("cloudFiles.schemaLocation", "/Volumes/main/checkpoints/orders_schema")
    .load("/Volumes/main/landing/orders/")
    .writeStream
    .option("checkpointLocation", "/Volumes/main/checkpoints/orders_stream")
    .trigger(availableNow=True)
    .toTable("main.bronze.orders")
)
```

Production points:

| Topic | Correct approach |
|-------|------------------|
| Schema tracking in SDP | Let SDP manage checkpoint/schema locations when using pipeline-managed Auto Loader |
| Schema tracking in classic streaming | Provide a durable `cloudFiles.schemaLocation` |
| Checkpointing | Use a unique checkpoint per stream/query |
| File location | Prefer UC volumes or governed cloud paths |
| Batch-like file drops | Use `Trigger.AvailableNow` so the job processes current backlog and stops |
| Continuous low latency | Use continuous/regular streaming only when SLA requires it |
| Rate limiting | Use `cloudFiles.maxFilesPerTrigger` or `cloudFiles.maxBytesPerTrigger` to control microbatch size |
| Overwritten input files | Avoid unless explicitly configured and understood; append-only is simpler and safer |

## CDC with AUTO CDC / APPLY CHANGES

Use this when the source provides change events with:

- A stable key or composite key.
- A sequence column such as timestamp, version, or operation number.
- Operation indicators for insert/update/delete.
- A requirement for SCD Type 1 or SCD Type 2 behavior.

Current SQL style:

```sql
CREATE OR REFRESH STREAMING TABLE customers_silver;

CREATE FLOW customers_cdc_flow AS AUTO CDC INTO customers_silver
FROM stream(bronze_customer_cdc)
KEYS (customer_id)
APPLY AS DELETE WHEN operation = 'DELETE'
SEQUENCE BY operation_ts
COLUMNS * EXCEPT (operation, operation_ts)
STORED AS SCD TYPE 1;
```

Exam wording may use the older name:

```sql
APPLY CHANGES INTO LIVE.customers_silver
FROM STREAM(LIVE.bronze_customer_cdc)
KEYS (customer_id)
APPLY AS DELETE WHEN operation = 'DELETE'
SEQUENCE BY operation_ts
COLUMNS * EXCEPT (operation, operation_ts);
```

SCD choice:

| Requirement | Correct CDC behavior |
|-------------|----------------------|
| Keep only the latest version of each key | SCD Type 1 |
| Preserve historical versions for audit or point-in-time reporting | SCD Type 2 |
| Ignore null fields in sparse updates | Use ignore-null update behavior where available |
| Deletes appear as operation records | Use `APPLY AS DELETE WHEN` |
| Multiple events arrive out of order | Use `SEQUENCE BY` so the pipeline applies the correct order |
| No CDC feed, only snapshots | Use AUTO CDC FROM SNAPSHOT where supported |

## Expectations and bad data handling

Expectations define data quality rules on pipeline datasets. The action should match the business requirement.

| Requirement | Expectation behavior |
|-------------|----------------------|
| Track quality metrics but keep records | Warn/expect only |
| Remove bad records from curated output | Drop invalid records |
| Stop the pipeline when invalid data is unacceptable | Fail the update |
| Keep bad records for remediation | Quarantine to a separate table or path |

Python pattern:

```python
from pyspark import pipelines as dp


@dp.table(name="silver_orders")
@dp.expect_or_drop("valid_order_id", "order_id IS NOT NULL")
@dp.expect_or_drop("valid_amount", "amount >= 0")
def silver_orders():
    return (
        spark.readStream.table("bronze_orders")
        .select("order_id", "customer_id", "amount", "order_ts")
    )
```

SQL pattern:

```sql
CREATE OR REFRESH STREAMING TABLE silver_orders (
  CONSTRAINT valid_order_id EXPECT (order_id IS NOT NULL) ON VIOLATION DROP ROW,
  CONSTRAINT valid_amount EXPECT (amount >= 0) ON VIOLATION DROP ROW
)
AS SELECT order_id, customer_id, amount, order_ts
FROM STREAM(bronze_orders);
```

Professional angle: expectations are not just syntax. The exam may ask which action fits reliability: fail for strict regulated data, drop for curated analytics, or quarantine when analysts need clean data but engineers must inspect rejected records.

## Pipeline code with control flow

Pipeline code can use Python control flow when defining datasets. Keep it deterministic and easy to reason about.

Parameterized branch:

```python
from pyspark import pipelines as dp

LOAD_SAMPLE = spark.conf.get("pipeline.load_sample", "false") == "true"


@dp.table(name="bronze_orders")
def bronze_orders():
    path = "/Volumes/main/sample/orders/" if LOAD_SAMPLE else "/Volumes/main/landing/orders/"
    return (
        spark.readStream.format("cloudFiles")
        .option("cloudFiles.format", "json")
        .load(path)
    )
```

For-each style table generation:

```python
from pyspark import pipelines as dp


def make_region_table(region: str):
    @dp.table(name=f"silver_orders_{region}")
    def region_orders():
        return (
            spark.readStream.table("bronze_orders")
            .filter(f"region = '{region}'")
        )


for region_name in ["amer", "emea", "apac"]:
    make_region_table(region_name)
```

Control-flow choice:

| Need | Correct surface |
|------|-----------------|
| Define similar pipeline datasets from a list | Python loop in pipeline source |
| Switch logic based on environment parameter | Pipeline parameter / Spark conf in source |
| Branch a workflow after a task returns a value | Lakeflow Jobs if/else condition task |
| Run a task once per table/region/date | Lakeflow Jobs for-each task |
| Run downstream task only after upstream failure or success | Job dependency `run_if` |

## Automating ETL with Lakeflow Jobs

Lakeflow Jobs are the orchestration layer. They can be created and managed in the UI, REST API, CLI, or bundles.

| Surface | Use for |
|---------|---------|
| UI | Build DAGs, inspect runs, repair failures, configure notifications |
| REST API | Programmatic run, repair, monitoring, and integration with external systems |
| CLI | Local and CI/CD automation, including bundle-based deploy and run |
| Bundles | Source-controlled job, pipeline, task, parameter, library, and target config |

Task design:

| Requirement clue | Correct job design |
|------------------|--------------------|
| One pipeline refresh after files arrive | Pipeline task in a job, possibly with file arrival trigger |
| Same transform for many regions | For-each task over region list |
| Run validation only if ingest produced data | If/else condition based on task value |
| Retry transient read failures | Configure retries with backoff if operation is idempotent |
| Do not run a task more than once | Disable retries/auto-optimization retries for non-idempotent task |
| Failed task needs rerun with fixed parameter | Use repair run or parameter override |
| Production job needs alerts | Configure job/task notifications and thresholds |

## Compute, environment, and dependency configs

| Config area | Correct approach |
|-------------|------------------|
| Jobs compute | Use serverless for supported task types; use classic jobs compute when unsupported features require it |
| All-purpose compute | Avoid for production jobs unless there is a specific reason |
| SQL task | Use a SQL warehouse |
| Pipeline task | Use serverless pipeline where supported; classic pipeline when required |
| Notebook high memory | Use high memory only for notebook tasks that need it |
| Dependencies | Declare task/job/pipeline dependencies; do not rely on ad hoc interactive installs |
| Shared job compute | Useful to reduce startup time; remember libraries and driver state can persist across tasks in one run |
| Separate task compute | Use when isolation matters more than startup-time savings |
| Serverless retries | Leave auto-optimization on for critical idempotent tasks; disable when the task must run at most once |

Exam cue about retries:

| Workload behavior | Retry setting |
|-------------------|---------------|
| Idempotent read/transform/write with safe overwrite or merge | Retries can improve reliability |
| Non-idempotent external side effect, such as sending emails or posting API transactions | Disallow retries or design idempotency keys first |
| Maintenance operation already handled by pipeline predictive optimization | Do not add duplicate custom maintenance retries |

## Unit testing Spark transforms

Use `DataFrame.transform` to keep logic composable and easy to test. It is especially useful for chaining transformations in a clean, linear, readable sequence without nesting.

```python
from pyspark.sql import DataFrame
from pyspark.sql import functions as F


def clean_orders(df: DataFrame) -> DataFrame:
    return (
        df.filter(F.col("order_id").isNotNull())
        .withColumn("order_date", F.to_date("order_ts"))
        .dropDuplicates(["order_id"])
    )
```

Pytest example:

```python
from datetime import date

from pyspark.testing.utils import assertDataFrameEqual, assertSchemaEqual

from orders_etl.transforms import clean_orders


def test_clean_orders(spark):
    input_df = spark.createDataFrame(
        [
            ("o1", 10, "2026-01-01 10:00:00"),
            ("o1", 10, "2026-01-01 10:00:00"),
            (None, 5, "2026-01-02 10:00:00"),
        ],
        schema="order_id STRING, amount INT, order_ts STRING",
    )

    expected = spark.createDataFrame(
        [("o1", 10, "2026-01-01 10:00:00", date(2026, 1, 1))],
        schema="order_id STRING, amount INT, order_ts STRING, order_date DATE",
    )

    result = input_df.transform(clean_orders)

    assertSchemaEqual(result.schema, expected.schema)
    assertDataFrameEqual(result, expected)
```

Chaining example:

```python
def cast_datatypes(df: DataFrame) -> DataFrame:
    return (
        df.withColumn("order_ts", F.to_timestamp("order_ts_raw"))
        .withColumn("amount", F.col("amount_raw").cast("decimal(12,2)"))
    )


def rename_columns(df: DataFrame) -> DataFrame:
    return df.withColumnRenamed("cust_id", "customer_id")


def filter_invalid_records(df: DataFrame) -> DataFrame:
    return df.filter(
        F.col("order_id").isNotNull()
        & F.col("customer_id").isNotNull()
        & F.col("amount").isNotNull()
        & (F.col("amount") >= 0)
    )


clean_orders = (
    raw_orders
    .transform(cast_datatypes)
    .transform(rename_columns)
    .transform(filter_invalid_records)
)
```

Testing focus:

| Test type | What it catches |
|-----------|-----------------|
| Unit test on transform function | Logic errors in filters, joins, columns, and deduplication |
| Schema assertion | Unexpected column type, nullability, or structure changes |
| DataFrame equality assertion | Wrong rows or values |
| Expectation test with malformed records | Whether bad data is dropped, failed, or quarantined correctly |
| Integration test in dev catalog | Whether bundle config, permissions, libraries, pipeline refresh, and tables work together |

## Integration tests and debugger

Integration tests should run in a dev target, not directly against production tables.

```text
databricks bundle validate -t dev
databricks bundle deploy -t dev
databricks bundle run -t dev silver_pipeline
run validation query against dev catalog
```

Good integration checks:

| Check | Example |
|-------|---------|
| Target table exists | Query `catalog.schema.table` in dev |
| Schema is compatible | Compare expected schema against actual |
| Row counts make sense | Bronze input count >= silver valid count |
| Expectations fired as expected | Inspect expectation metrics |
| CDC applied correctly | Verify latest version or SCD history for fixture keys |
| Job status is correct | Use Jobs UI/API/CLI run result |

Debugger use:

- Use the built-in debugger for Python files or notebooks when stepping through transform logic is faster than reading logs.
- Keep production transform logic in importable modules so the same code can be unit tested, debugged, and called from jobs.
- Use small fixtures when debugging; do not debug by repeatedly running against large production tables.

## SQL essentials

Streaming table from files:

```sql
CREATE OR REFRESH STREAMING TABLE main.bronze.events
AS SELECT * FROM STREAM read_files(
  '/Volumes/main/landing/events/',
  format => 'json'
);
```

Materialized view:

```sql
CREATE OR REPLACE MATERIALIZED VIEW main.gold.daily_revenue AS
SELECT order_date, SUM(amount) AS revenue
FROM main.silver.orders
GROUP BY order_date;
```

Quality rule:

```sql
CREATE OR REFRESH STREAMING TABLE main.silver.orders (
  CONSTRAINT valid_order EXPECT (order_id IS NOT NULL) ON VIOLATION DROP ROW
)
AS SELECT *
FROM STREAM(main.bronze.orders);
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Using a materialized view when every new event must be processed with low latency | Use a streaming table or Structured Streaming pipeline |
| Using a streaming table for a static monthly reference file overwritten in place | Use batch processing or a materialized view over the current reference data |
| Hand-writing complex CDC `MERGE` logic when the source has keys and a sequence column | Use AUTO CDC / APPLY CHANGES |
| Ignoring deletes in a CDC feed | Use `APPLY AS DELETE WHEN` or the equivalent CDC delete clause |
| Running continuous Auto Loader for hourly file drops with no low-latency SLA | Schedule Auto Loader with `Trigger.AvailableNow` in a job |
| Retrying a non-idempotent task that sends external side effects | Disable retries or make the operation idempotent before enabling retries |
| Testing directly in the production catalog | Run unit tests locally/in workspace and integration tests in an isolated dev target |
| Putting transformation logic only in notebooks | Move reusable logic into Python modules and call it from notebooks, jobs, or pipelines |
| Confusing Job `for each` with SQL window functions | Use Job `for each` for workflow loops; use SQL windows for analytical calculations |
| Using all-purpose compute for normal production jobs | Use serverless jobs or jobs compute unless a specific unsupported feature requires otherwise |

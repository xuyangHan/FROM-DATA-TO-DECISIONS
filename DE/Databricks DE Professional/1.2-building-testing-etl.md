# 1.2 Building and testing ETL with Lakeflow SDP, SQL, and Spark

## Exam objective

> Build and manage reliable, production-ready data pipelines for batch and streaming data using Lakeflow Spark Declarative Pipelines and Autoloader.
>
> Create and Automate ETL workloads using Jobs via UI/APIs/CLI.
>
> Explain the advantages and disadvantages of streaming tables compared to materialized views.
>
> Use APPLY CHANGES APIs to simplify CDC in Lakeflow Spark Declarative Pipelines.
>
> Compare Spark Structured Streaming and Lakeflow Spark Declarative Pipelines to determine the optimal approach for building scalable ETL pipelines.
>
> Create a pipeline component that uses control flow operators (e.g., if/else, for/each, etc.).
>
> Choose the appropriate configs for environments and dependencies, high memory for notebook tasks, and auto-optimization to disallow retries.
>
> Develop unit and integration tests using assertDataFrameEqual, assertSchemaEqual, DataFrame.transform, and testing frameworks, to ensure code correctness, including a built-in debugger.

## Must know

- **Lakeflow Spark Declarative Pipelines (SDP)** = declarative, managed ETL with streaming tables, materialized views, expectations, and event logs.
- **Auto Loader** (`cloudFiles`) is the standard incremental file ingestion path inside SDP or classic Structured Streaming jobs.
- **Lakeflow Jobs** orchestrate notebooks, Python, SQL, pipelines, and control flow across environments.
- **Streaming tables** ingest/process incrementally; good for append-heavy, low-latency paths.
- **Materialized views** cache precomputed query results; good for expensive repeated aggregates.
- **APPLY CHANGES** (and `APPLY CHANGES INTO`) simplifies CDC/SCD in SDP without hand-written `MERGE` in every layer.
- **Structured Streaming** offers maximum control; SDP offers managed operations, DQ, and UC integration with less boilerplate.
- **Pipeline control flow** includes conditional datasets and parameterized pipeline configuration; Jobs add `if/else` and `for each` at orchestration level.
- **Testing** should validate transforms in isolation (`transform`), schema, and full DataFrame equality before promotion.

## Streaming tables vs materialized views

| Dimension | Streaming table | Materialized view |
|-----------|-----------------|-------------------|
| Primary use | Incremental ingest/transform on arriving data | Precompute expensive queries for BI |
| Latency | Lower for append pipelines | Depends on refresh schedule/trigger |
| Update model | Continuous or triggered incremental processing | Refresh recomputes cached result |
| Cost profile | Steady streaming cost | Pay on refresh + storage of result |
| Best exam clue | "Records arrive continuously" | "Dashboard hits same aggregation 100x/day" |
| Caveat | Not ideal for arbitrary ad hoc rewrites of full history | Stale until refresh completes |

## SDP vs Structured Streaming

| Factor | Lakeflow SDP | Structured Streaming |
|--------|--------------|------------------------|
| Declarative tables/MVs | Yes | No — you write sinks/checkpoints |
| Built-in expectations/DQ | Yes | Manual |
| Event log / lineage UI | Yes | Jobs/cluster logs primarily |
| Fine-grained stream control | Less | More (triggers, state, foreachBatch) |
| Exam default for managed medallion | SDP | Classic jobs when custom stream logic required |

```
Cloud files / sources
  -> Auto Loader or streaming read
  -> SDP streaming table (bronze/silver)
  -> APPLY CHANGES / transforms
  -> gold table or materialized view
  -> Lakeflow Job schedules/refreshes pipeline
```

## APPLY CHANGES (CDC in SDP)

`APPLY CHANGES` tracks insert/update/delete from a change feed and applies them to a target table with less boilerplate than manual `MERGE`.

```sql
-- Simplified exam pattern
APPLY CHANGES INTO LIVE.customers
FROM STREAM(LIVE.customer_cdc)
KEYS (customer_id)
APPLY AS DELETE WHEN cdc_operation = 'DELETE'
SEQUENCE BY event_time
COLUMNS * EXCEPT (cdc_operation)
```

Exam model:

| Need | API |
|------|-----|
| Upsert + deletes from CDC stream | `APPLY CHANGES` / `APPLY CHANGES INTO` |
| One-off batch upsert in notebook job | Delta `MERGE INTO` |
| Track row versions over time | SCD Type 2 columns + `APPLY CHANGES` |

## Automate ETL with Lakeflow Jobs

| Surface | Use for |
|---------|---------|
| UI | Design DAG, retries, notifications, manual runs |
| REST API | CI/CD integration, programmatic repair/rerun |
| CLI | `databricks bundle run`, job submit from pipelines |

Task config exam clues:

| Clue | Config |
|------|--------|
| Notebook OOM on wide/shuffle data | Increase driver/executor memory or use high-memory instance type |
| Idempotent maintenance must not rerun | `max_retries: 0` on that task |
| Predictive optimization / auto-optimize already applied | Do not retry destructive duplicate maintenance |
| Environment-specific catalog | Job parameters or bundle variables |

## Pipeline and job control flow

**Jobs (orchestration layer):**

- `if/else` condition tasks — branch on boolean expression.
- `for each` — loop over JSON array (regions, tables, dates).
- `run_if` — react to upstream success/failure.
- Task values — pass small outputs between tasks.

**SDP (pipeline layer):**

- Parameterized pipeline settings per environment.
- Conditional logic via separate datasets or SQL/Python branches where supported.
- Expectations for data-driven gating (fail/drop on bad data).

For exam questions: **orchestration branching** → Jobs; **dataset-level CDC/DQ** → SDP.

## Testing patterns

```python
from pyspark.sql import DataFrame
from pyspark.testing.utils import assertDataFrameEqual, assertSchemaEqual
from pyspark.sql.types import StructType, StructField, StringType, IntegerType

def clean_orders(df: DataFrame) -> DataFrame:
    return (
        df.filter("order_id IS NOT NULL")
        .dropDuplicates(["order_id"])
    )

def test_clean_orders(spark):
    schema = StructType([
        StructField("order_id", StringType()),
        StructField("amount", IntegerType()),
    ])
    input_df = spark.createDataFrame(
        [("o1", 10), ("o1", 10), (None, 5)],
        schema=schema,
    )
    expected = spark.createDataFrame([("o1", 10)], schema=schema)
    result = input_df.transform(clean_orders)
    assertDataFrameEqual(result, expected)
    assertSchemaEqual(result.schema, expected.schema)
```

| Tool | Purpose |
|------|---------|
| `DataFrame.transform(fn)` | Chain testable pure functions |
| `assertDataFrameEqual` | Row + column equality (order may matter) |
| `assertSchemaEqual` | Schema-only validation |
| `pytest` + `spark` fixture | CI unit tests locally or in GitHub Actions |
| Databricks debugger | Step through notebook/Python in workspace |

Integration tests: run against dev catalog, small fixture tables, or bundle `databricks bundle run` in CI after deploy to dev.

## Auto Loader in production SDP

```python
from pyspark import pipelines as dp

@dp.table
def bronze_orders():
    return (
        spark.readStream.format("cloudFiles")
        .option("cloudFiles.format", "json")
        .option("cloudFiles.schemaLocation", "/Volumes/main/bronze/_schemas/orders")
        .load("/Volumes/main/landing/orders/")
    )
```

Key production settings: unique **checkpoint** per stream, **schema location** for evolution, UC **volume** or governed path, `Trigger.AvailableNow` for micro-batch schedules.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Managed medallion with DQ metrics | Lakeflow SDP |
| Custom `foreachBatch` with external API side effects | Structured Streaming in classic job |
| CDC from change feed into curated table | `APPLY CHANGES` in SDP |
| Nightly file drop, process all new files, stop | Auto Loader + `Trigger.AvailableNow` |
| Loop ETL over 50 regional tables | Job `for each` task |
| Validate transform before merge to main | `pytest` + `assertDataFrameEqual` |

## PySpark / API essentials

```python
# Expectations in SDP (Python)
from pyspark import pipelines as dp

@dp.table
@dp.expect_or_drop("valid_id", "order_id IS NOT NULL")
def silver_orders():
    return spark.readStream.table("LIVE.bronze_orders")
```

## SQL (exam)

```sql
CREATE OR REFRESH STREAMING TABLE main.bronze.events
AS SELECT * FROM STREAM read_files(
  '/Volumes/main/landing/events/',
  format => 'json'
);

CREATE MATERIALIZED VIEW main.gold.daily_revenue AS
SELECT order_date, SUM(amount) AS revenue
FROM main.silver.orders
GROUP BY order_date;
```

- **Catalog Explorer**: streaming tables, materialized views, permissions.

## Exam traps

- Using materialized view when every row must reflect source within seconds of arrival.
- Using streaming table for a static monthly reference file overwritten in place — batch read instead.
- Hand-writing complex `MERGE` when question describes CDC with sequence column — `APPLY CHANGES`.
- Retrying idempotent `OPTIMIZE`/`VACUUM` tasks that should run once (`max_retries: 0`).
- Testing only in production catalog instead of isolated dev + assertions.
- Confusing **Job** `for each` with **SQL** window functions.

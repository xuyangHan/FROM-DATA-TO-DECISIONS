# 3.1 Data transformation, cleansing, and quality

## Exam objective

> Write efficient Spark SQL and PySpark code to apply advanced data transformations, including window functions, joins, and aggregations, to manipulate and analyze large Datasets.
>
> Develop a quarantining process for bad data with Lakeflow Spark Declarative Pipelines, or autoloader in classic jobs.

## Naming note

- **Lakeflow Spark Declarative Pipelines (SDP)** is the current name for Databricks' declarative pipeline framework. Older notes and examples may say **Delta Live Tables (DLT)**.
- The exam says **Autoloader**; current docs usually write **Auto Loader**.
- Current Lakeflow examples use `from pyspark import pipelines as dp`.

## Must know

- Efficient transformations minimize data scanned, shuffled, and materialized: push filters early, select only needed columns, and aggregate before joining when possible.
- Window functions calculate row-level results across related rows without collapsing the dataset grain.
- Joins are often the most expensive transformation; join order, join type, statistics, broadcast eligibility, and skew all matter.
- Aggregations must match the required business grain. A wrong grouping key can silently change the meaning of the dataset.
- Cleansing should standardize types, parse semi-structured fields, normalize values, remove invalid records from curated outputs, and preserve enough raw data for replay.
- Quarantine means isolating invalid records with reason codes, source metadata, and timestamps so they can be audited and fixed.
- Lakeflow SDP expectations can warn, drop, or fail. They also emit quality metrics to the pipeline event log.
- Classic Auto Loader or Structured Streaming jobs implement quarantine by splitting valid and invalid DataFrames and writing both paths intentionally.
- Reader parse modes such as `PERMISSIVE`, `DROPMALFORMED`, and `FAILFAST` control malformed input handling before business quality rules run.
- Quality gates usually belong before silver/gold consumption, not after bad data has already been joined into downstream models.

## Professional focus vs Associate

Associate notes cover the mechanics of joins, deduplication, aggregation, and basic data quality. For Professional, focus on production tradeoffs:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know join types | Choose join type, join order, broadcast strategy, and skew handling |
| Know window syntax | Use windows for deterministic latest-row, ranking, running totals, and lag/lead logic |
| Know aggregates | Preserve grain, pre-aggregate for cost, and choose exact vs approximate functions |
| Know `dropDuplicates` | Define which duplicate wins and preserve rejected rows when needed |
| Know quality checks | Decide whether to warn, drop, fail, or quarantine |
| Know expectations exist | Use expectation metrics and event logs as operational observability |

## Transformation design principles

| Principle | Correct production shape |
|-----------|--------------------------|
| Preserve grain intentionally | Know whether output is one row per event, order, customer, or day |
| Reduce data early | Filter and project before joins, windows, and aggregations |
| Avoid hidden non-determinism | Use explicit ordering for deduplication and latest-row logic |
| Use built-in functions | Prefer Spark SQL/PySpark functions over Python UDFs |
| Split raw and curated logic | Bronze captures source reality; silver cleans and validates |
| Keep bad data observable | Quarantine or emit metrics instead of silently dropping rows |
| Make rules testable | Put transform logic in functions and assert output rows/schema |
| Monitor expensive operations | Use Query Profile, Spark UI, and pipeline event logs |

## Window functions

Window functions operate over a partition of rows and return a value for each input row. They are ideal when the output keeps row-level detail.

| Need | Window function pattern |
|------|-------------------------|
| Latest row per key | `row_number()` over key ordered by event time/version descending |
| Top N per group | `rank()` or `dense_rank()` over group ordered by metric |
| Running total | Aggregate window with frame from unbounded preceding to current row |
| Previous or next value | `lag()` / `lead()` |
| Dedup with deterministic winner | `row_number()` with complete tie-breaker order |
| Compare row to group metric | `avg()` / `sum()` over partition |

PySpark example:

```python
from pyspark.sql import functions as F
from pyspark.sql.window import Window


latest_order_window = (
    Window
    .partitionBy("order_id")
    .orderBy(F.col("event_ts").desc(), F.col("_ingest_ts").desc())
)

latest_orders = (
    orders
    .withColumn("_rn", F.row_number().over(latest_order_window))
    .filter(F.col("_rn") == 1)
    .drop("_rn")
)

running_window = (
    Window
    .partitionBy("region")
    .orderBy("order_date")
    .rowsBetween(Window.unboundedPreceding, Window.currentRow)
)

daily = orders.withColumn(
    "running_revenue",
    F.sum("amount").over(running_window),
)
```

SQL example:

```sql
SELECT
  order_id,
  customer_id,
  amount,
  ROW_NUMBER() OVER (
    PARTITION BY order_id
    ORDER BY event_ts DESC, _ingest_ts DESC
  ) AS rn,
  LAG(amount, 1) OVER (
    PARTITION BY customer_id
    ORDER BY event_ts
  ) AS previous_amount
FROM main.silver.orders;
```

Window design checks:

| Check | Why it matters |
|-------|----------------|
| Does the window need `PARTITION BY`? | Without it, Spark may process the whole dataset as one group |
| Is the `ORDER BY` deterministic? | Ties can produce inconsistent winners |
| Is the frame correct? | Running totals need a frame; ranking functions do not use frame clauses |
| Is input already reduced? | Window operations can be expensive after wide joins |
| Is latest by event time or ingest time? | Business correctness depends on the chosen ordering |

## Joins at scale

Join selection is both a correctness and performance decision.

| Join type | Use when |
|-----------|----------|
| Inner join | Keep only matching records |
| Left join | Preserve all fact rows while adding dimension attributes |
| Left semi join | Keep rows from left side that have a match; useful existence filter |
| Left anti join | Find missing foreign keys, duplicates to reject, or orphan records |
| Full outer join | Reconcile two datasets and inspect differences |
| Cross join | Rare; only when Cartesian product is truly required |
| Stream-static join | Enrich each streaming microbatch with a static Delta dimension |
| Stream-stream join | Match two streams; requires careful event-time and watermark design |

Performance rules:

| Technique | Correct use |
|-----------|-------------|
| Filter before join | Reduce shuffle input size |
| Select needed columns | Avoid carrying wide payload columns through joins |
| Join smaller tables first | Reduce intermediate result size |
| Broadcast small dimensions | Avoid shuffle when the dimension safely fits in memory |
| Use fresh statistics | Let the optimizer choose better plans |
| Rely on AQE where possible | Databricks can adapt join strategies and handle skew at runtime |
| Persist intermediate results selectively | Useful when query planning or recomputation becomes expensive |
| Avoid cross joins | They multiply row counts and cost quickly |

PySpark examples:

```python
from pyspark.sql import functions as F
from pyspark.sql.functions import broadcast


clean_fact = (
    fact
    .filter(F.col("order_ts") >= "2026-01-01")
    .select("order_id", "customer_id", "product_id", "amount", "order_ts")
)

enriched = clean_fact.join(
    broadcast(dim_customers.select("customer_id", "segment")),
    on="customer_id",
    how="left",
)

missing_customers = clean_fact.join(
    dim_customers.select("customer_id"),
    on="customer_id",
    how="left_anti",
)
```

SQL examples:

```sql
-- Foreign-key quality check
SELECT o.*
FROM main.silver.orders o
LEFT ANTI JOIN main.silver.customers c
ON o.customer_id = c.customer_id;
```

```sql
-- Keep only orders with known customers, without adding customer columns
SELECT o.*
FROM main.silver.orders o
LEFT SEMI JOIN main.silver.customers c
ON o.customer_id = c.customer_id;
```

Join decision notes:

| Requirement clue | Correct choice |
|------------------|----------------|
| "Find orders with no matching customer" | `left_anti` |
| "Filter orders to only known customers" | `left_semi` |
| "Keep all orders even when dimension is missing" | `left` join |
| "Small dimension table" | Broadcast dimension if it safely fits |
| "Highly skewed join key" | Let AQE handle skew; consider salting only when needed |
| "Timestamp range join is slow" | Consider range join optimization/hints |
| "Streaming facts with slowly changing dimension" | Stream-static join, with awareness that static side version can change |

## Aggregations and grain

Aggregation changes row grain. Before writing code, state the output grain explicitly.

| Need | Pattern |
|------|---------|
| Daily revenue | Group by `order_date` |
| Revenue by customer and day | Group by `customer_id`, `order_date` |
| Distinct users estimate | `approx_count_distinct` when exactness is not required |
| Percentiles at scale | Approximate percentile functions when acceptable |
| BI acceleration | Materialized view for repeated expensive aggregates |
| Reduce join cost | Pre-aggregate fact table before joining dimensions |

PySpark:

```python
daily_revenue = (
    orders
    .filter(F.col("order_status") == "paid")
    .groupBy("order_date")
    .agg(
        F.sum("amount").alias("revenue"),
        F.countDistinct("order_id").alias("order_count"),
        F.approx_count_distinct("customer_id").alias("approx_customers"),
    )
)
```

SQL:

```sql
CREATE OR REPLACE MATERIALIZED VIEW main.gold.daily_revenue AS
SELECT
  order_date,
  SUM(amount) AS revenue,
  COUNT(DISTINCT order_id) AS order_count,
  approx_count_distinct(customer_id) AS approx_customers
FROM main.silver.orders
WHERE order_status = 'paid'
GROUP BY order_date;
```

Aggregation checks:

| Check | Why it matters |
|-------|----------------|
| Is the grouping key complete? | Missing a key can over-aggregate |
| Is exact distinct required? | Exact distinct can be expensive |
| Is `collect_list` bounded? | High-cardinality arrays can pressure executors |
| Should aggregate happen before join? | Reduces fact table size before dimension enrichment |
| Is this repeated by dashboards? | Materialized view can reduce repeated compute |

## Cleansing patterns

Cleansing converts source-shaped data into reliable, typed, standardized silver data.

| Cleansing need | Common implementation |
|----------------|-----------------------|
| Type conversion | `cast`, `to_timestamp`, `to_date`, `try_cast` in SQL |
| Standard values | `upper`, `lower`, `trim`, mapping tables |
| Null handling | `coalesce`, required-field checks, default values only when valid |
| Semi-structured parsing | `from_json`, `schema_of_json`, `explode`, `get_json_object` |
| Duplicate resolution | Window with explicit order, then quarantine non-winners if needed |
| Referential checks | Anti join against dimension/reference table |
| Range checks | Boolean validation rules or Delta `CHECK` constraints |
| PII shaping | Mask, hash, tokenize, or separate restricted columns according to governance |

Example:

```python
from pyspark.sql import functions as F


staged_orders = (
    bronze_orders
    .withColumn("order_id", F.trim("order_id"))
    .withColumn("customer_id", F.trim("customer_id"))
    .withColumn("order_ts", F.to_timestamp("order_ts_raw"))
    .withColumn("amount", F.col("amount_raw").cast("decimal(12,2)"))
    .withColumn("order_status", F.upper(F.trim("order_status")))
)
```

Avoid using placeholder values such as `-1` or `"UNKNOWN"` as a substitute for a quality rule unless the business explicitly defines them and downstream users understand the semantics.

## Quality rule categories

| Category | Example rule | Typical implementation |
|----------|--------------|------------------------|
| Completeness | `order_id IS NOT NULL` | Expectation, filter, `NOT NULL` |
| Validity | `amount >= 0` | Expectation, `CHECK`, quarantine |
| Conformity | `status IN (...)` | Standardize then validate |
| Type validity | Timestamp string parses successfully | Cast then check for null |
| Uniqueness | One current row per business key | Window dedup + duplicate report |
| Referential integrity | Every `customer_id` exists | Left anti join |
| Freshness | Max event time within SLA | Aggregate assertion/monitor |
| Volume | Row count not unexpectedly zero or 10x | Job assertion/metric |
| Distribution | Null/error rate within threshold | Aggregate quality metrics |

Action decision:

| Requirement clue | Correct action |
|------------------|----------------|
| Need full source audit trail | Warn/track metrics and keep raw in bronze |
| Bad rows must not reach silver | Drop or quarantine before silver write |
| Engineers must inspect rejected records | Quarantine invalid rows with reason codes |
| Any violation indicates upstream incident | Fail the pipeline/job |
| Rule must apply to all future writes | Delta `CHECK` or `NOT NULL` constraint |
| Managed pipeline needs quality metrics | Lakeflow expectations |

## Quarantine table design

A useful quarantine table is not just a copy of failed rows. It explains why the row was rejected and where it came from.

Recommended quarantine columns:

| Column | Purpose |
|--------|---------|
| Raw source columns | Preserve the failed input |
| `_quarantine_reason` | Human-readable rule failure |
| `_quarantine_rule` | Stable rule identifier |
| `_quarantined_at` | Time rejected |
| `_source_file` / `_source_system` | Trace back to source |
| `_ingest_ts` | Compare arrival and rejection timing |
| `_batch_id` / `_commit_version` | Tie failure to a run or source version |
| `_rescued_data` | Preserve unexpected schema fields |

## Malformed record parsing modes

File reader options handle structurally bad input records at read/parse time. They are related to, but not the same as, business data quality rules.

| Mode | Behavior | Use when |
|------|----------|----------|
| `PERMISSIVE` | Keeps malformed records and places corrupt text in a corrupt-record column when configured | Bronze ingestion should preserve evidence for audit/replay |
| `DROPMALFORMED` | Silently skips malformed records | Rarely, when the source is noisy and losing bad rows is acceptable |
| `FAILFAST` | Fails the read/job as soon as malformed data is found | Any malformed input means the batch should stop |

Common related options:

| Option | Purpose |
|--------|---------|
| `.option("mode", "PERMISSIVE")` | Default-style behavior for CSV/JSON readers; preserve malformed rows instead of failing |
| `.option("mode", "DROPMALFORMED")` | Drops malformed rows before they enter the DataFrame |
| `.option("mode", "FAILFAST")` | Raises an error on malformed records |
| `.option("columnNameOfCorruptRecord", "_corrupt_record")` | Names the column that stores the bad raw record in permissive parsing |
| `.option("badRecordsPath", "...")` | Stores bad records and parse failure details outside the main output path |
| `.option("rescuedDataColumn", "_rescued_data")` | Auto Loader rescued-data column for fields that do not match the target schema |

Example:

```python
raw_orders = (
    spark.readStream.format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("cloudFiles.schemaLocation", "/Volumes/main/bronze/_schemas/orders")
    .option("mode", "PERMISSIVE")
    .option("columnNameOfCorruptRecord", "_corrupt_record")
    .option("rescuedDataColumn", "_rescued_data")
    .load("/Volumes/main/landing/orders/")
)
```

Exam lens:

- Prefer `PERMISSIVE` plus `_corrupt_record`, `_rescued_data`, or a quarantine table when bad records must remain auditable.
- Be cautious with `DROPMALFORMED`: it can make quality metrics look better by removing evidence before you count it.
- Use `FAILFAST` for strict feeds where malformed input should trigger incident handling.
- Parser modes catch malformed records such as invalid JSON/CSV structure. They do not replace checks like `amount >= 0`, required columns, duplicate handling, or referential integrity.
- Built-in parsing functions such as `from_json`/`from_csv` have more limited mode support than file readers; do not assume `DROPMALFORMED` works everywhere.

## Classic Auto Loader quarantine pattern

Classic jobs split valid and invalid records explicitly, then write each output.

```python
from pyspark.sql import functions as F


raw_orders = (
    spark.readStream.format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("cloudFiles.schemaLocation", "/Volumes/main/bronze/_schemas/orders")
    .option("mode", "PERMISSIVE")
    .option("columnNameOfCorruptRecord", "_corrupt_record")
    .option("rescuedDataColumn", "_rescued_data")
    .load("/Volumes/main/landing/orders/")
    .withColumn("_ingest_ts", F.current_timestamp())
    .withColumn("_source_file", F.input_file_name())
)

staged = (
    raw_orders
    .withColumn("order_ts", F.to_timestamp("order_ts_raw"))
    .withColumn("amount", F.col("amount_raw").cast("decimal(12,2)"))
)

valid_condition = (
    F.col("order_id").isNotNull()
    & F.col("order_ts").isNotNull()
    & F.col("amount").isNotNull()
    & (F.col("amount") >= 0)
)

valid_orders = staged.filter(valid_condition)

invalid_orders = (
    staged
    .filter(~valid_condition)
    .withColumn("_quarantine_rule", F.lit("orders_required_fields_v1"))
    .withColumn(
        "_quarantine_reason",
        F.concat_ws(
            "; ",
            F.when(F.col("order_id").isNull(), F.lit("order_id is null")),
            F.when(F.col("order_ts").isNull(), F.lit("order_ts is invalid")),
            F.when(F.col("amount").isNull(), F.lit("amount is invalid")),
            F.when(F.col("amount") < 0, F.lit("amount is negative")),
        ),
    )
    .withColumn("_quarantined_at", F.current_timestamp())
)
```

Use `foreachBatch` when one input stream must write to multiple Delta sinks.

```python
def write_valid_and_quarantine(batch_df, batch_id):
    staged = batch_df.withColumn("_batch_id", F.lit(batch_id))

    valid = staged.filter(valid_condition)
    invalid = (
        staged
        .filter(~valid_condition)
        .withColumn("_quarantine_rule", F.lit("orders_required_fields_v1"))
        .withColumn("_quarantined_at", F.current_timestamp())
    )

    valid.write.mode("append").saveAsTable("main.silver.orders")
    invalid.write.mode("append").saveAsTable("main.quarantine.orders_invalid")


(
    staged.writeStream
    .foreachBatch(write_valid_and_quarantine)
    .option("checkpointLocation", "/Volumes/main/checkpoints/orders_quality")
    .trigger(availableNow=True)
    .start()
)
```

Professional caution: when using `foreachBatch`, make writes idempotent or use the `batch_id` to avoid duplicates if a batch is retried.

## Lakeflow SDP expectations and quarantine

Expectations attach quality rules to pipeline datasets and emit metrics. Choose the violation policy based on the requirement.

| Policy | Result | Use when |
|--------|--------|----------|
| Warn / `expect` | Keep rows and record metrics | Need visibility but full capture |
| Drop / `expect_or_drop` | Remove invalid rows and record metrics | Invalid rows should not continue |
| Fail / `expect_or_fail` | Stop the update | Any invalid row is unacceptable |
| Quarantine table | Write invalid rows separately | Need audit/remediation path |

```python
from pyspark import pipelines as dp


@dp.table(name="silver_orders")
@dp.expect_or_drop("valid_order_id", "order_id IS NOT NULL")
@dp.expect_or_drop("valid_amount", "amount >= 0")
def silver_orders():
    return spark.readStream.table("bronze_orders_raw")
```

Quarantine with separate valid/invalid datasets:

```python
from pyspark import pipelines as dp
from pyspark.sql import functions as F


def staged_orders():
    return (
        spark.readStream.table("bronze_orders_raw")
        .withColumn("order_ts", F.to_timestamp("order_ts_raw"))
        .withColumn("amount", F.col("amount_raw").cast("decimal(12,2)"))
    )


@dp.table(name="silver_orders")
def silver_orders():
    return staged_orders().filter(
        "order_id IS NOT NULL AND order_ts IS NOT NULL AND amount >= 0"
    )


@dp.table(name="orders_quarantine")
def orders_quarantine():
    return (
        staged_orders()
        .filter("order_id IS NULL OR order_ts IS NULL OR amount IS NULL OR amount < 0")
        .withColumn("_quarantine_rule", F.lit("orders_required_fields_v1"))
        .withColumn("_quarantined_at", F.current_timestamp())
    )
```

Use expectations for metrics and enforcement. Use a quarantine table when the invalid rows themselves must be retained for investigation.

## Delta constraints

Delta constraints enforce simple table-level guarantees at write time.

```sql
ALTER TABLE main.silver.orders
ADD CONSTRAINT positive_amount CHECK (amount >= 0);
```

```sql
ALTER TABLE main.silver.orders
ALTER COLUMN order_id SET NOT NULL;
```

Important points:

- Existing rows must satisfy the constraint before it can be added.
- `CHECK` and `NOT NULL` constraints fail future writes that violate them.
- Informational primary key and foreign key constraints are useful metadata but do not enforce uniqueness or referential integrity.
- Constraints complement pipeline quality checks; they do not replace quarantine when rejected rows must be preserved.

## Observability and performance

| Surface | Use it for |
|---------|------------|
| Lakeflow SDP event log | Expectation counts, failed updates, flow progress, data quality trends |
| Lakeflow Pipelines UI | Graph, expectation metrics, refresh status |
| Query Profile | Expensive joins, windows, aggregations, spills, skew |
| Spark UI | Shuffle size, task skew, stage failures |
| Delta history | Writes, schema changes, operation metrics |
| Catalog Explorer | Constraints, lineage, table properties, comments, permissions |

Performance reminders:

- AQE is enabled by default and can adapt join strategies, coalesce shuffle partitions, and handle skew.
- Broadcast hints are useful only when the small side truly fits in memory.
- Avoid `collect()` for quality checks on large datasets; compute counts/metrics in Spark.
- Cache only when reused and only after reducing data.
- Materialize intermediate results when a very complex query repeatedly re-plans or recomputes expensive branches.
- Prefer materialized views for repeated business aggregates used by dashboards.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Latest record per business key | Window + `row_number` with deterministic ordering |
| Running total by group | Window aggregate with explicit frame |
| Missing dimension keys | `left_anti` join into quarantine/report |
| Keep only matching keys without adding columns | `left_semi` join |
| Small dimension enrichment | Broadcast join if the dimension fits |
| Expensive repeated aggregate | Materialized view |
| Bad rows must be auditable | Quarantine Delta table |
| Pipeline must stop on critical invalid data | `expect_or_fail`, assertion, or Delta constraint |
| Managed quality metrics | Lakeflow expectations |
| High-volume Auto Loader job with invalid rows | Split valid/quarantine outputs, often with `foreachBatch` |

## SQL essentials

```sql
-- Latest row per key
CREATE OR REPLACE TABLE main.silver.orders_latest AS
SELECT * EXCEPT (rn)
FROM (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY order_id
      ORDER BY event_ts DESC, _ingest_ts DESC
    ) AS rn
  FROM main.bronze.orders_raw
)
WHERE rn = 1;
```

```sql
-- Quarantine invalid rows with a reason
INSERT INTO main.quarantine.orders_invalid
SELECT
  *,
  CASE
    WHEN order_id IS NULL THEN 'order_id is null'
    WHEN TRY_CAST(amount_raw AS DECIMAL(12,2)) IS NULL THEN 'amount is invalid'
    WHEN TRY_CAST(amount_raw AS DECIMAL(12,2)) < 0 THEN 'amount is negative'
    ELSE 'unknown quality failure'
  END AS _quarantine_reason,
  CURRENT_TIMESTAMP() AS _quarantined_at
FROM main.bronze.orders_raw
WHERE order_id IS NULL
   OR TRY_CAST(amount_raw AS DECIMAL(12,2)) IS NULL
   OR TRY_CAST(amount_raw AS DECIMAL(12,2)) < 0;
```

```sql
-- Find referential quality failures
SELECT o.*
FROM main.silver.orders o
LEFT ANTI JOIN main.silver.customers c
ON o.customer_id = c.customer_id;
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Using `dropDuplicates` without defining which duplicate wins | Use `row_number` with a deterministic ordering rule |
| Running a window over the whole table when logic is per customer/order | Add the correct `PARTITION BY` |
| Broadcasting a dimension that is too large for executor memory | Let the optimizer choose or reduce/filter the dimension first |
| Silently filtering invalid rows | Quarantine them or emit expectation metrics and alerts |
| Checking quality only after gold tables are built | Validate before data reaches silver/gold consumers |
| Using `collect()` to validate large datasets | Compute quality metrics with Spark aggregations |
| Applying business merges directly to raw bronze | Keep bronze replayable and apply curated logic in silver |
| Adding a Delta `CHECK` constraint before cleaning existing violations | Clean or quarantine existing bad rows, then add the constraint |
| Assuming primary key constraints enforce uniqueness | Use deduplication checks/windows; treat PK/FK metadata as informational |
| Failing the whole pipeline for non-critical known bad rows | Quarantine/drop invalid rows and alert when that satisfies the requirement |

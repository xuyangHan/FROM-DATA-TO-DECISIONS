# 3.1 Data transformation, cleansing, and quality

## Exam objective

> Write efficient Spark SQL and PySpark code to apply advanced data transformations, including window functions, joins, and aggregations, to manipulate and analyze large Datasets.
>
> Develop a quarantining process for bad data with Lakeflow Spark Declarative Pipelines, or autoloader in classic jobs.

## Must know

- **Efficient transforms** push filters early, avoid unnecessary shuffles, broadcast small dimensions, and use column pruning.
- **Window functions** compute per-partition rankings, running totals, and lag/lead without collapsing grain.
- **Joins** choice (broadcast, sort-merge, skew hints) drives cost on large datasets.
- **Aggregations** should match required grain; pre-aggregate before joining when possible.
- **Quarantine** = isolate invalid rows with reason codes; do not silently drop without observability.
- **SDP expectations** can drop, fail, or warn; classic jobs use side tables + job failure.
- Quality gates belong in **silver**, before gold consumption.

## Advanced transformations

### Window functions

```python
from pyspark.sql.window import Window
from pyspark.sql.functions import col, row_number, sum as _sum, lag

w = Window.partitionBy("customer_id").orderBy(col("order_ts").desc())

ranked = (
    orders
    .withColumn("rn", row_number().over(w))
    .filter(col("rn") == 1)  # latest order per customer
)

w_running = Window.partitionBy("region").orderBy("order_date").rowsBetween(Window.unboundedPreceding, 0)
daily = orders.withColumn("running_revenue", _sum("amount").over(w_running))
```

```sql
SELECT
  customer_id,
  order_id,
  amount,
  ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_ts DESC) AS rn,
  LAG(amount, 1) OVER (PARTITION BY customer_id ORDER BY order_ts) AS prev_amount
FROM main.silver.orders;
```

### Join efficiency

| Technique | When |
|-----------|------|
| `broadcast(small_df)` | Dimension fits in memory |
| Filter before join | Reduce shuffle input size |
| Salting / AQE skew join | Extreme key skew |
| `left_anti` | Find orphans / missing FKs for DQ |

```python
from pyspark.sql.functions import broadcast

fact.join(broadcast(dim_customers), on="customer_id", how="inner")
```

### Aggregations at scale

- Aggregate to required grain **before** joining wide fact to many dimensions.
- Use `approx_count_distinct` when exact distinct is unnecessary.
- Avoid `collect_list` on high-cardinality groups (driver/executor pressure).

## Quarantine patterns

| Pattern | Mechanism | Best for |
|---------|-----------|----------|
| SDP expect or drop | `@dp.expect_or_drop` | Managed pipeline, metric in event log |
| SDP expect or fail | `@dp.expect_or_fail` | Hard stop on violation |
| Classic split + append | Filter valid/invalid DataFrames | Auto Loader + notebook job |
| SQL quarantine table | `INSERT` invalid rows with reason | SQL-heavy teams |

```python
from pyspark.sql.functions import col, current_timestamp, lit

valid_cond = (
    col("order_id").isNotNull()
    & col("amount").isNotNull()
    & (col("amount") >= 0)
)

valid = staged.filter(valid_cond)
invalid = (
    staged.filter(~valid_cond)
    .withColumn("_quarantine_reason", lit("null id or negative amount"))
    .withColumn("_quarantined_at", current_timestamp())
)

invalid.write.format("delta").mode("append").saveAsTable("main.quarantine.orders_invalid")
valid.write.format("delta").mode("overwrite").saveAsTable("main.silver.orders")
```

```python
from pyspark import pipelines as dp

@dp.table
@dp.expect_or_drop("valid_amount", "amount >= 0")
@dp.expect_or_fail("valid_order_id", "order_id IS NOT NULL")
def silver_orders():
    return spark.readStream.table("LIVE.bronze_orders_raw")
```

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Rank/latest row per key | Window + `row_number` |
| Running total by day | Window with frame |
| Find orders with no customer | `left_anti` join |
| Audit rejected records | Quarantine Delta table |
| Pipeline must stop on bad data | `expect_or_fail` or job exception |
| High-volume file ingest + bad rows | Auto Loader + split write to quarantine |

```python
# Skew-aware join hint (when exam mentions skewed key)
fact.hint("skew", "customer_id").join(dim, on="customer_id")
```

## SQL (exam)

```sql
ALTER TABLE main.silver.orders
ADD CONSTRAINT positive_amount CHECK (amount >= 0);

CREATE OR REPLACE TABLE main.quarantine.orders_invalid AS
SELECT *, 'failed amount check' AS quarantine_reason, current_timestamp() AS quarantined_at
FROM main.bronze.orders_raw
WHERE order_id IS NULL OR amount < 0;
```

## UI / product surfaces

- **SDP event log**: expectation pass/fail counts in `details:flow_progress.data_quality.expectations`.
- **Query profile**: expensive joins/windows after deployment.
- **Catalog Explorer**: quarantine table lineage and constraints on silver.

## Exam traps

- `dropDuplicates` without defining which duplicate wins.
- Window without `partitionBy` when question implies per-group logic.
- Broadcasting a large dimension table.
- Silently filtering bad rows with no quarantine or metric.
- Running quality only at gold while bad data already propagated through joins.
- Using `collect()` to validate quality on large datasets.

# 5.1 Monitoring

## Exam objective

> Use system tables for observability over resource utilization, cost, auditing and workload monitoring.
>
> Use Query Profiler UI and Spark UI to monitor workloads.
>
> Use the Databricks REST APIs/Databricks CLI for monitoring jobs and pipelines.
>
> Use Lakeflow Spark Declarative Pipelines Event Logs to monitor pipelines.

## Must know

- **System tables** in `system` catalog expose billing, audit, jobs, pipelines, and compute history (availability varies by account/cloud).
- **`system.billing.usage`**: DBU/cost attribution — tie spend to workspace, SKU, tags.
- **`system.access.audit`**: who did what — compliance and security investigations.
- **Jobs / pipeline system tables**: run history, task outcomes, durations at account scale.
- **Query Profile** (SQL warehouse / DBSQL): operator tree, time per node, I/O — best for SQL bottlenecks.
- **Spark UI**: stages, tasks, shuffle, spill, skew — best for PySpark/Spark jobs.
- **REST API / CLI**: automate monitoring (`jobs/runs/list`, `pipelines/events`, `bundle` status).
- **SDP event log**: pipeline updates, flow progress, expectation metrics, errors.

## System tables (exam awareness)

| Table (examples) | Purpose |
|------------------|---------|
| `system.billing.usage` | Cost/DBU monitoring |
| `system.access.audit` | Audit trail |
| `system.lakeflow.jobs` | Job run metadata |
| `system.lakeflow.job_task_run_timeline` | Task-level timing |
| `system.lakeflow.pipelines` | Pipeline updates |
| `system.compute.clusters` | Cluster lifecycle |

```sql
SELECT
  usage_date,
  sku_name,
  SUM(usage_quantity) AS dbus
FROM system.billing.usage
WHERE usage_date >= current_date() - 7
GROUP BY usage_date, sku_name
ORDER BY usage_date DESC;
```

## Query Profile vs Spark UI

| Tool | Best for | Look for |
|------|----------|----------|
| Query Profile | DBSQL / Photon SQL | Scan size, join order, broadcast vs shuffle |
| Spark UI | Notebooks, jobs, pipelines | Stage skew, spill, GC, task duration |
| Driver log | Failures before Spark plan | Import errors, auth, OOM stack traces |
| SDP event log | Pipeline DQ and flow errors | Expectation failures, upstream flow |

## SDP event log

Key fields for data quality:

```sql
-- Pattern: inspect pipeline events (exact table name per account docs)
SELECT timestamp, message, details
FROM event_log(TABLE main.silver.orders_pipeline)
WHERE message LIKE '%data_quality%'
ORDER BY timestamp DESC;
```

Expectation metrics path: `details:flow_progress.data_quality.expectations` with `passed_records` / `failed_records`.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Finance asks daily DBU by workspace | `system.billing.usage` |
| Security asks who dropped a table | `system.access.audit` |
| SQL dashboard query suddenly slow | Query Profile |
| PySpark job straggler tasks | Spark UI → Stages |
| Pipeline expectation failures | SDP event log |
| CI must fail if last job run failed | Jobs REST API in pipeline |

## UI / product surfaces

- **Lakeflow Jobs → Run history**: duration trends, failures, task timeline.
- **Lakeflow Pipelines → Event log**: per-update status, DQ metrics.
- **SQL → Query history / Query profile**: SQL performance.
- **Compute → Spark UI**: linked from job run or cluster page.

## Exam traps

- Using only notebook `display()` for production observability.
- Ignoring task-level timeline when driver says success but SLA missed (slow task).
- Checking Spark UI for pure SQL warehouse query — use Query Profile instead.
- Assuming all system tables exist without account enablement.
- Looking only at cluster CPU while root cause is massive shuffle (Spark UI stages).

# 5.1 Monitoring

## Exam objective

> Use system tables for observability over resource utilization, cost, auditing and workload monitoring.
>
> Use Query Profiler UI and Spark UI to monitor workloads.
>
> Use the Databricks REST APIs/Databricks CLI for monitoring jobs and pipelines.
>
> Use Lakeflow Spark Declarative Pipelines Event Logs to monitor pipelines.

## Naming note

- Current docs use **Lakeflow Jobs** and **Lakeflow Spark Declarative Pipelines (SDP)**. Older material may say Workflows and Delta Live Tables.
- The jobs system tables live under `system.lakeflow`. Databricks docs note that `system.workflow` was the older schema name with equivalent content.
- The exam says **Query Profiler UI**; Databricks docs usually call it **Query Profile**.

## Must know

- **System tables** in the `system` catalog provide account-level observability for billing, audit, jobs, pipelines, compute, query history, lineage, and more.
- System tables are read-only Unity Catalog objects. Users need `USE CATALOG`, `USE SCHEMA`, and `SELECT` privileges on the relevant system schemas.
- Some system tables are global and some are regional; some are public preview or cloud-dependent.
- **`system.billing.usage`** tracks billable usage and is the main source for cost/DBU attribution.
- **`system.access.audit`** tracks audit events for security, compliance, and forensic questions.
- **`system.lakeflow.job_run_timeline`** and **`system.lakeflow.job_task_run_timeline`** support account-wide job run and task monitoring.
- **`system.compute.node_timeline`**, **`system.compute.clusters`**, **`system.compute.warehouses`**, and **`system.compute.warehouse_events`** support compute and warehouse monitoring.
- **Query Profile** is the right performance tool for SQL warehouse and SQL query bottlenecks.
- **Spark UI** is the right performance tool for Spark stages, tasks, shuffle, spill, skew, and executor behavior.
- **Lakeflow SDP event logs** are the primary observability source for pipeline updates, flow progress, data quality expectation metrics, errors, and lineage.
- **REST APIs and CLI** are used to automate monitoring, fetch run status, integrate with CI/CD, and build external dashboards or alerting workflows.

## Professional focus vs Associate

Associate troubleshooting notes cover run history, Jobs UI views, and Spark UI basics. For Professional, focus on choosing the correct monitoring source and building repeatable observability:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Open Jobs run history | Query account-wide job trends with `system.lakeflow` tables |
| Inspect one failed run | Build monitoring across jobs, tasks, workspaces, and time windows |
| Use Spark UI for a slow job | Choose Query Profile vs Spark UI based on workload type |
| Read task logs | Correlate logs with system tables, cost, audit events, and pipeline event logs |
| Know pipeline event logs exist | Extract expectation metrics and failure patterns from event log JSON |
| Know CLI/API commands | Automate status checks in CI/CD and operations workflows |

## Monitoring decision map

| Question | Best source |
|----------|-------------|
| How much did Databricks cost by day, workspace, SKU, or tag? | `system.billing.usage` |
| Who changed/dropped/accessed an object? | `system.access.audit` |
| Which jobs are slow or failing across the account? | `system.lakeflow.job_run_timeline` |
| Which task is the bottleneck inside a job? | Jobs UI timeline or `system.lakeflow.job_task_run_timeline` |
| Why did a SQL query get slower? | Query Profile and `system.query.history` |
| Why did a Spark task spill or skew? | Spark UI stages/tasks/executors |
| Are pipeline expectations failing? | Lakeflow SDP event log |
| Is compute underused or saturated? | `system.compute.node_timeline`, Spark UI, warehouse metrics |
| Did a bundle-deployed job drift in the UI? | Bundle config plus Jobs UI read-only/deployment metadata |
| Does CI need to fail when a job/pipeline fails? | Jobs/Pipelines REST API or CLI |

## System tables overview

System tables are useful for historical, account-wide observability. UI pages are better for interactive investigation of one job, query, or pipeline.

| Table | Purpose | Common questions |
|-------|---------|------------------|
| `system.billing.usage` | Billable usage records | DBUs by workspace, SKU, job tag, product |
| `system.billing.list_prices` | SKU price history | Estimate cost by joining usage to prices |
| `system.access.audit` | Audit events | Who did what, from where, and when |
| `system.access.table_lineage` | Table/path lineage events | Which jobs or queries read/wrote a table |
| `system.access.column_lineage` | Column-level lineage | Which downstream columns depend on sensitive inputs |
| `system.lakeflow.jobs` | Job definitions over time | Which jobs exist, are paused, or changed |
| `system.lakeflow.job_tasks` | Job task definitions over time | What tasks exist and how dependencies changed |
| `system.lakeflow.job_run_timeline` | Job run timing and outcome | Duration, queue/setup/execution time, result state |
| `system.lakeflow.job_task_run_timeline` | Task run timing and outcome | Slow/failed tasks, retries, compute used |
| `system.lakeflow.pipelines` | Pipeline definitions | Which pipelines exist and changed |
| `system.lakeflow.pipeline_update_timeline` | Pipeline update timing/outcome | Update duration, failures, compute used |
| `system.compute.clusters` | Cluster configuration history | Cluster settings and lifecycle context |
| `system.compute.node_timeline` | Node utilization metrics | CPU/memory utilization on jobs/all-purpose compute |
| `system.compute.warehouses` | SQL warehouse configuration history | Warehouse settings over time |
| `system.compute.warehouse_events` | SQL warehouse events | Starts, stops, scaling events |
| `system.query.history` | Query execution metadata | Slow SQL queries and warehouse/serverless query history |

Access/retention notes:

| Topic | What to remember |
|-------|------------------|
| Enablement | System tables require Unity Catalog-enabled workspaces and account setup |
| Permissions | Grant `USE CATALOG`, `USE SCHEMA`, and `SELECT` on needed system schemas |
| Read-only | Users cannot modify system tables |
| Regional scope | Many operational tables are regional; query from the workspace/region you need |
| Retention | Many system tables have 365-day retention; some compute/query tables differ |
| Compute metrics retention | `system.compute.node_timeline` is shorter-lived than most job tables |
| UI run history | Jobs UI completed run history is shorter, commonly 60 days |
| Preview tables | Some tables are public preview and may vary by cloud/account |

## Cost monitoring with system tables

Daily usage by SKU:

```sql
SELECT
  usage_date,
  workspace_id,
  sku_name,
  SUM(usage_quantity) AS usage_quantity
FROM system.billing.usage
WHERE usage_date >= current_date() - INTERVAL 14 DAYS
GROUP BY usage_date, workspace_id, sku_name
ORDER BY usage_date DESC, usage_quantity DESC;
```

Usage by product/workload tags:

```sql
SELECT
  usage_date,
  usage_metadata.job_id,
  custom_tags.team,
  custom_tags.environment,
  sku_name,
  SUM(usage_quantity) AS usage_quantity
FROM system.billing.usage
WHERE usage_date >= current_date() - INTERVAL 30 DAYS
GROUP BY
  usage_date,
  usage_metadata.job_id,
  custom_tags.team,
  custom_tags.environment,
  sku_name
ORDER BY usage_date DESC;
```

Professional interpretation:

| Observation | Likely meaning | Next step |
|-------------|----------------|-----------|
| Cost spike tied to one SKU | Product/compute usage changed | Break down by workspace, job, tag, or usage metadata |
| Cost spike tied to one job | Job duration, retries, input volume, or compute size changed | Join to job run/task timelines |
| Cost has no useful tags | Attribution is weak | Enforce tags with policies/bundle config |
| Serverless background cost appears | Feature may create serverless usage | Check usage metadata/product fields |

## Job monitoring with system tables

Account-wide job reliability:

```sql
SELECT
  workspace_id,
  job_id,
  COUNT(DISTINCT run_id) AS runs,
  COUNT(DISTINCT CASE WHEN result_state = 'FAILED' THEN run_id END) AS failed_runs,
  ROUND(
    COUNT(DISTINCT CASE WHEN result_state = 'FAILED' THEN run_id END)
    / COUNT(DISTINCT run_id),
    3
  ) AS failure_rate
FROM system.lakeflow.job_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 7 DAYS
  AND result_state IS NOT NULL
GROUP BY workspace_id, job_id
ORDER BY failure_rate DESC, failed_runs DESC;
```

Job duration trend:

```sql
SELECT
  workspace_id,
  job_id,
  to_date(period_start_time) AS run_date,
  COUNT(DISTINCT run_id) AS runs,
  AVG(run_duration_seconds) AS avg_run_seconds,
  PERCENTILE(run_duration_seconds, 0.9) AS p90_run_seconds
FROM system.lakeflow.job_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 30 DAYS
  AND result_state IS NOT NULL
GROUP BY workspace_id, job_id, to_date(period_start_time)
ORDER BY run_date DESC, p90_run_seconds DESC;
```

Task-level bottlenecks:

```sql
SELECT
  workspace_id,
  job_id,
  task_key,
  COUNT(*) AS task_periods,
  AVG(execution_duration_seconds) AS avg_execution_seconds,
  PERCENTILE(execution_duration_seconds, 0.95) AS p95_execution_seconds
FROM system.lakeflow.job_task_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 14 DAYS
GROUP BY workspace_id, job_id, task_key
ORDER BY p95_execution_seconds DESC;
```

Monitoring logic:

| Metric | Why it matters |
|--------|----------------|
| `queue_duration_seconds` | Concurrency, capacity, or scheduling delay |
| `setup_duration_seconds` | Cluster/serverless startup or environment setup |
| `execution_duration_seconds` | Actual task/job execution work |
| `cleanup_duration_seconds` | Shutdown/output cleanup time |
| `result_state` / `termination_code` | Outcome and failure class |
| `trigger_type` | Manual, scheduled, file-arrival, or table-update runs may have different baselines |

## Audit monitoring

Use `system.access.audit` when the question asks who performed an action, what changed, or whether access occurred.

```sql
SELECT
  event_time,
  workspace_id,
  service_name,
  action_name,
  user_identity.email AS user_email,
  request_params,
  response
FROM system.access.audit
WHERE event_time >= current_timestamp() - INTERVAL 7 DAYS
  AND (
    action_name ILIKE '%delete%'
    OR action_name ILIKE '%drop%'
    OR action_name ILIKE '%grant%'
    OR action_name ILIKE '%revoke%'
  )
ORDER BY event_time DESC;
```

Security investigation examples:

| Question | Monitoring source |
|----------|-------------------|
| Who dropped a table? | Audit table filtered by action/resource |
| Who changed a job? | Audit table plus `system.lakeflow.jobs` change history |
| Which principal ran the production job? | Job run metadata and audit events |
| Was a secret or permission changed? | Audit table |
| Which downstream assets depend on this table? | Lineage system tables and Catalog Explorer |

## Query Profile vs Spark UI

| Tool | Best for | Look for |
|------|----------|----------|
| Query Profile | SQL warehouse, Databricks SQL, SQL task bottlenecks | Scan size, operator time, join strategy, spills, pruning, remote I/O |
| Spark UI | Spark jobs, PySpark notebooks, Python wheel tasks, Spark pipelines | Stages, tasks, skew, shuffle read/write, spill, executor memory, GC |
| Driver logs | Failures before or outside query execution | Import errors, auth failures, library install, OOM stack traces |
| Executor logs | Distributed task failures | Serialization errors, executor OOM, repeated task failures |
| Jobs UI timeline | Multi-task workflow timing | Critical path, skipped tasks, retries, queue/setup gaps |
| Pipeline event log | Lakeflow SDP health | Flow progress, expectation failures, update errors |

Decision rule:

| Scenario | Correct tool |
|----------|--------------|
| SQL dashboard query suddenly slow | Query Profile |
| SQL warehouse scales repeatedly | Query history plus warehouse events |
| PySpark job has straggler tasks | Spark UI stages/tasks |
| Driver failed before Spark stages started | Driver logs |
| Job succeeded but missed SLA | Jobs timeline and task run system tables |
| Pipeline dropped many rows by expectation | Lakeflow event log |

## Query Profile workflow

Use Query Profile when SQL is the main workload.

What to inspect:

| Signal | Meaning |
|--------|---------|
| Longest operator | Where query time is spent |
| Large scan bytes/rows | Missing filters, partition pruning, or overly broad query |
| Join strategy | Broadcast vs shuffle/sort-merge decision |
| Spilled data | Memory pressure or large shuffle/aggregation |
| Remote I/O | Data read cost and table layout effects |
| Repeated subqueries | Opportunity to materialize or simplify |

Exam patterns:

| Observation | Correct next step |
|-------------|-------------------|
| Huge table scan for selective query | Check filters, partition pruning, clustering, stats |
| Slow join node dominates | Check join keys, table sizes, broadcast eligibility |
| Aggregation spills | Reduce input, increase warehouse size, or pre-aggregate |
| Dashboard query repeated often | Consider materialized view or optimized gold table |

## Spark UI workflow

Use Spark UI when the workload runs on Spark compute.

What to inspect:

| Tab / signal | What it tells you |
|--------------|-------------------|
| Jobs/stages | Which stage is slow or failed |
| Task duration distribution | Skew or straggler tasks |
| Shuffle read/write | Expensive joins, aggregates, repartitions |
| Spill memory/disk | Executor memory pressure |
| Executors | Lost executors, OOM, uneven work |
| SQL tab | Physical query plan for Spark SQL/DataFrame operations |
| Storage tab | Cached data size and use |

Spark diagnosis:

| Observation | Likely issue |
|-------------|--------------|
| One task much longer than others | Data skew |
| Large shuffle read/write | Join/aggregation/repartition cost |
| High spill | Memory pressure or too-wide shuffle |
| Many tiny tasks | Small files or over-partitioning |
| Long GC time | Memory pressure/object churn |
| Stage retries | Data, code, executor, or infrastructure instability |

## Lakeflow SDP event logs

The pipeline event log is the primary observability primitive for Lakeflow SDP.

Query recent pipeline events:

```sql
SELECT
  timestamp,
  event_type,
  message,
  details
FROM event_log('<pipeline-id>')
ORDER BY timestamp DESC
LIMIT 100;
```

Flow progress and expectation metrics:

```sql
SELECT
  timestamp,
  details:flow_progress.status AS flow_status,
  details:flow_progress.metrics AS flow_metrics,
  details:flow_progress.data_quality.expectations AS expectations
FROM event_log('<pipeline-id>')
WHERE event_type = 'flow_progress'
ORDER BY timestamp DESC;
```

Failed events:

```sql
SELECT
  timestamp,
  event_type,
  message,
  details
FROM event_log('<pipeline-id>')
WHERE event_type = 'error'
   OR message ILIKE '%fail%'
ORDER BY timestamp DESC;
```

Event log monitoring:

| Need | Event log data |
|------|----------------|
| Pipeline update status | Update and flow progress events |
| Data quality pass/fail counts | `details:flow_progress.data_quality.expectations` |
| Failed dataset/flow | Error event details and flow progress status |
| Lineage and source/target behavior | Event log lineage details and Catalog Explorer |
| Audit of pipeline user actions | `user_action` events plus `system.access.audit` |

Important: do not delete the pipeline event log or the parent catalog/schema where it is published. Future pipeline updates can fail if the event log is removed.

## REST API and CLI monitoring

Use APIs/CLI for automation, CI/CD, external observability, and runbooks.

CLI examples:

```bash
databricks jobs list
databricks jobs list-runs --job-id 12345 --limit 20
databricks jobs get-run --run-id 67890
databricks jobs get-run-output --run-id 67890
databricks pipelines list-pipelines
databricks pipelines get --pipeline-id abc123
databricks pipelines list-updates --pipeline-id abc123
databricks bundle validate -t prod
```

REST API monitoring patterns:

| Need | API pattern |
|------|-------------|
| Check recent job runs | Jobs `runs/list` |
| Inspect one run | Jobs `runs/get` |
| Fetch notebook/task output | Jobs `runs/get-output` |
| Trigger validation job | Jobs `run-now` or bundle run |
| Check pipeline metadata | Pipelines `get` |
| Check pipeline update history | Pipelines update/list events endpoints |
| Cancel failed/stuck run | Jobs/Pipelines cancel endpoint |

CI/CD use cases:

| Requirement | Correct monitoring automation |
|-------------|-------------------------------|
| Deployment should fail if validation job fails | Run job and poll run state via API/CLI |
| Pipeline should be healthy after deploy | Trigger update and inspect final update status |
| Production bundle should be valid before deploy | `databricks bundle validate -t prod` |
| External dashboard needs job SLA metrics | Query system tables or Jobs API |
| Incident response needs run details | Link to run URL plus API output/logs |

## Building monitoring dashboards

Common dashboard tiles:

| Tile | Source |
|------|--------|
| Daily DBU/cost by workspace/SKU/team | `system.billing.usage` |
| Top cost jobs | Billing joined to job run/task tables |
| Job failure rate by week | `system.lakeflow.job_run_timeline` |
| Longest p95 task durations | `system.lakeflow.job_task_run_timeline` |
| Queue/setup time trends | Job run/task timeline columns |
| SQL warehouse start/scale events | `system.compute.warehouse_events` |
| Query duration and scan volume | `system.query.history` |
| Pipeline update failures | `system.lakeflow.pipeline_update_timeline` and event logs |
| Expectation failure trends | Lakeflow event log |
| Security-sensitive actions | `system.access.audit` |

Dashboard design:

| Principle | Correct approach |
|-----------|------------------|
| Separate signals | Cost, reliability, performance, and data quality are different views |
| Use comparable baselines | Compare same job, trigger, environment, and input class |
| Track p95/p99, not only average | Averages hide tail latency |
| Add tags | Cost and ownership dashboards need team/service/env tags |
| Link to drill-down | Dashboard should point to Jobs run, Query Profile, Spark UI, or pipeline event |
| Account for retention | Persist derived metrics if you need longer history |

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Finance asks for daily DBUs by team | `system.billing.usage` with custom tags |
| Security asks who granted access or dropped a table | `system.access.audit` |
| Platform team needs account-wide job failure rates | `system.lakeflow.job_run_timeline` |
| One job missed SLA because a task was slow | Jobs UI timeline and job task timeline table |
| SQL warehouse query is slow | Query Profile and query history |
| PySpark stage has stragglers | Spark UI stages/tasks |
| Lakeflow SDP quality dropped records | Pipeline event log expectation metrics |
| CI needs to verify deployed workflow health | REST API or Databricks CLI |
| Need compute utilization trend | `system.compute.node_timeline` |
| Need warehouse start/stop/scale events | `system.compute.warehouse_events` |

## PySpark / API essentials

Monitoring queries can be run from notebooks when the principal has access:

```python
job_runs = spark.sql("""
SELECT
  workspace_id,
  job_id,
  run_id,
  result_state,
  run_duration_seconds
FROM system.lakeflow.job_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 1 DAY
  AND result_state IS NOT NULL
""")
```

Capture lightweight task metrics:

```python
rows_written = silver_df.count()
dbutils.jobs.taskValues.set(key="rows_written", value=rows_written)
print(f"rows_written={rows_written}")
```

Do not add expensive `count()` actions just for logging if they materially increase job cost. Prefer metrics already available from Delta history, event logs, system tables, or task outputs.

## SQL essentials

Find recent failed jobs:

```sql
SELECT
  workspace_id,
  job_id,
  run_id,
  period_end_time,
  result_state,
  termination_code
FROM system.lakeflow.job_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 1 DAY
  AND result_state = 'FAILED'
ORDER BY period_end_time DESC;
```

Find high queue/setup time:

```sql
SELECT
  workspace_id,
  job_id,
  run_id,
  queue_duration_seconds,
  setup_duration_seconds,
  execution_duration_seconds
FROM system.lakeflow.job_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 7 DAYS
  AND result_state IS NOT NULL
ORDER BY queue_duration_seconds DESC, setup_duration_seconds DESC
LIMIT 50;
```

Check compute utilization:

```sql
SELECT
  workspace_id,
  cluster_id,
  date_trunc('hour', start_time) AS hour,
  AVG(cpu_user_percent + cpu_system_percent) AS avg_cpu_percent
FROM system.compute.node_timeline
WHERE start_time >= current_timestamp() - INTERVAL 1 DAY
GROUP BY workspace_id, cluster_id, date_trunc('hour', start_time)
ORDER BY hour DESC, avg_cpu_percent DESC;
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Using only notebook output or `display()` for production monitoring | Use Jobs UI, system tables, event logs, query profiles, and dashboards |
| Checking Spark UI for a pure SQL warehouse query | Use Query Profile and query history |
| Looking only at total job duration | Inspect task timeline, queue/setup time, retries, and critical path |
| Treating a single failed run as the whole trend | Compare against a baseline over comparable runs |
| Ignoring task-level tables when job-level status is successful but SLA missed | Use `job_task_run_timeline` or Jobs timeline to find the slow task |
| Assuming all system tables are available in every account/cloud | Verify enablement, region, preview status, and permissions |
| Looking only at cluster CPU for a Spark regression | Check shuffle, spill, skew, stage retries, and input size |
| Deleting Lakeflow SDP event logs | Keep event logs; they are required for pipeline observability and can affect future updates |
| Building cost dashboards without tags | Enforce team/environment/service tags for useful attribution |
| Adding expensive `count()` operations only for monitoring | Prefer existing metrics from system tables, Delta history, event logs, or lightweight counters |

# 6.1 Cost and performance optimisation

## Exam objective

> Understand how / why using Unity Catalog managed tables reduces operations overhead and maintenance burden.
>
> Understand delta optimization techniques, such as deletion vectors and liquid clustering.
>
> Understand the optimization techniques used by Databricks to ensure the performance of queries on large datasets (data skipping, file pruning, etc.).
>
> Apply Change Data Feed (CDF) to address specific limitations of streaming tables and enhance latency.
>
> Use the query profile to analyze the query and identify bottlenecks, such as bad data skipping, inefficient types of joins, and data shuffling.

## Must know

- **UC managed tables**: Databricks/UC manage storage location and support **predictive optimization** — less manual `OPTIMIZE`/`VACUUM` ops burden.
- **Data skipping**: min/max stats per file let engines skip irrelevant files.
- **File pruning**: metadata and layout reduce files read.
- **Deletion vectors**: row-level deletes/updates without rewriting whole files immediately — faster DML, deferred physical cleanup via `OPTIMIZE`.
- **Liquid clustering** (`CLUSTER BY`): adaptive layout; preferred over partition/Z-order for many new tables.
- **CDF (Change Data Feed)**: exposes row changes between versions — downstream consumers read deltas without re-scanning full table.
- **Query profile**: identifies scan volume, shuffle, join strategy, broadcast misuse.

## UC managed tables — ops reduction

| Managed table benefit | Exam angle |
|-----------------------|------------|
| UC controls storage path | Simpler governance + lifecycle |
| Predictive optimization | Auto `OPTIMIZE`, `VACUUM`, `ANALYZE` |
| Less manual file layout tuning | "Reduce operational overhead" |
| Lineage/permissions centralized | Fewer ad hoc path fixes |

External tables: you manage files and layout; UC governs access only.

## Layout and optimization

| Technique | What it does |
|-----------|--------------|
| `OPTIMIZE` | Compacts small files, applies clustering |
| `VACUUM` | Removes files no longer in table state (retention aware) |
| `ANALYZE TABLE` | Updates column statistics for better plans |
| Liquid clustering | Colocates related rows by `CLUSTER BY` keys |
| Deletion vectors | Efficient MERGE/DELETE metadata |
| Partitioning | Directory pruning; risky with high cardinality |
| Z-order | Legacy multi-column colocation; not mixed with liquid clustering |

```sql
CREATE TABLE main.silver.events (
  event_id STRING,
  event_date DATE,
  country STRING
)
USING DELTA
CLUSTER BY (event_date, country);

OPTIMIZE main.silver.events;
```

## Data skipping and file pruning

Exam symptoms of **bad skipping**:

- Query scans TB but returns few rows — missing stats, no clustering on filter columns, too many small files.
- High `numFiles` in `DESCRIBE DETAIL` with poor filter selectivity.

Fix path: `OPTIMIZE` + clustering keys matching `WHERE` columns + `ANALYZE` + reduce small files on ingest.

## Change Data Feed (CDF)

Enable on Delta table:

```sql
ALTER TABLE main.silver.customers
SET TBLPROPERTIES (delta.enableChangeDataFeed = true);
```

Read changes:

```sql
SELECT * FROM table_changes('main.silver.customers', 10, 20);
```

| Use CDF when | Why |
|--------------|-----|
| Downstream needs row-level inserts/updates/deletes | Avoid full table scan each micro-batch |
| Streaming table latency limited by full refresh | Incremental consumer on CDF |
| Audit what changed between versions | Compliance/debug |

CDF addresses **incremental change propagation** — not a substitute for liquid clustering on read performance.

## Query profile bottlenecks

| Symptom in profile | Likely cause | Remediation |
|--------------------|--------------|-------------|
| Huge shuffle read/write | Wide join, `groupBy`, `orderBy` | Broadcast small side, filter early, fix skew |
| Broadcast exchange too large | Broadcast big table | Remove broadcast hint |
| Scan bytes >> output rows | Poor pruning/skipping | Cluster keys, optimize, partition filters |
| Sort-merge join on skewed key | Hot keys | Salting, AQE skew join |
| Spill to disk | Insufficient memory | Increase shuffle partitions/memory, reduce wide columns |

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| New large Delta table, filters on date + region | Liquid clustering |
| High-cardinality column as partition | Avoid — use liquid clustering |
| Frequent row-level MERGE/DELETE | Deletion vectors (default on newer Delta) + periodic OPTIMIZE |
| Downstream incremental on changes only | Enable CDF |
| Reduce manual maintenance on UC tables | Managed table + predictive optimization |
| Find why SQL suddenly scans 10x data | Query profile + `DESCRIBE HISTORY` |

## SQL (exam)

```sql
DESCRIBE DETAIL main.silver.events;
DESCRIBE HISTORY main.silver.events;

ALTER TABLE main.silver.events CLUSTER BY (event_date, country);
OPTIMIZE main.silver.events;
VACUUM main.silver.events RETAIN 168 HOURS;
ANALYZE TABLE main.silver.events COMPUTE STATISTICS FOR ALL COLUMNS;
```

## Exam traps

- Partitioning on high-cardinality `user_id` — file explosion.
- Combining liquid clustering with partitioning/Z-order on same table.
- `VACUUM` with too-short retention breaking time travel.
- Enabling CDF but downstream still reads full table every run.
- Scaling workers only when profile shows massive shuffle from bad join order.
- External table expecting predictive optimization on provider-managed files.

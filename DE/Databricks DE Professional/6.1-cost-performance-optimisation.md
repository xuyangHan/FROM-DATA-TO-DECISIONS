# 6.1 Cost and performance optimisation

## Exam objective

> Understand how / why using Unity Catalog managed tables reduces operations overhead and maintenance burden.
>
> Understand delta optimization techniques, such as deletion vectors and liquid clustering.
>
> Understand the optimization techniques used by Databricks to ensure the performance of queries on large datasets (data skipping, file pruning, etc.).
>
> Apply Change Data Feed (CDF) to address specific limitations of streaming tables and enhance latency.
>
> Use the query profile to analyze the query and identify bottlenecks, such as bad data skipping, inefficient types of joins, and data shuffling.

## Naming note

- Databricks docs mostly use **optimization**. This note keeps the file's exam-guide spelling **optimisation** in the title.
- **Predictive optimization** is Databricks-managed table maintenance for Unity Catalog managed tables.
- **Predictive I/O** is a separate Photon optimization family for reads and updates. Do not confuse it with predictive optimization.
- **Liquid clustering** is the current recommended layout strategy for many new large Delta tables; older material may emphasize partitioning and Z-order.

## Must know

- Unity Catalog **managed tables** reduce operations overhead because Databricks controls table storage lifecycle and can apply predictive optimization.
- Predictive optimization can run `ANALYZE`, `OPTIMIZE`, and `VACUUM` on supported UC managed tables using serverless jobs compute.
- **Data skipping** uses per-file statistics such as min/max values, null counts, and record counts to skip irrelevant files.
- **File pruning** avoids reading files/directories that cannot satisfy query predicates.
- **Liquid clustering** uses `CLUSTER BY` to colocate related rows without rigid partition directories.
- **Deletion vectors** make `DELETE`, `UPDATE`, and `MERGE` faster by marking modified rows instead of rewriting full Parquet files immediately.
- **Change Data Feed (CDF)** exposes row-level changes between Delta versions so downstream consumers can process changes instead of rescanning full tables.
- **Query Profile** is the main SQL performance diagnostic tool; use it to find bad skipping, expensive scans, inefficient joins, shuffle, spill, and skew.
- Tuning should follow a measurement loop: baseline, identify bottleneck, change one lever, rerun comparable workload, and compare metrics.

## Professional focus vs Associate

Associate notes cover Spark tuning basics, Spark UI bottlenecks, and liquid clustering basics. For Professional, focus on choosing the right optimization lever:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know shuffle/broadcast configs | Diagnose with Query Profile before changing cluster size or Spark configs |
| Know liquid clustering syntax | Choose clustering keys, predictive optimization, and maintenance strategy |
| Know Spark UI symptoms | Connect profile symptoms to table layout, statistics, joins, and data volume |
| Know managed/external tables | Explain why managed tables reduce operational maintenance |
| Know CDF syntax | Use CDF for incremental downstream processing and understand retention limits |
| Know `OPTIMIZE` and `VACUUM` | Decide when predictive optimization should replace manual maintenance |

## Optimization decision map

| Symptom or requirement | First lever |
|------------------------|-------------|
| New UC table with recurring filters on date/customer/region | Managed table + liquid clustering or automatic clustering |
| Too many small files | Predictive optimization or `OPTIMIZE` |
| Query scans many files but returns few rows | Data skipping stats, clustering keys, predicates, `OPTIMIZE` |
| Large joins/aggregations shuffle heavily | Query rewrite, broadcast small side, pre-aggregate, skew handling |
| Repeated `MERGE`/`DELETE`/`UPDATE` is slow | Deletion vectors and Photon-enabled compute |
| Downstream table refresh scans full source every run | CDF + Structured Streaming or incremental batch |
| Manual maintenance jobs are expensive/noisy | UC managed table + predictive optimization |
| SQL warehouse query is slow | Query Profile |
| PySpark stage has skew/spill | Spark UI + Query Profile/physical plan |
| External table needs performance maintenance | Manual `OPTIMIZE`, `ANALYZE`, layout design; predictive optimization may not apply |

## UC managed tables and operations overhead

Managed tables are the preferred default for new governed Delta tables because Unity Catalog and Databricks manage more of the lifecycle.

| Managed table benefit | Why it reduces overhead |
|-----------------------|-------------------------|
| UC-managed storage location | Fewer path-management mistakes and simpler governance |
| Predictive optimization support | Databricks can run maintenance only when beneficial |
| Automatic statistics collection | Better query planning and data skipping without manual `ANALYZE` everywhere |
| Managed data lifecycle | Table drop/undrop and cleanup behavior is platform-managed |
| Centralized permissions/lineage | Less reliance on raw cloud path access |
| Works well with serverless and Photon | Lets platform optimizations apply more consistently |

External tables are still Unity Catalog-governed, but the cloud files remain externally managed. That means more responsibility for file lifecycle, layout, compaction, statistics, and operational consistency.

## Predictive optimization

Predictive optimization automatically identifies UC managed tables that would benefit from maintenance and queues the work.

| Operation | Purpose |
|-----------|---------|
| `ANALYZE` | Collect statistics used by the optimizer and data skipping |
| `OPTIMIZE` | Compact files and improve layout/clustering |
| `VACUUM` | Remove old unreferenced files after retention rules allow cleanup |

Important points:

- Databricks recommends predictive optimization for Unity Catalog managed tables.
- Predictive optimization uses serverless jobs compute, so it can create billable usage.
- Avoid duplicating predictive optimization with broad scheduled manual `OPTIMIZE`/`VACUUM` jobs unless there is a special case.
- You can enable, disable, or inherit predictive optimization settings at supported account/catalog/schema levels.
- Use the predictive optimization system table to inspect operations, costs, and impact where available.

SQL:

```sql
ALTER CATALOG main ENABLE PREDICTIVE OPTIMIZATION;
ALTER SCHEMA main.silver INHERIT PREDICTIVE OPTIMIZATION;

SELECT
  start_time,
  end_time,
  catalog_name,
  schema_name,
  table_name,
  operation_type,
  status
FROM system.storage.predictive_optimization_operations_history
ORDER BY start_time DESC;
```

## Delta layout techniques

| Technique | What it does | Use when | Watch out |
|-----------|--------------|----------|-----------|
| `OPTIMIZE` | Compacts files and applies clustering layout | Small files, clustering, after large load | Costs compute; may be redundant with predictive optimization |
| `VACUUM` | Deletes old unreferenced files after retention | Storage cleanup | Too-short retention can break time travel or CDF readers |
| `ANALYZE` | Collects optimizer/data-skipping stats | Bad estimates, missing stats, external tables | For managed tables, predictive optimization may handle it |
| Liquid clustering | Colocates rows by query filter keys | Large/growing tables, high-cardinality filters | Choose keys based on query predicates |
| Automatic liquid clustering | Lets Databricks choose clustering keys | Access patterns change or platform should manage layout | Requires supported table/runtime behavior |
| Partitioning | Directory-level pruning | Low-cardinality stable columns on very large tables | High cardinality causes file/directory explosion |
| Z-order | Legacy Delta colocation layout | Existing non-liquid Delta tables | Do not combine with liquid clustering |
| Deletion vectors | Marks row changes without immediate full file rewrite | Frequent DML/`MERGE`/`DELETE`/`UPDATE` | Client compatibility and later physical cleanup |

## Liquid clustering

Liquid clustering defines clustering keys with `CLUSTER BY` and lets `OPTIMIZE` or predictive optimization maintain layout incrementally.

Good clustering keys:

- Frequently appear in selective `WHERE` filters.
- Are common join/filter dimensions on large tables.
- Have enough cardinality to help skipping but not so many keys that layout becomes noisy.
- Match real query patterns, not just primary key aesthetics.

SQL:

```sql
CREATE TABLE main.silver.events (
  event_id STRING,
  event_ts TIMESTAMP,
  event_date DATE,
  country STRING,
  customer_id STRING,
  amount DECIMAL(12, 2)
)
USING DELTA
CLUSTER BY (event_date, country);
```

```sql
ALTER TABLE main.silver.events
CLUSTER BY (event_date, country);

OPTIMIZE main.silver.events;

-- Where supported, force a full reclustering instead of incremental maintenance.
OPTIMIZE main.silver.events FULL;

DESCRIBE DETAIL main.silver.events;
```

Automatic clustering where supported:

```sql
CREATE TABLE main.silver.events_auto (
  event_id STRING,
  event_ts TIMESTAMP,
  event_date DATE,
  country STRING
)
USING DELTA
CLUSTER BY AUTO;
```

Decision table:

| Requirement clue | Better choice |
|------------------|---------------|
| Large table filtered by `event_date` and `country` | Liquid clustering on those columns |
| High-cardinality `user_id` filter used often | Liquid clustering, not partitioning |
| Tiny table scanned fully | No layout tuning needed |
| Existing table already using Z-order | Migrate thoughtfully; do not layer liquid clustering and Z-order |
| Access patterns change | Automatic liquid clustering or adjust clustering keys |

`OPTIMIZE` usually performs incremental clustering work for liquid-clustered tables. Use full reclustering only when the table needs a complete layout rewrite and the runtime supports it.

## Data skipping and file pruning

Data skipping works when useful statistics exist for columns used in predicates. Databricks can skip files whose statistics prove they cannot contain matching rows.

| Concept | Meaning |
|---------|---------|
| File statistics | Min/max values, null counts, and record counts stored per file |
| Data skipping | Skip files based on column statistics and query predicates |
| File pruning | Avoid reading files/directories that cannot match |
| Dynamic file pruning | Runtime pruning based on join/filter values |
| Column statistics | Optimizer inputs used for planning and skipping |

Managed vs external stats:

| Table type | Stats behavior |
|------------|----------------|
| UC managed table with predictive optimization | Databricks can choose file-skipping stats intelligently and run `ANALYZE` |
| UC external table | Stats collection is more manual/default-limited; you may need explicit stats columns and `ANALYZE` |
| Long string columns | Stats can be less useful or truncated; avoid relying on long strings for skipping unless needed |

SQL:

```sql
ALTER TABLE main.silver.events
SET TBLPROPERTIES (
  'delta.dataSkippingStatsColumns' = 'event_date,country,customer_id'
);

ANALYZE TABLE main.silver.events COMPUTE DELTA STATISTICS;
```

Bad skipping symptoms:

| Query Profile symptom | Likely cause | Correct response |
|-----------------------|--------------|------------------|
| Scan bytes high, output rows low | Poor clustering/stats/predicate pruning | Cluster/filter columns, collect stats, optimize |
| Many files read for selective filter | Small files or layout not aligned to query | `OPTIMIZE`, liquid clustering |
| Partition filters missing | Query does not filter on layout columns | Add proper predicate or choose better layout |
| Filter column has no stats | Data skipping cannot use it | Configure stats columns and run `ANALYZE` |

## Deletion vectors

Deletion vectors accelerate DML by avoiding immediate full-file rewrites.

How they work:

```text
DELETE / UPDATE / MERGE
  -> mark affected rows in deletion vectors
  -> reads apply deletion vectors to return current table state
  -> later maintenance rewrites/purges files when needed
```

Benefits:

| Benefit | Explanation |
|---------|-------------|
| Faster row-level modifications | Avoid rewriting entire Parquet files for small changes |
| Better MERGE/UPDATE/DELETE performance | Especially with Photon/predictive I/O support |
| Row-level concurrency | Supported in newer runtimes for enabled tables |
| Lower write amplification | Fewer immediate rewritten files |

Tradeoffs:

| Concern | Correct handling |
|---------|------------------|
| Physical files still contain old rows | Use maintenance/`REORG`/`OPTIMIZE`/`VACUUM` patterns when physical purge matters |
| Client compatibility | Some older clients cannot read DV-enabled tables |
| Table protocol upgrade | Enabling table features can change compatibility |
| External sharing/clients | Verify downstream support before enabling |

SQL:

```sql
ALTER TABLE main.silver.customers
SET TBLPROPERTIES ('delta.enableDeletionVectors' = true);

DELETE FROM main.silver.customers
WHERE is_test_customer = true;

OPTIMIZE main.silver.customers;
```

Exam idea: deletion vectors speed up modifications. They do not replace data skipping, clustering, statistics, or cost monitoring.

## Change Data Feed (CDF)

CDF records row-level changes after it is enabled. It is useful when downstream consumers need only changed rows.

Enable:

```sql
ALTER TABLE main.silver.customers
SET TBLPROPERTIES (delta.enableChangeDataFeed = true);
```

Read in SQL:

```sql
SELECT *
FROM table_changes('main.silver.customers', 10, 20);
```

Read in PySpark batch:

```python
changes = (
    spark.read.format("delta")
    .option("readChangeFeed", "true")
    .option("startingVersion", 10)
    .option("endingVersion", 20)
    .table("main.silver.customers")
)
```

Read in Structured Streaming:

```python
(
    spark.readStream
    .option("readChangeFeed", "true")
    .table("main.silver.customers")
    .writeStream
    .option("checkpointLocation", "/Volumes/main/checkpoints/customers_cdf")
    .trigger(availableNow=True)
    .toTable("main.downstream.customers_changes")
)
```

CDF metadata columns:

| Column | Meaning |
|--------|---------|
| `_change_type` | `insert`, `delete`, `update_preimage`, `update_postimage` |
| `_commit_version` | Delta table version containing the change |
| `_commit_timestamp` | Timestamp of the commit |

Use CDF when:

| Requirement | Why CDF fits |
|-------------|--------------|
| Downstream needs only changed rows | Avoid full scans on each run |
| Low-latency propagation to target table | Stream table changes incrementally |
| Audit row-level changes | Capture inserts/updates/deletes after enablement |
| Replicate changes to external system | Process CDF into outbound feed |

CDF limits:

- Only changes after CDF is enabled are recorded.
- CDF is not a permanent audit archive by itself; retention follows table history and `VACUUM`.
- Starting from a version no longer in history fails.
- If permanent change history is required, write CDF output to a durable audit table.
- Do not query CDF files directly; use Delta APIs such as `table_changes` or `readChangeFeed`.

## CDF and streaming table limitations

CDF helps when a downstream process would otherwise re-read a large table or wait for a full refresh.

| Limitation / symptom | CDF-based design |
|----------------------|------------------|
| Consumer scans full source table every microbatch | Read CDF changes only |
| Downstream target needs near-real-time change propagation | Structured Streaming over CDF |
| Need restartable incremental processing | CDF stream with checkpoint |
| Need version-bounded backfill | Batch `table_changes` between versions |
| Need full historical audit forever | Archive CDF to a durable audit table |

Do not use CDF as a general query speed feature. Use CDF for incremental change propagation; use clustering/skipping/statistics for read performance.

## Query Profile bottleneck analysis

Query Profile is the primary tool for SQL performance diagnosis.

| Symptom in Query Profile | Likely cause | Correct response |
|--------------------------|--------------|------------------|
| Scan bytes much larger than output | Poor pruning/skipping or broad query | Add predicates, improve clustering/stats, optimize layout |
| Many files scanned | Small files or bad layout | `OPTIMIZE`, liquid clustering, better ingestion file sizing |
| Shuffle read/write large | Join, groupBy, distinct, window, orderBy | Filter/project early, pre-aggregate, improve join strategy |
| Broadcast exchange too large | Broadcast side too big | Remove broadcast hint or reduce dimension |
| Sort-merge join where dimension is small | Missing stats or broadcast not selected | `ANALYZE`, broadcast hint if safe |
| One side of join explodes rows | Join key issue or many-to-many join | Validate grain and join keys |
| Spill to disk | Memory pressure or too much shuffled data | Reduce columns/rows, tune partitions/warehouse size |
| Skewed stage/tasks | Hot keys or uneven file sizes | AQE skew join, salting, better layout |
| UDF node dominates | Python UDF overhead | Use built-in Spark SQL/PySpark functions |

Diagnosis sequence:

```text
1. Check scan size and files read.
2. Check filters, pruning, and data skipping.
3. Check join type and join order.
4. Check shuffle, spill, and skew.
5. Check aggregation/window grain.
6. Change query/table layout/compute, then re-measure.
```

## Cost levers

Cost optimization is not only choosing smaller clusters. It is matching compute, layout, and maintenance to workload.

| Lever | Cost impact |
|-------|-------------|
| Serverless SQL / Jobs where appropriate | Reduces idle infrastructure and improves startup behavior |
| Right-size SQL warehouses | Avoid overlarge always-on warehouses |
| Auto-stop / schedules | Reduce idle compute cost |
| Job compute over all-purpose compute | Avoid long-lived interactive clusters for production jobs |
| Photon | Faster SQL/Spark execution can reduce runtime and cost |
| Predictive optimization | Avoid unnecessary maintenance runs; improves layout automatically |
| Liquid clustering | Reduce scanned data for common selective queries |
| CDF | Process changes instead of full table scans |
| Tags | Attribute spend to team/product/environment |
| Query rewrites | Reduce scan/shuffle/spill before adding hardware |

Use `system.billing.usage`, query history, job run timelines, and warehouse/compute system tables to verify whether a change improves total cost, not just runtime.

## Manual maintenance vs managed maintenance

| Scenario | Better answer |
|----------|---------------|
| New UC managed table | Enable/inherit predictive optimization |
| External table | Plan manual `OPTIMIZE`, `ANALYZE`, and `VACUUM` as needed |
| Large one-time backfill just landed | Manual `OPTIMIZE` may be justified |
| Predictive optimization already handles table | Avoid duplicative scheduled maintenance |
| Strict physical deletion requirement after DVs | Use documented purge/maintenance pattern, then `VACUUM` after retention |
| Long time travel/CDF retention required | Do not run aggressive `VACUUM`; set retention intentionally |

SQL essentials:

```sql
DESCRIBE DETAIL main.silver.events;
DESCRIBE HISTORY main.silver.events;

ALTER TABLE main.silver.events CLUSTER BY (event_date, country);
OPTIMIZE main.silver.events;

ANALYZE TABLE main.silver.events COMPUTE DELTA STATISTICS;

VACUUM main.silver.events RETAIN 168 HOURS;
```

## Performance measurement loop

Before changing settings:

| Step | What to record |
|------|----------------|
| Baseline | Runtime, input rows, output rows, data scanned, files read |
| Query profile | Scan, join, shuffle, spill, skew, operator time |
| Table details | Size, file count, clustering columns, stats, table properties |
| Workload context | Same data volume, same warehouse/cluster, same code version |
| Change | One lever at a time |
| Rerun | Comparable input and compute |
| Compare | Runtime, cost, scan bytes, shuffle, spill, correctness |

Examples of poor measurement:

- Changing cluster size and table layout at the same time.
- Comparing dev and prod with different data volumes.
- Measuring only wall-clock time while cost doubled.
- Tuning Spark configs before confirming the bottleneck.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| New large table with repeated selective filters | UC managed table + liquid clustering |
| Reduce manual `OPTIMIZE`/`VACUUM` work | Predictive optimization |
| External table query planner has bad estimates | `ANALYZE` and explicit stats columns |
| Frequent `MERGE`/`DELETE` workloads | Deletion vectors + Photon where supported |
| Downstream job needs changed rows only | Enable CDF and read change feed |
| Dashboard query scans too much data | Query Profile, clustering/stats/data skipping |
| Huge shuffle in query profile | Filter/project early, join strategy, pre-aggregate, skew handling |
| High-cardinality filter column | Liquid clustering rather than partitioning |
| Tiny lookup table repeatedly joined | Broadcast if it safely fits |
| Storage cleanup without breaking time travel | `VACUUM` with correct retention |

## SQL essentials

Create a managed clustered table:

```sql
CREATE TABLE main.silver.events (
  event_id STRING,
  event_ts TIMESTAMP,
  event_date DATE,
  country STRING,
  customer_id STRING,
  amount DECIMAL(12, 2)
)
USING DELTA
CLUSTER BY (event_date, country);
```

Enable CDF and deletion vectors:

```sql
ALTER TABLE main.silver.customers
SET TBLPROPERTIES (
  delta.enableChangeDataFeed = true,
  delta.enableDeletionVectors = true
);
```

Read CDF:

```sql
SELECT *
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Partitioning a large table by high-cardinality `user_id` | Use liquid clustering when queries filter by `user_id` |
| Scheduling manual `OPTIMIZE` everywhere while predictive optimization manages tables | Let predictive optimization handle UC managed tables unless a special case requires manual work |
| Expecting predictive optimization to manage external table files | Use manual maintenance or migrate to UC managed tables where appropriate |
| Running `VACUUM` with very short retention | Respect time travel, CDF, and downstream reader retention needs |
| Enabling CDF and still reading the full table downstream | Read the change feed with `readChangeFeed` or `table_changes` |
| Treating CDF as permanent audit history | Archive CDF output to a durable table if permanent history is required |
| Using Z-order on a new table where liquid clustering fits | Prefer liquid clustering for new Databricks table layout scenarios |
| Adding workers before checking Query Profile | Diagnose scan, skipping, joins, shuffle, and spill first |
| Broadcasting a dimension that does not fit in memory | Let optimizer choose or reduce/filter the dimension before broadcasting |
| Optimizing columns not used in predicates | Choose clustering/statistics columns from real query filters |

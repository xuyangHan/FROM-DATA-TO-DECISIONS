# 2.1 Data ingestion and acquisition

## Exam objective

> Design and implement data ingestion pipelines to efficiently ingest a variety of data formats including Delta Lake, Parquet, ORC, AVRO, JSON, CSV, XML, Text and Binary from diverse sources such as message buses and cloud storage.
>
> Create an append-only data pipeline capable of handling both batch and streaming data using Delta.

## Naming note

- The exam says **Autoloader**; current Databricks docs usually write **Auto Loader**.
- The source API is `cloudFiles`, and SQL often uses `read_files`.
- Lakeflow Spark Declarative Pipelines is the current managed pipeline framework; older material may say Delta Live Tables.

## Must know

- Production ingestion should land data in **Unity Catalog-governed Delta tables** using `catalog.schema.table`.
- A bronze ingestion table is normally **append-only**: keep raw facts, source metadata, ingestion timestamp, and rescued/unparsed data for replay and audit.
- Use **Auto Loader** for scalable incremental file ingestion from cloud object storage or Unity Catalog volumes.
- Use **streaming tables** or Structured Streaming for continuous ingestion from files, Delta tables, and message buses.
- Use **Lakeflow Connect** when a managed SaaS/database connector fits the source; drop down to JDBC, APIs, or custom streams when it does not.
- **COPY INTO** is idempotent and useful for SQL-driven or low-volume file loads, but Auto Loader/streaming tables are the scalable default for incremental ingestion.
- Format choice affects schema behavior: Delta, Parquet, ORC, and Avro are schema-aware; JSON, CSV, XML, text, and binary need more parsing care.
- A single Delta table can accept batch and streaming appends when writers are compatible and streaming queries have unique checkpoints.
- Streaming output mode matters: append is the usual bronze default; complete mode is for stateful aggregations, not raw ingestion.

## Professional focus vs Associate

Associate notes cover basic ingestion patterns and Auto Loader mechanics. For Professional, focus on production decisions:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know Auto Loader ingests files | Choose Auto Loader vs COPY INTO vs streaming table vs custom Structured Streaming |
| Know common file formats | Decide schema, rescue, parsing, and bronze-to-silver strategy by format |
| Know Delta is the table format | Design append-only bronze tables that support batch and streaming writers |
| Know message bus ingestion exists | Choose Structured Streaming or Lakeflow SDP and manage checkpoints, output mode, and schema |
| Know UC tables and volumes exist | Land data in governed tables/volumes instead of DBFS root |
| Know Lakeflow Connect exists | Use managed connectors when available before building custom ingestion code |

## Ingestion architecture

```text
Cloud object storage / UC volumes
  -> Auto Loader or COPY INTO
  -> bronze append-only Delta table
  -> silver parsing, deduplication, CDC, quality rules

Message bus
  -> Structured Streaming or Lakeflow SDP streaming table
  -> bronze append-only Delta table
  -> silver parsing, ordering, watermarks, deduplication

SaaS / database
  -> Lakeflow Connect managed connector, JDBC, CDC feed, or API extraction
  -> bronze append-only Delta table
  -> silver conformed model
```

Production principles:

| Principle | Correct production shape |
|-----------|--------------------------|
| Land raw first | Bronze keeps source-aligned data and ingestion metadata |
| Preserve replayability | Do not overwrite bronze unless the requirement explicitly says full refresh |
| Govern access | Use UC catalogs, schemas, volumes, external locations, and permissions |
| Separate ingestion from transformation | Ingest raw/near-raw data first; cleanse, deduplicate, and merge in silver |
| Use managed features first | Prefer Lakeflow Connect and Auto Loader before custom code when they fit |
| Checkpoint every stream | One unique checkpoint per streaming query |
| Make schemas observable | Track schema evolution and rescued/unexpected columns |
| Choose cost by latency | Use scheduled `Trigger.AvailableNow` when continuous streaming is not required |

## Source selection

| Source type | Recommended ingestion path | Use when | Watch out |
|-------------|----------------------------|----------|-----------|
| Cloud files in object storage or UC volume | Auto Loader / streaming table | Files arrive repeatedly or at large scale | Need checkpoint/schema tracking and file immutability assumptions |
| One-time or small SQL file load | `COPY INTO` | Controlled batch loads with SQL | Less scalable than Auto Loader for high-volume continuous arrival |
| Delta table source | Batch read or `readStream.table` | Upstream is already Delta | Prefer change data feed for changed rows where possible |
| Kafka | Structured Streaming or SDP | Event bus ingestion | Parse binary `key`/`value`, manage offsets and schema |
| Kinesis / Pub/Sub | Structured Streaming connector | Cloud-native stream source | Connector options and cloud networking matter |
| SaaS app | Lakeflow Connect managed connector | Supported enterprise app source | Understand sync mode, destination tables, and permissions |
| RDBMS | Lakeflow Connect, JDBC, CDC tool, or federation + materialization | Database extraction | JDBC batch is not the same as reliable CDC |
| REST API | Custom job or connector | No managed connector exists | Rate limits, retries, idempotency, and pagination |
| Binary documents/images | UC volume + `binaryFile` metadata table | PDFs, images, logs, ML documents | Store metadata and content intentionally; parse downstream |

## Data format matrix

| Format | Read API | Schema behavior | Professional note |
|--------|----------|-----------------|-------------------|
| Delta Lake | `spark.table()` / `read.format("delta")` | Table schema enforced | Preferred governed storage for bronze/silver/gold |
| Parquet | `read.parquet()` / `format("parquet")` | Embedded columnar schema | Good source format; convert to Delta for ACID tables |
| ORC | `read.format("orc")` | Embedded columnar schema | Efficient columnar format; Auto Loader required for streaming ORC files |
| Avro | `read.format("avro")` | Avro schema and logical types | Common with message buses and schema registry workflows |
| JSON | `read.json()` / `cloudFiles.format = json` | Infer or explicit schema | Use schema evolution and rescued data for changing payloads |
| CSV | `read.csv()` / `cloudFiles.format = csv` | Weak schema unless explicit | Configure header, delimiter, quotes, multiline, nulls, corrupt records |
| XML | `read.format("xml")` / Auto Loader XML | Requires row tag / structure choices | Land raw and parse carefully; nested structures can drift |
| Text | `read.text()` | Single `value` string column | Logs and line-delimited raw data; parse in silver |
| Binary | `read.format("binaryFile")` | `path`, `modificationTime`, `length`, `content` | Images/PDFs/audio; create metadata table and parse content separately |

Format decision rules:

| Requirement clue | Correct choice |
|------------------|----------------|
| Source data is already transactional Delta | Read Delta directly or stream from Delta/change data feed |
| Source sends nested JSON with evolving fields | Auto Loader with schema evolution and rescued data |
| Source sends CSV with inconsistent headers | Explicit schema and format options; rescue or quarantine bad rows |
| Source sends compact event payloads through Kafka | Avro/JSON decode in streaming job; write raw and parsed fields |
| Source sends PDFs/images | Store files in UC volume and ingest metadata/content with `binaryFile` |
| Source is huge set of columnar files | Auto Loader for incremental Parquet/ORC/Avro ingestion |

## Append-only bronze design

Bronze should preserve what arrived and when it arrived. It is a landing layer, not the place to fix business correctness.

Common bronze columns:

| Column | Purpose |
|--------|---------|
| `_ingest_ts` | When Databricks ingested the record |
| `_source_file` | File path or source identifier |
| `_source_system` | Source application/system name |
| `_batch_id` or `_load_id` | Batch/job/run correlation |
| `_rescued_data` | Unexpected or unparsed fields |
| Raw payload column | Useful for JSON, XML, text, or message bus payload replay |

Append-only rules:

| Rule | Why it matters |
|------|----------------|
| Use `append` for batch writes into bronze | Keeps full audit trail |
| Use `writeStream` with a checkpoint for streaming | Enables recovery and exactly-once behavior with Delta sinks |
| Deduplicate in silver | Bronze should capture source reality, including duplicates if they arrived |
| Avoid overwriting bronze | Overwrite destroys replay and can break stream readers |
| Use MERGE/CDC in silver+ | Curated layers can maintain latest-state or SCD behavior |

## Batch and streaming writers to Delta

Delta Lake supports ACID transactions, so multiple compatible writers can append to the same table. The design must keep operations compatible.

| Mode | Pattern | Checkpoint | Good for |
|------|---------|------------|----------|
| Batch append | `write.mode("append").saveAsTable(...)` | No stream checkpoint | Backfills, daily file drops, one-time loads |
| Streaming append | `writeStream.outputMode("append").toTable(...)` | Required | Continuous or scheduled incremental ingestion |
| Auto Loader available-now | `trigger(availableNow=True)` | Required | Scheduled micro-batch ingestion |
| COPY INTO | SQL `COPY INTO target FROM source` | Managed by command history | SQL batch loads with idempotent file tracking |

Safe mixed-writer pattern:

```text
Historical backfill job
  -> batch append files from historical path

Daily incremental job
  -> Auto Loader Trigger.AvailableNow
  -> same bronze Delta table

Silver job
  -> deduplicate, parse, enforce quality, apply CDC/merge
```

What to coordinate:

| Concern | Correct approach |
|---------|------------------|
| Backfill overlaps current stream | Use source metadata and silver deduplication keys |
| Two streams write to same table | Use separate checkpoints and confirm append-only compatibility |
| Source files can change after arrival | Prefer immutable files; avoid overwrites or configure intentionally |
| Stream query changes | Changing source, sink, state schema, or topic usually requires a new checkpoint |
| Complete/update outputs | Avoid for bronze ingestion; use append for raw event records |

## Auto Loader production pattern

Python Structured Streaming:

```python
from pyspark.sql import functions as F


raw_orders = (
    spark.readStream.format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("cloudFiles.schemaLocation", "/Volumes/main/bronze/_schemas/orders")
    .option("cloudFiles.schemaEvolutionMode", "rescue")
    .load("/Volumes/main/landing/orders/")
    .withColumn("_ingest_ts", F.current_timestamp())
    .withColumn("_source_file", F.input_file_name())
)

(
    raw_orders.writeStream
    .outputMode("append")
    .option("checkpointLocation", "/Volumes/main/bronze/_checkpoints/orders")
    .trigger(availableNow=True)
    .toTable("main.bronze.orders_raw")
)
```

Lakeflow SDP streaming table:

```sql
CREATE OR REFRESH STREAMING TABLE main.bronze.orders_raw
AS SELECT
  *,
  current_timestamp() AS _ingest_ts,
  _metadata.file_path AS _source_file
FROM STREAM read_files(
  '/Volumes/main/landing/orders/',
  format => 'json'
);
```

Important settings:

| Setting / concept | Use |
|-------------------|-----|
| `cloudFiles.format` | Source file format: JSON, CSV, XML, PARQUET, AVRO, ORC, TEXT, BINARYFILE |
| `cloudFiles.schemaLocation` | Durable schema tracking for non-SDP Auto Loader streams |
| `cloudFiles.schemaEvolutionMode` | Control whether new fields are added, rescued, ignored, or fail the stream |
| `_rescued_data` | Capture fields that do not match the expected schema |
| `checkpointLocation` | Durable stream progress and exactly-once recovery |
| `Trigger.AvailableNow` | Process all currently available files, then stop |
| `cloudFiles.maxFilesPerTrigger` | Limit files processed per microbatch |
| `cloudFiles.maxBytesPerTrigger` | Limit bytes processed per microbatch |

## COPY INTO pattern

`COPY INTO` is SQL-first, idempotent for already-loaded files, and good for controlled file loads.

```sql
CREATE TABLE IF NOT EXISTS main.bronze.events_raw (
  event_id STRING,
  event_ts TIMESTAMP,
  payload STRING
);

COPY INTO main.bronze.events_raw
FROM '/Volumes/main/landing/events/'
FILEFORMAT = JSON
FORMAT_OPTIONS (
  'inferSchema' = 'true'
)
COPY_OPTIONS (
  'mergeSchema' = 'true'
);
```

Use it when:

| Requirement | Fit |
|-------------|-----|
| SQL-only team needs simple file ingestion | Good fit |
| Small or periodic batch file loads | Good fit |
| Need idempotent skipping of already-loaded files | Good fit |
| Continuous high-volume file arrival | Prefer Auto Loader or streaming tables |
| Complex streaming state and checkpoint control | Prefer Structured Streaming |

## Message bus ingestion

Message buses usually provide records as bytes with offsets and metadata. Bronze should capture both payload and metadata.

Kafka example:

```python
from pyspark.sql import functions as F


kafka_orders = (
    spark.readStream.format("kafka")
    .option("kafka.bootstrap.servers", "broker:9092")
    .option("subscribe", "orders")
    .option("startingOffsets", "latest")
    .load()
    .select(
        F.col("topic"),
        F.col("partition"),
        F.col("offset"),
        F.col("timestamp").alias("_source_ts"),
        F.col("key").cast("string").alias("message_key"),
        F.col("value").cast("string").alias("payload"),
        F.current_timestamp().alias("_ingest_ts"),
    )
)

(
    kafka_orders.writeStream
    .outputMode("append")
    .option("checkpointLocation", "/Volumes/main/bronze/_checkpoints/kafka_orders")
    .toTable("main.bronze.orders_kafka_raw")
)
```

Message bus design points:

| Concern | Correct approach |
|---------|------------------|
| Payload encoding | Decode Avro/JSON/Protobuf in silver or controlled parsing step |
| Offset recovery | Use a durable checkpoint |
| Late or duplicate events | Add event-time, keys, watermarks, and deduplication in silver |
| Backpressure | Use source-specific rate options such as `maxOffsetsPerTrigger` |
| Reprocessing | Reset checkpoint intentionally or replay from a controlled starting offset |
| External side effects | Use `foreachBatch` carefully with idempotency keys |

## Lakeflow Connect and database ingestion

Use the most managed supported connector first.

| Source | First choice | Fallback |
|--------|--------------|----------|
| Supported SaaS app | Lakeflow Connect managed connector | Partner connector or custom API job |
| Supported database | Lakeflow Connect managed connector | CDC tool, JDBC batch, or custom extraction |
| Unsupported file source | Auto Loader through cloud storage/volume | Custom file transfer then Auto Loader |
| Unsupported event stream | Structured Streaming connector | Custom job with durable offsets |

Professional checks:

- Confirm the connector writes to the intended Unity Catalog destination.
- Understand whether the connector provides full refresh, incremental sync, or CDC behavior.
- Grant the connector/runtime identity only the required source and target permissions.
- Do not confuse Lakehouse Federation with ingestion: federation queries external data in place; ingestion copies/materializes data into Databricks.

## Schema handling

| Source pattern | Correct schema approach |
|----------------|-------------------------|
| Stable Parquet/ORC/Avro | Trust embedded schema, still validate in silver |
| Evolving JSON/XML | Use Auto Loader schema tracking and rescued data |
| CSV from external partner | Provide explicit schema and parsing options |
| Text logs | Land raw `value`; parse downstream with versioned logic |
| Binary files | Store metadata/content first; extract features/text later |
| Message bus with registry | Decode using agreed schema and handle compatibility rules |

Bad data options:

| Requirement | Correct handling |
|-------------|------------------|
| Keep every source record | Append raw payload plus `_rescued_data` |
| Analysts need only valid records | Filter/drop invalid records in silver |
| Engineers must inspect failures | Quarantine bad rows to a separate table |
| Invalid records must stop the pipeline | Fail the job/pipeline with expectations or assertions |

## Output modes and checkpoints

| Concept | Correct understanding |
|---------|-----------------------|
| Append mode | Default and best for raw ingestion records |
| Complete mode | Rewrites all aggregate results each trigger; not a bronze ingestion default |
| Update mode | Emits changed rows for stateful queries; not supported by all Delta/UC sink patterns |
| Checkpoint | Stores offsets, commits, and state so the query can resume correctly |
| Unique checkpoint | Each streaming query needs its own checkpoint path |
| New checkpoint | Required when changing source type, topics, sink, or state schema in many cases |

For append-only ingestion, keep the stream simple:

```text
source stream -> parse only enough to land raw safely -> append Delta table
```

Then handle deduplication, watermarks, joins, merges, and CDC in silver where the business logic is easier to test.

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Overwriting bronze on every run when the requirement is append-only | Append raw records and handle deduplication or corrections in silver |
| Using `COPY INTO` for continuous high-volume file arrival | Use Auto Loader or a streaming table |
| Running Auto Loader continuously for daily drops with no low-latency SLA | Use `Trigger.AvailableNow` on a schedule |
| Reusing one checkpoint for multiple streaming queries | Give each query a unique checkpoint location |
| Changing Kafka topics or stream state while reusing the same checkpoint | Start a new checkpoint when the query identity/state changes |
| Reading PDFs or images as CSV/JSON | Use `binaryFile` and store file metadata/content intentionally |
| Landing new governed workloads in DBFS root | Use UC volumes, external locations, and UC Delta tables |
| Applying business merges directly in bronze | Keep bronze append-only and apply MERGE/CDC in silver or gold |
| Inferring CSV schemas in production without controls | Define explicit schema and parsing options |
| Treating federation as ingestion | Use federation for remote query access; ingest/materialize when data must land in Delta |

# 7.2 Ensuring compliance

## Exam objective

> Implement a compliant batch & streaming pipeline that detects and applies masking of PII to ensure data privacy.
>
> Develop a data purging solution ensuring compliance with data retention policies.

## Naming note

- The exam guide says **APPLY CHANGES** in Lakeflow Spark Declarative Pipelines. Current docs increasingly use **AUTO CDC** for the same CDC pattern.
- The exam may say **masking** broadly. Distinguish query-time masks from irreversible ETL transformations such as suppression, hashing, and tokenization.
- Current Databricks docs distinguish logical deletes from physical purge. For soft-deleted data, `REORG TABLE ... APPLY (PURGE)` plus `VACUUM` can be needed.

## Must know

- Compliance pipelines should detect sensitive data early, restrict raw access, transform or mask PII before broad consumption, and preserve audit evidence.
- Bronze can contain raw PII when needed for replay, but access should be highly restricted.
- Silver should standardize, validate, tag, mask, tokenize, hash, or suppress sensitive fields before most users can consume the data.
- Gold should expose only approved fields and should not leak raw PII unless explicitly governed.
- Batch and streaming paths must apply the same privacy policy. A streaming job that bypasses batch masking logic is a compliance gap.
- Query-time controls such as row filters and column masks enforce role-dependent visibility, but raw data still exists in the protected table.
- ETL-time controls such as suppression, tokenization, hashing, and generalization create safer downstream tables and shared datasets.
- Retention policies define how long data should remain. Purging means logically deleting records and, when required, physically removing files after retention rules allow it.
- `DELETE` removes rows from the current Delta table state. `VACUUM` removes old unreferenced files after the retention threshold.
- With deletion vectors or other soft-deletes, old values may remain in current files until `REORG TABLE ... APPLY (PURGE)` rewrites affected files, followed by `VACUUM`.
- CDF consumers must handle delete changes if the source table supports privacy deletes or right-to-erasure workflows.

## Professional focus vs Associate

Associate notes cover managed/external tables, masks, row filters, and ABAC. For Professional, focus on end-to-end compliance design:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know column masks exist | Decide query-time mask vs irreversible transformation |
| Know PII should not leak to gold | Design batch and streaming pipelines that enforce the same policy |
| Know `DELETE` and `VACUUM` exist | Sequence logical delete, purge/rewrite, retention-aware vacuum, and audit evidence |
| Know grants restrict access | Restrict raw bronze, token vaults, quarantine tables, and purge jobs by role |
| Know CDF exists | Propagate privacy deletes to downstream consumers incrementally |
| Know tags exist | Use tags, comments, lineage, and audit logs to prove policy coverage |

## Compliance architecture

```text
Source systems
  -> bronze raw tables / volumes
       restricted access, source metadata, raw replay
  -> PII detection and classification
       schema rules, regex, tags, data quality checks
  -> silver protected tables
       masking, tokenization, hashing, suppression, row filters, expectations
  -> gold consumer tables
       approved fields, aggregated/generalized outputs, governed sharing
  -> audit and monitoring
       job runs, system.access.audit, lineage, event logs, quality metrics
  -> retention and purge workflows
       delete, propagate, reorg purge, vacuum, evidence
```

Design principles:

| Principle | Correct production shape |
|-----------|--------------------------|
| Minimize raw access | Raw PII is restricted to small groups and service principals |
| Apply controls before wide consumption | Silver/gold should not rely only on BI-layer masking |
| Keep batch and streaming consistent | Shared transform functions or shared policy definitions |
| Preserve audit evidence | Log who ran masking/purge jobs and what tables/records were affected |
| Separate token vaults | Reversible mappings are not stored beside broadly readable analytics tables |
| Propagate deletes | Downstream tables and CDF consumers must handle delete events |
| Respect retention | Do not vacuum aggressively when time travel, legal hold, or CDF readers require history |
| Test with restricted users | Verify masks, filters, grants, and transformed outputs from consumer identity |

## PII detection and classification

PII detection is not one mechanism. Use schema knowledge, metadata, rules, and profiling together.

| Detection method | Use when | Example |
|------------------|----------|---------|
| Source contract | Known source columns contain PII | `email`, `ssn`, `phone`, `card_number` |
| Schema/name rules | Column names imply sensitivity | column name matches `%ssn%` |
| Regex/pattern checks | Free text or poorly modeled columns | email regex, SSN pattern |
| Data profiling | Unknown or changing data | detect unexpected PII in comment fields |
| Governed tags | Central policy targeting | `pii = email`, `sensitivity = high` |
| Expectations/assertions | PII must be absent from a layer | fail if `ssn IS NOT NULL` in gold |
| Lineage | Understand where sensitive data flows | Catalog Explorer/system lineage |

PII categories:

| Category | Examples | Common control |
|----------|----------|----------------|
| Direct identifiers | Name, email, phone, SSN, national ID | Suppress, hash, tokenize, mask |
| Quasi-identifiers | ZIP, birth date, age, location | Generalize, aggregate, row filter |
| Financial identifiers | PAN, bank account | Tokenize, suppress, strict access |
| Health or regulated attributes | Diagnosis, claim, benefit data | Row filters, masks, restricted gold |
| Free text | Notes, comments, tickets | Detection/profiling, redaction, quarantine |

## Batch and streaming privacy consistency

The same rule must apply regardless of processing mode.

| Pipeline mode | Correct approach |
|---------------|------------------|
| Batch backfill | Apply the same masking/tokenization functions used in streaming |
| Streaming ingestion | Mask/tokenize as data is transformed into silver, not weeks later |
| Lakeflow SDP | Use expectations, shared functions, and CDC/delete propagation |
| Classic Structured Streaming | Use shared transform modules and checkpoints |
| Gold refresh | Validate that raw PII columns are absent or policy-protected |

Shared transform example:

```python
from pyspark.sql import DataFrame
from pyspark.sql import functions as F


def protect_customer_pii(df: DataFrame, salt: str) -> DataFrame:
    normalized_email = F.lower(F.trim(F.col("email")))

    return (
        df
        .withColumn("email_hash", F.sha2(F.concat_ws(":", F.lit(salt), normalized_email), 256))
        .withColumn("email_masked", F.regexp_replace(normalized_email, "(?<=.).(?=.*@)", "*"))
        .withColumn("age_band", (F.floor(F.col("age") / 10) * 10).cast("int"))
        .drop("email", "ssn")
    )
```

Batch use:

```python
salt = dbutils.secrets.get("security", "hash_salt")

silver = protect_customer_pii(
    spark.table("main.bronze.customers_raw"),
    salt,
)

silver.write.mode("overwrite").saveAsTable("main.silver.customers_protected")
```

Streaming use:

```python
salt = dbutils.secrets.get("security", "hash_salt")

query = (
    protect_customer_pii(spark.readStream.table("main.bronze.customers_raw"), salt)
    .writeStream
    .option("checkpointLocation", "/Volumes/main/checkpoints/customers_protected")
    .trigger(availableNow=True)
    .toTable("main.silver.customers_protected")
)
```

## Lakeflow SDP compliance pattern

Use Lakeflow SDP for declarative enforcement and observability when it fits the pipeline.

```python
from pyspark import pipelines as dp
from pyspark.sql import functions as F


@dp.table(name="silver_customers_protected")
@dp.expect_or_fail("no_raw_ssn", "ssn IS NULL")
def silver_customers_protected():
    salt = dbutils.secrets.get("security", "hash_salt")

    raw = spark.readStream.table("bronze_customers_raw")
    email_normalized = F.lower(F.trim(F.col("email")))

    return (
        raw
        .withColumn("email_hash", F.sha2(F.concat_ws(":", F.lit(salt), email_normalized), 256))
        .withColumn("email_masked", F.regexp_replace(email_normalized, "(?<=.).(?=.*@)", "*"))
        .drop("email")
    )
```

Use expectations for compliance checks:

| Requirement | Expectation action |
|-------------|--------------------|
| Gold must never contain raw SSN | `expect_or_fail` |
| Invalid PII format should be excluded but investigated | quarantine plus alert |
| Track percentage of rows with email | `expect` / warn metric |
| Drop rows with prohibited values from silver | `expect_or_drop` only when dropping satisfies policy |

## Query-time controls vs ETL-time controls

| Requirement | Better control |
|-------------|----------------|
| HR can see salary; analysts see masked salary | UC column mask |
| EU team sees only EU rows | UC row filter |
| All columns tagged `pii = email` need same mask | ABAC column mask policy |
| Export must contain no raw email at all | Hash/suppress email before writing export table |
| Shared dataset must be anonymous enough for broad access | Generalization, suppression, aggregation |
| Authorized workflow needs to recover original ID | Tokenization with separate vault |

Professional shortcut:

```text
Different users see different values in the same table -> UC mask/filter/ABAC.
Downstream table must not contain raw PII -> transform before writing.
Right-to-erasure requires removal -> delete everywhere, propagate, purge files when required.
```

## Masking and pseudonymization methods

| Method | Reversible? | Use when | Watch out |
|--------|-------------|----------|-----------|
| Column mask | Depends on function | Same table, role-dependent visibility | Raw value still exists in source table |
| Hashing | No | Stable join key without raw identifier | Unsalted common values can be dictionary attacked |
| Salted/keyed hash | No without secret | More robust pseudonymous join key | Protect the salt/key in secrets |
| Tokenization | Yes with vault | Authorized reversal is required | Vault must be separately governed |
| Suppression | No | Data should not be present downstream | May reduce analytical utility |
| Generalization | No | Public/broad data release | Too much detail can still re-identify |
| Aggregation | No at row level | Reporting without individual records | Small groups can still be sensitive |

SQL transformation example:

```sql
CREATE OR REPLACE TABLE main.silver.customers_protected AS
SELECT
  customer_id,
  sha2(concat(secret('security', 'hash_salt'), ':', lower(trim(email))), 256) AS email_hash,
  regexp_replace(lower(trim(email)), '(?<=.).(?=.*@)', '*') AS email_masked,
  floor(age / 10) * 10 AS age_band,
  substr(zip_code, 1, 3) AS zip3,
  current_timestamp() AS protected_at
FROM main.bronze.customers_raw;
```

Query-time dynamic mask:

```sql
ALTER TABLE main.silver.customers
ALTER COLUMN email SET MASK main.security.mask_email;
```

## Data retention and purge concepts

Retention policies define how long data remains available. Purge workflows enforce those policies.

| Concept | Meaning |
|---------|---------|
| Logical delete | Remove rows from current table state with `DELETE` |
| Soft-delete | Old values may remain in files because of deletion vectors or column mapping |
| Physical purge | Rewrite affected files and remove old files from storage |
| Time travel retention | How long old table versions remain queryable |
| CDF retention | How long change feed data remains readable |
| Legal hold | Exception that blocks deletion/purge despite normal retention |
| Audit evidence | Proof that purge ran and what it affected |

Basic retention workflow:

```text
1. Identify records subject to retention policy.
2. Confirm no legal hold or exception applies.
3. DELETE matching rows from every affected Delta table.
4. Propagate deletes to downstream tables and systems.
5. If soft-deleted data must be physically purged, run REORG TABLE ... APPLY (PURGE).
6. Run VACUUM after the retention threshold.
7. Record job run, row counts, table versions, and audit evidence.
```

## DELETE, REORG, and VACUUM

`DELETE` changes the current table state:

```sql
DELETE FROM main.silver.events
WHERE event_date < current_date() - INTERVAL 7 YEAR;
```

`REORG TABLE ... APPLY (PURGE)` rewrites files that contain soft-deleted data:

```sql
REORG TABLE main.silver.events
APPLY (PURGE);
```

`VACUUM` removes old unreferenced files after the retention threshold:

```sql
VACUUM main.silver.events RETAIN 168 HOURS;
```

Decision table:

| Requirement clue | Correct action |
|------------------|----------------|
| Remove rows from current table result | `DELETE` |
| Physically remove soft-deleted values from current files | `REORG TABLE ... APPLY (PURGE)` |
| Remove old files no longer referenced by retained table versions | `VACUUM` |
| Preserve time travel or CDF for 30 days | Do not vacuum below that retention need |
| Managed table should get routine cleanup | Predictive optimization can run `VACUUM` |
| Prove purge happened | Audit logs, job run logs, table history, purge summary |

Important: `VACUUM` before `DELETE` does not remove rows that are still part of the current table state. For compliance deletion, the logical delete comes first.

## Right-to-erasure workflow

For a subject erasure request, delete from all copies and propagate the delete.

```text
Input: customer_id = c-123

1. Find all tables containing the subject through lineage/catalog inventory.
2. Delete matching rows from bronze/silver/gold where policy requires it.
3. Delete or update token vault mappings if policy requires.
4. Propagate deletes through CDC/CDF consumers.
5. Run purge/vacuum when physical deletion is required and retention permits.
6. Store evidence in a restricted compliance audit table.
```

SQL:

```sql
DELETE FROM main.bronze.customers_raw
WHERE customer_id = 'c-123';

DELETE FROM main.silver.customers_protected
WHERE customer_id = 'c-123';

DELETE FROM main.gold.customer_analytics
WHERE customer_id = 'c-123';
```

PySpark orchestration:

```python
from delta.tables import DeltaTable
from pyspark.sql import functions as F


customer_id = dbutils.widgets.get("customer_id")

for table_name in [
    "main.bronze.customers_raw",
    "main.silver.customers_protected",
    "main.gold.customer_analytics",
]:
    DeltaTable.forName(spark, table_name).delete(F.col("customer_id") == customer_id)
```

In production, restrict the table list to approved constants and record row counts/table versions as evidence.

## Streaming and CDC compliance

Append-only pipelines are not enough when policy requires deletion or correction.

| Scenario | Correct design |
|----------|----------------|
| Source sends delete events | Use AUTO CDC / APPLY CHANGES with delete condition |
| Downstream consumes Delta changes | Enable CDF and handle `_change_type = 'delete'` |
| Gold table maintains latest customer state | Use `MERGE` or AUTO CDC, not append-only duplicates |
| Erasure request arrives after data is already published | Delete from all affected layers and downstream systems |
| External replica must mirror deletes | Send CDF delete records or a controlled delete feed |

Lakeflow CDC pattern:

```sql
CREATE OR REFRESH STREAMING TABLE customers_silver;

CREATE FLOW customers_cdc_flow AS AUTO CDC INTO customers_silver
FROM stream(bronze_customer_cdc)
KEYS (customer_id)
APPLY AS DELETE WHEN operation = 'DELETE'
SEQUENCE BY operation_ts
COLUMNS * EXCEPT (operation, operation_ts)
STORED AS SCD TYPE 1;
```

CDF consumer check:

```sql
SELECT _change_type, COUNT(*) AS records
FROM table_changes('main.silver.customers', 100, 120)
GROUP BY _change_type;
```

Delete rows have `_change_type = 'delete'`. Consumers that ignore deletes can recreate non-compliant downstream copies.

## Compliance observability

Compliance controls must be measurable.

| Evidence | Source |
|----------|--------|
| Who changed grants, tags, masks, or filters | `system.access.audit` |
| Which tables/columns contain PII tags | Catalog Explorer, information schema, governed tags |
| Which pipeline enforced PII checks | Lakeflow SDP event log and job run logs |
| Which records were quarantined | Quarantine Delta tables |
| Which purge job ran | Lakeflow Jobs run history and task output |
| Which Delta version contains deletion | `DESCRIBE HISTORY` |
| Whether downstream delete propagation happened | CDF audit table or downstream reconciliation |
| Which datasets depend on raw PII | Lineage in Catalog Explorer/system lineage |

Audit query:

```sql
SELECT
  event_time,
  action_name,
  user_identity.email AS user_email,
  request_params
FROM system.access.audit
WHERE event_time >= current_timestamp() - INTERVAL 30 DAYS
  AND (
    action_name ILIKE '%delete%'
    OR action_name ILIKE '%vacuum%'
    OR action_name ILIKE '%grant%'
    OR action_name ILIKE '%revoke%'
    OR action_name ILIKE '%tag%'
    OR action_name ILIKE '%mask%'
  )
ORDER BY event_time DESC;
```

Purge summary table pattern:

```sql
CREATE TABLE IF NOT EXISTS main.compliance.purge_runs (
  purge_run_id STRING,
  requested_by STRING,
  subject_id STRING,
  table_name STRING,
  deleted_rows BIGINT,
  delta_version BIGINT,
  purge_ts TIMESTAMP
);
```

## Data sharing and compliance

Before sharing or exporting data, decide whether the recipient should see raw, masked, or transformed data.

| Requirement | Correct approach |
|-------------|------------------|
| Recipient should never receive direct identifiers | Share pre-masked/pre-hashed table |
| Recipient's access varies by group | Share policy-protected object where supported and tested |
| Public/broad release | Aggregate, suppress, and generalize |
| Need contractual deletion propagation | Provide CDF/delete feed or downstream purge process |
| Partner uses open sharing | Verify how masks/filters and credentials behave for that recipient pattern |

For high-risk sharing, prefer a curated export/share table that contains only approved columns and generalized fields.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| PII must never reach gold | Suppress/hash/tokenize in silver and test gold schema |
| Different users need different visibility | UC masks, row filters, or ABAC |
| Need reversible identifier mapping | Tokenization with separate restricted vault |
| Need stable joins without raw identifier | Salted/keyed hash |
| Public demographic release | Generalization + aggregation + small-group suppression |
| GDPR/right-to-erasure request | Delete all copies, propagate deletes, purge/vacuum as policy requires |
| Storage cleanup after retention | `VACUUM` with correct retention |
| Physically purge soft-deleted data | `REORG TABLE ... APPLY (PURGE)` then `VACUUM` |
| Prove compliance action happened | Audit logs, job logs, table history, purge summary |
| Downstream must process deletes incrementally | CDF or AUTO CDC / APPLY CHANGES |

## SQL essentials

Mask at query time:

```sql
ALTER TABLE main.silver.customers
ALTER COLUMN email SET MASK main.security.mask_email;
```

Delete by retention:

```sql
DELETE FROM main.silver.events
WHERE event_date < current_date() - INTERVAL 7 YEAR;
```

Purge soft-deleted data and old files:

```sql
REORG TABLE main.silver.events APPLY (PURGE);
VACUUM main.silver.events RETAIN 168 HOURS;
```

Inspect history:

```sql
DESCRIBE HISTORY main.silver.events;
```

Read CDF delete events:

```sql
SELECT *
FROM table_changes('main.silver.customers', 100, 120)
WHERE _change_type = 'delete';
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Masking only in a BI tool while raw PII remains broadly readable in gold | Enforce UC policies or transform PII before writing gold |
| Applying privacy logic in batch but not streaming | Use shared transform functions or shared Lakeflow policy logic |
| Running `VACUUM` before deleting rows | `DELETE` first, then purge/rewrite if needed, then `VACUUM` after retention |
| Assuming `DELETE` physically removes all old values immediately | Use `REORG TABLE ... APPLY (PURGE)` plus `VACUUM` when physical purge is required |
| Using zero or very short `VACUUM` retention without considering time travel/CDF | Align retention with compliance, audit, CDF, and downstream reader needs |
| Deleting only bronze while silver/gold still contain subject data | Delete or anonymize every downstream copy required by policy |
| Ignoring CDF delete events in downstream consumers | Process `_change_type = 'delete'` and reconcile targets |
| Storing token vault mappings in a widely readable analytics table | Store token mappings in a separate restricted vault/table |
| Publishing detailed quasi-identifiers in public data | Generalize, aggregate, and suppress small groups |
| Claiming compliance with no audit trail | Store job run evidence, table history, audit logs, and purge summaries |

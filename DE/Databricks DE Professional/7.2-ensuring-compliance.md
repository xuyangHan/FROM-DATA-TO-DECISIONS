# 7.2 Ensuring compliance

## Exam objective

> Implement a compliant batch & streaming pipeline that detects and applies masking of PII to ensure data privacy.
>
> Develop a data purging solution ensuring compliance with data retention policies.

## Must know

- **Compliance pipelines** detect PII early, mask/tokenize before gold, and audit transformations.
- Apply controls in **silver** before wide consumption; gold should not leak raw PII unintentionally.
- **Batch and streaming** both need the same policy: streaming masks on ingest/transform, not only on batch backfill.
- **Retention policies** define how long data lives; purging = delete rows + remove obsolete files per policy.
- **`DELETE`** removes logical rows; **`VACUUM`** removes physical files after retention window.
- Legal hold / audit requirements may block purge — exam scenarios usually state policy clearly.

## Compliant PII pipeline

```
Bronze (raw, restricted access)
  -> detect PII (regex, tags, classification)
  -> mask/hash/tokenize in silver
  -> gold exposes only approved columns
  -> audit log of processing
```

| Stage | Control |
|-------|---------|
| Bronze | Restricted catalog; few principals |
| Silver | Masking transform + expectations |
| Gold | No raw email/SSN unless approved |
| Sharing | Delta Sharing + masks on provider or pre-masked tables |

```python
from pyspark.sql.functions import col, sha2, regexp_replace, when

pii_detected = col("email").rlike(r"^[^@]+@[^@]+\.[^@]+$")

silver = (
    bronze
    .withColumn(
        "email",
        when(pii_detected, regexp_replace(col("email"), "(?<=.).(?=.*@)", "*"))
        .otherwise(col("email")),
    )
    .withColumn("email_hash", sha2(col("email"), 256))
    .drop("ssn")  # suppression — do not carry to gold
)
```

SDP with expectations:

```python
@dp.table
@dp.expect_or_fail("no_raw_ssn", "ssn IS NULL")
def gold_customers():
    return spark.table("LIVE.silver_customers_masked")
```

UC dynamic enforcement (query-time):

```sql
ALTER TABLE main.silver.customers
ALTER COLUMN email SET MASK main.security.mask_email;
```

## Data purging and retention

| Step | Action |
|------|--------|
| 1 | Identify data subject to policy (table, partition key, `event_date`) |
| 2 | `DELETE` rows older than retention |
| 3 | `VACUUM` with appropriate `RETAIN` hours (respect time travel needs) |
| 4 | Document run in audit (`system.access.audit`) |

```sql
-- Delete customer data past retention (example: 7 years)
DELETE FROM main.silver.events
WHERE event_date < current_date() - INTERVAL 7 YEAR;

-- Remove orphaned files (after confirming legal/audit requirements)
VACUUM main.silver.events RETAIN 168 HOURS;
```

Streaming compliance:

- Use **`APPLY CHANGES`** or `MERGE` for right-to-erasure updates (delete propagates).
- CDF consumers must handle `delete` change type.
- Do not only append if policy requires deletion.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| GDPR erasure request | `DELETE` + downstream replicate deletes |
| Reduce storage after retention window | `DELETE` then `VACUUM` |
| PII must never reach gold | Mask in silver pipeline + fail expectation |
| Role-based live masking in SQL | UC column masks |
| Prove who purged data | `system.access.audit` + job run logs |

## PySpark / API essentials

```python
# Subject erasure — delete by key across related tables (orchestrate in job)
spark.sql("DELETE FROM main.silver.orders WHERE customer_id = 'c-123'")
spark.sql("DELETE FROM main.silver.customers WHERE customer_id = 'c-123'")
```

## SQL (exam)

`DELETE`, `VACUUM`, mask/row filter DDL, `MERGE` for coordinated updates.

## UI / product surfaces

- **SDP event log**: failed expectations on PII rules.
- **Catalog Explorer**: tags (`pii`), column comments, lineage.
- **Jobs**: scheduled purge job with notifications on failure.

## Exam traps

- Masking only in BI tool while raw PII remains in gold table.
- `VACUUM` before `DELETE` — wrong order.
- `VACUUM` with zero retention breaking compliance audit time travel needs.
- Append-only gold for regulated delete-rights scenarios.
- Purging bronze without silver/gold — orphaned PII copies remain.
- Storing reversible tokens in same widely readable table as analytics data.

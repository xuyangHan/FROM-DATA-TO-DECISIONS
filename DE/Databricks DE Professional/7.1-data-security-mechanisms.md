# 7.1 Applying data security mechanisms

## Exam objective

> Use ACLs to secure Workspace Objects, enforcing the principle of least privilege, including enforcing principles like least privilege, policy enforcement.
>
> Use row filters and column masks to filter and mask sensitive table data.
>
> Apply anonymization and pseudonymization methods, such as Hashing, Tokenization, Suppression, and generalisation, to confidential data.

## Naming note

- The exam says **ACLs** for workspace objects. In current docs this is usually called **workspace object access control** or **workspace ACLs**.
- Unity Catalog uses **GRANT** and **REVOKE**. `DENY` is not supported for Unity Catalog objects; it applies to legacy `hive_metastore` objects.
- Current Databricks spelling is often **generalization**. The exam guide uses **generalisation**.

## Must know

- **Least privilege** means granting only the permissions required for the work, preferably to groups for humans and service principals for production jobs.
- **Workspace ACLs** secure workspace-level objects such as folders, notebooks, files, dashboards, alerts, experiments, jobs, and some compute resources.
- **Unity Catalog privileges** secure data and governance securables such as catalogs, schemas, tables, views, volumes, functions, models, shares, recipients, storage credentials, and external locations.
- Users usually need parent privileges such as `USE CATALOG` and `USE SCHEMA` plus object-level privileges such as `SELECT`.
- **Row filters** restrict which rows a user can see by applying a boolean SQL UDF at query time.
- **Column masks** transform sensitive column values by applying a SQL UDF at query time.
- Row filters and column masks restrict users who already have table access; they do not grant access by themselves.
- Use **ABAC policies** when the same row-filter or masking rule must apply centrally across many tagged objects.
- **Anonymization** irreversibly removes or generalizes identifying information.
- **Pseudonymization** replaces identifiers with tokens or pseudonyms so records can still be linked, sometimes reversibly through a protected mapping.
- Hashing, tokenization, suppression, and generalization solve different privacy problems; the exam often asks which one fits a scenario.

## Professional focus vs Associate

Associate notes cover GRANT/REVOKE, row filters, masks, and ABAC mechanics. For Professional, focus on designing layered security controls:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know `GRANT SELECT` syntax | Apply least privilege across workspace, compute, UC objects, jobs, and service principals |
| Know row filters and masks exist | Choose query-time enforcement vs irreversible pipeline transformation |
| Know ABAC scales policies | Decide when table-level controls become too hard to maintain |
| Know groups are preferred | Separate human groups, job service principals, admin groups, and exception groups |
| Know PII should be masked | Choose hashing, tokenization, suppression, or generalization based on reversibility and utility |
| Know workspace ACLs exist | Distinguish workspace object ACLs from Unity Catalog data access |

## Security layers

| Layer | Secures | Examples | Professional rule |
|-------|---------|----------|-------------------|
| Identity and groups | Who can authenticate and belong to access groups | Users, groups, service principals | Manage through account/IdP groups, not ad hoc users |
| Workspace ACLs | Workspace objects and execution artifacts | Folders, notebooks, jobs, dashboards, alerts | Grant to groups; organize folders by ownership/security boundary |
| Unity Catalog privileges | Governed data and AI objects | Catalogs, schemas, tables, views, volumes, functions | Use least privilege with parent `USE` privileges |
| Fine-grained UC controls | Row/column visibility inside tables | Row filters, column masks, ABAC | Apply sensitive-data controls close to the table |
| Compute permissions and policies | Who can use compute and what compute can be created | Cluster policies, SQL warehouses, job compute | Prevent uncontrolled privileged compute paths |
| Secrets and credentials | Access to external systems | Secret scopes, storage credentials, external locations | Do not hard-code credentials in notebooks or configs |
| Pipeline transformations | Irreversible privacy transformations | Hashing, suppression, generalization | Use when data should be safe even outside query-time controls |

## Workspace ACLs vs Unity Catalog

| Question | Workspace ACL | Unity Catalog |
|----------|---------------|---------------|
| What does it secure? | Workspace objects and interactive assets | Data, files, functions, models, shares, credentials |
| Examples | Notebook, folder, job, dashboard, alert | Table, view, volume, catalog, schema, external location |
| Common permissions | `CAN READ`, `CAN RUN`, `CAN EDIT`, `CAN MANAGE`, job-specific permissions | `USE CATALOG`, `USE SCHEMA`, `SELECT`, `MODIFY`, `CREATE`, `EXECUTE`, `MANAGE` |
| Best for | Controlling who can view/run/edit workspace artifacts | Controlling who can read/write governed data |
| Important limitation | Notebook access is not a substitute for data access control | UC table access does not automatically allow notebook/job editing |

Workspace ACL examples:

| Object | Least-privilege pattern |
|--------|-------------------------|
| Shared production folder | Engineers get `CAN READ`; deployment service principal gets edit/manage through CI/CD path |
| Notebook used by analysts | Analysts get `CAN RUN`; only owners get `CAN EDIT` |
| Job | Operators get `CAN MANAGE RUN`; platform owners get `CAN MANAGE` |
| Dashboard/query/alert | Consumers get view/run; editors are limited to owning group |
| Cluster policy | Users get `CAN USE`, not free-form cluster creation |

Important Jobs detail: a job run uses the job owner's or configured run-as identity for permissions. Grant production data privileges to the service principal or run-as identity, not to every user who can click Run Now.

## Unity Catalog least privilege

Read one table:

```sql
GRANT USE CATALOG ON CATALOG main TO `finance_analysts`;
GRANT USE SCHEMA ON SCHEMA main.gold TO `finance_analysts`;
GRANT SELECT ON TABLE main.gold.daily_revenue TO `finance_analysts`;
```

Production job writes one table:

```sql
GRANT USE CATALOG ON CATALOG main TO `sp_orders_job`;
GRANT USE SCHEMA ON SCHEMA main.silver TO `sp_orders_job`;
GRANT MODIFY ON TABLE main.silver.orders TO `sp_orders_job`;
```

Production job creates managed tables in a schema:

```sql
GRANT USE CATALOG ON CATALOG main TO `sp_lakeflow_jobs`;
GRANT USE SCHEMA ON SCHEMA main.silver TO `sp_lakeflow_jobs`;
GRANT CREATE TABLE ON SCHEMA main.silver TO `sp_lakeflow_jobs`;
GRANT MODIFY ON SCHEMA main.silver TO `sp_lakeflow_jobs`;
```

Least-privilege choices:

| Need | Correct grant |
|------|---------------|
| Read one table | Parent `USE` plus table `SELECT` |
| Read all current/future tables in schema | Parent `USE` plus `SELECT` on schema |
| Write existing table | Parent `USE` plus table `MODIFY` |
| Create tables in schema | Parent `USE` plus schema `CREATE TABLE` |
| Execute masking/filter function | Parent `USE` plus function `EXECUTE` |
| Read files in volume | Parent `USE` plus `READ VOLUME` |
| Manage permissions without reading data | `MANAGE` where appropriate, not `SELECT` |

Unity Catalog does not support `DENY`. To remove access, revoke direct grants, remove inherited group membership, narrow group design, apply row filters/masks, or use ABAC.

## Row filters and column masks

| Mechanism | UDF return type | Effect |
|-----------|-----------------|--------|
| Row filter | `BOOLEAN` | Keeps rows when the function returns `TRUE`; hides rows on `FALSE` or `NULL` |
| Column mask | Same type as target column, or castable | Replaces visible values whenever the column is referenced |

Requirements:

- Unity Catalog table.
- A SQL UDF registered in Unity Catalog.
- Users querying the table still need `USE CATALOG`, `USE SCHEMA`, and `SELECT`.
- To assign policies, the administrator needs privileges on the function and table, such as `EXECUTE` on the UDF plus ownership or `MANAGE` on the target table.
- Prefer simple SQL UDFs; complex policy functions can hurt performance and become hard to audit.

Column mask:

```sql
CREATE SCHEMA IF NOT EXISTS main.security;

CREATE OR REPLACE FUNCTION main.security.mask_email(email STRING)
RETURN
  CASE
    WHEN is_account_group_member('pii_readers') THEN email
    WHEN email IS NULL THEN NULL
    ELSE regexp_replace(email, '(?<=.).(?=.*@)', '*')
  END;

ALTER TABLE main.gold.customers
ALTER COLUMN email
SET MASK main.security.mask_email;
```

Row filter:

```sql
CREATE OR REPLACE FUNCTION main.security.region_filter(region STRING)
RETURN
  CASE
    WHEN is_account_group_member('global_analysts') THEN true
    WHEN is_account_group_member('eu_analysts') THEN region = 'EU'
    WHEN is_account_group_member('na_analysts') THEN region = 'NA'
    ELSE false
  END;

ALTER TABLE main.gold.orders
SET ROW FILTER main.security.region_filter ON (region);
```

Remove policies before dropping their functions:

```sql
ALTER TABLE main.gold.customers
ALTER COLUMN email DROP MASK;

ALTER TABLE main.gold.orders
DROP ROW FILTER;
```

## Design choices for masks and filters

| Requirement clue | Correct mechanism |
|------------------|-------------------|
| Same table, salary hidden from contractors | Column mask |
| Same table, EU users see only EU rows | Row filter |
| Hundreds of tagged PII columns need same mask | ABAC policy |
| Separate curated read-only object with joins/projections | View or dynamic view |
| Users should not access table at all | Do not grant `SELECT`; revoke inherited grants |
| Raw value must be removed before sharing/export | Pipeline suppression/hash/tokenization, not only a query-time mask |

Policy design guidance:

- Include privileged exception groups such as `pii_readers`, compliance, or admins.
- Keep UDF arguments minimal; each argument can force additional source column access.
- Do not implement masking only in notebooks, dashboards, or BI tools when the requirement is governed access control.
- Test with both restricted and privileged principals.
- Remember that predicates, joins, and ordering use masked values where masks apply.
- For many tables, prefer ABAC with governed tags over manually applying masks table by table.
- Use `is_account_group_member()` for Unity Catalog account groups; use `is_member()` only when the scenario is explicitly about workspace-local groups or legacy Hive metastore behavior.

Fine-grained access controls have compute/runtime requirements and some limitations. For exam purposes, remember that they require Unity Catalog, supported Databricks compute/runtime behavior, and table access by name rather than bypassing policies through raw path access.

## ABAC policy fit

Attribute-based access control uses governed tags and central policies to apply masks or filters across many objects.

| Situation | Table-level mask/filter | ABAC policy |
|-----------|-------------------------|-------------|
| One sensitive table | Good fit | Usually unnecessary |
| A few special table-specific rules | Good fit | Optional |
| Hundreds of PII columns across schemas | Hard to maintain | Best fit |
| New tagged assets should be protected automatically | Manual updates required | Best fit |
| Central governance team owns policy | Possible but fragmented | Best fit |

ABAC does not replace `GRANT`. Users still need base privileges such as `USE CATALOG`, `USE SCHEMA`, and `SELECT`.

## Query-time controls vs transformation-time controls

| Control type | Examples | Best for | Limitation |
|--------------|----------|----------|------------|
| Query-time control | Row filter, column mask, ABAC | Role-dependent visibility on the same table | Raw data still exists in the protected table |
| Transformation-time control | Hashing, tokenization, suppression, generalization | Producing a safer silver/gold/share table | Less flexible if raw access is later needed |
| Access control | UC grants, workspace ACLs, compute permissions | Preventing access paths entirely | Does not transform data |
| Governance metadata | Tags, comments, lineage, classifications | Discovery and policy targeting | Not enforcement by itself unless used by ABAC/policies |

Professional shortcut:

```text
Need different users to see different values in the same table -> mask/filter/ABAC.
Need downstream table/export to contain no raw PII at all -> transform data before writing it.
Need nobody in a group to access the object -> revoke/do not grant access.
```

## Anonymization and pseudonymization

| Method | Description | Reversible? | Common use |
|--------|-------------|-------------|------------|
| Hashing | One-way digest such as `sha2` | No, if done correctly | Stable joins without exposing raw identifier |
| Salted/keyed hashing | Hash with a secret salt or key | No without secret | Reduce dictionary/re-identification risk |
| Tokenization | Replace value with token stored in secure mapping/vault | Yes, with vault | Payment card, customer identifier, regulated workflows |
| Suppression | Remove, null, redact, or drop value | No | Hide SSN, drop raw email, redact free text |
| Generalization | Reduce precision or bucket values | No | Age band, ZIP prefix, month instead of exact date |
| Masking | Show partial or transformed value | Usually no | Last four digits, masked email |

Technique selection:

| Requirement clue | Prefer |
|------------------|--------|
| Need stable join key but not raw email | Salted/keyed hash |
| Need to reverse value for authorized workflow | Tokenization with separate vault |
| Analysts should never receive SSN | Suppression/drop column |
| Public demographic release | Generalization plus aggregation |
| Support team can see last four digits only | Masking |
| Role-dependent live visibility | UC column mask or row filter |

PySpark examples:

```python
from pyspark.sql import functions as F


secret_salt = dbutils.secrets.get("security", "hash_salt")

protected = (
    customers
    .withColumn("email_normalized", F.lower(F.trim("email")))
    .withColumn("email_hash", F.sha2(F.concat_ws(":", F.lit(secret_salt), "email_normalized"), 256))
    .withColumn("age_band", (F.floor(F.col("age") / 10) * 10).cast("int"))
    .withColumn("zip3", F.substring("zip_code", 1, 3))
    .drop("email", "ssn")
)
```

SQL examples:

```sql
SELECT
  customer_id,
  sha2(concat(secret('security', 'hash_salt'), ':', lower(trim(email))), 256) AS email_hash,
  floor(age / 10) * 10 AS age_band,
  substr(zip_code, 1, 3) AS zip3
FROM main.silver.customers_raw;
```

Keep salts, token vaults, and reversible mappings separate from broadly readable analytics tables.

## Data security design patterns

| Scenario | Recommended pattern |
|----------|---------------------|
| Analysts need aggregate revenue only | Grant `SELECT` on gold aggregate; no raw table access |
| Contractors can query customers but not email | Column mask on `email` or masked view |
| Regional teams see only their region | Row filter on `region` |
| Enterprise PII policy across many tables | Governed tags + ABAC column mask policy |
| Shared external dataset cannot include direct identifiers | Suppress raw PII and publish hashed/generalized fields |
| Production job writes silver tables | Service principal with scoped UC privileges and job run-as |
| Users should run but not edit notebooks | Workspace ACL `CAN RUN`, not `CAN EDIT` |
| Users should use only approved compute sizes | Cluster policies / SQL warehouse permissions |

## Auditing and validation

Security controls should be inspectable and testable.

| Check | Tool |
|-------|------|
| Who has access? | `SHOW GRANTS`, Catalog Explorer permissions |
| Which masks exist? | `DESCRIBE EXTENDED`, information schema, Catalog Explorer column metadata |
| Which row filter exists? | `DESCRIBE EXTENDED`, Catalog Explorer table overview |
| Who changed permissions? | `system.access.audit` |
| Which tags drive ABAC? | Catalog Explorer tags, governed tag definitions |
| Does policy work? | Test queries as restricted and privileged users |
| Does workspace ACL match intended groups? | Workspace permissions UI or Permissions API |

Useful SQL:

```sql
SHOW GRANTS ON TABLE main.gold.customers;
DESCRIBE EXTENDED main.gold.customers;
```

Audit sensitive security actions:

```sql
SELECT
  event_time,
  action_name,
  user_identity.email AS user_email,
  request_params
FROM system.access.audit
WHERE event_time >= current_timestamp() - INTERVAL 7 DAYS
  AND (
    action_name ILIKE '%grant%'
    OR action_name ILIKE '%revoke%'
    OR action_name ILIKE '%permission%'
    OR action_name ILIKE '%mask%'
    OR action_name ILIKE '%filter%'
  )
ORDER BY event_time DESC;
```

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Secure notebooks/folders/jobs | Workspace ACLs |
| Secure table read/write access | Unity Catalog grants |
| Enforce approved compute choices | Cluster policies and compute permissions |
| Hide sensitive column values by user group | Column mask |
| Restrict visible rows by region/tenant/department | Row filter |
| Apply same sensitive-data control across many tagged columns | ABAC policy |
| Remove raw PII from curated table | Suppression/hash/tokenization in pipeline |
| Preserve joins without exposing raw identifier | Salted/keyed hash |
| Reversible identifier replacement | Tokenization with separate vault |
| Public or broad data release | Generalization, suppression, aggregation |

secret_salt = dbutils.secrets.get("security", "hash_salt")

safe_customers = (

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Granting `ALL PRIVILEGES` when users only need to read | Grant parent `USE` privileges plus `SELECT` only |
| Giving production jobs a personal user's permissions | Use a service principal or configured `run_as` identity |
| Assuming a column mask grants table access | Grant `SELECT` separately; the mask only restricts visible values |
| Using `DENY` for Unity Catalog objects | Use least privilege, `REVOKE`, group design, row filters, masks, or ABAC |
| Masking PII only in a notebook or BI dashboard | Enforce with UC masks/filters or transform data before writing curated tables |
| Storing reversible token mappings in the same readable analytics table | Store token vault/mapping separately with restricted access |
| Hashing unsalted common identifiers for privacy-sensitive use | Use salted/keyed hashing or tokenization depending on reversibility needs |
| Applying manual masks to hundreds of tables | Use governed tags and ABAC policies |
| Creating a row filter UDF that returns a string | Return `BOOLEAN` |
| Dropping a policy UDF before removing masks/filters | Drop the mask/filter first, then drop the function |

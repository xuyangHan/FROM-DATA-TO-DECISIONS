# 1.1 Using Python and tools for development

## Exam objective

> Design and implement a scalable Python project structure optimized for Databricks Asset Bundles (DABs), enabling modular development, deployment automation, and CI/CD integration.
>
> Manage and troubleshoot external third-party library installations and dependencies in Databricks, including PyPI packages, local wheels, and source archives.
>
> Develop User-Defined Functions (UDFs) using Pandas/Python UDF.

## Naming note

The exam guide uses **Databricks Asset Bundles (DABs)**. Current Databricks docs increasingly use **Declarative Automation Bundles**, described as the newer name for the same bundle concept. For the exam, recognize both names and map them to `databricks.yml`, bundle targets, resources, artifacts, validation, deployment, and CI/CD automation.

## Must know

- **Bundles** package Databricks resources, code, artifacts, permissions, variables, and target-specific configuration as source-controlled files.
- A professional-grade Python project separates **resource definitions** (`resources/`), **business logic** (`src/`), **tests** (`tests/`), and **deployment config** (`databricks.yml`).
- **Targets** such as `dev`, `staging`, and `prod` let the same reviewed commit deploy to different workspaces, catalogs, schemas, permissions, and run identities.
- **Artifacts** build deployable files, commonly Python wheels, during bundle deployment so jobs and pipelines can use versioned internal code.
- **Libraries** can be installed at notebook, compute, job, or pipeline scope; scope controls reproducibility, blast radius, and troubleshooting path.
- **PyPI packages**, **local wheels**, and **source archives** are common Python dependency forms; production should prefer pinned versions and wheel artifacts.
- **Pandas UDFs** use Apache Arrow batches and are usually faster than row-at-a-time Python UDFs, but native Spark SQL/PySpark functions are still the first choice.
- UDFs introduce serialization overhead and can block optimizer visibility, so use them only when native functions cannot express the logic.

## Professional focus vs Associate

Associate-level notes already cover what bundles, CLI commands, and variable overrides are. For Professional, expect questions framed around **production design decisions**:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Identify bundle structure | Design a maintainable project that supports CI/CD and environment promotion |
| Run `databricks bundle deploy` | Decide how resources, targets, artifacts, variables, and permissions fit together |
| Use `%pip install` | Choose dependency scope and debug library failures in jobs or pipelines |
| Know Python UDF syntax | Decide whether native Spark, Python UDF, Pandas UDF, or UC governed UDF is appropriate |

## Scalable DAB Python project layout

```text
orders_bundle/
  databricks.yml
  pyproject.toml              # package metadata and wheel build config
  resources/
    jobs.yml                  # Lakeflow Jobs definitions
    pipelines.yml             # Lakeflow Spark Declarative Pipelines definitions
    schemas.yml               # optional UC schema resources
  src/
    orders_etl/
      __init__.py
      transforms.py           # pure, testable Spark DataFrame logic
      quality.py
      udf.py
  notebooks/
    exploration.py            # optional; not the only home for production logic
  tests/
    test_transforms.py
    test_quality.py
  .databricks/
    bundle/dev/variable-overrides.json
```

Production design principles:

| Principle | Why it matters |
|-----------|----------------|
| One repo, many targets | The same reviewed commit promotes through environments |
| Logic in `src/`, orchestration in `resources/` | Code becomes testable, importable, and CI-friendly |
| Notebooks are thin wrappers | Avoids copy-pasted production logic hidden in interactive notebooks |
| Bundle variables for environment differences | Catalogs, schemas, warehouse IDs, and cluster policies vary without code edits |
| Wheel artifacts for shared code | Jobs and pipelines run a packaged version of internal code |
| Permissions and `run_as` in config | Makes least privilege auditable and repeatable |
| Tests beside code | CI catches schema and transformation regressions before deployment |

## Bundle config pattern

`databricks.yml` is the root bundle file. It should define the bundle identity, include resource files, declare variables, build artifacts, and define targets.

```yaml
bundle:
  name: orders_etl

include:
  - resources/*.yml

variables:
  catalog:
    description: Target Unity Catalog catalog
    default: dev_main
  schema:
    description: Target schema
    default: orders_silver
  warehouse_id:
    description: SQL warehouse used by validation queries

artifacts:
  orders_etl:
    type: whl
    path: .
    build: python -m build

targets:
  dev:
    default: true
    mode: development
    workspace:
      host: https://adb-dev.example.com
    variables:
      catalog: dev_main
      schema: orders_silver_dev

  prod:
    mode: production
    workspace:
      host: https://adb-prod.example.com
    variables:
      catalog: prod_main
      schema: orders_silver
    run_as:
      service_principal_name: sp-orders-etl-prod
```

Resource file example:

```yaml
resources:
  jobs:
    orders_daily_job:
      name: orders-daily-${bundle.target}
      tasks:
        - task_key: clean_orders
          job_cluster_key: orders_cluster
          python_wheel_task:
            package_name: orders_etl
            entry_point: clean_orders
            parameters:
              - --catalog
              - ${var.catalog}
              - --schema
              - ${var.schema}
          libraries:
            - whl: ../dist/*.whl
            - pypi:
                package: great-expectations==0.18.22
      job_clusters:
        - job_cluster_key: orders_cluster
          new_cluster:
            spark_version: 15.4.x-scala2.12
            node_type_id: i3.xlarge
            num_workers: 2
```

Exam interpretation:

| Config element | Exam meaning |
|----------------|--------------|
| `include` | Split large bundles into maintainable resource files |
| `variables` | Parameterize values that change by target |
| `targets` | Promote the same bundle to dev/test/prod workspaces |
| `mode: development` | Dev-friendly names/settings, often safer defaults |
| `mode: production` | Production-oriented deployment behavior |
| `artifacts` | Build and deploy wheel or other package artifacts |
| `resources.jobs` / `resources.pipelines` | Jobs and pipelines as code |
| `permissions` | Codify who can view, run, or manage deployed resources |
| `run_as` | Separate deployer identity from runtime identity |

## CI/CD flow

Typical pipeline:

```text
Pull request
  -> lint + unit tests
  -> databricks bundle validate -t dev
  -> databricks bundle deploy -t dev
  -> databricks bundle run -t dev orders_daily_job
  -> integration checks
  -> approval
  -> databricks bundle deploy -t prod
```

Professional exam cues:

| Requirement clue | Strong answer |
|------------------|---------------|
| Same project must deploy to multiple workspaces | Bundle targets and variables |
| Shared transform logic across many tasks | Package logic under `src/` and deploy a wheel artifact |
| Prod must run as a service principal | Configure `run_as` for prod target |
| Different schema per environment | `${var.catalog}` and `${var.schema}` |
| CI should catch invalid resource YAML | `databricks bundle validate` |
| CI should run deployed workflow in dev | `databricks bundle run -t dev <job_or_pipeline>` |

## Library management scopes

| Method | Scope | Best for | Watch out |
|--------|-------|----------|-----------|
| Job task libraries | One job task or job workload | Production ETL dependencies | Pin versions; keep close to workload definition |
| Pipeline libraries | One Lakeflow pipeline | Pipeline-specific package needs | Must be compatible with pipeline compute/runtime |
| Compute-scoped libraries | All notebooks/jobs using that compute | Shared interactive or team compute | Drift between clusters; broader blast radius |
| Notebook-scoped `%pip` | Current notebook session | Exploration and isolated notebook experiments | Not persistent; avoid as hidden prod setup |
| Bundle wheel artifact | Internal project code | Reusable tested code across tasks | Rebuild/redeploy when code changes |
| Init scripts | Cluster startup setup | OS-level setup or rare unsupported cases | Harder to audit; avoid for ordinary Python deps |

Databricks library source choices:

| Dependency form | Example | Production guidance |
|-----------------|---------|---------------------|
| PyPI package | `pandas==2.2.2` | Pin exact versions for prod workloads |
| Private PyPI / VCS | internal package index or Git URL | Use secrets for credentials; avoid plain tokens in Git |
| Local wheel | `orders_etl-1.0.0-py3-none-any.whl` | Best format for internal Python code |
| Source archive | `.tar.gz` or `.zip` source distribution | Use when needed, but prefer wheels to avoid build-time surprises |
| Requirements file | `requirements.txt` on supported DBR | Useful for groups of Python packages, still pin versions |

Important production note: DBFS root library storage is legacy/deprecated in newer patterns. Prefer Unity Catalog volumes, workspace files, package repositories, or bundle-managed artifacts depending on the workload and access mode.

## Troubleshooting library failures

Start with this decision tree:

```text
Import failed?
  -> Was the library declared at the correct scope?
  -> Is the package installed on the job/pipeline compute, not only your notebook?
  -> Does the runtime/access mode support the library source?
  -> Does the runtime identity have READ access to the wheel/file?
  -> Is there a version conflict with DBR built-in libraries or another installed package?
  -> Is Python/Spark/DBR version compatible with the wheel?
  -> Did import precedence load a local module with the same name?
```

Common failures:

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `ModuleNotFoundError` in job but not notebook | Installed with `%pip` only in notebook | Declare job library or bundle wheel |
| Works on dev cluster, fails on prod job cluster | Different DBR, Python version, access mode, or library scope | Align runtime and declare dependencies as code |
| Permission error reading `.whl` | Job runtime identity cannot read workspace file, UC volume, or cloud path | Grant read access or move artifact to approved location |
| Package install hangs/fails | Network/index access or source archive build step fails | Use approved repository, prebuilt wheel, or mirror |
| Wrong package version imported | Library precedence or unpinned transitive dependency | Pin direct/transitive deps; inspect `sys.path` and installed versions |
| Native extension error | Wheel not compatible with cluster Python/OS architecture | Use a compatible wheel or pure Python package |
| Conflicting dependency versions | Two packages require incompatible transitive versions | Isolate workloads or pin compatible versions |

Practical checks:

```python
import sys
import importlib.metadata as md

print(sys.version)
print(sys.executable)
print(md.version("pandas"))
print(sys.path[:5])
```

In UI, check **Compute -> Libraries**, job task library status, pipeline configuration, and driver logs. In CI, make dependency installation explicit so failures happen before production runs.

## Source code pattern for testable transforms

Keep Spark transformation logic as functions that accept and return DataFrames. This is easier to unit test and reuse from notebooks, jobs, and pipelines.

```python
from pyspark.sql import DataFrame
from pyspark.sql import functions as F


def clean_orders(df: DataFrame) -> DataFrame:
    return (
        df.filter(F.col("order_id").isNotNull())
        .withColumn("order_date", F.to_date("order_ts"))
        .dropDuplicates(["order_id"])
    )
```

Thin entry point:

```python
import argparse

from pyspark.sql import SparkSession

from orders_etl.transforms import clean_orders


def main() -> None:
    parser = argparse.ArgumentParser()
    parser.add_argument("--catalog", required=True)
    parser.add_argument("--schema", required=True)
    args = parser.parse_args()

    spark = SparkSession.builder.getOrCreate()
    source = spark.table(f"{args.catalog}.bronze.orders_raw")
    result = clean_orders(source)
    result.write.mode("overwrite").saveAsTable(f"{args.catalog}.{args.schema}.orders")
```

Exam idea: production code should not rely on hard-coded catalog/schema names, hidden notebook state, or manually installed libraries.

## UDF decision framework

Use this order of preference:

| Choice | Performance | Optimizer visibility | Use when |
|--------|-------------|----------------------|----------|
| Native Spark SQL/PySpark functions | Best | Best | Logic can be expressed with built-ins |
| SQL built-in or UC function | Best/good | Good | Shared SQL logic or governed reusable logic |
| Pandas UDF | Medium | Limited | Custom vectorized Python logic is unavoidable |
| Python scalar UDF | Slowest | Limited | Small data or simple unavoidable Python logic |

Exam trap: a question may offer a Python UDF for logic that can be done with `regexp_replace`, `sha2`, `coalesce`, `when`, `array`, `struct`, `from_json`, or window functions. Pick the native Spark option.

## Python UDF vs Pandas UDF

| Feature | Python scalar UDF | Pandas UDF |
|---------|-------------------|------------|
| Input model | One row/value at a time | Batches as pandas Series/DataFrames |
| Transport | Python/JVM serialization per row | Arrow columnar batches |
| Typical speed | Slower | Faster for vectorized logic |
| Return type | Explicit Spark SQL type | Explicit Spark SQL type or type hints |
| Best use | Small or rare custom logic | Numeric/string/vectorized batch logic |
| Still worse than built-ins? | Yes | Usually yes |

Python scalar UDF:

```python
from pyspark.sql import functions as F
from pyspark.sql.types import StringType


@F.udf(StringType())
def mask_email(email: str) -> str:
    if email and "@" in email:
        local, domain = email.split("@", 1)
        return local[:1] + "***@" + domain
    return email


df = df.withColumn("masked_email", mask_email(F.col("email")))
```

Pandas scalar UDF:

```python
import pandas as pd
from pyspark.sql import functions as F


@F.pandas_udf("double")
def dollars_to_cents(amount: pd.Series) -> pd.Series:
    return (amount * 100).round(0)


df = df.withColumn("amount_cents", dollars_to_cents(F.col("amount")))
```

Native Spark expression, usually preferred:

```python
from pyspark.sql import functions as F


df = df.withColumn(
    "masked_email",
    F.concat(
        F.substring("email", 1, 1),
        F.lit("***@"),
        F.split("email", "@").getItem(1),
    ),
)
```

Pandas iterator UDF pattern for heavier setup:

```python
from typing import Iterator

import pandas as pd
from pyspark.sql import functions as F


@F.pandas_udf("double")
def normalize_batches(values: Iterator[pd.Series]) -> Iterator[pd.Series]:
    for batch in values:
        yield (batch - batch.mean()) / batch.std()
```

Use an iterator form when expensive Python state can be reused across batches, but still prefer native Spark if the operation can be expressed there.

## Unity Catalog governed UDFs vs session-scoped UDFs

| Type | Scope | Use when |
|------|-------|----------|
| Session-scoped Python/Pandas UDF | Current Spark session | Job-local or notebook-local logic |
| Unity Catalog governed Python UDF | Catalog/schema function | Shared, discoverable, governed reusable function |
| Batch UC Python UDF | UC function using pandas-style batch processing | Governed batch UDF logic with lower overhead than row-at-a-time |

Professional angle: choose UC governed functions when governance, reuse, and discoverability matter. Choose session-scoped functions for workload-local code packaged with the job.

## UDF limitations and design risks

- UDFs can hide logic from Catalyst optimization.
- UDFs add JVM/Python boundary overhead.
- Python dependencies used inside a UDF must be available on the compute where it executes.
- Pandas UDFs require Arrow-compatible data exchange and type-compatible inputs/outputs.
- Scalar Pandas UDFs generally preserve 1:1 input/output row count.
- Avoid side effects, network calls, non-deterministic behavior, and external writes inside UDFs.
- For streaming workloads, make sure UDF logic is deterministic, fast, and compatible with the pipeline/runtime.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Promote same job through dev/test/prod | Bundle targets and variables |
| Shared business logic across workflows | `src/` package plus wheel artifact |
| Production dependency on external package | Job/pipeline library with pinned version |
| Short notebook experiment | `%pip install` in notebook |
| Internal reusable library | Build wheel and deploy with bundle |
| Private dependency | Approved repository or UC volume/workspace file with secrets-managed access |
| Custom row logic on huge table | Native Spark expression if possible; otherwise Pandas UDF |
| Shared governed custom function | Unity Catalog governed UDF |
| Library works interactively but job fails | Scope/identity/runtime mismatch |

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Keeping all production logic inside notebooks | Put reusable logic in importable modules under `src/`; keep notebooks as thin orchestration or exploration layers |
| Installing production dependencies with hidden `%pip install` cells | Declare dependencies in job/pipeline libraries or package internal code as a bundle wheel artifact |
| Hard-coding `main.prod` tables in code | Use bundle variables for target-specific catalog and schema values |
| Building a wheel but not attaching it to the workload | Reference the wheel from the job or pipeline task libraries |
| Deploying as one user and assuming jobs run with that user's permissions | Configure the production target with an explicit `run_as` identity, usually a service principal |
| Installing a dependency cluster-wide when only one job needs it | Attach the library at job or task scope to reduce blast radius |
| Storing production libraries in DBFS root in modern UC-governed setups | Use package repositories, Unity Catalog volumes, workspace files, or bundle-managed artifacts |
| Writing a Python UDF for logic available in Spark SQL/PySpark functions | Use native Spark expressions such as `regexp_replace`, `sha2`, `when`, `coalesce`, or window functions |
| Treating Pandas UDFs as automatically fast | Use Pandas UDFs only when vectorized Python logic is necessary; native Spark remains the default |
| Defining UDF code that imports packages unavailable on the workload compute | Install UDF dependencies at the job, pipeline, or compute scope where the UDF executes |

# 1.1 Using Python and tools for development

## Exam objective

> Design and implement a scalable Python project structure optimized for Databricks Asset Bundles (DABs), enabling modular development, deployment automation, and CI/CD integration.
>
> Manage and troubleshoot external third-party library installations and dependencies in Databricks, including PyPI packages, local wheels, and source archives.
>
> Develop User-Defined Functions (UDFs) using Pandas/Python UDF.

## Must know

- **Databricks Asset Bundles (DAB)** package jobs, pipelines, dashboards, artifacts, permissions, and environment config as code in `databricks.yml`.
- A **scalable Python project** separates deployable resources (`resources/`) from reusable business logic (`src/`) and automated tests (`tests/`).
- **Targets** (`dev`, `staging`, `prod`) map the same bundle to different workspaces, catalogs, schemas, and permissions.
- **Artifacts** (especially Python wheels) are built during `databricks bundle deploy` and referenced by job/pipeline tasks.
- **Libraries** can be installed at cluster, job, pipeline, or notebook scope; scope affects reproducibility and blast radius.
- **PyPI**, **wheels**, and **source archives** are the three common dependency forms on Databricks.
- **Pandas UDFs** (vectorized UDFs) are faster than row-at-a-time Python UDFs for many workloads but still slower than native Spark expressions.
- Prefer **built-in Spark functions** over UDFs when possible; UDFs block Catalyst optimization and add serialization overhead.

## Recommended DAB project layout

```
my_bundle/
  databricks.yml          # bundle name, targets, variables, includes
  resources/
    jobs.yml              # Lakeflow Jobs
    pipelines.yml         # Lakeflow Spark Declarative Pipelines
  src/
    my_package/           # importable Python modules
      transforms.py
      __init__.py
  tests/
    test_transforms.py
  setup.py                # wheel build for artifacts
  .databricks/            # local bundle state / overrides
```

Production principles:

| Principle | Why it matters |
|-----------|----------------|
| One repo, many targets | Same reviewed commit promotes through environments |
| Logic in `src/`, not notebooks only | Unit-testable, importable, CI-friendly |
| Variables per target | `catalog`, `schema`, `env` differ; code stays identical |
| Permissions in bundle YAML | Least-privilege service principals per environment |
| Wheel artifacts for shared code | Avoid copy-pasting transform logic across tasks |

## Minimal `databricks.yml` pattern

```yaml
bundle:
  name: orders_etl

variables:
  catalog:
    default: dev_main
  schema:
    default: silver

include:
  - resources/*.yml

targets:
  dev:
    default: true
    workspace:
      host: https://adb-dev.example.com
    variables:
      catalog: dev_main
  prod:
    workspace:
      host: https://adb-prod.example.com
    variables:
      catalog: prod_main
```

## Library management

| Method | Scope | Best for | Watch out |
|--------|-------|----------|-----------|
| Cluster libraries | All tasks on that cluster | Shared baseline deps | Drift between clusters |
| Job/pipeline `libraries` | That workload only | Reproducible ETL | Pin versions for prod |
| `%pip install` in notebook | Notebook session | Ad hoc exploration | Not ideal for production jobs |
| Init scripts | Cluster startup | OS-level or custom setup | Harder to audit than bundle libs |
| Wheel in `artifacts` | Bundle-deployed package | Shared internal code | Must rebuild on code change |

Troubleshooting dependency issues:

- **Version conflict**: two libraries require incompatible transitive deps — pin versions or isolate workloads.
- **Driver vs executor**: library must be available where code runs; cluster-scoped install usually covers both.
- **Private PyPI / wheel**: use secure repos; store credentials in secrets, not in Git.
- **Reproducibility**: pin `package==1.2.3` in prod; floating `package` installs can break CI unexpectedly.

```yaml
# Job task libraries in bundle YAML
libraries:
  - pypi:
      package: pandas==2.2.2
  - whl: ./dist/my_package-1.0.0-py3-none-any.whl
```

## UDF types

| UDF type | API | Performance | Use when |
|----------|-----|-------------|----------|
| Python UDF | `@udf` / `udf()` | Slowest (row serialization) | Simple logic, small data |
| Pandas UDF (scalar) | `@pandas_udf` | Better (Arrow batches) | Vectorized per-batch logic |
| Pandas UDF (grouped/map) | `@pandas_udf` with `GROUPED_MAP` | Good for group-wise pandas | Per-key pandas transforms |
| Spark SQL built-ins | `col()`, `when()`, etc. | Best | Always prefer when possible |

```python
from pyspark.sql.functions import col, udf, pandas_udf
from pyspark.sql.types import StringType
import pandas as pd

# Avoid for large data — exam knows this is slow
@udf(StringType())
def mask_email(email: str) -> str:
    if email and "@" in email:
        local, domain = email.split("@", 1)
        return local[0] + "***@" + domain
    return email

# Pandas UDF — processes batches via Arrow
@pandas_udf(StringType())
def mask_email_batch(series: pd.Series) -> pd.Series:
    return series.str.replace(r"(?<=.).(?=.*@)", "*", regex=True)

# Preferred — no UDF
df.withColumn("masked", concat(substring(col("email"), 1, 1), lit("***@"), split(col("email"), "@")[1]))
```

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Promote same job through dev/test/prod | DAB with targets and variables |
| Shared transform logic across tasks | Python wheel in `src/` + artifacts |
| Pin exact dependency versions in prod | Job/pipeline `libraries` with pinned PyPI |
| One-off notebook experiment | `%pip install` |
| Complex row logic on huge tables | Native Spark SQL functions, not Python UDF |
| Custom scalar logic unavoidable | Pandas UDF over Python UDF |

## UI / product surfaces

- **Bundle deploy**: `databricks bundle validate`, `deploy`, `run` from CLI or CI.
- **Compute → Libraries**: inspect cluster-attached libraries.
- **Job / pipeline task config**: per-task library list.
- **Catalog Explorer**: verify deployed resources landed in the right catalog/schema.

## Exam traps

- Putting all production logic in untested notebooks instead of importable `src/` modules.
- Using `%pip install` in scheduled production notebooks instead of declared job libraries.
- Choosing Python UDF when `regexp_replace`, `hash`, `sha2`, or `when`/`otherwise` would work.
- Forgetting that UDFs cannot be optimized by Catalyst like native expressions.
- Deploying to prod without target-specific variables for catalog/schema names.
- Mixing legacy DBFS library paths with UC-governed production patterns without a reason.

# 5.2 Alerting

## Exam objective

> Use SQL Alerts to monitor data quality.
>
> Use the Lakeflow Jobs UI and Jobs API to set up notifications for job status and performance issues.

## Naming note

- Current docs call workflow orchestration **Lakeflow Jobs**. Older material may say Workflows or Jobs.
- Current docs call Databricks SQL alert management APIs/CLI **alerts-v2**.
- The exam says **SQL Alerts**; current docs also emphasize alerts on metric views, data quality monitoring output, anomaly detection output, and scheduled SQL queries.

## Must know

- **SQL alerts** run a SQL query on a schedule, evaluate a condition against the query result, and notify people or destinations when the condition is met.
- SQL alerts require a **SQL warehouse**. They do not run on all-purpose clusters or notebook output.
- SQL alerts are best for **data quality**, **freshness**, **volume**, **business KPIs**, **cost trends**, and **operational checks expressed as SQL**.
- **Job notifications** fire on job or task events such as start, success, failure, duration threshold, or streaming backlog threshold.
- Job notifications are best for **operational workflow state**: failed ETL, slow tasks, stuck streams, or SLA breaches.
- Notifications can target email addresses or configured system destinations such as Slack, Microsoft Teams, PagerDuty, and HTTP webhooks.
- Admins configure system notification destinations; jobs and alerts reference those destinations.
- Alerting should separate **detection**, **notification**, **ownership**, and **remediation**.
- Use meaningful thresholds, owners, schedules, and runbooks to avoid alert fatigue.

## Professional focus vs Associate

Associate notes cover Jobs UI health, data quality, and triggers. For Professional, focus on which alerting surface matches the failure mode:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know Jobs can notify on failure | Choose job-level vs task-level notifications and retry behavior |
| Know SQL alerts exist | Use SQL alerts for data quality, freshness, volume, and KPI thresholds |
| Know DQ checks exist | Alert from quarantine tables, expectation metrics, quality monitors, or metric views |
| Know triggers and schedules | Align alert schedule to pipeline SLA and expected data arrival |
| Know webhooks exist | Use system destinations or user-defined webhooks safely |
| Know run history exists | Use Jobs API/system tables when alerting must work across many jobs/workspaces |

## Alerting decision map

| Requirement clue | Best alerting mechanism |
|------------------|-------------------------|
| Invalid rows appeared in quarantine table | SQL alert |
| Source data is stale past SLA | SQL alert on freshness query |
| Gold revenue dropped below threshold | SQL alert on gold table or metric view |
| ETL job failed | Lakeflow Job notification on failure |
| A specific task failed after all retries | Task notification or Jobs API monitor |
| Job exceeded expected duration | Job or task duration warning notification |
| Streaming backlog exceeded threshold | Streaming backlog notification or stream metrics monitor |
| Need to page on every failed retry attempt | Task notification, not only job-level notification |
| Need account-wide failure alerting | System tables + scheduled monitor job/API |
| CI/CD must fail on unhealthy deployment | Jobs/Pipelines API or CLI polling |

Decision shortcut:

```text
Bad data or business metric -> SQL alert
Workflow state or task runtime -> Job notification
Cross-workspace/platform monitoring -> System tables + scheduled monitor
Automated deployment validation -> Jobs/Pipelines API or CLI
```

## SQL alerts mental model

```text
SQL query
  -> runs on SQL warehouse on a schedule
  -> returns result
  -> alert evaluates condition
  -> notification goes to users or notification destinations
  -> alert history stores previous evaluations
```

Common SQL alert use cases:

| Data quality check | Example condition |
|--------------------|-------------------|
| Freshness | `max_event_ts < current_timestamp() - INTERVAL 2 HOURS` |
| Volume | `row_count = 0` or `row_count > expected_max` |
| Validity | `invalid_rows > 0` |
| Uniqueness | `duplicate_key_count > 0` |
| Referential integrity | `missing_fk_count > 0` |
| Quarantine growth | `new_quarantine_rows > 0` |
| SLA breach | `latest_success_ts < expected_cutoff` |
| Cost trend | `daily_dbus > threshold` |
| Business KPI | `daily_revenue < target` |

## SQL alert query patterns

Quarantine rows in last hour:

```sql
SELECT
  COUNT(*) AS invalid_rows
FROM main.quarantine.orders_invalid
WHERE _quarantined_at >= current_timestamp() - INTERVAL 1 HOUR;
```

Freshness check:

```sql
SELECT
  MAX(event_ts) AS latest_event_ts,
  CASE
    WHEN MAX(event_ts) < current_timestamp() - INTERVAL 2 HOURS THEN 1
    ELSE 0
  END AS is_stale
FROM main.silver.orders;
```

Duplicate business keys:

```sql
SELECT
  COUNT(*) AS duplicate_key_count
FROM (
  SELECT order_id
  FROM main.silver.orders
  GROUP BY order_id
  HAVING COUNT(*) > 1
);
```

Unexpected volume:

```sql
WITH today AS (
  SELECT COUNT(*) AS row_count
  FROM main.silver.orders
  WHERE order_date = current_date()
),
baseline AS (
  SELECT AVG(daily_count) AS avg_7d_count
  FROM (
    SELECT order_date, COUNT(*) AS daily_count
    FROM main.silver.orders
    WHERE order_date BETWEEN current_date() - INTERVAL 7 DAYS
                         AND current_date() - INTERVAL 1 DAY
    GROUP BY order_date
  )
)
SELECT
  today.row_count,
  baseline.avg_7d_count,
  CASE
    WHEN today.row_count = 0 THEN 1
    WHEN today.row_count > baseline.avg_7d_count * 2 THEN 1
    ELSE 0
  END AS volume_anomaly
FROM today CROSS JOIN baseline;
```

SQL alert configuration:

| Setting | Correct approach |
|---------|------------------|
| Warehouse | Use a SQL warehouse sized for the query; serverless reduces cold-start latency |
| Schedule | Run after the pipeline/table is expected to be updated |
| Condition | Alert on a clear scalar result, such as `invalid_rows > 0` |
| Destination | Use owner group email or approved notification destination |
| Permissions | Alert owner and viewers need access to the query and data |
| Query cost | Keep alert queries cheap; summarize from quality tables when possible |
| History | Use alert history to review past evaluations and noise |
| Ownership | Transfer ownership when people move teams |

## SQL alerts on quality monitors and metric views

Databricks SQL alerts are not limited to hand-written table checks.

| Source | Use when |
|--------|----------|
| Quarantine table | You explicitly write bad rows with reason codes |
| Lakeflow expectation metrics | You want alerts from managed pipeline DQ results |
| Unity Catalog data quality monitor output | You use Databricks quality monitoring/profiling |
| Anomaly detection output | You want alerts on detected anomalies |
| Metric view | You want governed business KPI alerts |
| System tables | You want SQL-based alerts on cost, jobs, or audit trends |

Professional pattern:

```text
Pipeline writes quality metrics
  -> SQL alert reads compact metrics table
  -> alert fires when threshold is crossed
  -> notification links to runbook and owning team
```

Avoid rescanning a huge bronze table every five minutes when a compact quality summary or quarantine table already exists.

## Lakeflow Job notifications

Job notifications are configured in the Lakeflow Jobs UI or Jobs API. They can be set at job level or task level.

| Notification event | Use when |
|--------------------|----------|
| Start | A dependent team needs to know a critical job began |
| Success | Downstream handoff or "Succeeded with failures" visibility |
| Failure | On-call or owner must respond to failed workflow |
| Duration warning | Job or task exceeded expected duration |
| Streaming backlog | Streaming task is falling behind |

Important behavior:

| Behavior | Exam-safe interpretation |
|----------|--------------------------|
| Job-level failure notifications | Sent when the job run stops in an unsuccessful state |
| Task-level notifications | Use when you need notifications for specific task failures |
| Retries | Job-level notifications are not sent for failed tasks that are retried before job completion |
| Last retry noise control | Task notifications can mute notifications until the last retry |
| Succeeded with failures | Treated as a successful state for job notification purposes |
| Duration warnings | Require an expected duration or duration threshold |
| Skipped/canceled filtering | Can be muted, but job-level filters do not automatically filter task-level notifications |
| System destinations | Each job/task supports a limited number of destinations per event type |

## Job notification JSON/API pattern

Conceptual job settings:

```json
{
  "name": "orders-daily",
  "email_notifications": {
    "on_failure": ["data-eng-oncall@example.com"],
    "on_duration_warning_threshold_exceeded": ["data-eng-oncall@example.com"],
    "no_alert_for_skipped_runs": true
  },
  "webhook_notifications": {
    "on_failure": [
      {
        "id": "slack-data-eng-prod"
      }
    ]
  },
  "notification_settings": {
    "no_alert_for_canceled_runs": true,
    "alert_on_last_attempt": true
  },
  "timeout_seconds": 7200
}
```

CLI/API examples:

```bash
databricks jobs get --job-id 12345
databricks jobs update --json @job-with-notifications.json
databricks jobs run-now --job-id 12345
databricks jobs get-run --run-id 67890
databricks jobs repair-run --run-id 67890 --rerun-tasks transform_orders
```

Use JSON/API when notifications must be version-controlled in bundles, Terraform, CI/CD, or standard job templates.

## System notification destinations and webhooks

Notification destinations are workspace/admin-level integrations that jobs and alerts can reuse.

| Destination | Use when |
|-------------|----------|
| Email | Simple team or owner notifications |
| Slack | Team channel awareness and triage |
| Microsoft Teams | Enterprise chat notifications |
| PagerDuty | On-call paging and incident routing |
| HTTP webhook | Custom integration or stable payload handling |

Security and reliability:

- Admins should configure approved destinations instead of embedding raw webhook URLs in many jobs.
- Use separate credentials/secrets per destination so one integration can be revoked without breaking all alerts.
- Do not build automation that depends on Slack/Teams message formatting; use a user-defined webhook if the receiver needs a stable schema.
- Include enough context in alert destinations or linked runbooks: job, table, environment, owner, severity, and first action.

## Alert routing and severity

Not every alert should page someone.

| Severity | Example | Destination |
|----------|---------|-------------|
| Info | Job succeeded, daily metric published | Optional email or dashboard |
| Warning | Freshness approaching SLA, invalid rows below threshold | Team channel |
| High | Production job failed, table stale past SLA | On-call channel or PagerDuty |
| Critical | Regulated data quality failure, customer-facing outage | PagerDuty/escalation policy |

Good alert contents:

| Field | Why it matters |
|-------|----------------|
| Environment | Distinguish dev/test/prod |
| Object/workflow | Job, task, pipeline, table, query, alert name |
| Condition | The threshold that fired |
| Observed value | Current value such as `invalid_rows = 25` |
| Owner | Team responsible for response |
| Runbook | First diagnostic/remediation steps |
| Link | Job run, query, dashboard, table, or alert history |

## Data quality alerting design

Data quality alerting should be based on curated checks, not scattered one-off dashboard logic.

Recommended pattern:

```text
Bronze/Silver pipeline
  -> expectations and/or validation logic
  -> quarantine table and quality summary table
  -> SQL alert checks compact summary
  -> notification to owner with runbook
  -> optional job repair or remediation workflow
```

Quality alert examples:

| Check | Query source | Alert condition |
|-------|--------------|-----------------|
| Quarantine rows | `main.quarantine.orders_invalid` | `invalid_rows > 0` |
| Freshness | `MAX(event_ts)` from silver/gold | `is_stale = 1` |
| Duplicate keys | quality summary table | `duplicate_key_count > 0` |
| Referential failures | `orders_missing_customer` table | `missing_fk_count > threshold` |
| Lakeflow expectations | event log / quality metrics table | `failed_records > 0` |
| Volume anomaly | quality monitor output or summary | `anomaly_detected = true` |

Choose alert action:

| Requirement | Correct action |
|-------------|----------------|
| Bad rows can be excluded but investigated later | Quarantine + SQL alert |
| Any bad row makes downstream output invalid | Fail pipeline + job notification |
| Metric drift should be visible but not page | SQL alert to team channel |
| Regulated dataset violates a hard rule | Fail update and page owner |

## SLA and performance alerting

SLA alerts should match the SLA type.

| SLA type | Correct alert |
|----------|---------------|
| Workflow must finish by a time | Job duration warning, job failure notification, or scheduled SLA SQL alert |
| Table must contain data by cutoff | SQL freshness/volume alert |
| Streaming job must keep up | Streaming backlog notification and stream metrics |
| SQL query/dashboard must stay fast | Query history/system table monitor or external APM |
| Cost must stay below threshold | SQL alert on `system.billing.usage` |

Example table freshness SLA:

```sql
SELECT
  CASE
    WHEN MAX(_ingest_ts) < current_timestamp() - INTERVAL 90 MINUTES THEN 1
    ELSE 0
  END AS freshness_breach
FROM main.silver.orders;
```

Example cost threshold:

```sql
SELECT
  SUM(usage_quantity) AS today_dbus
FROM system.billing.usage
WHERE usage_date = current_date()
  AND custom_tags.environment = 'prod';
```

Set alert condition: `today_dbus > <approved_threshold>`.

## Remediation and runbooks

Alerting is incomplete without a response path.

| Alert fires | First response |
|-------------|----------------|
| Job failed | Open run, find first failed task, inspect logs, repair after fix |
| Duration warning | Check task timeline, queue/setup time, input volume, Query Profile or Spark UI |
| Quarantine rows | Inspect reason codes, source file/system, notify source owner if needed |
| Freshness breach | Check upstream ingestion, file arrival, source system, and latest job run |
| Duplicate keys | Inspect dedup rule, source replay/backfill, and event ordering |
| Cost spike | Break down by job, workspace, SKU, and tags |
| Streaming backlog | Check source input rate, processing rate, state size, and scaling |

Remediation tools:

| Tool | Use |
|------|-----|
| Repair run | Rerun failed/skipped tasks after fixing cause |
| Retry policy | Handle transient failures when operation is idempotent |
| Quarantine workflow | Keep valid data flowing while bad data is investigated |
| Task values / condition tasks | Branch to notifications or remediation logic |
| Incident system | Track ownership and resolution for production incidents |

## Alert fatigue controls

| Noise source | Better design |
|--------------|---------------|
| Alert every minute on same unresolved issue | Use appropriate schedule, stateful incident routing, or notification grouping |
| Alert on dev/test noise | Route dev/test to team channels, page only production |
| Alert before pipeline is expected to finish | Schedule after SLA cutoff or upstream completion |
| Alert on every retry attempt | Mute until last retry unless each failed attempt matters |
| Alert on tiny expected variance | Use thresholds, baselines, or anomaly detection |
| Alert with no owner | Add owner/team tags and runbook links |
| Alert query is expensive | Precompute quality metrics or use summary tables |

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Analysts monitor daily revenue dip | SQL alert on metric view or gold aggregate |
| ETL failure should page engineer | Job notification on failure |
| Invalid row count after silver load | SQL alert on quarantine or quality summary table |
| Job run exceeds two hours | Job/task duration warning |
| Streaming ingestion backlog grows | Streaming backlog notification |
| Need every task failure after retries | Task-level notification with last-attempt behavior |
| Need cross-workspace failure monitoring | System tables + scheduled monitoring job |
| Need API-managed alert definitions | Alerts API or `databricks alerts-v2` |
| Need stable webhook payload for automation | User-defined webhook destination |
| Need automated recovery after failure | Jobs API repair-run or remediation job after root cause is known |

## SQL essentials

SQL alerts are defined in Databricks SQL using a query, schedule, warehouse, condition, and destination.

Example alert query:

```sql
SELECT
  COUNT(*) AS failed_expectations
FROM main.observability.pipeline_quality_summary
WHERE event_date = current_date()
  AND failed_records > 0;
```

Example condition:

```text
failed_expectations > 0
```

Example cost alert query:

```sql
SELECT
  SUM(usage_quantity) AS prod_dbus_today
FROM system.billing.usage
WHERE usage_date = current_date()
  AND custom_tags.environment = 'prod';
```

Example condition:

```text
prod_dbus_today > 500
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Using a job failure notification to detect gradual data quality drift | Use a SQL alert on quality metrics, quarantine rows, or monitor output |
| Creating a SQL alert against notebook output or an all-purpose cluster | Use a SQL query that runs on a SQL warehouse |
| Alerting on every retry attempt for noisy transient failures | Mute until last retry unless every attempt is actionable |
| Paging on dev/test failures the same way as production | Route by environment and severity |
| Sending alerts with no owner or runbook | Include owner, context, threshold, link, and remediation steps |
| Running an expensive alert query over raw bronze every few minutes | Alert from compact quality summary or quarantine tables |
| Assuming job-level failure notifications fire for every failed retried task | Use task-level notifications when task attempts matter |
| Forgetting that `Succeeded with failures` is treated as success for job notifications | Configure success notifications or separate monitoring if that state matters |
| Using Slack/Teams message formatting as an automation contract | Use a user-defined webhook for stable machine-readable payloads |
| Repairing a job automatically without understanding root cause | Use repair after diagnosis or design idempotent remediation workflows |

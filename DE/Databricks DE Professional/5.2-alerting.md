# 5.2 Alerting

## Exam objective

> Use SQL Alerts to monitor data quality.
>
> Use the Lakeflow Jobs UI and Jobs API to set up notifications for job status and performance issues.

## Must know

- **SQL Alerts** run scheduled queries against SQL warehouses; trigger on condition (threshold, row count, null rate).
- SQL Alerts are ideal for **data quality** and **business metric** monitoring consumed by analysts.
- **Job notifications** fire on run success, failure, duration thresholds, or policy-based rules via UI/API.
- Notifications can target **email**, **webhooks**, **Slack**, **Microsoft Teams**, **PagerDuty** (integration-dependent).
- Separate **detection** (alert/query) from **remediation** (job repair, quarantine pipeline).
- Alert fatigue: use meaningful thresholds and ownership tags.

## SQL Alerts (data quality)

```
Scheduled SQL query on SQL warehouse
  -> evaluates condition (e.g., invalid_count > 0)
  -> sends notification to subscribers
```

| DQ check | Example alert condition |
|----------|-------------------------|
| Freshness | `max(event_ts) < current_timestamp() - interval 2 hours` |
| Volume | `count(*) = 0` or `count(*) > 2 * avg_7d` |
| Validity | `sum(CASE WHEN amount < 0 THEN 1 ELSE 0 END) > 0` |
| Uniqueness | duplicate key subquery returns rows |

```sql
-- Alert query example: rows in quarantine in last hour
SELECT COUNT(*) AS invalid_rows
FROM main.quarantine.orders_invalid
WHERE quarantined_at >= current_timestamp() - INTERVAL 1 HOUR;
```

Alert configuration concepts:

- **Schedule**: cron or interval aligned to pipeline SLA.
- **Destination**: email list or webhook URL.
- **Threshold**: trigger when `invalid_rows > 0`.

## Job notifications (operational)

| Event | Typical notification |
|-------|----------------------|
| Run failed | Page on-call / Slack channel |
| Run succeeded | Optional info for downstream trigger |
| Run duration exceeded | Performance regression alert |
| Task skipped due to `run_if` | May need custom monitoring |

Jobs UI: **Job → Settings → Notifications** — configure on failure/success, optionally per task.

API pattern (conceptual JSON fields):

```json
{
  "email_notifications": {
    "on_failure": ["data-eng-oncall@example.com"],
    "on_success": []
  },
  "webhook_notifications": {
    "on_failure": [
      {
        "id": "slack-webhook",
        "url": "https://hooks.slack.com/services/..."
      }
    ]
  }
}
```

```bash
databricks jobs update --json @job-with-notifications.json
```

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| Analysts monitor daily revenue dip | SQL Alert on gold aggregate |
| ETL job failure pages engineer | Job notification on failure |
| Invalid row count after silver load | SQL Alert on quarantine table |
| SLA breach: job > 2 hours | Job duration alert or external APM |
| Cross-workspace audit of failures | System tables + scheduled job, not SQL Alert alone |

## PySpark / API essentials

Job notification configuration is primarily JSON/API/UI; remediation may call:

```bash
databricks jobs repair-run --run-id 67890 --rerun-tasks transform_orders
```

## SQL (exam)

SQL Alerts are defined in Databricks SQL UI with a SQL query body — exam tests when to choose them vs job notifications.

## UI / product surfaces

- **SQL → Alerts**: create, schedule, test alert queries.
- **Lakeflow Jobs → Notifications**: failure/success hooks.
- **Query history**: validate alert query cost before enabling.

## Exam traps

- Using job failure notification to detect gradual DQ drift — SQL Alert fits better.
- SQL Alert on all-purpose cluster notebook output — alerts need SQL warehouse.
- No owner/action when alert fires — exam expects notification + runbook/remediation path.
- Alerting on raw cluster metrics only when question is invalid **data** in tables.
- Forgetting quarantine table makes DQ alerts testable without re-scanning entire bronze.

# 4.1 Data sharing and federation

## Exam objective

> Demonstrate delta sharing securely between Databricks deployments using Databricks to Databricks Sharing (D2D) or to external platforms using the open sharing protocol (D2O).
>
> Configure Lakehouse Federation with proper governance across the supported source Systems.
>
> Use Delta Share to share live data from Lakehouse to any computing platform.

## Must know

- **Delta Sharing** provides secure access to live data **without copying** the full dataset to the recipient's storage.
- **D2D (Databricks to Databricks)**: share to another Databricks account/workspace via Unity Catalog shares.
- **D2O (open protocol)**: recipients on non-Databricks platforms use open Delta Sharing clients.
- **Shares** bundle tables/volumes; **recipients** get grants on share objects.
- **Lakehouse Federation** queries external databases **in place** through Unity Catalog connections — different from replication.
- Federation requires **UC governance** (connections, foreign tables/credentials, grants).
- Choose **share** when consumer needs a governed copy-less live read; **federate** when data must stay in source RDBMS; **replicate** when you need Delta-native performance inside your lakehouse.

## Delta Sharing mental model

```
Provider UC metastore
  -> CREATE SHARE + ADD TABLE/VOLUME
  -> GRANT recipient access to share

Recipient
  D2D: mount share in recipient UC catalog
  D2O: open sharing client reads shared tables remotely
```

| Mode | Recipient | Protocol |
|------|-----------|----------|
| D2D | Another Databricks workspace | Native UC Delta Sharing integration |
| D2O | Spark, Pandas, Power BI, etc. | Open Delta Sharing REST protocol |

Benefits for exam:

- Live reads — provider updates visible per protocol/recipient refresh rules.
- Centralized governance on provider side.
- No full duplicate export pipeline required for every consumer.

## Lakehouse Federation

```
Unity Catalog connection (JDBC/API to source)
  -> foreign table or managed federation object
  -> SQL/PySpark query pushes down where possible
  -> grants via UC on foreign tables
```

| Aspect | Delta Sharing | Lakehouse Federation |
|--------|---------------|----------------------|
| Data location | Provider Delta storage | Source system (SQL Server, etc.) |
| Primary pattern | Share lakehouse tables | Query remote DB in place |
| Networking | Sharing endpoints | Often VNet/private connectivity to source |
| Best clue | "Partner Databricks workspace needs sales table" | "Query on-prem SQL without copying nightly" |

Governance checklist:

- UC **connection** with least-privilege source credentials.
- **Storage credential** / network path for private sources.
- `GRANT SELECT` on foreign tables to consumer groups.
- Lineage shows federation source in Catalog Explorer.

## When to use what

| Requirement clue | Best choice |
|------------------|-------------|
| External partner on Databricks needs your Delta tables | D2D Delta Sharing |
| Tableau on non-Databricks cloud needs live lakehouse data | D2O Delta Sharing |
| Avoid nightly JDBC copy of operational DB | Lakehouse Federation |
| Heavy transformations on same data repeatedly inside lakehouse | Replicate to Delta (ingestion), not federation |
| Govern file assets (PDFs) cross-org | Share UC volumes via Delta Sharing |

## PySpark / API essentials

```python
# Recipient D2D — read shared table mounted in recipient catalog
shared_orders = spark.table("partner_catalog.shared_schema.orders")

# Provider side is primarily SQL/UC DDL; PySpark reads shared data like any UC table
```

## SQL (exam)

```sql
-- Provider: create share and add table
CREATE SHARE sales_share;
ALTER SHARE sales_share ADD TABLE main.gold.daily_revenue;

-- Grant recipient
GRANT SELECT ON SHARE sales_share TO RECIPIENT partner_corp;

-- Recipient D2D: create catalog from share (pattern varies by UI/DDL version)
-- CREATE CATALOG partner_sales USING SHARE provider.share_name;

-- Federation: foreign table example pattern
CREATE CONNECTION sqlserver_conn TYPE JDBC
OPTIONS (host 'sqlserver.internal', port '1433', user secret('jdbc-user'), password secret('jdbc-pass'));

CREATE FOREIGN TABLE main.federated_sqlserver.orders
USING CONNECTION sqlserver_conn
OPTIONS (dbtable 'dbo.orders');
```

## UI / product surfaces

- **Catalog Explorer → Sharing**: create shares, recipients, provider/recipient flows.
- **Data sharing UI**: D2D recipient activation.
- **Connections**: Lakehouse Federation setup and test.

## Exam traps

- Calling federation "ingestion" — it queries in place unless you explicitly copy results.
- Sharing without UC — production sharing is UC-governed.
- Copying full tables nightly when question asks for live cross-platform access without duplication — Delta Sharing.
- Forgetting recipient still needs appropriate grants on share.
- Using public internet JDBC to on-prem DB when scenario requires private connectivity (VNet injection, private link).

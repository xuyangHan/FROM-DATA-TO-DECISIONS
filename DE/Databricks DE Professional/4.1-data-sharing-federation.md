# 4.1 Data sharing and federation

## Exam objective

> Demonstrate delta sharing securely between Databricks deployments using Databricks to Databricks Sharing (D2D) or to external platforms using the open sharing protocol (D2O).
>
> Configure Lakehouse Federation with proper governance across the supported source Systems.
>
> Use Delta Share to share live data from Lakehouse to any computing platform.

## Naming note

- The exam says **D2D** and **D2O**. Current Databricks docs usually call these **Databricks-to-Databricks sharing** and **open sharing**.
- **Delta Sharing** is the protocol and Unity Catalog feature for sharing lakehouse data and assets.
- **Lakehouse Federation** is query federation and catalog federation through Unity Catalog. It is not the same as ingestion.

## Must know

- **Delta Sharing** securely shares live lakehouse assets without building a copy pipeline for each recipient.
- **Databricks-to-Databricks sharing (D2D)** is for recipients who also use a Unity Catalog-enabled Databricks workspace.
- **Open sharing (D2O)** is for recipients outside Databricks who use open Delta Sharing clients, connectors, or BI tools.
- A **share** is the provider-side securable object that contains shared assets.
- A **recipient** represents the external organization, user, or Databricks metastore that receives access.
- A **provider** is created on the recipient side for Databricks-to-Databricks sharing; recipients create catalogs from provider shares.
- Provider-side grants control which recipients can access the share.
- Recipient-side grants control which local users can access the shared catalog/tables.
- **Lakehouse Federation** lets Databricks query external systems in place through Unity Catalog connections and foreign catalogs.
- Use sharing for cross-organization lakehouse access, federation for live queries against external sources, and ingestion/replication for repeated heavy transformations or Delta-native performance.

## Professional focus vs Associate

Associate notes cover Unity Catalog objects, table types, and basic privileges. For Professional, focus on architecture and governance decisions:

| Associate baseline | Professional emphasis |
|--------------------|-----------------------|
| Know Delta Sharing exists | Choose D2D vs open sharing and configure provider/recipient objects |
| Know UC governs tables | Apply least privilege across shares, recipients, providers, catalogs, and foreign catalogs |
| Know managed/external tables | Decide what asset can be shared and what lifecycle remains with the provider |
| Know external systems exist | Choose federation vs Lakeflow Connect ingestion vs direct replication |
| Know grants are required | Distinguish provider-side recipient access from recipient-side user access |
| Know volumes store files | Share UC volumes when governed file assets need cross-org access |

## Delta Sharing mental model

```text
Provider Databricks account / UC metastore
  -> create recipient
  -> create share
  -> add tables, views, materialized views, schemas, volumes, or notebooks
  -> grant recipient SELECT on share
  -> monitor share/recipient activity

Recipient
  D2D: create catalog from provider.share, then grant local users access
  D2O: use activation URL or credential profile with an open sharing client/tool
```

Core objects:

| Object | Side | Purpose |
|--------|------|---------|
| Share | Provider | Container of assets being shared |
| Recipient | Provider | External party authorized to access shares |
| Provider | Recipient | Representation of the organization that shared data |
| Shared catalog | Recipient | Local UC catalog created from a provider share in D2D |
| Credential profile | Recipient | Open sharing file or OAuth/client credential details used by non-Databricks tools |

Important property: recipients read shared data, but they do not get write/update privileges on the provider's source tables through Delta Sharing.

## D2D vs open sharing

| Dimension | Databricks-to-Databricks (D2D) | Open sharing (D2O) |
|-----------|--------------------------------|--------------------|
| Recipient platform | Databricks workspace with Unity Catalog | Any supported client/tool |
| Recipient identity | Recipient's UC metastore sharing identifier | Token/OAuth credential profile |
| Provider setup | `CREATE RECIPIENT ... USING ID ...` | `CREATE RECIPIENT ...` and share activation/credential details |
| Recipient experience | Create a UC catalog from provider share | Read with Delta Sharing connector/client |
| Local governance | Recipient can grant local users access to shared catalog objects | Governance depends on recipient platform/tool |
| Typical clue | "Partner also uses Databricks" | "Power BI, pandas, Spark, external platform, or non-Databricks tool" |
| Operational risk | Cross-metastore privileges and provider/recipient setup | Credential file/token handling and rotation |

## Provider workflow for Delta Sharing

Provider-side steps:

| Step | What happens |
|------|--------------|
| 1. Identify recipient type | Databricks-to-Databricks or open sharing |
| 2. Create recipient | D2D uses recipient sharing identifier; open sharing creates activation/credential flow |
| 3. Create share | Share is a UC securable object |
| 4. Add assets | Add tables, views, schemas, materialized views, volumes, or notebooks as supported |
| 5. Grant access | `GRANT SELECT ON SHARE ... TO RECIPIENT ...` |
| 6. Monitor and revoke | Use audit logs, system tables, Catalog Explorer, `SHOW GRANTS`, and `REVOKE` |

SQL pattern:

```sql
-- D2D recipient: recipient provides sharing identifier from CURRENT_METASTORE()
CREATE RECIPIENT partner_corp
USING ID 'aws:us-west-2:11111111-2222-3333-4444-555555555555';

-- Open sharing recipient: Databricks generates activation/credential details
CREATE RECIPIENT external_bi_partner
COMMENT 'Non-Databricks BI partner';

CREATE SHARE sales_share
COMMENT 'Curated sales data for approved partners';

ALTER SHARE sales_share
ADD TABLE main.gold.daily_revenue;

ALTER SHARE sales_share
ADD TABLE main.gold.customer_segments;

GRANT SELECT ON SHARE sales_share TO RECIPIENT partner_corp;

SHOW GRANTS ON SHARE sales_share;
```

Provider permissions:

| Task | Typical required privilege |
|------|----------------------------|
| Create share | `CREATE SHARE` on metastore |
| Create recipient | `CREATE RECIPIENT` on metastore |
| Add table/view to share | Share ownership plus `USE SCHEMA` and `SELECT` on the asset |
| Grant share to recipient | Share ownership, plus ability to use recipient/share where required |
| Remove access | Share ownership or sufficient management privilege |

## Recipient workflow

D2D recipient:

```sql
-- Recipient side: inspect shared providers/shares in Catalog Explorer or SQL
-- Then create a local catalog from the provider share.
CREATE CATALOG partner_sales
USING SHARE provider_name.sales_share;

GRANT USE CATALOG ON CATALOG partner_sales TO `analytics`;
GRANT USE SCHEMA ON SCHEMA partner_sales.default TO `analytics`;
GRANT SELECT ON CATALOG partner_sales TO `analytics`;
```

PySpark read:

```python
shared_orders = spark.table("partner_sales.default.daily_revenue")
```

Open sharing recipient:

```python
import delta_sharing


profile = "/secure/path/to/open-sharing.profile"
client = delta_sharing.SharingClient(profile)

for share in client.list_shares():
    print(share.name)

df = delta_sharing.load_as_pandas(
    f"{profile}#sales_share.gold.daily_revenue"
)
```

Recipient-side principles:

| Concern | Correct approach |
|---------|------------------|
| Local Databricks users need access | Grant privileges on the shared catalog/schema/table |
| Recipient wants to update provider table | Not supported through Delta Sharing; use a separate write process |
| Open sharing credential is exposed | Rotate/revoke recipient credentials and reissue securely |
| Partner no longer needs access | Provider revokes recipient access to the share |
| Local consumers need stable names | Create catalog/schema naming standards for shared data |

## What can be shared

| Asset | Sharing guidance |
|-------|------------------|
| Delta tables | Primary Delta Sharing use case |
| Views | Useful to expose filtered/projected data instead of base tables |
| Materialized views | Share curated/precomputed results where supported |
| Schemas | Useful when many objects need the same recipient access |
| Volumes | Use for governed file assets such as PDFs or images |
| Notebooks | Useful for provider documentation/examples |
| Row filters / column masks / ABAC-protected data | Possible with current constraints; verify policy behavior and recipient enforcement model |

Design choice:

| Requirement clue | Better shared asset |
|------------------|---------------------|
| Partner needs only approved columns/rows | Share a view or policy-protected object |
| Partner needs a whole curated domain | Share a schema or multiple tables |
| Partner needs PDF/image files | Share a UC volume |
| Partner needs examples and usage instructions | Share a notebook along with data |
| Partner needs write-back collaboration | Delta Sharing alone is not enough; design a separate write or clean-room pattern |

## Delta Sharing governance

Provider governance:

- Share only curated assets, not raw unrestricted bronze by default.
- Use views, row filters, column masks, or ABAC when the recipient should not see full base tables.
- Grant access to recipients, not broad external identities.
- Keep ownership and management privileges limited.
- Monitor share changes, recipient grants, and recipient access events through audit logs/system tables.
- Revoke share access when the contract or use case ends.

Recipient governance:

- Do not assume everyone in the recipient workspace can read a shared catalog.
- Grant `USE CATALOG`, `USE SCHEMA`, and `SELECT` to local groups.
- Treat shared catalogs as read-only external data sources.
- Document source/provider, allowed use, freshness expectations, and contact path.

Security checklist:

| Question | Correct answer |
|----------|----------------|
| Does the recipient need direct cloud storage credentials? | No for normal Delta Sharing access |
| Can provider stop access centrally? | Yes, revoke recipient access to the share |
| Can recipient grant write access back to provider tables? | No |
| Should activation links or credential files be emailed broadly? | No, send through secure channels and store securely |
| Does sharing replace table masking/filtering design? | No, use views/policies when different recipients need different data |
| Is sharing UC-governed? | Yes, shares, recipients, and providers are UC securables |

## Lakehouse Federation mental model

```text
Unity Catalog connection
  -> authenticates to external source
  -> foreign catalog exposes external schemas/tables in UC
  -> users query foreign tables with SQL/PySpark
  -> optimizer pushes down supported filters/projections/aggregations
  -> UC grants control who can discover and query the foreign objects
```

Federation does not copy source data into Delta. It queries the external system directly or accesses externally governed catalog/storage depending on federation type.

## Query federation vs catalog federation

| Dimension | Query federation | Catalog federation |
|-----------|------------------|--------------------|
| Source access | JDBC/API-style remote query to external database/warehouse | External catalog metadata plus direct storage access where supported |
| Common objects | Connection and foreign catalog | Connection, external location/storage credential, foreign catalog |
| Query execution | Some work pushed to source, some in Databricks | Databricks reads external table storage governed by imported catalog metadata |
| Best for | Operational DBs and warehouses | External catalogs such as Glue/Hive/Snowflake catalog federation where supported |
| Watch out | Source load, pushdown limits, network latency | Storage credentials, external locations, catalog compatibility |

Supported query-federation source examples include MySQL, PostgreSQL, Teradata, Oracle, Redshift, SQL Server, Azure Synapse, BigQuery, Snowflake, Salesforce Data 360, and Databricks. Supported sources change over time, so the exam usually tests the concept rather than a rare connector edge case.

## Lakehouse Federation setup

SQL pattern:

```sql
CREATE CONNECTION postgresql_connection
TYPE POSTGRESQL
OPTIONS (
  host 'postgres.internal.company.com',
  port '5432',
  user secret('db-secrets', 'postgres-user'),
  password secret('db-secrets', 'postgres-password')
);

CREATE FOREIGN CATALOG postgresql_catalog
USING CONNECTION postgresql_connection
OPTIONS (database = 'salesdb');

GRANT USE CATALOG ON CATALOG postgresql_catalog TO `analysts`;
GRANT USE SCHEMA ON SCHEMA postgresql_catalog.public TO `analysts`;
GRANT SELECT ON SCHEMA postgresql_catalog.public TO `analysts`;
```

Query pattern:

```sql
SELECT
  order_date,
  SUM(amount) AS revenue
FROM postgresql_catalog.public.orders
WHERE order_date >= DATE '2026-01-01'
GROUP BY order_date;
```

PySpark pattern:

```python
orders = spark.table("postgresql_catalog.public.orders")
```

Federation governance checklist:

| Area | Correct approach |
|------|------------------|
| Connection credentials | Use least-privilege source identity and secrets |
| UC privileges | Grant only required foreign catalog/schema/table privileges |
| Network path | Use required private connectivity for private sources |
| Source impact | Avoid broad scans on production OLTP systems |
| Performance | Push filters/projections; materialize to Delta for repeated heavy workloads |
| Lineage | Use Catalog Explorer/system lineage where supported |
| Ownership | Separate connection owners from broad data consumers |

## Sharing vs federation vs ingestion

| Requirement clue | Best choice |
|------------------|-------------|
| External partner needs live access to your curated Delta table | Delta Sharing |
| Partner also uses Databricks and Unity Catalog | D2D Delta Sharing |
| Partner uses pandas, Power BI, Spark, or another non-Databricks tool | Open Delta Sharing |
| Users need to query operational database tables without copying them | Lakehouse Federation |
| Remote source is queried heavily every day for transformations | Ingest/replicate to Delta |
| Need low-latency managed sync from supported SaaS/database source | Lakeflow Connect |
| Need BI acceleration over external data | Materialize into Delta or create Databricks-side serving tables |
| Need collaborative write-back | Delta Sharing is read-oriented; use another architecture |

Decision shortcut:

```text
Share lakehouse data outward -> Delta Sharing
Query external data in place -> Lakehouse Federation
Copy external data into lakehouse -> Lakeflow Connect / ingestion / replication
```

## Performance and operational considerations

Delta Sharing:

- Provider updates are available to recipients without a separate export pipeline.
- D2D uses Unity Catalog integration and avoids long-lived bearer tokens.
- Open sharing requires careful credential/activation handling.
- Shared data is usually read-only to recipients.
- Partition filtering and views can reduce what recipients scan.

Lakehouse Federation:

- Query pushdown is not unlimited; unsupported operations may execute in Databricks after data is returned.
- Source systems can become bottlenecks or be impacted by analytical queries.
- Network latency matters.
- Repeated heavy joins, aggregations, or transformations usually belong in Delta tables.
- Query federation is excellent for discovery, light joins, and selective access, but not a replacement for a curated medallion pipeline when data is reused heavily.

## SQL essentials

Provider-side Delta Sharing:

```sql
CREATE SHARE IF NOT EXISTS finance_share;

ALTER SHARE finance_share
ADD TABLE main.gold.monthly_revenue;

CREATE RECIPIENT IF NOT EXISTS finance_partner
USING ID 'aws:us-east-1:00000000-0000-0000-0000-000000000000';

GRANT SELECT ON SHARE finance_share TO RECIPIENT finance_partner;

SHOW GRANTS ON SHARE finance_share;
```

Recipient-side D2D:

```sql
CREATE CATALOG finance_partner_catalog
USING SHARE provider_name.finance_share;

SELECT *
FROM finance_partner_catalog.default.monthly_revenue;
```

Federation:

```sql
CREATE CONNECTION sqlserver_conn
TYPE SQLSERVER
OPTIONS (
  host 'sqlserver.internal.company.com',
  port '1433',
  user secret('db-secrets', 'sqlserver-user'),
  password secret('db-secrets', 'sqlserver-password')
);

CREATE FOREIGN CATALOG sqlserver_sales
USING CONNECTION sqlserver_conn
OPTIONS (database = 'SalesDW');

SELECT COUNT(*)
FROM sqlserver_sales.dbo.orders;
```

## Wrong vs correct approach

| What is wrong | Correct approach |
|---------------|------------------|
| Calling Lakehouse Federation an ingestion pipeline | Treat federation as live query-in-place; ingest separately when you need Delta copies |
| Building nightly exports when the requirement is live copy-less external access | Use Delta Sharing |
| Using open sharing for a recipient that has Unity Catalog Databricks | Use D2D with the recipient sharing identifier |
| Sharing base tables when each recipient should see different rows or columns | Share filtered views or policy-protected objects |
| Granting a recipient access to a share but forgetting local recipient-side grants | On the recipient workspace, grant users access to the shared catalog/schema/table |
| Expecting a recipient to update provider tables through Delta Sharing | Design a separate write-back or collaboration pattern |
| Running heavy repeated transformations directly against a federated OLTP database | Materialize the needed data into Delta with an ingestion pipeline |
| Using broad source database credentials for federation | Use least-privilege credentials and UC grants |
| Ignoring private networking requirements for external sources | Configure the required private connectivity before federation |
| Treating credential files or activation links as harmless | Store and transmit open sharing credentials securely and rotate/revoke when needed |

# 5.3 Bundle deploy and promotion

## Exam objective

> Deploy Declarative Automation Bundles (formerly Databricks Asset Bundles) to package, configure, and promote Lakeflow Jobs, Lakeflow Spark Declarative Pipelines, and other workspace assets across dev, test, and prod environments.

## Big idea

Declarative Automation Bundles are Databricks infrastructure as code. A bundle packages source files and resource definitions so the same project can be validated, deployed, and promoted across environments.

For the exam, remember:

> Define jobs, pipelines, dashboards, artifacts, permissions, and environment settings as code. Promote the same reviewed commit through dev, test, and prod with target-specific configuration.

## What a bundle can contain

- Source files: notebooks, Python files, SQL files, libraries.
- Lakeflow Jobs.
- Lakeflow Spark Declarative Pipelines.
- Dashboards and alerts.
- Python wheel artifacts.
- Permissions.
- Workspace paths and deployment state.
- Tests and scripts used by CI/CD.

## Core files

| File or folder | Purpose |
|---|---|
| `databricks.yml` | Required bundle root configuration |
| `resources/*.yml` | Jobs, pipelines, dashboards, schemas, volumes, and other resources |
| `src/` | Business logic used by jobs or pipelines |
| `tests/` | Unit or integration tests |
| `.databricks/` | Local bundle metadata and optional variable overrides |

## Minimal deployment lifecycle

1. Developer edits code in a Git branch.
2. Developer opens a PR.
3. CI runs tests and `databricks bundle validate`.
4. PR is merged.
5. CI deploys to dev or test.
6. Integration checks run.
7. Prod deployment runs after approval.
8. Prod workflows run as a controlled identity, often a service principal.

## Key commands

```bash
databricks bundle validate -t dev
databricks bundle deploy -t dev
databricks bundle run orders_daily -t dev
databricks bundle summary -t dev
databricks bundle deploy -t prod
```

## Example job resource

```yaml
resources:
  jobs:
    orders_daily:
      name: orders_daily_${bundle.target}
      parameters:
        - name: catalog
          default: ${var.catalog}
      tasks:
        - task_key: transform_orders
          notebook_task:
            notebook_path: ../src/transform_orders.py
            base_parameters:
              catalog: ${var.catalog}
          job_cluster_key: etl_cluster
      job_clusters:
        - job_cluster_key: etl_cluster
          new_cluster:
            spark_version: 15.4.x-scala2.12
            node_type_id: i3.xlarge
            num_workers: 2
```

## Example pipeline resource

```yaml
resources:
  pipelines:
    silver_pipeline:
      name: silver_pipeline_${bundle.target}
      catalog: ${var.catalog}
      schema: ${var.schema}
      libraries:
        - notebook:
            path: ../src/pipelines/silver_orders.py
      configuration:
        bundle.sourcePath: ${workspace.file_path}/src
```

## Promotion pattern

| Environment | Typical behavior |
|---|---|
| Dev | Fast iteration, personal or shared dev workspace, paused schedules |
| Test/staging | Integration tests, production-like settings, controlled data |
| Prod | Approved deploy, service principal `run_as`, schedules enabled, tighter permissions |

Promotion should normally use the same Git commit. The environment changes through target configuration, not through editing code differently for prod.

## Resource drift

If a job is managed by a bundle, manual UI edits can be overwritten by the next bundle deploy. The bundle is the desired state. If production settings need to change, change the bundle YAML, review it in Git, and redeploy.

## Bundle identity and state

Databricks tracks deployed bundle resources by bundle identity and deployment state. Important identity pieces include:

- Bundle name.
- Target.
- Workspace.
- Workspace root path/state path.

This is why accidentally deploying the wrong target or changing identity fields can create or affect unexpected resources.

## `run_as`

`run_as` controls the identity used when a bundle workflow runs. It can be configured at the top level, inside a target, or for specific supported resources such as jobs.

Production best practice:

```yaml
targets:
  prod:
    run_as:
      service_principal_name: 00000000-0000-0000-0000-000000000000
```

This separates "who deployed the bundle" from "who runs the production workflow."

## What to choose in exam scenarios

| Scenario | Best answer |
|---|---|
| Same job in dev/test/prod | Bundle resource with targets |
| Promote Lakeflow Job | `databricks bundle deploy -t <target>` |
| Promote Lakeflow Spark Declarative Pipeline | Define pipeline resource in bundle and deploy |
| Prod job should not run as a human | Configure `run_as` service principal |
| UI job changed but bundle deploy reverted it | Bundle is source of truth |
| Need review before prod deployment | PR plus CI/CD pipeline |

## Exam traps

- Deploying manually changed UI jobs instead of bundle-managed resources.
- Maintaining separate YAML files with copied job definitions for each environment when variables/targets would work.
- Running production jobs as the developer who deployed them.
- Forgetting to enable/disable schedules by environment.
- Deploying from an unreviewed feature branch.
- Assuming `bundle deploy` runs the job. Deploy creates/updates resources; `bundle run` starts a workflow.
- Using `bundle destroy` casually in production.

## Short memory hook

**Bundles define resources. Targets promote them. Deploy updates them. Run executes them.**

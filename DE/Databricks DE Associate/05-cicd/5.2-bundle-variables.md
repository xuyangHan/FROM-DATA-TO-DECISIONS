# 5.2 Bundle variables and target overrides

## Exam objective

> Understand environment-specific configuration using Automation Bundle (formerly Databricks Asset Bundles) variables and overrides while promoting the same codebase across dev, test, and prod targets.

## Big idea

Declarative Automation Bundles let you keep one codebase and one set of resource definitions, then change environment-specific values by target. The bundle should describe the job or pipeline once, while `dev`, `test`, and `prod` provide different values such as workspace host, catalog name, schema name, service principal, schedules, cluster settings, and permissions.

The exam usually wants this answer:

> Do not hardcode environment names in notebooks or maintain separate job definitions for each environment. Use bundle variables, substitutions, and target overrides.

## Must know

- **`databricks.yml`**: Required root bundle configuration file.
- **`include`**: Lets `databricks.yml` include resource files such as `resources/jobs.yml`.
- **`targets`**: Named deployment environments such as `dev`, `test`, and `prod`.
- **`variables`**: Custom deployment-time values referenced as `${var.name}`.
- **`type: complex`**: Keyword for variables that hold structured data such as objects/maps or arrays.
- **Substitutions**: Built-in values such as `${bundle.name}`, `${bundle.target}`, `${workspace.host}`, and `${workspace.current_user.userName}`.
- **Target override**: Target-specific replacement or extension of top-level bundle settings.
- **Default target**: One target can be marked `default: true`.
- **Deployment-time variable**: Bundle variables are resolved during validation/deployment. For per-run values, use job parameters instead.
- **`workspace.root_path`**: Target-specific workspace deployment directory. Use it to override the default bundle deployment path.

## Common variable examples

| Requirement | Bundle feature |
|---|---|
| `dev_main` catalog in dev and `prod_main` catalog in prod | `variables.catalog` with per-target values |
| Different workspace URLs | `targets.<target>.workspace.host` |
| Different service principals | `targets.<target>.run_as` |
| Prod schedule enabled, dev schedule paused | Target-specific resource override |
| Different cluster policy IDs | Variable or target resource override |
| Reusable cluster object or list of settings | `type: complex` variable |
| Same notebook path across environments | Top-level resource definition with `${var.catalog}` parameter |
| Shared production deployment directory | `targets.prod.workspace.root_path` |

## Example bundle structure

```text
my_project/
  databricks.yml
  resources/
    jobs.yml
  src/
    transform_orders.py
```

## Example `databricks.yml`

```yaml
# databricks.yml
# Root config file for the bundle

bundle:
  name: orders_pipeline
  databricks_cli_version: ">= 0.230.0"

include:
  - resources/*.yml

variables:
  catalog:
    description: "Unity Catalog catalog name"
    default: dev_main

  schema:
    description: "Schema/database name"
    default: silver

  warehouse_id:
    description: "SQL warehouse used by SQL tasks"

  job_cluster_node_type:
    default: Standard_DS3_v2

  job_cluster:
    description: "Reusable job cluster definition"
    type: complex
    default:
      spark_version: 15.4.x-scala2.12
      node_type_id: Standard_DS3_v2
      num_workers: 2
      spark_conf:
        spark.databricks.delta.schema.autoMerge.enabled: "true"

workspace:
  root_path: /Workspace/Users/${workspace.current_user.userName}/.bundle/${bundle.name}/${bundle.target}

run_as:
  user_name: ${workspace.current_user.userName}

resources:
  jobs:
    orders_etl_job:
      name: ${bundle.target}_orders_etl_job

      tags:
        project: orders
        env: ${bundle.target}

      schedule:
        quartz_cron_expression: "0 0 6 * * ?"
        timezone_id: America/Toronto
        pause_status: PAUSED

      parameters:
        - name: catalog
          default: ${var.catalog}
        - name: schema
          default: ${var.schema}

      job_clusters:
        - job_cluster_key: orders_job_cluster
          new_cluster: ${var.job_cluster}

      tasks:
        - task_key: bronze_ingest
          job_cluster_key: orders_job_cluster
          notebook_task:
            notebook_path: ./src/bronze_ingest.py
            base_parameters:
              catalog: ${var.catalog}
              schema: ${var.schema}

        - task_key: validate_orders
          depends_on:
            - task_key: bronze_ingest
          job_cluster_key: orders_job_cluster
          notebook_task:
            notebook_path: ./src/validate_orders.py

        - task_key: silver_transform
          depends_on:
            - task_key: validate_orders
          condition_task:
            op: EQUAL_TO
            left: "{{tasks.validate_orders.values.status}}"
            right: "ok"

        - task_key: gold_aggregate
          depends_on:
            - task_key: silver_transform
          job_cluster_key: orders_job_cluster
          spark_python_task:
            python_file: ./src/gold_aggregate.py
            parameters:
              - "--catalog"
              - ${var.catalog}
              - "--schema"
              - ${var.schema}

        - task_key: refresh_dashboard_table
          depends_on:
            - task_key: gold_aggregate
          sql_task:
            warehouse_id: ${var.warehouse_id}
            query:
              query_id: 00000000-0000-0000-0000-000000000000

  pipelines:
    orders_dlt_pipeline:
      name: ${bundle.target}_orders_dlt_pipeline
      catalog: ${var.catalog}
      schema: ${var.schema}
      target: dlt_orders
      development: true
      continuous: false
      libraries:
        - notebook:
            path: ./src/dlt/orders_dlt.py

targets:
  dev:
    default: true
    mode: development
    workspace:
      host: https://dev.cloud.databricks.com
      root_path: /Workspace/Users/${workspace.current_user.userName}/.bundle/${bundle.name}/${bundle.target}
    variables:
      catalog: dev_main
      schema: silver_dev
      warehouse_id: dev_warehouse_id
    run_as:
      user_name: ${workspace.current_user.userName}

  prod:
    mode: production
    workspace:
      host: https://prod.cloud.databricks.com
      root_path: /Shared/production_pipelines/${bundle.name}/${bundle.target}
    variables:
      catalog: prod_main
      schema: silver
      warehouse_id: prod_warehouse_id
      job_cluster_node_type: Standard_DS4_v2
      job_cluster:
        spark_version: 15.4.x-scala2.12
        node_type_id: Standard_DS4_v2
        num_workers: 4
        spark_conf:
          spark.databricks.delta.schema.autoMerge.enabled: "true"
    run_as:
      service_principal_name: 00000000-0000-0000-0000-000000000000
```

## Complex variables

By default, custom bundle variables are treated like simple string values. In Databricks Asset Bundles / Declarative Automation Bundles, `complex` is the specific keyword used in the `variables` block for variables that hold structured data.

Use `type: complex` for objects/maps of key-value pairs or arrays.

Example object/map variable:

```yaml
variables:
  job_cluster:
    description: "Reusable cluster settings"
    type: complex
    default:
      spark_version: 15.4.x-scala2.12
      node_type_id: Standard_DS3_v2
      num_workers: 2

resources:
  jobs:
    orders_daily:
      job_clusters:
        - job_cluster_key: orders_cluster
          new_cluster: ${var.job_cluster}
```

Example array variable:

```yaml
variables:
  notification_emails:
    type: complex
    default:
      - data-eng@example.com
      - platform@example.com
```

Exam model:

```text
Simple scalar variable -> no type needed
Structured variable    -> type: complex
```

Important restriction: if `type` is set to `complex`, the default value must also be structured. Bundle validation fails if a `complex` variable has a single scalar default value.

## How variables can be set

Databricks supports several ways to provide variable values:

- Top-level `variables` default.
- Per-target `targets.<target>.variables`.
- Environment variables beginning with `BUNDLE_VAR_`.
- `--var` values on `databricks bundle validate`, `deploy`, or `run`.
- `.databricks/bundle/<target>/variable-overrides.json`.

For exam purposes, the most important one is **per-target variables**.

## Variables vs job parameters

| Need | Use |
|---|---|
| Different catalog per environment | Bundle variable |
| Different workspace per environment | Bundle target |
| Different run date for today's job run | Job parameter |
| Row count discovered by an upstream task | Task value |
| Different service principal in prod | `run_as` target override |

Bundle variables are deployment-time configuration. Job parameters are run-time configuration.

## Target overrides

Targets can override many mappings:

- `workspace`
- `variables`
- `resources`
- `permissions`
- `presets`
- `sync`
- `artifacts`
- `run_as`

Example: define one job at the top level, then override only the prod schedule:

```yaml
resources:
  jobs:
    orders_daily:
      name: orders_daily
      schedule:
        quartz_cron_expression: "0 0 6 * * ?"
        timezone_id: UTC
        pause_status: PAUSED

targets:
  prod:
    resources:
      jobs:
        orders_daily:
          schedule:
            quartz_cron_expression: "0 0 6 * * ?"
            timezone_id: UTC
            pause_status: UNPAUSED
```

## Overriding the deployment path

The `workspace.root_path` property inside a specific target block is the standard way to override the default deployment directory in the Databricks workspace.

This is common when moving from user-specific development deployments to shared production locations. For example, dev might use the default user-scoped bundle path, while prod deploys to a shared folder such as `/Shared/production_pipelines/`.

```yaml
targets:
  dev:
    default: true
    workspace:
      host: https://dev.cloud.databricks.com

  prod:
    workspace:
      host: https://prod.cloud.databricks.com
      root_path: /Shared/production_pipelines/${bundle.name}/${bundle.target}
```

Exam model:

```text
workspace.host      -> which workspace to deploy to
workspace.root_path -> where bundle files/state/resources are deployed in that workspace
```

## Exam traps

- Hardcoding `prod_main` in a notebook and then trying to reuse it in dev.
- Creating three unrelated job definitions instead of one resource with target overrides.
- Confusing bundle variables with job parameters.
- Forgetting `-t prod` and accidentally deploying to the default target.
- Putting secrets directly in `databricks.yml`; use secret scopes or CI secret stores.
- Assuming target overrides copy everything manually. Unspecified values can inherit from the top-level definition.
- Trying to change production deployment location with notebook paths or variables only. Use `targets.prod.workspace.root_path`.
- Defining an object or array variable without `type: complex`.
- Setting `type: complex` but giving the variable a single scalar default value.

## Short memory hook

**Same code, different targets. Variables for environment values; parameters for run values.**

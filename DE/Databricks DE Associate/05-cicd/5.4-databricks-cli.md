# 5.4 Databricks CLI for CI/CD

## Exam objective

> Understand the Databricks CLI to validate, deploy, and manage Declarative Automation Bundles (formerly Databricks Asset Bundles) and other workspace assets in automated CI/CD workflows.

## Big idea

The Databricks CLI is the command-line tool used by developers and CI/CD systems to validate, deploy, run, summarize, and manage Declarative Automation Bundles. In exam scenarios, the CLI is the bridge between Git and Databricks deployment.

## Must know

- Command name: `databricks`.
- Bundle commands require the Databricks CLI.
- CLI authentication can use profiles, OAuth, PATs, or service principal based automation depending on the environment.
- CI/CD should store credentials in secret stores, not in the repo.
- `databricks bundle validate` checks bundle syntax and resource schema.
- `databricks bundle deploy` creates or updates resources in the target workspace.
- `databricks bundle run` starts a defined job or pipeline.
- `databricks bundle summary` shows bundle identity and resources.
- `databricks bundle destroy` removes deployed resources and should be treated carefully.
- Bundle commands support the global `--output` or `-o` option. Use `--output json` when CI/CD needs machine-readable output.
- Bundle commands can use `--var="<key>=<value>"` to temporarily override bundle variable values.

## Core bundle commands

```bash
# Check CLI version
databricks --version

# Create a starter bundle
databricks bundle init

# Validate bundle config for the default target
databricks bundle validate

# Validate a specific target
databricks bundle validate -t dev

# Validate and return JSON for CI/CD parsing
databricks bundle validate -t dev --output json
databricks bundle validate -t dev -o json

# Deploy to a target
databricks bundle deploy -t test
databricks bundle deploy --target test

# Deploy with JSON output for automation
databricks bundle deploy -t test --output json

# Deploy while temporarily overriding a bundle variable
databricks bundle deploy -t test --var="catalog=test_main"

# Run a deployed job or pipeline
databricks bundle run orders_daily -t test

# Show deployed resources and links
databricks bundle summary -t test

# Show summary as JSON for jq or GitHub Actions outputs
databricks bundle summary -t test -o json

# Preview deployment changes where available
databricks bundle plan -t prod
databricks bundle plan -t prod -o json

# Delete deployed resources; dangerous in prod
databricks bundle destroy -t dev
```

## JSON output for automation

The Databricks CLI, including bundle commands, supports the global `--output` option, also written as `-o`. Specifying `--output json` makes the command return JSON-formatted output instead of human-readable text.

This is ideal in CI/CD because later steps can parse deployment metadata with tools like `jq`, then pass values into GitHub Actions outputs, logs, approvals, or follow-up commands.

```bash
databricks bundle validate -t dev --output json
databricks bundle summary -t test -o json
databricks bundle plan -t prod -o json | jq .
```

Memory hook:

```text
-t / --target -> where to deploy
-o / --output -> how to format command output
--json        -> JSON request/input payload for many non-bundle commands
```

## Temporary variable overrides

Use `--var="<key>=<value>"` with bundle commands such as `validate`, `deploy`, and `run` to temporarily override a bundle variable's default value.

```bash
databricks bundle validate -t dev --var="catalog=dev_main"
databricks bundle deploy -t test --var="catalog=test_main"
databricks bundle deploy -t prod --var="catalog=prod_main,warehouse_id=abc123"
```

According to the Databricks variable resolution order, `--var` has the highest precedence. If the same variable is defined in multiple places, the CLI uses the first value it finds in this order:

1. `--var` options on the bundle command.
2. Environment variables beginning with `BUNDLE_VAR_`.
3. `.databricks/bundle/<target>/variable-overrides.json`.
4. `variables` mappings under `targets` in bundle configuration files.
5. Top-level variable `default` values in bundle configuration files.

Exam model:

```text
--var is temporary and highest precedence.
target variables and defaults live in databricks.yml.
BUNDLE_VAR_ values are useful in CI/CD, but --var still wins.
```

## What `validate` catches

`databricks bundle validate` helps catch:

- Invalid YAML.
- Missing required bundle fields.
- Resource settings that do not match the expected schema.
- Missing variables without defaults.
- Target configuration problems.

It does not prove business logic is correct. You still need unit tests, integration tests, and/or test job runs.

## What `deploy` does

`databricks bundle deploy -t prod` applies the bundle configuration to the target workspace. It can:

- Upload/sync source files and artifacts.
- Create or update Lakeflow Jobs.
- Create or update Lakeflow Spark Declarative Pipelines.
- Apply configured permissions.
- Track deployment state for future updates.

Deploying does not automatically mean the workflow has run.

## What `run` does

`databricks bundle run <resource_key> -t dev` starts a deployed job, pipeline, or script by bundle resource key.

Use `--params` when passing job parameters at run time:

```bash
databricks bundle run -t dev --params process_date=2026-05-21 orders_daily
```

## CLI in CI/CD

Example flow:

```text
Pull request:
  1. Install Databricks CLI.
  2. Authenticate to Databricks.
  3. Run unit tests.
  4. Run databricks bundle validate -t dev.

Merge to main:
  1. Run tests again.
  2. Deploy to test.
  3. Run integration checks.
  4. Deploy to prod after approval.
```

## Authentication exam points

| Situation | Good practice |
|---|---|
| Local developer | User profile or OAuth user-to-machine auth |
| CI/CD | Service principal or workload identity/OIDC where supported |
| Secret handling | CI secret store, not committed files |
| Multiple workspaces | Use targets and/or profiles carefully |
| Production run identity | Bundle `run_as` service principal |

## Target flag

The `-t` or `--target` flag selects the bundle target:

```bash
databricks bundle validate -t prod
databricks bundle deploy --target prod
```

If no target is specified, the default target can be used. This is convenient in dev but risky in automation. CI/CD should usually be explicit.

## Other common bundle commands

These are useful to recognize even if the exam mainly emphasizes `validate`, `deploy`, `run`, and `destroy`:

```bash
# Display command help
databricks -h
databricks bundle -h
databricks bundle deploy -h

# Generate bundle config from existing workspace resources
databricks bundle generate job --existing-job-id 12345
databricks bundle generate pipeline --existing-pipeline-id abc-123

# Bind an existing workspace resource to a bundle resource key
databricks bundle deployment bind my_job 12345 -t dev

# Remove the binding between a bundle resource and workspace resource
databricks bundle deployment unbind my_job -t dev

# Print the bundle configuration JSON schema
databricks bundle schema

# Open a deployed bundle resource in the workspace UI
databricks bundle open my_job -t dev

# Sync local bundle files to the remote workspace
databricks bundle sync --dry-run
databricks bundle sync --watch
```

Exam notes:

- `bundle generate` helps create YAML for an existing job, pipeline, dashboard, or app.
- `bundle deployment bind` lets a bundle manage an existing workspace resource instead of creating a new one.
- `bundle schema` returns the bundle configuration schema.
- `bundle sync` synchronizes local files to the workspace; `--dry-run` previews and `--watch` keeps syncing.
- `bundle open` is convenience for opening resources, not deploying or running them.

## Other useful CLI awareness

You may see non-bundle CLI commands too:

```bash
databricks auth profiles
databricks current-user me
databricks workspace list /Workspace
databricks workspace mkdirs /Workspace/Shared/demo
databricks fs ls dbfs:/
databricks jobs list
databricks jobs get 12345
databricks jobs run-now 12345
databricks clusters list
databricks pipelines list-pipelines
databricks secrets list-scopes
databricks repos list
```

For the certification objective, prefer bundle commands when the scenario asks about validating, deploying, and promoting CI/CD assets.

## Exam traps

- Running only tests but never `bundle validate`, so invalid YAML reaches deployment.
- Running `bundle validate` but assuming the job was deployed.
- Running `bundle deploy` but assuming the job executed.
- Omitting `-t prod` and deploying to the default target.
- Putting PATs or service principal secrets in Git.
- Using a human user as the production workflow identity.
- Treating `bundle destroy` as a normal cleanup step for prod.
- Thinking `BUNDLE_VAR_` always wins. `--var` has higher precedence.
- Using old workspace import scripts when the question clearly asks for bundle-based CI/CD.

## Short memory hook

**Validate checks. Deploy applies. Run executes. Summary shows. Destroy removes.**

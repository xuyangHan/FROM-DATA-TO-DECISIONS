# 5.4 Databricks CLI for CI/CD

## Exam objective

> Understand the Databricks CLI to validate, deploy, and manage Declarative Automation Bundles (formerly Databricks Asset Bundles) and other workspace assets in automated CI/CD workflows.

## Big idea

The Databricks CLI is the command-line tool used by developers and CI/CD systems to authenticate, validate bundle configuration, deploy resources, run jobs or pipelines, inspect deployed resources, and clean up environments.

For the exam, think of the CLI as the bridge between Git and Databricks deployment.

```text
Git repo -> Databricks CLI -> target workspace
```

## Must know

- Command name: `databricks`.
- Bundle commands require the Databricks CLI.
- Bundle configuration usually lives in `databricks.yml`.
- CLI authentication can use profiles, OAuth, PATs, or service principal based automation depending on the environment.
- CI/CD should store credentials in secret stores, not in the repo.
- Use `-t` or `--target` to select the bundle target, such as `dev`, `test`, or `prod`.
- Use `--output json` or `-o json` when CI/CD needs machine-readable output.
- Use `--var="<key>=<value>"` to temporarily override bundle variables.

## Main bundle lifecycle

Memorize this flow first:

```text
init -> validate -> deploy -> run -> summary

plan = preview
destroy = delete
```

### 1. Initialize

Create a starter bundle project:

```bash
databricks bundle init
```

Purpose:

- Scaffolds a new bundle project.
- Helps create the first `databricks.yml` and related files.

### 2. Validate

Validate the default target:

```bash
databricks bundle validate
```

Validate a specific target:

```bash
databricks bundle validate -t dev
```

Validate and return JSON:

```bash
databricks bundle validate -t dev --output json
databricks bundle validate -t dev -o json
```

What `validate` checks:

- YAML syntax.
- Missing required bundle fields.
- Variable resolution.
- References.
- Resource definitions.
- Resource settings that do not match the expected schema.
- Missing variables without defaults.
- Target configuration problems.

Important: `validate` does not prove business logic is correct. You still need unit tests, integration tests, and/or test job runs.

### 3. Deploy

Deploy to a target:

```bash
databricks bundle deploy -t dev
databricks bundle deploy --target dev
```

Deploy with JSON output:

```bash
databricks bundle deploy -t test --output json
```

Deploy with a temporary variable override:

```bash
databricks bundle deploy \
  -t dev \
  --var="catalog=dev_main"
```

What `deploy` does:

- Applies the bundle configuration to the target workspace.
- Uploads or syncs source files and artifacts.
- Creates or updates Lakeflow Jobs.
- Creates or updates Lakeflow Spark Declarative Pipelines.
- Applies configured permissions.
- Tracks deployment state for future updates.

Important: `deploy` creates or updates resources. It does not automatically run the workflow.

### 4. Run

Run a deployed resource by bundle resource key:

```bash
databricks bundle run orders_daily -t dev
```

Use `--params` when passing job parameters at run time:

```bash
databricks bundle run -t dev --params process_date=2026-05-21 orders_daily
```

What `run` does:

- Starts a deployed job, pipeline, or script.
- Triggers execution after resources already exist.

Important: `deploy` applies resources; `run` executes them.

### 5. Summary

Inspect deployed resources:

```bash
databricks bundle summary -t dev
```

Show summary as JSON for CI/CD parsing:

```bash
databricks bundle summary -t test -o json
```

What `summary` shows:

- Deployed jobs.
- Deployed pipelines.
- Bundle resource keys.
- Links to workspace resources.

### 6. Plan

Preview deployment changes:

```bash
databricks bundle plan -t prod
```

Return plan output as JSON:

```bash
databricks bundle plan -t prod -o json
```

Mental model:

```text
databricks bundle plan = terraform plan for Databricks bundles
```

It shows what would change if deployed now, without applying the changes. This is useful for CI/CD approval gates.

### 7. Destroy

Remove deployed resources from a workspace:

```bash
databricks bundle destroy -t dev
```

Use for cleanup in dev/test. Treat as dangerous in prod.

## Key CLI flags

### Target flag

The `-t` or `--target` flag selects the bundle target:

```bash
databricks bundle validate -t prod
databricks bundle deploy --target prod
```

If no target is specified, the default target can be used. This is convenient in dev but risky in automation. CI/CD should usually be explicit.

### JSON output flag

The Databricks CLI, including bundle commands, supports the global `--output` option, also written as `-o`.

Use `--output json` or `-o json` when later CI/CD steps need to parse command output with tools like `jq`, GitHub Actions outputs, logs, approvals, or follow-up commands.

```bash
databricks bundle validate -t dev --output json
databricks bundle summary -t test -o json
databricks bundle plan -t prod -o json | jq .
```

Memory hook:

```text
-t / --target -> where to deploy
-o / --output -> how to format command output
--json        -> JSON request/input payload for many non-bundle commands
```

### Temporary variable override flag

Use `--var="<key>=<value>"` with bundle commands such as `validate`, `deploy`, and `run` to temporarily override a bundle variable's default value.

```bash
databricks bundle validate -t dev --var="catalog=dev_main"
databricks bundle deploy -t test --var="catalog=test_main"
databricks bundle deploy -t prod --var="catalog=prod_main,warehouse_id=abc123"
```

According to the Databricks variable resolution order, `--var` has the highest precedence. If the same variable is defined in multiple places, the CLI uses the first value it finds in this order:

1. `--var` options on the bundle command.
2. Environment variables beginning with `BUNDLE_VAR_`.
3. `.databricks/bundle/<target>/variable-overrides.json`.
4. `variables` mappings under `targets` in bundle configuration files.
5. Top-level variable `default` values in bundle configuration files.

Exam model:

```text
--var is temporary and highest precedence.
target variables and defaults live in databricks.yml.
BUNDLE_VAR_ values are useful in CI/CD, but --var still wins.
```

## Authentication

Authenticate the CLI:

```bash
databricks auth login
```

This is the modern authentication command to know. Older docs or examples may show:

```bash
databricks configure
```

Show configured authentication profiles:

```bash
databricks auth profiles
```

Profiles are useful when switching between:

```text
dev
prod
personal
ci
```

Authentication exam points:

| Situation | Good practice |
|---|---|
| Local developer | User profile or OAuth user-to-machine auth |
| CI/CD | Service principal or workload identity/OIDC where supported |
| Secret handling | CI secret store, not committed files |
| Multiple workspaces | Use targets and/or profiles carefully |
| Production run identity | Bundle `run_as` service principal |

## CI/CD flow

Example flow:

```text
Pull request:
  1. Install Databricks CLI.
  2. Authenticate to Databricks.
  3. Run unit tests.
  4. Run databricks bundle validate -t dev.

Merge to main:
  1. Run tests again.
  2. Run databricks bundle plan -t prod for review if needed.
  3. Deploy to test.
  4. Run integration checks.
  5. Deploy to prod after approval.
```

For CI/CD, prefer:

```bash
databricks bundle validate -t dev -o json
databricks bundle deploy -t test -o json
databricks bundle summary -t test -o json
```

## Helper bundle commands

These are useful to recognize even if the exam mainly emphasizes `init`, `validate`, `deploy`, `run`, `summary`, `plan`, and `destroy`.

Display command help:

```bash
databricks -h
databricks --version
databricks bundle -h
databricks bundle deploy -h
```

Generate bundle configuration from existing workspace resources:

```bash
databricks bundle generate job --existing-job-id 12345
databricks bundle generate pipeline --existing-pipeline-id abc-123
```

Bind or unbind an existing workspace resource:

```bash
databricks bundle deployment bind my_job 12345 -t dev
databricks bundle deployment unbind my_job -t dev
```

Print the bundle configuration schema:

```bash
databricks bundle schema
```

Open deployed resources in the workspace UI:

```bash
databricks bundle open -t dev
databricks bundle open orders_daily -t dev
databricks bundle open my_job -t dev
```

Sync local bundle files to the remote workspace:

```bash
databricks bundle sync --dry-run
databricks bundle sync --watch
```

Helper command notes:

- `bundle generate` helps create YAML for an existing job, pipeline, dashboard, or app.
- `bundle deployment bind` lets a bundle manage an existing workspace resource instead of creating a new one.
- `bundle deployment unbind` removes the binding between a bundle resource and workspace resource.
- `bundle schema` returns the bundle configuration schema and can help with autocomplete or IDE tooling.
- `bundle open` opens resources in the workspace UI; it does not deploy or run them.
- `bundle sync --dry-run` previews sync changes.
- `bundle sync --watch` keeps syncing local files.

## Exam traps

- Running only tests but never `bundle validate`, so invalid YAML reaches deployment.
- Running `bundle validate` but assuming the job was deployed.
- Running `bundle deploy` but assuming the job executed.
- Omitting `-t prod` and deploying to the default target.
- Putting PATs or service principal secrets in Git.
- Using a human user as the production workflow identity.
- Treating `bundle destroy` as a normal cleanup step for prod.
- Thinking `BUNDLE_VAR_` always wins. `--var` has higher precedence.
- Confusing `--output json` with `--json`. `--output json` controls response format; `--json` is commonly used for JSON request/input payloads.
- Using old workspace import scripts when the question clearly asks for bundle-based CI/CD.

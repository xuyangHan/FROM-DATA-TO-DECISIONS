# 4.3 Job schedules and triggers

## Exam objective

> Implement job schedules using Lakeflow Jobs with an understanding of trigger types (scheduled, file arrival, and table update).

## Must know

- Lakeflow Jobs can be started manually, by schedule, by file arrival, by table update, continuously, or programmatically through API/external orchestration.
- The exam objective focuses on **scheduled**, **file arrival**, and **table update** triggers.
- **Scheduled triggers** start a job based on time. They are best when the business process is tied to a clock.
- **File arrival triggers** start a job when new files arrive in a monitored Unity Catalog volume or external location path.
- **Table update triggers** start a job when source tables are updated. They are best for downstream jobs that should react to Delta/Iceberg/table commits.
- **Continuous triggers** keep a job always running by starting a new run after the previous one completes or fails. Know that this exists, but do not choose it when the question asks for scheduled, file arrival, or table update.
- **Paused triggers** do not start new runs. Any active run continues.
- By default, only one run of a job can be active at a time unless max concurrent runs is changed. Runs can be skipped when concurrency is exceeded.

## Trigger types

| Trigger type | Starts job when | Best for | Avoid when |
|--------------|-----------------|----------|------------|
| Scheduled | A configured time/interval arrives | Nightly ETL, daily reports, fixed SLAs | Data arrives unpredictably and empty runs are costly |
| File arrival | New files appear in a UC volume/external location path | Irregular cloud-file drops | You need downstream reaction to table commits instead of raw files |
| Table update | Monitored table(s) update | Silver/gold jobs after upstream table writes | Source readiness is represented only by raw files |
| Continuous | Previous run completes or fails | Always-on workflows | Periodic batch or event-driven batch |
| None/manual | User/API starts the job | Ad hoc backfills, external orchestrator control | Required automatic schedule/trigger |

## Scheduled triggers

Use a scheduled trigger for time-based orchestration.

Schedule options:

| Schedule type | Meaning |
|---------------|---------|
| Simple | Run periodically by interval, such as every 12 hours |
| Advanced | Specify period, start time, time zone, and optional Quartz cron syntax |

Important details:

- Advanced schedules allow a time zone selection.
- UTC avoids daylight saving time surprises.
- Time zones with daylight saving time can skip or delay hourly jobs when the clock changes.
- The scheduler is not for very low-latency orchestration; runs can occasionally be delayed by several minutes.
- Databricks enforces a minimum interval between scheduled runs even if cron syntax is more aggressive.

Example use cases:

| Requirement | Trigger |
|-------------|---------|
| Run every day at 2 AM after source systems close | Scheduled |
| Refresh executive dashboard every weekday morning | Scheduled |
| Monthly financial close process | Scheduled |
| Hourly polling when source does not emit events | Scheduled, but consider data-driven options if available |

Exam shortcut: if the requirement names a fixed clock time, a scheduled trigger is usually the answer.

## File arrival triggers

Use a file arrival trigger when files land irregularly and the job should run when new files appear.

Key requirements and behavior:

- Requires Unity Catalog.
- The monitored location must be a Unity Catalog volume or external location, including a subpath.
- The trigger recursively checks subdirectories under the configured path.
- It checks for new files on a best-effort basis, often around every minute, subject to storage performance.
- For best performance, enable file events on the external location.
- Only **new files** trigger runs. Overwriting an existing file with the same name does not reliably represent a new arrival.
- The trigger path cannot contain wildcards such as `*` or `?`.
- Use **Wait after last change** when files arrive as a batch and the job should wait until the batch quiets down.

Typical file-driven pattern:

```
Vendor drops files into /Volumes/main/landing/orders/
  -> file arrival trigger starts job
  -> Auto Loader notebook/pipeline incrementally reads new files
  -> writes main.bronze.orders_raw
```

File arrival trigger plus Auto Loader:

```python
(spark.readStream
  .format("cloudFiles")
  .option("cloudFiles.format", "json")
  .option("cloudFiles.schemaLocation", "/Volumes/main/landing/_schemas/orders")
  .load("/Volumes/main/landing/orders")
  .writeStream
  .option("checkpointLocation", "/Volumes/main/landing/_checkpoints/orders")
  .trigger(availableNow=True)
  .toTable("main.bronze.orders_raw"))
```

Exam shortcut: file arrival starts the job; Auto Loader processes the files exactly-once/incrementally.

## Table update triggers

Use a table update trigger when table commits represent readiness for downstream work.

Key behavior:

- The trigger checks monitored tables for updates and starts the job when updates are detected.
- A trigger can run when **one** monitored table updates or when **all** monitored tables update, depending on configuration.
- It supports data changes such as updates, merges, deletes, and appends on supported tables.
- Supported sources include Unity Catalog Delta managed tables, some external Delta-backed UC tables, managed Iceberg tables, materialized views, streaming tables, and supported UC views/metric views.
- For best performance and scalability, enable file events on the external location where the tables are stored.
- A trigger can monitor up to a limited number of supported tables. The current docs call out a 10-table limit per trigger.
- Use **Wait after last change** when upstream writes arrive in several commits and downstream should wait for the batch to finish.

Typical table-driven pattern:

```
main.bronze.orders_raw updated
  -> table update trigger starts silver job
  -> main.silver.orders updated
  -> table update trigger starts gold job
```

Dynamic references available with table update triggers include:

| Reference pattern | Meaning |
|-------------------|---------|
| `{{job.trigger.table_update.updated_tables}}` | JSON list of tables updated since last run |
| `{{job.trigger.table_update.commit_timestamp.iso_datetime}}` | Commit timestamp when one table is monitored |
| `{{job.trigger.table_update.version}}` | Commit version when one table is monitored |

Exam shortcut: choose table update when the question says "run downstream when table X is updated" or "only after upstream Delta table has new data."

## Trigger configuration mental model

Ask: "What signal proves the input is ready?"

| Readiness signal | Trigger |
|------------------|---------|
| Clock time | Scheduled |
| New raw file in governed storage | File arrival |
| New table commit/update | Table update |
| External tool decides when | None/manual plus API run |
| Always-on processing | Continuous |

Then ask: "Should one job contain multiple dependent tasks instead of multiple triggers?"

If the workflow is one logical pipeline, prefer one multi-task job:

```
scheduled daily trigger
  -> ingest
  -> transform
  -> refresh dashboard
```

If jobs are separately owned and table readiness is the contract, use table update triggers:

```
bronze job updates main.bronze.orders_raw
silver job starts on table update
gold job starts on silver table update
```

## External orchestration

When an external orchestrator manages dependencies across platforms, let that tool explicitly trigger the Databricks job only after the upstream dependency succeeds.

Best pattern:

```
mainframe job succeeds
  -> external orchestrator calls Databricks Jobs REST API Run Now endpoint
  -> Databricks Lakeflow Job starts
  -> downstream Databricks task DAG runs in order
```

This is more reliable than guessing with a Databricks schedule because sequencing stays with the system that knows when the upstream cross-platform work actually finished. Use **None/manual** as the Databricks trigger style when an external orchestrator will start runs programmatically.

## PySpark / API essentials

Triggers are configured in Lakeflow Jobs, not in PySpark. The task code should assume the trigger selected the right time to run.

Scheduled run date from a job parameter:

```python
process_date = dbutils.widgets.get("process_date")
```

File arrival job using Auto Loader:

```python
source_path = dbutils.widgets.get("source_path")
target_table = dbutils.widgets.get("target_table")

(spark.readStream
  .format("cloudFiles")
  .option("cloudFiles.format", "csv")
  .option("cloudFiles.schemaLocation", f"{source_path}/_schemas")
  .option("header", "true")
  .load(source_path)
  .writeStream
  .option("checkpointLocation", f"{source_path}/_checkpoints")
  .trigger(availableNow=True)
  .toTable(target_table))
```

Table update trigger can pass metadata into a task parameter:

```text
updated_tables = {{job.trigger.table_update.updated_tables}}
```

Notebook reads it:

```python
updated_tables = dbutils.widgets.get("updated_tables")
```

## UI / product surfaces

- **Jobs & Pipelines > Jobs > Job details > Schedules & Triggers**: add Scheduled, File arrival, Table update, or Continuous trigger.
- **Trigger status**: Active or Paused.
- **Advanced schedule**: time zone and Quartz cron syntax.
- **File arrival trigger**: storage location, wait after last change, test connection.
- **Table update trigger**: monitored tables, one-table/all-tables readiness behavior, wait after last change, test trigger.
- **Run history**: see whether runs started by schedule, trigger, manual action, or API.

## Exam traps

- Choosing cron when the question says files arrive unpredictably. File arrival is usually better.
- Choosing file arrival when the readiness signal is an upstream Delta table commit. Use table update.
- Forgetting Unity Catalog requirements for file arrival paths.
- Using wildcard paths in file arrival triggers.
- Expecting file arrival triggers to re-run when an existing file is overwritten with the same name.
- Ignoring daylight saving time when scheduling hourly jobs in a local time zone.
- Assuming a trigger can start unlimited overlapping runs. Max concurrent runs still matters.
- Confusing **Paused trigger** with a failed or disabled job. Paused means no new automatic runs.
- Splitting tightly related sequential tasks into separate scheduled jobs when one DAG with dependencies would express readiness better.
- Scheduling a Databricks job to "probably" run after an external mainframe job. If an external orchestrator owns the dependency chain, have it call the Jobs REST API Run Now endpoint after upstream success.

# 4.2 Job tasks and DAG dependencies

## Exam objective

> Configure common tasks (notebook, SQL query, dashboard, and pipeline tasks) and their dependencies using Lakeflow Jobs and its DAG-based task graph.

## Must know

- A **Lakeflow Job** is an orchestrated unit of work. It can contain one task or a multi-task DAG.
- A **task** is one runnable step, such as a notebook, SQL query/file, dashboard refresh, or Lakeflow Spark Declarative Pipeline.
- A **task key** uniquely identifies a task within a job. Downstream dependencies and dynamic value references use task keys.
- A **DAG** is a directed acyclic graph. It allows parallel execution when tasks have no dependency between them and prevents cycles.
- By default, a downstream task starts after all configured upstream dependencies succeed.
- Use **job parameters** for run-level values shared across tasks, such as `process_date`, `env`, `catalog`, and `schema`.
- Use **task parameters** to pass values into the task's code or asset.
- Use **task values** to pass small outputs from one running task to later tasks.
- Use **job clusters** or serverless jobs compute for task execution when available; use SQL warehouses for SQL and dashboard tasks.
- A **Run Job** task triggers another existing Databricks job from the parent DAG. The parent task waits for the triggered job run to complete before downstream tasks proceed.

## Common task types

| Task type | Runs | Compute requirement | Best for |
|-----------|------|---------------------|----------|
| Notebook | Workspace/Git notebook | Jobs compute, serverless, or existing cluster depending on setup | PySpark/SQL ETL, validations, custom logic |
| SQL query | Saved SQL query | Serverless or Pro SQL warehouse | SQL transforms, checks, DML |
| SQL file | `.sql` file from workspace or Git | Serverless or Pro SQL warehouse | Versioned SQL assets in jobs |
| Dashboard | Published dashboard refresh | Serverless or Pro SQL warehouse | Refresh BI outputs after data is ready |
| Pipeline | Lakeflow Spark Declarative Pipeline | Pipeline configuration | Managed medallion pipelines, streaming tables, materialized views |
| Run Job | Another Lakeflow Job | Target job's config | Reuse an existing job as a step |
| If/else | Boolean condition | No data compute for user code | Branching |
| For each | Loop over nested task | Nested task's compute | Metadata-driven repetition |

The exam objective names notebook, SQL query, dashboard, and pipeline tasks. Other task types may appear as distractors, but those four are the core set for this subdomain.

## Run Job tasks

Use a **Run Job** task when one Lakeflow Job should call another existing Databricks job. This is useful for modularity and reuse: shared ingestion, validation, or publishing jobs can be maintained once and triggered from multiple parent DAGs.

Key behavior:

- The Run Job task triggers the target job as a separate job run.
- The parent DAG waits for that triggered job run to finish before starting dependent downstream tasks.
- Downstream behavior depends on the triggered job's result, just like other task dependencies.
- The target job keeps its own task graph, parameters, compute, permissions, and run history.

Exam shortcut: choose **Run Job** when the requirement says to trigger or reuse another existing Databricks job, not when you need to run a single notebook or SQL query directly.

## Other task types

Common task types beyond the core notebook/SQL/dashboard/pipeline set include:

| Task type | Use when |
|-----------|----------|
| Run Job | Trigger another existing Databricks job and wait for it in the parent DAG |
| If/else condition | Branch based on a boolean expression, task value, or parameter |
| For each | Run a nested task repeatedly over a list of inputs |
| Python wheel | Run packaged Python code as a reusable project artifact |
| JAR | Run JVM/Scala/Java application code |
| dbt | Run dbt commands/projects where supported |

## Notebook tasks

Use a notebook task when the step needs PySpark, Python, Scala, R, SQL notebook cells, or custom logic.

Good fits:

- Bronze-to-silver cleaning in PySpark.
- Calling a custom REST API.
- Running validation logic and setting task values.
- Preparing parameters for downstream For each loops.

Notebook task parameters are exposed through widgets:

```python
process_date = dbutils.widgets.get("process_date")
catalog = dbutils.widgets.get("catalog")

orders = spark.table(f"{catalog}.bronze.orders_raw")
```

Notebook task can set a value for downstream tasks:

```python
row_count = orders.count()
dbutils.jobs.taskValues.set(key="row_count", value=row_count)
```

Exam traps:

- Forgetting to configure widgets or task parameters.
- Hardcoding catalog/schema names when the job should be environment-aware.
- Using all-purpose interactive clusters as the default production answer when jobs compute or serverless jobs compute is available.

## SQL tasks

Use a SQL task when the step is a SQL query, SQL alert, or SQL file.

SQL tasks require **Databricks SQL** and a **serverless or Pro SQL warehouse**.

Good fits:

- `INSERT INTO` / `MERGE INTO` gold tables.
- CTAS statements for simple transforms.
- Data quality checks expressed in SQL.
- Running versioned `.sql` files from Git.

Example SQL task body:

```sql
INSERT INTO main.gold.daily_revenue
SELECT
  order_date,
  SUM(amount) AS revenue,
  COUNT(*) AS order_count
FROM main.silver.orders
WHERE order_date = :process_date
GROUP BY order_date;
```

Exam traps:

- Choosing a SQL task but not configuring a SQL warehouse.
- Assuming SQL task parameters use notebook widgets. SQL tasks use configured SQL parameters.
- Running a dashboard refresh before the SQL transform dependency finishes.

## Dashboard tasks

Use a dashboard task to refresh a **published Databricks dashboard** after upstream data is ready.

Key points:

- The dashboard must be published and accessible to the user configuring the job.
- Dashboard tasks use a serverless or Pro SQL warehouse.
- The task can notify subscribers after the dashboard is refreshed.
- Access behavior depends on how the dashboard was published and whether shared data permissions are used.

Typical dependency:

```
ingest_bronze -> transform_silver -> update_gold -> refresh_dashboard
```

Exam shortcut: dashboards are usually **last** in the DAG because they consume curated outputs.

## Pipeline tasks

Use a pipeline task to run a **Lakeflow Spark Declarative Pipeline** as part of a job.

Pipeline tasks are for **triggered pipelines**. Continuous pipelines already run continuously, so triggering them from a job task is not the normal pattern.

Good fits:

- Running a managed medallion pipeline as one step in a broader workflow.
- Orchestrating a pipeline before a dashboard refresh.
- Coordinating a pipeline with non-pipeline tasks such as file checks, notifications, or downstream SQL.

Important distinction:

| Concept | Meaning |
|---------|---------|
| Pipeline task | A Jobs task that triggers/runs a pipeline |
| Pipeline definition | The Lakeflow Spark Declarative Pipeline with tables, materialized views, expectations, and settings |
| Job DAG | The orchestration layer around the pipeline and other tasks |

Exam trap: do not try to pass normal notebook-style parameters directly into a pipeline task unless the question describes a supported configuration pattern. Pipeline configuration is managed at the pipeline level.

## DAG dependency design

Use dependencies to express data readiness:

```
ingest_customers_raw
ingest_orders_raw
  -> clean_orders
  -> clean_customers
      -> build_customer_order_mart
      -> refresh_sales_dashboard
```

Parallel where possible:

- `ingest_customers_raw` and `ingest_orders_raw` can run at the same time if independent.
- `clean_orders` waits only for orders ingestion.
- `build_customer_order_mart` waits for both cleaned customers and cleaned orders.
- `refresh_sales_dashboard` waits for the gold mart.

Do not serialize independent tasks just because they are listed in that order. Lakeflow Jobs can run independent upstream tasks in parallel, which reduces total job duration.

## Dependency examples

Basic dependency:

```json
{
  "task_key": "silver_orders",
  "depends_on": [
    { "task_key": "bronze_orders" }
  ]
}
```

Multiple dependencies:

```json
{
  "task_key": "gold_customer_orders",
  "depends_on": [
    { "task_key": "silver_customers" },
    { "task_key": "silver_orders" }
  ]
}
```

Failure handling task:

```json
{
  "task_key": "send_failure_alert",
  "depends_on": [
    { "task_key": "silver_customers" },
    { "task_key": "silver_orders" }
  ],
  "run_if": "AT_LEAST_ONE_FAILED"
}
```

The exact JSON shape can vary by API/tooling version, but the exam concept is stable: **dependencies define order; Run if defines outcome-based execution**.

## Parameters and dynamic references

Job parameter:

```text
process_date = 2026-05-21
catalog = main
```

Notebook task parameter values:

| Key | Value |
|-----|-------|
| `process_date` | `{{job.parameters.process_date}}` |
| `catalog` | `{{job.parameters.catalog}}` |

Downstream task parameter from upstream task value:

| Key | Value |
|-----|-------|
| `input_row_count` | `{{tasks.ingest_orders.values.row_count}}` |

Use job parameters for values known before the run. Use task values for values discovered during the run.

## End-to-end exam pattern

Scenario: "A pipeline ingests raw orders with PySpark, builds a daily revenue table with SQL, and refreshes a dashboard after the table is updated."

Best Lakeflow Jobs design:

| Order | Task | Type | Dependency | Compute |
|-------|------|------|------------|---------|
| 1 | `ingest_orders` | Notebook | None | Jobs/serverless compute |
| 2 | `build_daily_revenue` | SQL query or SQL file | `ingest_orders` | SQL warehouse |
| 3 | `refresh_sales_dashboard` | Dashboard | `build_daily_revenue` | SQL warehouse |

Why this is exam-safe:

- The job DAG expresses data readiness.
- Notebook handles PySpark ingestion.
- SQL task handles SQL transform.
- Dashboard refresh waits for curated output.
- Each task uses the correct compute type.

## PySpark / API essentials

Notebook task:

```python
from pyspark.sql.functions import col, current_timestamp

process_date = dbutils.widgets.get("process_date")

orders = (
    spark.table("main.bronze.orders_raw")
    .filter(col("_ingest_date") == process_date)
    .withColumn("_silver_loaded_at", current_timestamp())
)

(orders.write
  .mode("append")
  .saveAsTable("main.silver.orders"))

dbutils.jobs.taskValues.set(key="rows_written", value=orders.count())
```

Downstream notebook task can receive the value through a configured parameter:

```python
rows_written = int(dbutils.widgets.get("rows_written"))
```

## SQL (exam)

SQL query task:

```sql
MERGE INTO main.gold.daily_revenue AS target
USING (
  SELECT
    order_date,
    SUM(amount) AS revenue,
    COUNT(*) AS order_count
  FROM main.silver.orders
  WHERE order_date = :process_date
  GROUP BY order_date
) AS source
ON target.order_date = source.order_date
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;
```

## UI / product surfaces

- **Jobs & Pipelines > Jobs**: create/edit jobs and inspect run history.
- **Tasks tab**: add notebook, SQL, dashboard, pipeline, If/else, For each, and other tasks.
- **Task graph**: visual DAG, dependency lines, skipped/excluded tasks, parallel branches.
- **Task configuration**: task key, type, source, parameters, compute, retries, timeouts, notifications.
- **Job details pane**: schedules/triggers, job parameters, permissions, max concurrency, Run as.
- **Run details**: per-task logs, outputs, duration, retry attempts, failed/skipped tasks.

## Exam traps

- Missing dependency between transform and dashboard refresh.
- Duplicate task keys in one job.
- Creating a cycle in the DAG. DAGs cannot loop back to earlier tasks.
- Using a notebook task for a plain SQL file when a SQL task is the simpler and more direct answer.
- Forgetting SQL and dashboard tasks need an appropriate SQL warehouse.
- Confusing a pipeline task with individual notebook tasks inside a pipeline.
- Passing values through ad hoc tables or files when a small task value would be cleaner.
- Making all tasks depend on one root task when several ingestion steps could run in parallel.
- Using a time schedule to imply dependency readiness when explicit DAG dependencies are available inside a job.

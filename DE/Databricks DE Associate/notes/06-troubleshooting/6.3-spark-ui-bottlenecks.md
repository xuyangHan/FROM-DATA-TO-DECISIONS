# 6.3 Spark UI bottlenecks

## Exam objective

> Identify common performance bottlenecks such as data skew, shuffling, and disk spilling by interpreting stage-level metrics in the Spark UI.

## Must know

- **Shuffle**: Exchange shuffle read/write — wide transformations (`groupBy`, join, `repartition`).
- **Skew**: one task much longer than others in same stage.
- **Spill**: memory exhausted → disk spill (slow).
- **Stages**: DAG breakdown; click stage for task timeline.
- **GC time** high → memory pressure.

## When to use what

| Metric | Likely issue |
|--------|----------------|
| Huge shuffle read | Join/agg on bad keys; too many partitions |
| One straggler task | **Skew** — salting, broadcast, filter skewed key |
| Spill (memory/disk) | Increase memory or reduce cache; partition tuning |

## Exam traps

- Adding workers without fixing **skew** (straggler remains).
- **Cache** everything → OOM spill.
- Misreading **job duration** when only one stage is slow.

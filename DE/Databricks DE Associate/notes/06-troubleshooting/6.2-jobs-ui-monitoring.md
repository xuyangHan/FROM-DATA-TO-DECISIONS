# 6.2 Jobs UI pipeline health

## Exam objective

> Use the Lakeflow Jobs UI to monitor pipeline health by interpreting job statuses, viewing DAG-based task graphs to spot upstream blockers, and tracking pipeline run times and failure rates.

## Must know

- Lakeflow Jobs is the orchestration layer for repeatable data workflows.
- A multi-task job is displayed as a **DAG**: a directed acyclic graph of task dependencies.
- The Jobs UI can show a job run through **graph**, **timeline**, and **list** views.
- The graph view is best for dependency failures and upstream blockers.
- The timeline view is best for critical path, task overlap, and "what took longest?"
- The list view is best for sorting/filtering by status, type, duration, dependencies, or task name.
- A downstream task can be **Skipped** because an upstream dependency failed or because its `Run if` condition was not met.
- Databricks determines job success from the status of **leaf tasks**: tasks with no downstream dependencies.
- A job can be **Succeeded with failures** when some non-leaf tasks failed but all leaf tasks succeeded.
- For failed multi-task jobs, a **repair run** can rerun failed/skipped tasks and dependent tasks instead of rerunning everything.

## Statuses to interpret

| Status | Meaning | Exam-safe response |
|--------|---------|--------------------|
| Pending | Waiting to start, often for dependencies or resources | Check upstream tasks, concurrency, or compute acquisition |
| Running | Task/job is currently executing | Monitor duration, logs, metrics, Spark UI/query profile |
| Succeeded | Completed successfully | Use as baseline if input and config are comparable |
| Succeeded with failures | Some tasks failed, but leaf tasks succeeded | Inspect failed non-leaf tasks before calling the run healthy |
| Failed | One or more required/leaf tasks failed | Find the first failed upstream task |
| Skipped | Task did not run | Check dependency status and `Run if` condition |
| Disabled | Task was explicitly disabled or disabled by propagation | Check job settings and DAG impact |
| Timed Out | Task/job exceeded configured timeout | Check whether timeout is too low or task regressed |
| Canceled | Run was stopped by user/platform | Do not treat as performance baseline |

The exact casing in UI/API can vary, but the concepts above are stable for the exam.

## DAG reading pattern

When a job fails, read the DAG from upstream to downstream.

```text
extract_orders -> clean_orders -> build_gold -> refresh_dashboard
```

If `extract_orders` fails:

- `clean_orders` may be skipped.
- `build_gold` may be skipped.
- `refresh_dashboard` may be skipped.
- The first task to investigate is `extract_orders`, not the dashboard.

If `clean_orders` succeeds but `build_gold` fails:

- Start with `build_gold`.
- Do not spend time debugging extract unless the logs point back to bad input.

## Upstream blocker examples

| Observation | Likely blocker | First action |
|-------------|----------------|--------------|
| Many downstream tasks skipped | One upstream dependency failed | Find the earliest failed task in the graph |
| Branch did not run | `Run if` condition not met or If/else condition false | Inspect branch condition and upstream task values |
| For each task has failed iterations | One or more iteration inputs failed | Open For each run history and filter failed iterations |
| Dashboard refresh skipped | Gold table task failed or was skipped | Fix the data-producing task first |
| Task pending for long time | Waiting for dependencies, concurrency, or compute | Check timeline and job/cluster limits |
| Task failed after retries | Real code/data/compute problem, not transient | Open task logs and repair after fixing root cause |

## Pipeline health metrics

For exam scenarios, pipeline health means more than "did it run?"

| Health signal | What it tells you |
|---------------|-------------------|
| Failure rate | Reliability over a window, such as failed runs / total runs |
| Run duration trend | Whether the workflow is meeting SLA over time |
| Task duration trend | Which part of the DAG is changing |
| Skipped task count | Whether dependency failures are blocking downstream work |
| Retry frequency | Whether transient errors are becoming normal |
| Timeout count | Whether runtime growth exceeded configured limits |
| Streaming backlog metrics | Whether streaming tasks are keeping up, if available |

Example:

```text
20 runs this week
4 failed
Failure rate = 4 / 20 = 20%
```

That is a pipeline reliability issue even if the latest run succeeded.

## Graph vs timeline vs list

| View | Best use |
|------|----------|
| Graph | Understand task dependencies, blockers, skipped tasks, branch effects |
| Timeline | Find long-running tasks, overlap, queue/setup gaps, and critical path |
| List | Sort/filter by duration, status, task type, resource, and dependencies |
| Matrix run history | Compare task outcomes across many runs |

Exam shortcut:

- **Graph** answers "what blocked what?"
- **Timeline** answers "where did the time go?"
- **List** answers "which tasks match this status/type/duration?"
- **Run history** answers "is this normal?"

## Repair run behavior

A repair run is used after you identify and fix the cause of a failed multi-task job.

Key points:

- Repair reruns unsuccessful tasks and, when selected, dependent tasks.
- Successful tasks do not need to rerun.
- The repair uses the **current** job and task settings.
- If a job cluster is shared by tasks, repair can create a new job cluster version for the repair attempt.
- The displayed duration for a repaired run can include the original failed attempt plus the repair attempt.

Use repair for recovery. Do not use it as a substitute for finding the root cause.

## Common root causes from Jobs UI

| UI evidence | Likely root cause |
|-------------|-------------------|
| `CLUSTER_ERROR` / compute failed | Startup, capacity, policy, library, or init script issue |
| `LIBRARY_INSTALLATION_ERROR` | Bad package/version, missing repository, incompatible runtime |
| `UNAUTHORIZED_ERROR` | Missing Unity Catalog, workspace, source, or service principal permission |
| `WORKSPACE_RUN_LIMIT_EXCEEDED` | Too many active runs in workspace |
| `CLUSTER_REQUEST_LIMIT_EXCEEDED` | Too many cluster start/upsize requests |
| Task timeout | Runtime regression or timeout too strict |
| Skipped task | Upstream failed or run condition not met |

## UI / product surfaces

- **Jobs & Pipelines > Jobs**: list of jobs and pipelines.
- **Job details > Runs tab**: active and completed runs, matrix/list views.
- **Run details > Graph view**: task DAG with status colors.
- **Run details > Timeline view**: task duration and overlap.
- **Run details > List view**: sortable task table.
- **Task run details**: task metadata, source, logs, output, metrics, compute, query history.
- **Metrics tab**: task-level metrics; for streaming tasks, backlog metrics may be available.
- **Repair run**: rerun failed/skipped tasks after fixing the cause.

## PySpark / API essentials

Monitoring is mostly UI-first, but task values often explain DAG behavior.

```python
# Upstream validation task
bad_rows = invalid_df.count()
dbutils.jobs.taskValues.set(key="bad_rows", value=bad_rows)

if bad_rows > 0:
    raise ValueError(f"Data quality failed: bad_rows={bad_rows}")
```

A downstream If/else or conditional branch can then react to the validation result if configured in the job.

## SQL (exam)

System tables can support reporting across jobs, but UI interpretation is the core objective.

```sql
SELECT
  job_id,
  run_id,
  result_state,
  COUNT(*) AS task_periods
FROM system.lakeflow.job_task_run_timeline
WHERE period_start_time >= current_timestamp() - INTERVAL 7 DAYS
GROUP BY job_id, run_id, result_state;
```

Use this as awareness. If the exam says "Lakeflow Jobs UI," choose the UI surface.

## Exam traps

- Debugging a downstream skipped task instead of the first failed upstream task.
- Treating **Skipped** as success.
- Missing **Succeeded with failures** and assuming every task succeeded.
- Ignoring the timeline when a job is slow but no task failed.
- Rerunning the whole job when repair run would rerun only failed/skipped pieces.
- Forgetting that a repair run uses current task/job settings, not necessarily the original broken settings.
- Confusing pipeline health with only the latest run status; trend and failure rate matter.
- Overlooking concurrency, run limits, and compute acquisition as reasons for pending or skipped runs.

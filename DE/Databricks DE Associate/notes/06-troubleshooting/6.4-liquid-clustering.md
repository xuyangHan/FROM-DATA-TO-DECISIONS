# 6.4 Liquid clustering & predictive optimization

## Exam objective

> Understand the features of Liquid Clustering and predictive optimization.

## Must know

- **Liquid clustering**: `CLUSTER BY (cols)` — adaptive data layout; replaces rigid partitioning for many workloads.
- **OPTIMIZE**: compacts small files; with liquid clustering rewrites layout by cluster keys.
- **Predictive optimization**: Databricks auto-runs `OPTIMIZE` / vacuum-style maintenance based on workload patterns (enable on table).
- vs **partitioning**: static partition columns; can cause small files if high cardinality.
- vs **Z-order**: older multi-dimensional layout; liquid clustering preferred for new tables (exam context).

## When to use what

| Table pattern | Layout |
|---------------|--------|
| Filters on few high-traffic columns | **Liquid clustering** on those cols |
| Time-series daily reports | Cluster by `date` |
| Let platform maintain | **Predictive optimization** ON |

## PySpark / API essentials

```python
# Create table with liquid clustering (SQL often clearer)
spark.sql("""
  CREATE TABLE main.silver.events (...)
  USING DELTA
  CLUSTER BY (event_date, country)
""")

spark.sql("OPTIMIZE main.silver.events")
```

## SQL (exam)

```sql
CREATE TABLE main.silver.events (...)
USING DELTA
CLUSTER BY (event_date, country);

OPTIMIZE main.silver.events;

ALTER TABLE main.silver.events SET TBLPROPERTIES ('delta.enablePredictiveOptimization' = 'true');
```

## UI / product surfaces

- **Catalog** → table → **Optimization** tab / predictive optimization toggle.

## Exam traps

- **Over-partitioning** high-cardinality column (millions of dirs) vs liquid cluster.
- Running **OPTIMIZE** on huge table during peak without maintenance window awareness.
- Confusing **VACUUM** (delete old files) with **OPTIMIZE** (compaction/layout).

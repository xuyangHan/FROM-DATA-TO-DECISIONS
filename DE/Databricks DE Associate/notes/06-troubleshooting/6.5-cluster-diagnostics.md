# 6.5 Cluster failures & OOM

## Exam objective

> Diagnose cluster startup failures, library conflicts, and out-of-memory issues.

## Must know

- **Startup failure**: instance profile, spot capacity, init scripts, policy violation — check **event log**.
- **Library conflicts**: incompatible PyPI/Java jars; mixing cluster-installed vs notebook-scoped libraries.
- **Driver OOM**: `collect()`, `toPandas()`, large broadcast, too many partitions metadata.
- **Executor OOM**: skew, cache, UDF memory; increase executor memory or fix logic.
- **Spark UI** → Storage / Executors tabs for memory usage.

## When to use what

| Symptom | Fix |
|---------|-----|
| Won't start | Event log → policy / quota / init script |
| `NoSuchMethodError` / import error | Align library versions; reinstall on cluster |
| Driver OOM | Remove collect; increase `spark.driver.memory` |
| Executor OOM | Skew fix; reduce cache; `spark.executor.memory` |

## PySpark / API essentials

```python
# Avoid driver OOM
# BAD: df.collect() on billions of rows
# GOOD:
df.groupBy("country").count().show()  # or write to table

# Limit broadcast gone wrong
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)  # disable to test OOM cause
```

## UI / product surfaces

- **Compute** → cluster → **Event log**; **Spark UI** → Executors; driver log in notebook output.

## Exam traps

- Increasing **workers** when **driver** OOM.
- Installing libraries on **running** cluster without restart.
- **Init script** failure looks like mysterious startup error.

## Hands-on checklist

- [ ] Read event log for a failed cluster start (simulate with bad init script in dev).
- [ ] Trigger small driver OOM with `collect()` on large DF — then fix with aggregate.
- [ ] Document library install mode used on your cluster.

# 4.1 Lakeflow Jobs control flow

## Exam objective

> Implement control flows (retries and conditional tasks such as branching and looping) using Lakeflow Jobs for pipeline orchestration.

## Must know

- **Lakeflow Jobs** orchestrate one or more tasks as a **DAG**: a directed acyclic graph where dependencies decide what can run and in what order.
- **Control flow belongs in the job configuration**, not inside scattered notebook `if` statements when the decision affects which task should run next.
- **Dependencies** connect upstream and downstream tasks. By default, downstream tasks run only after all dependencies succeed.
- **Run if** conditions let a downstream task react to dependency outcomes: all succeeded, at least one succeeded, none failed, all done, at least one failed, or all failed.
- **Retries** are configured per task and are for transient failures such as temporary source downtime, cloud API throttling, or brief network errors.
- **Timeouts** prevent stuck or runaway tasks. If both retries and timeout are configured, the timeout applies to each retry attempt.
- **If/else condition tasks** evaluate a boolean expression and route downstream tasks through a true or false branch.
- **For each tasks** run one nested task repeatedly over a JSON array of inputs. This is Lakeflow Jobs' loop pattern.
- **Task values** allow one task to publish small outputs that later tasks can reference with dynamic value references.
- **Job parameters** are run-level inputs. They are useful for dates, environment names, catalog/schema names, or backfill ranges.

## Core control-flow primitives

| Primitive | What it controls | Exam clue |
|-----------|------------------|-----------|
| `Depends on` | Task order in the DAG | "Run silver after bronze" |
| `Run if` | Whether a task should run based on upstream status | "Always send alert if any upstream task fails" |
| Retry policy | Re-attempt failed task runs | "Source API occasionally times out" |
| Timeout | Mark long tasks as timed out | "Prevent job from hanging forever" |
| If/else condition | Branch on a boolean expression | "Only transform if row count is greater than zero" |
| For each | Loop over an input array | "Process the same logic for each table/region/date" |
| Task values | Pass small outputs between tasks | "Use row count from ingest task in a condition" |

## Retry and timeout

Retries should be narrow and intentional. They make sense when re-running the same task might succeed without a code change.

| Scenario | Retry? | Why |
|----------|--------|-----|
| API returns HTTP 429 or temporary 503 | Yes | Transient external-system issue |
| Object storage read fails briefly | Yes | Cloud services can have short blips |
| Cluster startup race or library download hiccup | Maybe | Retry may clear temporary infra issue |
| SQL has wrong column name | No | Code will fail every time |
| Permission denied on source table | No | Retry will not grant permissions |
| Data quality rule rejects bad records | Usually no | Route to quarantine/alert instead |

Common JSON-style fields:

```json
{
  "task_key": "ingest_orders",
  "max_retries": 2,
  "min_retry_interval_millis": 60000,
  "timeout_seconds": 3600
}
```

Exam shortcut: **retry transient failures; fix deterministic failures**.

## Dependency and `Run if` behavior

The default dependency behavior is **All succeeded**. Use `Run if` when the downstream task is a cleanup, notification, validation, or compensating task.

| `Run if` option | Meaning | Typical use |
|-----------------|---------|-------------|
| **All succeeded** | Every dependency succeeded | Normal ETL chain |
| **At least one succeeded** | Continue if any upstream path produced usable output | Merge optional branches |
| **None failed** | No dependency failed and at least one ran | Continue when skipped branches are acceptable |
| **All done** | Dependencies finished regardless of result | Cleanup, final status write |
| **At least one failed** | Run only when something failed | Failure alert, diagnostics |
| **All failed** | Run only when every dependency failed | Fallback path after all alternatives fail |

Important nuance: tasks skipped because a condition was unmet are often shown as **Excluded**. In dependency evaluation, excluded upstream tasks can be treated as successful, so read the question carefully when it asks about failure handling versus branch skipping.

## If/else branching

Use an **If/else condition** task when the workflow should choose one path based on a value.

Typical pattern:

```
ingest_orders
  -> check_new_rows        (If/else: {{tasks.ingest_orders.values.row_count}} > 0)
      -> transform_orders  (depends on check_new_rows true)
      -> no_op_or_notify   (depends on check_new_rows false)
```

The operands in an If/else condition can reference:

- Job parameters, such as `{{job.parameters.process_date}}`
- Task parameters
- Task values, such as `{{tasks.ingest_orders.values.row_count}}`

Comparison behavior to remember:

| Operator type | Behavior |
|---------------|----------|
| `==`, `!=` | String comparison |
| `>`, `>=`, `<`, `<=` | Numeric comparison |
| Task values | Should be numeric, string, or boolean for conditions |

Example notebook task that publishes a value:

```python
from pyspark.sql.functions import col

process_date = dbutils.widgets.get("process_date")

new_orders = (
    spark.table("main.bronze.orders_raw")
    .filter(col("_ingest_date") == process_date)
)

row_count = new_orders.count()
dbutils.jobs.taskValues.set(key="row_count", value=row_count)
```

Then configure the condition task as:

```text
{{tasks.ingest_orders.values.row_count}} > 0
```

## For each looping

Use a **For each** task when the same task logic should run once per item in an input array.

Examples:

- Run the same validation notebook for each source table.
- Backfill the same transform for a list of dates.
- Process the same ingestion task for each region.
- Apply the same SQL file to each tenant/schema.

For each requires:

| Setting | Meaning |
|---------|---------|
| Input array | JSON array supplied directly, from a job parameter, or from an upstream task value |
| Nested task | The task to execute for each item |
| Concurrency | How many iterations can run at the same time; default is 1 |
| `{{input}}` | Reference to the current loop item |
| `{{input.<key>}}` | Reference to a field when each input item is a JSON object |

Direct input array:

```json
["us", "ca", "uk"]
```

Object input array:

```json
[
  {"region": "us", "catalog": "main"},
  {"region": "ca", "catalog": "main"}
]
```

Nested notebook task parameters:

| Parameter key | Parameter value |
|---------------|-----------------|
| `region` | `{{input.region}}` |
| `catalog` | `{{input.catalog}}` |

Upstream generated array:

```python
tables_to_process = [
    {"catalog": "main", "schema": "bronze", "table": "orders_raw"},
    {"catalog": "main", "schema": "bronze", "table": "customers_raw"}
]

dbutils.jobs.taskValues.set(key="table_list", value=tables_to_process)
```

For each input:

```text
{{tasks.discover_tables.values.table_list}}
```

Exam shortcut: **For each is for looping one nested task over an array; If/else is for branching**.

## Task values vs parameters

| Feature | Set by | Read by | Best use |
|---------|--------|---------|----------|
| Job parameter | Job definition or run submitter | Any compatible task | Run date, environment, catalog, backfill flag |
| Task parameter | Individual task config | The task code | Notebook widget, SQL parameter, script arg |
| Task value | Running task code | Later tasks through dynamic refs | Row count, generated list, validation status |

Notebook task reads a parameter:

```python
process_date = dbutils.widgets.get("process_date")
```

Task publishes a value:

```python
dbutils.jobs.taskValues.set(key="status", value="ok")
```

Downstream task references it in job configuration:

```text
{{tasks.validate_orders.values.status}}
```

Do not confuse `dbutils.jobs.taskValues.get()` in notebook code with dynamic value references in the Jobs UI. On the exam, the clean orchestration answer is usually: set a task value upstream, then reference it in downstream task parameters or conditions.

## PySpark / API essentials

Control flow is configured in the Jobs UI, Jobs API, or Declarative Automation Bundles. The task code usually only reads parameters and writes task values.

```python
from pyspark.sql.functions import col

process_date = dbutils.widgets.get("process_date")

df = spark.table("main.bronze.orders_raw").filter(col("_ingest_date") == process_date)
count = df.count()

dbutils.jobs.taskValues.set(key="row_count", value=count)

if count > 0:
    (df.write
      .mode("append")
      .saveAsTable("main.silver.orders"))
```

Example task dependency snippet:

```json
{
  "task_key": "transform_orders",
  "depends_on": [
    {
      "task_key": "check_new_rows",
      "outcome": "true"
    }
  ],
  "notebook_task": {
    "notebook_path": "/Repos/team/de/transform_orders",
    "base_parameters": {
      "process_date": "{{job.parameters.process_date}}"
    }
  }
}
```

## UI / product surfaces

- **Jobs & Pipelines > Jobs > Tasks**: DAG graph, task keys, task configuration, dependencies, If/else, For each.
- **Task configuration panel**: retries, timeout, metric thresholds, parameters, notifications.
- **Depends on / Run if dependencies**: dependency status logic for downstream tasks.
- **Run details**: inspect branch decisions, skipped/excluded tasks, retry attempts, and task duration.
- **Declarative Automation Bundles**: version job definitions, including tasks, dependencies, parameters, and triggers.

## Exam traps

- Retrying a task that has deterministic code or permission errors.
- Building all branching inside a notebook when the question asks for Lakeflow Jobs orchestration.
- Confusing **If/else** with **Run if**. If/else branches on values; Run if reacts to upstream task outcomes.
- Using For each for dependent steps that must run in a custom sequence. For each is one nested task repeated over an array.
- Forgetting that the DAG cannot contain cycles.
- Branching on a task value that was never set because the upstream task failed or was disabled.
- Using string equality for numeric-looking values and expecting numeric equality semantics, such as `"12.0" == "12"`.
- Setting high concurrency in For each without considering source-system load, SQL warehouse capacity, or cluster limits.

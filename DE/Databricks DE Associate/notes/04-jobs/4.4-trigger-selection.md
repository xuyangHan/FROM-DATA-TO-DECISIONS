# 4.4 Time-based vs data-driven triggers

## Exam objective

> Choose between time-based and data-driven triggers based on data availability and pipeline dependencies.

## Must know

- **Time-based triggers** run because the clock says it is time.
- **Data-driven triggers** run because a data-readiness event happened.
- In Lakeflow Jobs, the main time-based trigger is **Scheduled**.
- In Lakeflow Jobs, the main data-driven triggers are **File arrival** and **Table update**.
- Choose based on the signal that best proves the input is ready.
- Data-driven triggers reduce empty runs when data arrives irregularly.
- Scheduled triggers are better when the business SLA is tied to a fixed time, even if the job sometimes no-ops.
- Dependencies inside a multi-task job are often better than chaining many jobs with separate schedules.

## Fast decision table

| Situation | Best trigger | Why |
|-----------|--------------|-----|
| Report must run every weekday at 7 AM | Scheduled | Wall-clock SLA |
| Vendor drops files unpredictably | File arrival | Raw data readiness is new files |
| Bronze table update should start silver | Table update | Readiness is a table commit |
| Silver table update should start gold | Table update | Downstream depends on curated table changes |
| Files arrive in batches over several minutes | File arrival with wait after last change | Avoid processing partial batch |
| Multiple upstream tables must all be updated | Table update configured for all monitored tables | Avoid partial downstream processing |
| Source has no event signal and only supports daily extract | Scheduled | Clock is the available signal |
| External orchestrator owns scheduling | None/manual plus API | Databricks job is externally triggered |
| Streaming job should run continuously | Continuous or Lakeflow pipeline mode | Not a batch schedule problem |

## Selection algorithm

Use this sequence in exam questions:

1. Does the requirement name a fixed time, interval, business cutoff, or SLA?
   Choose **Scheduled** unless the question says the job should run only when data arrives.

2. Is the source signal a file appearing in cloud storage, a UC volume, or an external location?
   Choose **File arrival**.

3. Is the source signal a Delta/Iceberg/Unity Catalog table being updated?
   Choose **Table update**.

4. Are multiple tasks part of one logical workflow?
   Use one job with a DAG and dependencies, then choose one trigger for the root job.

5. Are independently owned pipelines communicating through tables?
   Use **Table update** triggers between jobs.

6. Is there no reliable data-readiness signal?
   Use **Scheduled** with checks, retries, alerts, and idempotent/no-op behavior.

## Time-based triggers

Scheduled triggers are simple and predictable.

Good when:

- Data is guaranteed to be ready before a known time.
- Users expect outputs by a fixed time.
- Source systems publish on a reliable timetable.
- Regulatory or operational processes require wall-clock execution.
- External systems cannot emit file or table events.

Costs and risks:

- Can start before data is ready if upstream is late.
- Can run with no new data.
- Can waste jobs compute or SQL warehouse time.
- Can create stale outputs if downstream tasks run on yesterday's data.
- Needs careful time zone and daylight saving time handling.

Exam phrase mapping:

| Phrase | Likely answer |
|--------|---------------|
| "Every night at 2 AM" | Scheduled |
| "Every 15 minutes" | Scheduled |
| "By 8 AM each business day" | Scheduled, with monitoring and buffer |
| "After market close at 5 PM Eastern" | Scheduled with explicit time zone |

## Data-driven triggers

Data-driven triggers are better when data availability is irregular or upstream duration varies.

Good when:

- Data arrives 0 to many times per day.
- The upstream completion time is unpredictable.
- Empty runs are expensive or noisy.
- Downstream should process only after a specific data asset changes.
- You want lower latency after data arrival without running continuously.

Costs and risks:

- Requires a supported readiness signal.
- File arrival needs Unity Catalog volume/external location configuration.
- Table update triggers require supported table/view types.
- Event/checking behavior can be best effort, so monitoring is still needed.
- Poorly designed triggers can start too often if upstream writes many small commits/files.

Exam phrase mapping:

| Phrase | Likely answer |
|--------|---------------|
| "When a new file lands" | File arrival |
| "Files arrive unpredictably" | File arrival |
| "Only after the bronze table updates" | Table update |
| "Run when all source tables are refreshed" | Table update with all monitored tables |
| "Upstream job finishes at variable times" | Table update if it writes a table as readiness marker |

## File arrival vs table update

| Question | File arrival | Table update |
|----------|--------------|--------------|
| What is monitored? | Storage path | Table(s) or supported views |
| Readiness signal | New file exists | Table commit/update exists |
| Best source layer | Raw landing/bronze file ingestion | Downstream table-to-table pipelines |
| Common pairing | Auto Loader | Silver/gold Lakeflow Jobs |
| Governance requirement | UC volume or external location | Supported UC table/view |
| Common pitfall | Existing file overwrite is not a clean event | Unsupported source table/view dependencies |

Choose file arrival for **raw files**. Choose table update for **table dependencies**.

## Multi-task job vs multiple triggered jobs

Use one multi-task job when tasks are tightly coupled and owned together:

```
scheduled trigger at 2 AM
  -> ingest_orders
  -> clean_orders
  -> build_daily_revenue
  -> refresh_dashboard
```

Benefits:

- One run history for the whole workflow.
- Clear DAG dependencies.
- Easier repair of failed/skipped tasks.
- Fewer trigger interactions to reason about.

Use multiple jobs with data-driven triggers when pipelines are independent or table ownership is the contract:

```
source ingestion job
  -> updates main.bronze.orders_raw

silver job
  -> table update trigger on main.bronze.orders_raw
  -> updates main.silver.orders

gold job
  -> table update trigger on main.silver.orders
  -> updates main.gold.daily_revenue
```

Benefits:

- Teams can own separate jobs.
- Downstream jobs do not need to know the upstream schedule.
- Late upstream completion does not cause stale downstream runs.

## Handling late or missing data

| Problem | Better design |
|---------|---------------|
| Upstream often finishes late | Use table update trigger instead of fixed downstream cron |
| Files arrive in a batch over 10 minutes | File arrival trigger with wait after last change |
| Some days have no data | Data-driven trigger or scheduled job with explicit no-op logic |
| Dashboard must publish by 8 AM even if no new data | Scheduled job with data freshness check and alert |
| Multiple upstream tables arrive separately | Table update trigger that waits for all required tables |
| External source sometimes misses SLA | Scheduled job plus validation and notification |

Exam shortcut: **data-driven prevents stale/empty runs; scheduled protects wall-clock SLAs**.

## Cost and reliability tradeoffs

| Dimension | Scheduled | Data-driven |
|-----------|-----------|-------------|
| Empty runs | More likely | Less likely |
| Latency after data arrives | Depends on schedule interval | Usually lower |
| Predictability for users | High | Depends on arrival event |
| Setup complexity | Lower | Higher |
| Source support required | Minimal | Requires file/table readiness signal |
| Risk when upstream is late | Higher | Lower |
| Cost for irregular data | Potentially higher | Usually lower |

## Design patterns

### Nightly SLA pattern

Use when business expects a fixed output time.

```
Scheduled 2:00 AM
  -> ingest
  -> transform
  -> validate freshness
  -> refresh dashboard
  -> notify if freshness check fails
```

Why not purely data-driven? Because the requirement is a fixed delivery time. You can still add validation to catch missing or stale data.

### File-drop ingestion pattern

Use when external partners drop files irregularly.

```
File arrival on /Volumes/main/landing/orders/
  -> Auto Loader availableNow ingestion
  -> write main.bronze.orders_raw
```

Why not scheduled? Because the file might arrive 0, 1, or many times per day.

### Table-driven medallion pattern

Use when each layer should run only after the previous layer updates.

```
main.bronze.orders_raw update
  -> silver job trigger
  -> main.silver.orders update
  -> gold job trigger
```

Why not chained cron? Because upstream lateness can create stale downstream results.

### Batch-complete pattern

Use when many files or table commits arrive close together.

```
File arrival/table update trigger
  -> wait after last change
  -> run downstream job once for the full batch
```

Why? It avoids processing partial batches and reduces repeated cluster starts.

## UI / product surfaces

- **Schedules & Triggers**: compare Scheduled, File arrival, Table update, Continuous, and None/manual.
- **Advanced schedule**: choose time zone and optionally inspect Quartz cron syntax.
- **File arrival settings**: choose UC volume/external location path and wait after last change.
- **Table update settings**: choose monitored table(s), one/all update behavior, and wait after last change.
- **Run history**: confirm whether runs are useful or empty, and whether data-driven triggers reduce wasted runs.
- **Task graph**: decide whether dependencies belong inside one DAG instead of across multiple scheduled jobs.

## Exam traps

- Choosing Scheduled only because "daily" appears, even though the job should run only after a table is updated.
- Choosing File arrival for downstream silver/gold jobs when the readiness signal is a Delta table update.
- Choosing Table update for a raw landing folder before data has been loaded into a table.
- Ignoring the need for a fixed dashboard/report SLA. Data-driven triggers may be too late if data arrives late.
- Chaining bronze/silver/gold with separate cron schedules and creating stale reads when upstream is delayed.
- Forgetting to use wait-after-last-change for batches that arrive as many files or commits.
- Assuming data-driven triggers remove the need for idempotent processing, monitoring, and alerts.
- Using a data-driven trigger on an unsupported path, table, or view.

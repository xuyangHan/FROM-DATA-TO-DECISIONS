# 4.4 Time-based vs data-driven triggers

## Exam objective

> Choose between time-based and data-driven triggers based on data availability and pipeline dependencies.

## Must know

- **Time-based (scheduled)**: fixed latency SLA; runs even if no new data (may no-op).
- **Data-driven (file arrival, table update)**: event-aligned; reduces empty runs and cascading delays.
- **Dependency chain**: bronze table update → silver job → gold job (table update triggers).
- Cost: fewer wasted cluster starts with data-driven when arrivals are irregular.

## When to use what

| Situation | Trigger |
|-----------|---------|
| Regulatory batch at 2 AM daily | **Scheduled cron** |
| Files arrive unpredictably 0–N times/day | **File arrival** |
| Downstream only after upstream Delta commit | **Table update** |
| Strict wall-clock report by 8 AM | **Scheduled** (may need buffer) + monitoring |
| Upstream SaaS sync finishes at variable time | **Table update** on landing table |

## Exam traps

- **Scheduled** silver job when bronze sometimes has no data — wastes $ (data-driven better).
- **Table update** trigger on non-Delta or external table without update detection support.
- Chaining only with cron causing **stale data** when upstream is late (overlap window issues).

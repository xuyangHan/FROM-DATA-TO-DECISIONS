# 7.1 Managed vs external tables

## Exam objective

> Differentiate between managed and external tables in Unity Catalog and perform basic operations (create, modify, delete, and convert between managed and external tables) on them.

## Must know

- Unity Catalog governs both managed and external tables, but it only controls the file lifecycle for **managed** tables.
- A **managed table** is the default and recommended table type for most Databricks workloads.
- An **external table** registers data at a cloud storage path that you specify with `LOCATION`.
- Managed tables store data under the managed storage location configured at the schema, catalog, or metastore level.
- External tables store data in an **external location** registered in Unity Catalog.
- `DROP TABLE` on a managed table removes table metadata and schedules/removes the underlying data files according to UC lifecycle behavior.
- `DROP TABLE` on an external table removes table metadata only. The files in cloud storage remain.
- Both managed and external tables can be queried by name with three-level namespace: `catalog.schema.table`.
- For structured analytics data, prefer tables. For raw files, images, documents, logs, and other unstructured/semi-structured files, use Unity Catalog **volumes**.
- Unity Catalog also has **foreign tables**, but this objective focuses on managed and external tables.

## Managed vs external

| Feature | Managed table | External table |
|---------|---------------|----------------|
| Storage location | Chosen by Unity Catalog | Explicit `LOCATION` path |
| File lifecycle | Managed by Unity Catalog | Managed by you or external system |
| Default choice | Yes | No |
| `DROP TABLE` data behavior | Data files are deleted by UC lifecycle | Data files remain |
| Formats | Delta Lake, Apache Iceberg | Delta recommended; also CSV, JSON, Avro, Parquet, ORC, Text |
| Best fit | Production Delta/Iceberg tables | Existing data, legacy integrations, external path access |
| Requires external location | No, unless custom managed storage uses one | Yes |

Exam shortcut: **no `LOCATION` usually means managed; explicit `LOCATION` means external**.

## When to use what

| Need | Best choice |
|------|-------------|
| New bronze/silver/gold Delta table in UC | Managed table |
| Production analytics table with optimization/maintenance benefits | Managed table |
| Register existing files already stored in S3/ADLS/GCS | External table |
| Keep files after dropping table metadata | External table |
| Allow non-Databricks systems to keep managing the files | External table |
| Store raw files that are not tables | Managed or external volume |

## Required concepts

### Managed storage location

Managed tables live under the managed storage location. This can be configured at:

1. Metastore level.
2. Catalog level.
3. Schema level.

The most specific configured location wins. You do not provide a table `LOCATION` when creating a normal managed table.

### External location

An external table path must be covered by a Unity Catalog external location. The external location connects a cloud path to a storage credential and lets UC authorize access.

To create an external table, the principal typically needs:

- `USE CATALOG` on the parent catalog.
- `USE SCHEMA` on the parent schema.
- `CREATE TABLE` on the parent schema.
- `CREATE EXTERNAL TABLE` on the external location that covers the path.

## Create tables

### Managed table

```sql
CREATE TABLE main.silver.orders (
  order_id STRING,
  customer_id STRING,
  amount DOUBLE,
  order_ts TIMESTAMP
)
USING DELTA;
```

```python
(
    df.write
      .format("delta")
      .mode("overwrite")
      .saveAsTable("main.silver.orders")
)
```

Because no `LOCATION` is specified, Unity Catalog creates a managed table.

### External table

```sql
CREATE TABLE main.bronze.orders_external (
  order_id STRING,
  customer_id STRING,
  amount DOUBLE,
  order_ts TIMESTAMP
)
USING DELTA
LOCATION 's3://company-raw/bronze/orders/';
```

Cloud-specific paths might look like:

```sql
-- AWS
LOCATION 's3://bucket/path/';

-- Azure
LOCATION 'abfss://container@account.dfs.core.windows.net/path/';

-- GCP
LOCATION 'gs://bucket/path/';
```

## Modify tables

Common table modifications use `ALTER TABLE`.

```sql
ALTER TABLE main.silver.orders
ADD COLUMNS (ingest_file STRING);

ALTER TABLE main.silver.orders
ALTER COLUMN amount COMMENT 'Order amount in USD';

ALTER TABLE main.silver.orders
SET TBLPROPERTIES ('quality' = 'silver');

ALTER TABLE main.silver.orders
RENAME TO main.silver.orders_v2;
```

Important permission shortcut:

- `MODIFY` is needed for many data and schema changes such as adding/dropping columns and setting table properties.
- Ownership or `MANAGE` is needed for administrative operations such as changing owner or dropping the table.

## Delete tables

```sql
DROP TABLE main.silver.orders;
```

For a managed table, dropping the table removes the Unity Catalog table entry and the managed data files according to UC lifecycle behavior.

```sql
DROP TABLE main.bronze.orders_external;
```

For an external table, dropping the table removes only the table metadata. The cloud files still exist and must be deleted separately if you truly want to remove the data.

Exam shortcut: **external table drop is not a data deletion strategy**.

## Inspect table type and location

```sql
DESCRIBE EXTENDED main.silver.orders;
DESCRIBE DETAIL main.silver.orders;
```

Look for:

- `Type`: `MANAGED` or `EXTERNAL`.
- `Location`: physical storage path.
- `Provider`: usually `delta`.

## Convert tables

### External to managed

Databricks supports converting an external Delta table to a Unity Catalog managed table with `SET MANAGED`.

```sql
ALTER TABLE main.bronze.orders_external SET MANAGED;
```

Key points:

- Requires Delta format.
- Requires Databricks Runtime 17.0+ or serverless compute.
- Preserves table name, permissions, history, settings, and dependent views better than a CTAS migration.
- After conversion, the table type should show `MANAGED` in `DESCRIBE EXTENDED`.
- Streams and path-based jobs may need restart/review.

For UniForm/Iceberg-read enabled tables, current docs include:

```sql
ALTER TABLE main.bronze.orders_external
SET MANAGED TRUNCATE UNIFORM HISTORY;
```

### Roll back to external after conversion

After an external table is converted to managed, Databricks supports rollback for a limited time:

```sql
ALTER TABLE main.bronze.orders_external UNSET MANAGED;
```

Key points:

- Requires Databricks Runtime 17.0+ or serverless compute.
- Rollback is supported within the documented rollback window after conversion.
- Verify with `DESCRIBE EXTENDED`; type should show `EXTERNAL`.

### Migration pattern when direct conversion is not appropriate

If a direct in-place conversion is not available or not appropriate, use a migration pattern:

```sql
CREATE TABLE main.silver.orders_managed
AS SELECT * FROM main.bronze.orders_external;

CREATE TABLE main.bronze.orders_external_copy
USING DELTA
LOCATION 's3://company-curated/orders_external_copy/'
AS SELECT * FROM main.silver.orders_managed;
```

CTAS is simple, but it creates a new table and does not preserve all history, permissions, and dependencies automatically. Use it carefully and validate before swapping names.

## PySpark / API essentials

```python
# Read either table type by name. UC handles governance.
orders = spark.table("main.silver.orders")

# Managed write by table name.
orders.write.mode("overwrite").format("delta").saveAsTable("main.gold.orders_daily")

# External-style path writes should be intentional and governed by an external location.
orders.write.format("delta").mode("overwrite").save("s3://company-curated/orders_daily/")
```

Exam shortcut: prefer `saveAsTable("catalog.schema.table")` for UC-governed tables unless the question explicitly asks for path-based storage.

## Exam traps

- Assuming an external table's files are deleted by `DROP TABLE`.
- Assuming a managed table requires a `LOCATION`.
- Creating external tables at arbitrary paths that are not covered by UC external locations.
- Confusing **managed by Unity Catalog for governance** with **managed table storage lifecycle**. External tables are still governed by UC, but UC does not manage their file lifecycle.
- Using external tables for new production Delta tables when managed tables would be simpler and better optimized.
- Forgetting that direct path access to external table files from outside Databricks is not protected by UC table privileges.
- Thinking CTAS conversion preserves all table history, permissions, and dependent objects automatically.
- Dropping a managed table when the real intention is only to hide or unregister it.

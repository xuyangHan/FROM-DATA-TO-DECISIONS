# 7.3 Column masking & row filters

## Exam objective

> Understand column-level masking and row-level security to restrict data visibility based on user groups.

## Must know

- A **column mask** changes what values users see in a sensitive column.
- A **row filter** controls which rows users can see.
- Both are Unity Catalog fine-grained access controls enforced at query time.
- Consumers query the same table name; UC applies the mask/filter automatically.
- Both features use SQL UDFs registered in Unity Catalog.
- Column mask UDFs must return the same type, or a castable type, as the masked column.
- Row filter UDFs must return `BOOLEAN`.
- Use group membership functions such as `is_account_group_member()` to vary behavior by group.
- Table-level masks/filters are good for a small number of tables. Use ABAC policies for centralized tag-based control across many tables.

## Column masks

A column mask replaces references to a column with the output of a masking function.

Example use cases:

- Contractors can query `customers`, but email is redacted.
- Analysts can see last four digits of SSN but not full SSN.
- HR group can see salary; everyone else sees `NULL`.

Important rules:

- Each column can have one mask.
- The mask is applied as soon as rows are fetched.
- Predicates, joins, and ordering use the masked value, not the original value.
- You must drop the mask before dropping the mask function.

## Row filters

A row filter applies a UDF predicate whenever rows are fetched from the table.

Example use cases:

- EU analysts only see rows where `region = 'EU'`.
- Sales reps see only their assigned accounts.
- Contractors see only non-confidential records.

Important rules:

- Each table can have one row filter.
- The row filter UDF returns `TRUE` to keep a row.
- If the filter returns `FALSE` or `NULL`, the row is filtered out.
- You must drop the row filter before dropping the filter function.

## When to use what

| Requirement | Mechanism |
|-------------|-----------|
| Hide or transform sensitive column values | Column mask |
| Restrict rows by region, department, tenant, or account | Row filter |
| Restrict many tables based on governed tags | ABAC policy |
| Create a separate read-only object with custom SQL logic | Dynamic view |
| Block all access to a table | Do not grant `SELECT`, or revoke it |

## Required privileges

To assign a row filter or column mask:

- `EXECUTE` on the UDF.
- `USE SCHEMA` on the function's schema.
- `USE CATALOG` on the function's catalog.
- For a new table: `CREATE TABLE` on the target schema.
- For an existing table: ownership or `MANAGE` on the table.

To query the protected table, users still need base table access:

- `USE CATALOG`.
- `USE SCHEMA`.
- `SELECT`.

Masks and filters **restrict** visibility; they do not grant table access.

## SQL: column mask

```sql
CREATE SCHEMA IF NOT EXISTS main.security;

CREATE OR REPLACE FUNCTION main.security.mask_email(email STRING)
RETURN
  CASE
    WHEN is_account_group_member('pii_readers') THEN email
    ELSE '***'
  END;

ALTER TABLE main.gold.customers
ALTER COLUMN email
SET MASK main.security.mask_email;
```

Consumers query the table normally:

```sql
SELECT customer_id, email
FROM main.gold.customers;
```

Members of `pii_readers` see real email values. Other users see masked values.

Remove the mask:

```sql
ALTER TABLE main.gold.customers
ALTER COLUMN email
DROP MASK;
```

Then, if needed:

```sql
DROP FUNCTION main.security.mask_email;
```

## SQL: row filter

```sql
CREATE OR REPLACE FUNCTION main.security.region_filter(region STRING)
RETURN
  CASE
    WHEN is_account_group_member('admins') THEN true
    WHEN is_account_group_member('eu_analysts') THEN region = 'EU'
    WHEN is_account_group_member('na_analysts') THEN region = 'NA'
    ELSE false
  END;

ALTER TABLE main.gold.orders
SET ROW FILTER main.security.region_filter ON (region);
```

Consumers query normally:

```sql
SELECT order_id, region, amount
FROM main.gold.orders;
```

The returned rows depend on the querying user's group membership.

Remove the row filter:

```sql
ALTER TABLE main.gold.orders
DROP ROW FILTER;
```

Then, if needed:

```sql
DROP FUNCTION main.security.region_filter;
```

## SQL: create table with policies

```sql
CREATE TABLE main.gold.employees (
  employee_id STRING,
  region STRING,
  email STRING MASK main.security.mask_email
)
USING DELTA
ROW FILTER main.security.region_filter ON (region);
```

## PySpark / API essentials

Readers do not add special logic in PySpark. They read the table by name and UC applies the policy.

```python
customers = spark.table("main.gold.customers")
orders = spark.table("main.gold.orders")
```

Do not implement masking only in notebook code or BI semantic layers when the requirement is governed access control. That only protects one access path.

## Dynamic views vs row filters/masks

| Feature | Dynamic view | Row filter / column mask |
|---------|--------------|--------------------------|
| Object queried | A separate view name | Same table name |
| Managed with | SQL view definition | UDF plus table policy |
| Best for | Read-only curated views, joins, custom projections | Direct table protection |
| Scale pattern | Many views can become hard to manage | Use ABAC for many tables |

Dynamic views are still useful, but native UC row filters and column masks are the direct table-level controls.

## Performance and design notes

- Prefer simple SQL UDFs over Python UDFs for policy logic.
- Keep function arguments minimal.
- Avoid complex mapping-table lookups unless needed.
- Test realistic queries after applying masks/filters.
- Match UDF parameter types to table column types.
- Enable ANSI behavior where possible so bad casts fail visibly instead of silently turning into `NULL`.
- Avoid policy functions that can throw errors in ways that might leak information.

## Limitations to remember

- Requires Unity Catalog.
- Applies to Databricks SQL and supported Databricks Runtime versions.
- Row filters and column masks are not applied to views directly.
- Path-based access to files in tables with policies is not supported.
- Time travel and clone operations have limitations with row filters/masks.
- `REPLACE TABLE` retains existing row filters and column masks where possible; if referenced columns disappear or change, queries can fail until policies are updated.

## UI / product surfaces

- **Catalog Explorer > table > Overview > Row filter**: add or remove row filter.
- **Catalog Explorer > table > Columns**: add or remove column mask.
- **Catalog Explorer > Functions**: inspect policy UDFs.
- **SQL editor / notebook**: `CREATE FUNCTION`, `ALTER TABLE ... SET ROW FILTER`, `ALTER COLUMN ... SET MASK`.

## Exam traps

- Thinking masks/filters grant access. They only restrict after `SELECT` access exists.
- Masking in a notebook or BI dashboard instead of Unity Catalog.
- Forgetting parent `USE CATALOG` and `USE SCHEMA`.
- Returning the wrong data type from a mask UDF.
- Returning non-boolean values from a row filter UDF.
- Dropping a policy UDF before removing the mask/filter from the table.
- Applying table-by-table masks to hundreds of tables when the scenario asks for centralized control. That is ABAC.
- Forgetting privileged exception groups such as `admins` or `pii_readers`.

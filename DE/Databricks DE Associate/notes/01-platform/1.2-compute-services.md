# 1.2 Compute services & cost

## Exam objective

> Understand Databricks Data Intelligence Platform's compute services, including their characteristics, limitations, and cost models, and select the most suitable option for each workload use case.

## Must know

- **All-purpose cluster**: interactive notebooks/dev; shared or single-user; you start/stop; higher $ if left running.
- **Job cluster**: ephemeral cluster per job run; best for scheduled production workloads; terminates after task.
- **Serverless compute**: Databricks-managed scaling for supported notebook/SQL workloads; less cluster ops.
- **SQL warehouse**: BI/SQL analytics; Photon; concurrency slots; not for arbitrary PySpark ETL.
- **Cost drivers**: DBUs × runtime × instance family; autoscaling adds workers; storage billed separately (cloud + UC).
- **Autoscaling**: min/max workers; scale down when idle (all-purpose) saves cost.
- **Photon**: accelerated SQL/DF on supported runtimes.

## When to use what

| Workload | Choose |
|----------|--------|
| Scheduled ETL / Lakeflow Job | **Job cluster** (or pipeline-managed compute) |
| Ad hoc notebook exploration | **All-purpose** or **serverless** |
| Dashboards / BI SQL | **SQL warehouse** |
| Production pipeline (LDP) | **Pipeline** cluster policy (not shared all-purpose) |
| Low-latency SQL only | **SQL warehouse** (not generic cluster) |
| Long-running dev without ops | **Serverless** where supported |

## Exam traps

- **All-purpose cluster** attached to nightly production job (wastes $, wrong isolation).
- **SQL warehouse** for PySpark notebook ETL.
- Forgetting **auto-termination** on dev all-purpose clusters.
- Assuming **more workers always faster** without shuffle/skew diagnosis (see 6.3).

# 1.2 Compute services & cost

## Exam objective

> Understand Databricks Data Intelligence Platform's compute services, including their characteristics, limitations, and cost models, and select the most suitable option for each workload use case.

## Must know

- **Compute** is the processing layer for notebooks, jobs, pipelines, SQL queries, dashboards, and BI tools.
- Databricks groups compute into **serverless compute**, **classic compute**, and **SQL warehouses**.
- **Serverless compute**: Databricks-managed, on-demand compute with automatic provisioning and scaling. Used for supported notebooks, Lakeflow Jobs, and Lakeflow Spark Declarative Pipelines.
- **Classic compute**: customer-configured compute deployed in the customer's cloud account. Includes all-purpose compute, jobs compute, and classic pipeline compute.
- **All-purpose compute**: interactive cluster for exploration and development. Users start, share, attach notebooks, and terminate it.
- **Jobs compute**: compute created for automated job runs. It is preferred over all-purpose compute for production Lakeflow Jobs because it is isolated and can terminate after the run.
- **SQL warehouse**: compute optimized for SQL, BI, dashboards, dbt SQL tasks, and SQL analytics. It is not the right choice for arbitrary PySpark DataFrame ETL.
- **Serverless SQL warehouse**: recommended SQL warehouse type for most SQL/BI workloads because it starts quickly and scales rapidly.
- **Photon**: Databricks-native vectorized query engine. It accelerates SQL, DataFrame API calls, ETL, and supported streaming workloads. Photon is always enabled on serverless compute, SQL warehouses, and serverless pipelines.
- **Access mode** controls who can use classic compute and how data access is isolated. Common exam terms: standard/shared-style access for multiple users and dedicated/single-user-style access for one user or principal.
- **Cost drivers**: DBUs, runtime duration, node type or warehouse size, worker count, autoscaling range, Photon/serverless SKU, cloud infrastructure, storage, and network transfer.
- **Auto-termination** stops idle all-purpose compute and is one of the easiest cost controls.

## Compute mental model

```
Interactive development
  -> serverless notebook compute
  -> classic all-purpose compute when custom settings, R, or RDD APIs are needed

Production orchestration
  -> serverless jobs for supported tasks
  -> classic jobs compute when serverless does not support the workload

SQL and BI
  -> SQL warehouse
  -> serverless SQL warehouse when available

Declarative pipelines
  -> serverless Lakeflow Spark Declarative Pipelines
  -> classic pipeline compute when needed for unsupported features
```

## Compute types

| Compute type | Best for | Key characteristics | Watch out |
|--------------|----------|---------------------|-----------|
| Serverless notebook compute | Interactive Python/SQL notebooks | Fast start, autoscaling, Databricks-managed | Some Spark APIs and custom cluster features are unsupported |
| Classic all-purpose compute | Development, debugging, custom cluster needs | User-configurable cluster; can attach notebooks; Spark UI available | Expensive if left running; generally avoid for scheduled production jobs |
| Serverless jobs compute | Most automated notebook/Python/wheel jobs | No cluster config; Databricks provisions and scales | Not all task types and streaming triggers are supported |
| Classic jobs compute | Jobs needing custom settings, JAR, Spark submit, unsupported serverless features | Configured in job definition; can be shared by tasks in a job | More infrastructure choices to manage |
| SQL warehouse | SQL analytics, BI, dashboards, SQL tasks, dbt | Optimized for SQL; supports serverless, pro, and classic types | Not for arbitrary PySpark cluster workloads |
| Pipeline compute | Lakeflow Spark Declarative Pipelines | Serverless or classic pipeline compute | Separate from general all-purpose clusters |

## When to use what

| Scenario | Choose |
|----------|--------|
| Ad hoc notebook exploration with Python or SQL | Serverless notebook compute |
| Notebook needs R, RDD APIs, custom init scripts, or unsupported serverless features | Classic all-purpose compute |
| Scheduled production notebook/Python/wheel task | Serverless jobs compute when supported |
| Job task uses JAR or Spark submit | Classic jobs compute |
| Multi-task production job with non-SQL tasks and custom cluster settings | Classic jobs compute, shared across tasks if useful |
| SQL query task in Lakeflow Jobs | SQL warehouse, preferably serverless SQL warehouse |
| BI dashboards and external BI tools | SQL warehouse |
| dbt SQL models | SQL warehouse |
| Lakeflow Spark Declarative Pipeline | Serverless pipeline compute when supported |
| Continuous or unsupported streaming pattern on serverless jobs | Classic jobs compute or pipeline mode that supports it |
| Cost-sensitive dev cluster | Small all-purpose compute with auto-termination and autoscaling |

## SQL warehouse types

| Warehouse type | Use when |
|----------------|----------|
| Serverless | Default recommendation for BI, SQL analytics, exploratory SQL, and variable query demand |
| Pro | Serverless is not available or you need features/custom networking not supported by serverless |
| Classic | Basic SQL exploration when serverless or pro is not an option |

SQL warehouses scale by **warehouse size** and, depending on configuration, may scale clusters for concurrency. Larger warehouses are not automatically cheaper or faster overall; they help when the workload is CPU/IO bound and can use the extra capacity.

## Serverless limitations to recognize

Serverless is often the recommended default, but exam questions may test when **not** to choose it.

| Limitation area | What to remember |
|-----------------|------------------|
| Languages | Serverless notebooks support Python and SQL; R is not supported |
| Spark APIs | Spark Connect APIs are supported; RDD APIs are not supported |
| Customization | Most Spark configs, compute-scoped init scripts, compute-scoped libraries, instance pools, and custom containers are not supported |
| Storage access | Use Unity Catalog external locations and volumes; DBFS access is limited |
| Streaming | `Trigger.AvailableNow()` is recommended; continuous and processing-time triggers are not supported for serverless notebook/jobs streaming |
| Debugging | Spark UI and Spark logs are not available in the same way as classic compute; use query profile/client logs |
| Libraries | Notebook-scoped Python libraries are the usual pattern; Maven coordinates and notebook JAR libraries are limited |

## Cost model

| Cost factor | How it affects cost |
|-------------|---------------------|
| DBUs | Databricks usage unit; higher compute capability or SKU consumes more DBUs |
| Runtime | Longer-running compute costs more, especially all-purpose clusters left idle |
| Node type | Larger or specialized VMs usually cost more cloud infrastructure and may consume DBUs differently |
| Worker count | More workers increase parallelism and cost |
| Autoscaling | Can reduce idle capacity, but max workers can still raise peak cost |
| Warehouse size | Larger SQL warehouses consume more DBUs per hour |
| Photon | Can increase DBU rate on classic compute but may reduce total runtime |
| Serverless | Billed under serverless SKUs; Databricks manages infrastructure and scaling |
| Storage/network | Cloud storage and network transfer are separate from compute DBUs |

Exam shortcut: cost is roughly **rate x time**. A larger cluster can be cheaper only if it reduces runtime enough to offset the higher hourly rate.

## SQL (exam)

```sql
-- SQL warehouses are for SQL analytics and BI-style workloads.
SELECT
  order_date,
  COUNT(*) AS orders
FROM main.gold.orders
GROUP BY order_date
ORDER BY order_date;
```

```sql
-- Serverless/billing usage is commonly monitored through system tables
-- when your account has system table access.
SELECT
  billing_origin_product,
  SUM(usage_quantity) AS usage_quantity
FROM system.billing.usage
GROUP BY billing_origin_product;
```

## UI / product surfaces

- **Compute**: create and manage classic all-purpose compute, jobs compute settings, SQL warehouses, policies, libraries, event logs, and Spark UI.
- **Notebook compute picker**: attach a notebook to serverless compute, classic all-purpose compute, or supported SQL warehouses.
- **Lakeflow Jobs**: choose compute per task, including serverless jobs, classic jobs compute, SQL warehouse, and pipeline compute.
- **SQL Warehouses**: configure serverless/pro/classic warehouse type, size, scaling, auto-stop, permissions, and monitoring.
- **Account console / Admin settings**: manage serverless settings, quotas, policies, and billing-related controls.
- **System tables**: inspect billable usage when enabled, especially `system.billing.usage`.

## Exam traps

- Choosing all-purpose compute for a nightly production job. Use serverless jobs or jobs compute instead.
- Choosing a SQL warehouse for PySpark DataFrame ETL. Use notebook/jobs compute.
- Assuming serverless supports every Spark feature. It has API, language, library, storage, and streaming limitations.
- Forgetting auto-termination on interactive all-purpose compute.
- Scaling workers up before checking shuffle, skew, spills, file size, or query plan problems.
- Treating Photon as a separate compute type. It is an execution engine used by several compute types.
- Thinking a job always needs one cluster per task. Tasks can share a job compute resource when startup latency and reuse matter.
- Confusing SQL warehouse scaling with Spark worker autoscaling. They are configured differently and serve different workloads.

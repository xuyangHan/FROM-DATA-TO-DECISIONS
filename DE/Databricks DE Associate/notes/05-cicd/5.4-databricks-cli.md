# 5.4 Databricks CLI

## Exam objective

> Understand the Databricks CLI to validate, deploy, and manage Declarative Automation Bundles (formerly Databricks Asset Bundles) and other workspace assets in automated CI/CD workflows.

## Must know

- **CLI v0.2+** (`databricks` command) — auth via profile (`~/.databrickscfg`).
- **bundle validate** — schema check before deploy.
- **bundle deploy** — apply resources to target workspace.
- **bundle destroy** — remove deployed resources (careful in prod).
- CI: GitHub Actions / Azure DevOps runs CLI with OIDC or PAT secret.
- **Workspace import** legacy; bundles preferred for jobs/pipelines.

## When to use what

| Task | CLI |
|------|-----|
| Pre-merge check | `databricks bundle validate` |
| Release | `databricks bundle deploy -t prod` |
| Auth in CI | service principal + OIDC |

## PySpark / API essentials

```bash
# Configure profile
databricks configure --profile dev

# Bundle lifecycle
databricks bundle validate -t dev
databricks bundle deploy -t dev
databricks bundle run daily_etl -t dev

# Workspace (awareness)
databricks workspace list /Repos/
```

## Exam traps

- Old **`dbfs`**-only CLI workflows when question asks about **bundles**.
- Missing **`-t target`** → deploys wrong environment.
- Storing **PAT in repo** instead of secret store.

## Hands-on checklist

- [ ] Run validate + deploy for a sample bundle.
- [ ] Sketch a 3-step GitHub Actions pipeline (validate → deploy test → deploy prod).

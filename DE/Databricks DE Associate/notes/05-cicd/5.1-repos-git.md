# 5.1 Repos & Git workflow

## Exam objective

> Manage your code development workflow within the Databricks workspace UI, including creating and switching between branches in Databricks Repos, committing and pushing changes, and creating pull requests using Databricks Git integration.

## Must know

- **Repos**: Git folders in workspace linked to GitHub/GitLab/ADO/etc.
- **Branch**: create/switch in Repos UI; isolates notebook/job changes.
- **Commit + push**: saves to remote feature branch.
- **Pull request**: created from Repos UI or provider; review before merge to main.
- **Path**: `/Repos/<user>/<repo>/...` — jobs reference repo paths for notebooks.
- Don't edit merged production branch directly in UI without process.

## When to use what

| Task | Action |
|------|--------|
| New feature | New branch from `main` |
| Share code | Push branch + open PR |
| Deploy prod | Merge PR → bundle deploy from `main` (5.3) |

## UI / product surfaces

- **Repos** → clone → **Create branch** → edit → **Commit & push** → **Create pull request**.

## Exam traps

- Editing **main** directly in workspace without PR (governance anti-pattern in exam answers).
- Job pointing to **local workspace copy** not repo path after clone.
- Forgetting to **pull** latest before starting work (merge conflicts).

## Hands-on checklist

- [ ] Open a PR (or describe steps if solo).

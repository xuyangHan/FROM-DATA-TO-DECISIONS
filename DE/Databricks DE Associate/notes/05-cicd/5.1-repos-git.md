# 5.1 Repos / Git folders and Git workflow

## Exam objective

> Manage your code development workflow within the Databricks workspace UI, including creating and switching between branches in Databricks Repos, committing and pushing changes, and creating pull requests using Databricks Git integration.

## Big idea

Databricks Repos are now commonly called **Git folders** in the product docs. For the exam, treat these as the same idea: a workspace folder connected to a remote Git repository where notebooks, Python files, SQL files, and bundle configuration can be edited with Git operations from the Databricks UI.

The safe workflow is:

1. Clone or open the Git folder.
2. Pull the latest default branch.
3. Create a feature branch.
4. Edit notebooks or files.
5. Commit and push to the remote feature branch.
6. Open a pull request in the Git provider.
7. Merge after review.
8. Let CI/CD deploy from the approved branch.

## Must know

- **Git folder / Repo**: A Databricks workspace folder linked to a remote Git repository.
- **Remote repository**: GitHub, GitLab, Azure DevOps, Bitbucket, or another supported provider.
- **Branch**: Isolates work. Developers should normally work on feature branches, not directly on `main`.
- **Pull**: Updates the workspace Git folder from the remote branch.
- **Commit and push**: Saves local workspace changes and sends them to the remote branch.
- **Pull request (PR)**: Review gate before merging a feature branch into the deployment branch.
- **Production Git folder**: A restricted workspace Git folder that tracks a deployment branch and should usually be updated by automation or trusted admins/service principals.
- **Git credentials**: Users or service principals need credentials/permissions to access the remote repository.

## Developer workflow in the Databricks UI

1. Go to **Workspace** and create or open a Git folder.
2. Select the Git branch control.
3. Create a branch from `main` or switch to an existing branch.
4. Make notebook/file changes.
5. Open the Git dialog and review changed files.
6. Add a commit message.
7. Select **Commit & Push**.
8. Create a PR in the Git provider UI or from the Databricks integration if available.

## Branching pattern

| Situation | Best action |
|---|---|
| Starting new work | Pull latest `main`, create a feature branch |
| Sharing work | Commit and push the branch |
| Asking for review | Open a PR into `main` or the release branch |
| Fixing a merge conflict | Pull/rebase/merge carefully, then test again |
| Deploying to production | Deploy from merged, reviewed code, not an unreviewed workspace edit |

## Repos vs bundles

Git folders manage **source code collaboration**. Declarative Automation Bundles manage **deployment of Databricks resources as code**.

Typical exam relationship:

- Use Git folders/Repos for editing notebooks, Python files, SQL files, and YAML in branches.
- Use PRs for code review.
- Use bundle validation/deploy commands in CI after merge.
- Avoid manual production UI edits that drift away from Git.

## Production Git folder pattern

A production Git folder is not a personal working copy. It is usually:

- Created outside a user's personal workspace folder.
- Mapped to a stable deployment branch such as `main` or `release`.
- Locked down so most users have **Can Run** or **Can View**, not edit permissions.
- Updated by a service principal or CI/CD process after PR merge.

This pattern matters when a question asks how to avoid users changing production code by hand.

## Example scenario

Question: A developer needs to update a notebook used by a production job. The team requires review before deployment. What should the developer do?

Answer: Create a feature branch in the Databricks Git folder, edit the notebook, commit and push the branch, open a PR, and deploy only after the PR is reviewed and merged.

## Exam traps

- Choosing "edit `main` directly in the workspace" as the best workflow.
- Treating a Databricks workspace copy as the source of truth instead of remote Git.
- Forgetting to pull the latest branch before starting work.
- Assuming commit alone is enough. You must push to the remote for PR/CI.
- Using a personal user Git folder as the production deployment source.
- Manually changing production jobs in the UI after they are bundle-managed.

## Short memory hook

**Branch, edit, commit, push, PR, merge, deploy.**

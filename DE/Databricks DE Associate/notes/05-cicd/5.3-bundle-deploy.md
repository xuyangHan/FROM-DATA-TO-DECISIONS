# 5.3 Bundle deploy & promotion

## Exam objective

> Deploy Declarative Automation Bundles (formerly Databricks Asset Bundles) to package, configure, and promote Lakeflow Jobs, Lakeflow Spark Declarative Pipelines, and other workspace assets across dev, test, and prod environments.

## Must know

- Bundle **packages**: jobs, pipelines, notebooks, dashboards as resources.
- **deploy** pushes defined resources to target workspace.
- **Promotion flow**: dev validate → test integration → prod deploy (often via CI).
- **run_as**: service principal for production jobs.
- **Artifacts**: wheel/files uploaded as part of deploy.
- LDP (pipelines) defined in bundle same as jobs.

## When to use what

| Stage | Command / action |
|-------|------------------|
| Local check | `bundle validate` |
| Dev deploy | `bundle deploy -t dev` |
| Prod | CI pipeline with approval + `bundle deploy -t prod` |

## PySpark / API essentials

```yaml
# resources/jobs.yml
resources:
  jobs:
    daily_etl:
      name: daily_etl
      tasks:
        - task_key: transform
          notebook_task:
            notebook_path: ./src/transform.py
            base_parameters:
              catalog: ${var.catalog}
      job_clusters:
        - job_cluster_key: etl
          new_cluster:
            spark_version: 15.4.x-scala2.12
            node_type_id: i3.xlarge
            num_workers: 2
```

## UI / product surfaces

- After deploy: **Jobs** / **Pipelines** appear in target workspace with bundle-managed tags.

## Exam traps

- Manual UI job edits **drift** from bundle (next deploy overwrites).
- Deploying pipelines to **dev** workspace URL with **prod** target selected.
- Missing **permissions** for service principal `run_as`.

- [ ] Deploy a minimal bundle with one notebook job to dev.
- [ ] Change a variable and redeploy; confirm job parameter updated.

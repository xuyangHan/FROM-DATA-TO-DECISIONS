# 5.2 Bundle variables & overrides

## Exam objective

> Understand environment-specific configuration using Automation Bundle (formerly Databricks Asset Bundles) variables and overrides while promoting the same codebase across dev, test, and prod targets.

## Must know

- **Bundle**: `databricks.yml` + resources (`resources/jobs.yml`, etc.).
- **Targets**: `dev`, `test`, `prod` — each maps to a workspace/host.
- **Variables**: `${var.catalog}` placeholders in YAML.
- **Overrides**: per-target `variables` / `workspace` / `run_as` in `targets.prod`.
- Same git commit promoted across environments with different catalog names, paths, clusters.

## When to use what

| Config | Place in bundle |
|--------|-----------------|
| Catalog name per env | `variables.catalog` |
| Cluster policy ID | target override |
| Job schedule cron | prod-only override |

## PySpark / API essentials

```yaml
# databricks.yml (fragment)
bundle:
  name: etl_project

variables:
  catalog:
    default: dev_catalog

targets:
  dev:
    default: true
    workspace:
      host: https://dev.cloud.databricks.com
    variables:
      catalog: dev_main
  prod:
    workspace:
      host: https://prod.cloud.databricks.com
    variables:
      catalog: prod_main
```

```python
# Notebook reads widget or spark.conf set by job parameter from bundle
catalog = dbutils.widgets.get("catalog")
spark.table(f"{catalog}.silver.orders")
```

## Exam traps

- Hardcoding **prod catalog** in notebooks instead of variables.
- Different **job definitions** per env (should be one bundle, overrides).
- Deploying to **wrong target** without `-t prod` flag.

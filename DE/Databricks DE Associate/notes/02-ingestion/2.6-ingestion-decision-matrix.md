# 2.6 Ingestion method prioritization

## Exam objective

> Prioritize between Auto Loader, Lakeflow Connect (standard and managed connectors), partner connectors, and other ingestion methods based on technical requirements such as data volume, ingestion frequency, data types, and governance needs with Unity Catalog.

## Must know

- **Unity Catalog** should govern landing tables (`catalog.schema.table`) for lineage and access control.
- **Auto Loader**: incremental files in cloud object storage (S3/ADLS/GCS); schema evolution; very high file volume.
- **COPY INTO**: SQL-first batch incremental loads from cloud paths into Delta tables; simpler ops, less streaming-native.
- **Lakeflow Connect**: SaaS/DB/API sources with connectors; standard (you operate) vs managed (Databricks-operated sync).
- **JDBC/REST/notebooks**: custom sources, small volume, or one-off; orchestrate with Lakeflow Jobs.
- **Partner connectors**: third-party marketplace integrations when Connect does not cover the source.

## When to use what

| Requirement | Best fit |
|-------------|----------|
| Thousands+ new files/day in cloud storage | **Auto Loader** |
| Periodic batch of known files in S3/ADLS/GCS | **COPY INTO** or Auto Loader |
| Salesforce, Workday, SaaS with supported connector | **Lakeflow Connect** (managed if available) |
| Enterprise DB with JDBC, custom SQL extract | **JDBC** notebook/job or Connect |
| Strict schema, rare file drops, SQL team | **COPY INTO** |
| Schema drift, nested JSON files in storage | **Auto Loader** + `schemaEvolution` |
| Need Databricks-managed sync SLA | **Connect managed** connector |
| Prototype / low volume / bespoke API | **Notebook + Job** |
| UC governance + lineage required | Any path that **writes to UC Delta tables** (not unregistered paths) |

```
Source type?
├── Files in cloud storage
│   ├── High volume / continuous → Auto Loader
│   └── Scheduled batch, simple → COPY INTO
├── Enterprise app (connector exists) → Lakeflow Connect
└── Custom / JDBC / REST → Notebook → (optional) staging → UC table + Job
```

## PySpark / API essentials

```python
# Pattern: always land to UC-governed Delta (example path)
catalog, schema = "main", "bronze"
table = f"{catalog}.{schema}.events_raw"

# Auto Loader (incremental files)
(spark.readStream.format("cloudFiles")
  .option("cloudFiles.format", "json")
  .load("s3://bucket/incoming/")
  .writeStream
  .option("checkpointLocation", "s3://bucket/_checkpoints/events")
  .table(table))

# JDBC one-off (often scheduled as Job instead)
jdbc_df = (spark.read.format("jdbc")
  .option("url", jdbc_url)
  .option("dbtable", "public.orders")
  .option("user", dbutils.secrets.get("scope", "db_user"))
  .option("password", dbutils.secrets.get("scope", "db_password"))
  .load())
jdbc_df.write.format("delta").mode("append").saveAsTable(table)
```

## SQL (exam)

```sql
-- COPY INTO when source is cloud files (see 2.2)
COPY INTO main.bronze.events_raw
FROM 's3://bucket/incoming/'
FILEFORMAT = JSON;
```

## UI / product surfaces

- **Data ingestion** / **Lakeflow Connect** — create connections and syncs to UC tables.
- **Catalog Explorer** — verify governed target tables and lineage.

## Exam traps

- Choosing **COPY INTO** for millions of small daily files (Auto Loader is built for that).
- Using **all-purpose clusters** for production ingestion instead of **job clusters** or pipelines.
- Landing data only on **DBFS** without registering in **Unity Catalog** when governance is required.
- **Partner connector** vs **Connect** — exam wants the supported first-party path when listed.

# 3.7 Data quality

## Exam objective

> Apply data quality checks and validation rules to ensure reliable Silver and Gold datasets.

## Must know

- Validate at **silver/gold** — not only in BI tools.
- Techniques: row **filters**, `dropDuplicates`, **constraints** (Delta CHECK), **expectations** in LDP, quarantine tables for bad rows.
- **Great Expectations** / DQX (awareness) — optional frameworks; exam may reference LDP expectations.
- Track metrics: null rate, row count deltas, key uniqueness.
- Fail pipeline or route bad data — explicit business choice.

## When to use what

| Rule | Implementation |
|------|----------------|
| `order_id` not null | `filter` before write |
| Unique key | `dropDuplicates` or constraint |
| Amount >= 0 | `filter` or `CHECK` constraint |
| Bad rows auditable | Write to `main.quarantine.orders` |

## PySpark / API essentials

```python
bronze = spark.table("main.bronze.orders")

valid = bronze.filter(
  "order_id IS NOT NULL AND amount >= 0"
)

invalid = bronze.join(valid, "order_id", "left_anti")
invalid.write.format("delta").mode("append").saveAsTable("main.quarantine.orders")

if invalid.count() > 0:
    # Option: fail job for hard quality gate
    raise Exception(f"Quality failed: {invalid.count()} bad rows")

valid.dropDuplicates(["order_id"]).write.format("delta").mode("overwrite").saveAsTable("main.silver.orders")
```

```python
# LDP-style expectation (conceptual)
@dlt.expect_or_drop("valid_amount", "amount >= 0")
def silver_orders(): ...
```

## SQL (exam)

```sql
ALTER TABLE main.silver.orders ADD CONSTRAINT valid_amount CHECK (amount >= 0);
```

## UI / product surfaces

- LDP pipeline **Expectations** panel; job email on failure.

## Exam traps

- Only validating in **Power BI** layer (too late).
- Silently **dropping** bad rows with no quarantine or metric.
- **CHECK** constraint on table with existing violating data (add fails).

## Hands-on checklist

- [ ] Split valid/invalid rows into silver vs quarantine tables.
- [ ] Add a CHECK constraint or document why filter-only is enough.
- [ ] Log row counts before/after quality step in job output.

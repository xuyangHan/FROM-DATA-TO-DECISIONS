# 3.3 Column and row manipulation

## Exam objective

> Manipulate columns, rows, and table structures by adding, dropping, splitting, renaming column names, applying filters, and exploding arrays.

## Must know

- **`select`** chooses columns and can create renamed or derived columns.
- **`withColumn`** adds a new column or replaces an existing column with the same name.
- **`drop`** removes columns from a DataFrame.
- **`withColumnRenamed`** renames one column at a time.
- **`filter`** and **`where`** are equivalent row filters.
- **`split`** converts a string into an array.
- **`getItem`**, **`element_at`**, and array indexing extract values from arrays.
- **`explode`** turns each array or map element into a separate output row.
- **`explode_outer`** preserves a row when the array or map is null by returning one row with null values.
- DataFrame transformations are lazy. Nothing runs until an action or write.

## Operation map

| Goal | PySpark API | SQL equivalent |
|------|-------------|----------------|
| Add derived column | `withColumn("x", expr)` | `SELECT ..., expr AS x` |
| Replace column | `withColumn("x", new_expr)` | `SELECT new_expr AS x` |
| Rename column | `withColumnRenamed("old", "new")` | `SELECT old AS new` or `ALTER TABLE RENAME COLUMN` |
| Drop column | `drop("raw_payload")` | Select every column except it |
| Filter rows | `filter(col("status") == "ok")` | `WHERE status = 'ok'` |
| Split string | `split(col("path"), "/")` | `SPLIT(path, '/')` |
| Array to rows | `explode(col("tags"))` | `EXPLODE(tags)` |
| Keep null array rows | `explode_outer(col("tags"))` | `EXPLODE_OUTER(tags)` |

## PySpark / API essentials

```python
from pyspark.sql.functions import col, element_at, explode, explode_outer, lower, split, trim

events = spark.table("main.bronze.events_raw")

clean = (
    events
    .filter(col("event_time").isNotNull())
    .withColumnRenamed("userId", "user_id")
    .withColumn("event_type", lower(trim(col("event_type"))))
    .withColumn("path_parts", split(col("page_path"), "/"))
    .withColumn("top_level_path", element_at(col("path_parts"), 2))
    .drop("raw_payload")
)
```

```python
# Explode an array: one output row per tag.
tags = (
    clean
    .select("event_id", "user_id", explode(col("tags")).alias("tag"))
)

# Keep parent rows where tags is null.
tags_outer = (
    clean
    .select("event_id", "user_id", explode_outer(col("tags")).alias("tag"))
)
```

```python
# Select is often clearer than repeated drop/rename operations.
silver = clean.select(
    col("event_id"),
    col("user_id"),
    col("event_time").alias("event_ts"),
    col("event_type"),
    col("top_level_path")
)

silver.write.format("delta").mode("overwrite").saveAsTable("main.silver.events")
```

## SQL (exam)

```sql
CREATE OR REPLACE TABLE main.silver.events AS
SELECT
  event_id,
  userId AS user_id,
  LOWER(TRIM(event_type)) AS event_type,
  SPLIT(page_path, '/') AS path_parts,
  ELEMENT_AT(SPLIT(page_path, '/'), 2) AS top_level_path,
  event_time AS event_ts
FROM main.bronze.events_raw
WHERE event_time IS NOT NULL;
```

```sql
-- Explode array values to a new grain: one row per event-tag.
CREATE OR REPLACE TABLE main.silver.event_tags AS
SELECT
  event_id,
  userId AS user_id,
  EXPLODE(tags) AS tag
FROM main.bronze.events_raw;
```

## Grain awareness

Exploding arrays changes the grain of the dataset.

| Before explode | After explode |
|----------------|---------------|
| One row per event | One row per event-tag |
| `count(*)` counts events | `count(*)` counts event-tag combinations |
| Aggregates by event are safe | Aggregates may overcount events unless deduped |

Exam shortcut: if an array has three elements, `explode` returns three rows for that input row. If the array is empty or null, regular `explode` returns zero rows.

## Table structure changes

For persisted Unity Catalog tables, SQL can change table structure directly.

```sql
ALTER TABLE main.silver.events RENAME COLUMN userId TO user_id;
ALTER TABLE main.silver.events DROP COLUMN raw_payload;
```

For exam transformation questions, building a cleaned table with `CREATE OR REPLACE TABLE AS SELECT` or a DataFrame write is often simpler than mutating the bronze table.

## UI / product surfaces

- **Notebook schema display**: confirm arrays, structs, maps, and nested fields before using `explode`.
- **Data profile**: inspect nulls and distinct values before filtering or replacing.
- **Catalog Explorer**: verify final silver schema and column comments.

## Exam traps

- Assuming `withColumn` always adds a column. It replaces when the name already exists.
- Exploding an array and then interpreting row counts as if the original grain was preserved.
- Using regular `explode` when the business rule says to keep rows with missing arrays.
- Using DataFrame `union` after column manipulation without checking column order.
- Dropping a raw column that is still needed for audit or troubleshooting in bronze.
- Applying filters after an expensive explode when filtering could have reduced rows first.

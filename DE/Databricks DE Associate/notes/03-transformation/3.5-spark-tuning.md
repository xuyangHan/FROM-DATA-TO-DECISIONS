# 3.5 Spark tuning & re-measure

## Exam objective

> Understand the basic tuning parameters (spark.sql.shuffle.partitions, spark.default.parallelism, spark.executor/driver.memory, spark.sql.autoBroadcastJoinThreshold) and re-measure the performance.

## Must know

- **spark.sql.shuffle.partitions** — default partitions after shuffle (often 200); raise for large shuffles, lower for small.
- **spark.default.parallelism** — default RDD parallelism (often # cores).
- **spark.executor.memory** / **spark.driver.memory** — JVM heap per role; OOM if too low.
- **spark.sql.autoBroadcastJoinThreshold** — max bytes for auto broadcast (default ~10MB).
- **Re-measure**: Spark UI — stage duration, shuffle read/write, spill, skew.

## When to use what

| Symptom | Knob / action |
|---------|----------------|
| Many small shuffle tasks | Lower `shuffle.partitions` |
| Huge shuffle, long stages | Increase partitions or fix skew; repartition by key |
| Large-small join still shuffles | Increase broadcast threshold or `broadcast()` hint |
| Driver OOM (collect, wide ops) | Increase driver memory; avoid huge collects |
| Executor OOM | Increase executor memory; cache carefully |

## PySpark / API essentials

```python
# Set before heavy work (cluster or notebook)
spark.conf.set("spark.sql.shuffle.partitions", "64")
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 50 * 1024 * 1024)  # 50MB

# Inspect
print(spark.conf.get("spark.sql.shuffle.partitions"))

large = spark.table("main.silver.fact")
small = spark.table("main.silver.dim")
result = large.join(small, "key")  # re-check Spark UI after config change

# Optional repartition before write
result.repartition(32, "date").write.format("delta").mode("overwrite").saveAsTable("main.gold.fact")
```

## SQL (exam)

```sql
SET spark.sql.shuffle.partitions = 64;
```

## UI / product surfaces

- **Spark UI** → Stages: **Shuffle Read/Write**, **Spill (Memory/Disk)**, task skew (one task much longer).
- **Metrics** tab on job run — link from Lakeflow Jobs.

## Exam traps

- Tuning **without measuring** — exam expects check Spark UI after changes.
- Only increasing workers without fixing **data skew**.
- Lowering shuffle partitions on **very large** data → tiny number of huge tasks.

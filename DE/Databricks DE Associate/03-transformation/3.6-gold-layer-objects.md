# 3.6 Gold layer objects

## Exam objective

> Understand the difference between, and how to build, Gold layer objects such as materialized views, views, streaming tables, and tables for BI and analytics teams in Unity Catalog.

## Must know

- **Gold** is the serving layer for BI, reporting, dashboards, metric consumption, and analytics teams.
- Gold objects should have stable names, clear grain, documented metrics, and governed access in Unity Catalog.
- A **table** physically stores data. In Databricks, a managed Delta table is the default durable serving object.
- A **view** stores a query definition and computes results when queried. It does not store the result data.
- A **materialized view** stores cached/precomputed results and refreshes them, often incrementally, to speed repeated analytical queries.
- A **streaming table** is a table managed by Lakeflow Spark Declarative Pipelines or standalone pipeline syntax that is updated from streaming or incremental inputs.
- **Temporary views** are session-scoped and are not production gold assets.
- Gold is not only "aggregate tables"; it can include dimensional models, denormalized marts, materialized views, semantic serving tables, or streaming outputs.

## Object decision table

| Object | Stores data? | Freshness model | Best for | Avoid when |
|--------|--------------|-----------------|----------|------------|
| Managed Delta table | Yes | Updated by job/pipeline/write | Canonical BI datasets, facts, dimensions, marts | You only need a thin SQL alias |
| View | No | Always reads current source data | Lightweight logic, security abstraction, simple joins/renames | Query is expensive and run often |
| Materialized view | Yes | Refreshed by managed pipeline | Expensive repeated aggregations, BI acceleration | Need every row to update immediately |
| Streaming table | Yes | Incremental/streaming pipeline updates | Low-latency append-style gold outputs | Source is simple static batch and no low-latency need |
| Temporary view | No | Current Spark session only | Notebook exploration | Shared BI or governed production use |

## Gold design questions

Before choosing the object, ask:

- Who consumes it: BI dashboard, analysts, ML, operations, or downstream jobs?
- What is the grain: one row per order, customer, day, customer-day, or event?
- How fresh must it be: real-time, every few minutes, hourly, daily?
- Is the query expensive enough to materialize?
- Does the table need row-level/column-level security, grants, comments, or lineage?
- Should users query a stable abstraction while implementation changes underneath?

## PySpark / API essentials

```python
from pyspark.sql.functions import count, sum as sum_

orders = spark.table("main.silver.orders")

daily_revenue = (
    orders
    .groupBy("order_date")
    .agg(
        count("*").alias("order_count"),
        sum_("amount").alias("revenue")
    )
)

(
    daily_revenue
    .write
    .format("delta")
    .mode("overwrite")
    .saveAsTable("main.gold.daily_revenue")
)
```

```python
# Exploration only, not a governed gold object.
daily_revenue.createOrReplaceTempView("tmp_daily_revenue")
```

## SQL: gold table

```sql
CREATE OR REPLACE TABLE main.gold.daily_revenue AS
SELECT
  order_date,
  COUNT(*) AS order_count,
  SUM(amount) AS revenue
FROM main.silver.orders
GROUP BY order_date;
```

## SQL: view

```sql
CREATE OR REPLACE VIEW main.gold.orders_for_bi AS
SELECT
  o.order_id,
  o.order_date,
  o.amount,
  c.segment,
  c.region
FROM main.silver.orders AS o
LEFT JOIN main.silver.customers AS c
  ON o.customer_id = c.customer_id;
```

Use a view when the logic is lightweight or when you want a stable query interface over changing silver tables.

## SQL: materialized view

```sql
CREATE MATERIALIZED VIEW main.gold.daily_revenue_mv AS
SELECT
  order_date,
  COUNT(*) AS order_count,
  SUM(amount) AS revenue
FROM main.silver.orders
GROUP BY order_date;
```

```sql
REFRESH MATERIALIZED VIEW main.gold.daily_revenue_mv;
```

Materialized views are useful when many BI users repeatedly query the same expensive aggregation. In current Databricks, standalone materialized views are Unity Catalog managed objects maintained by a managed pipeline.

## SQL: streaming table concept

Streaming tables are commonly created in Lakeflow Spark Declarative Pipelines. They are used when records should be processed incrementally as data arrives.

```sql
CREATE OR REFRESH STREAMING TABLE main.gold.order_events_live AS
SELECT
  order_id,
  customer_id,
  amount,
  order_ts
FROM STREAM main.silver.orders_stream;
```

Exact syntax can vary by whether the object is defined in Lakeflow Spark Declarative Pipelines, Databricks SQL, or standalone pipeline syntax. The exam focus is usually the difference in behavior and use case.

## Unity Catalog serving practices

- Use three-part names: `catalog.schema.object`.
- Put gold objects in a gold schema such as `main.gold`.
- Grant BI groups read access on gold instead of broad access to bronze.
- Add comments to tables and columns for discoverability.
- Use lineage to trace gold metrics back to silver and bronze inputs.
- Avoid exposing raw PII in gold unless governed with masking, row filters, or restricted grants.

## UI / product surfaces

- **Catalog Explorer**: create/browse tables and views, inspect lineage, comments, grants, and history.
- **Databricks SQL Editor**: create views and materialized views for BI use.
- **Lakeflow Spark Declarative Pipelines UI**: define and monitor streaming tables, materialized views, pipeline graph, and refreshes.
- **Dashboards / BI connectors**: commonly query gold objects.

## Exam traps

- Choosing a view for an expensive dashboard query that many users run repeatedly.
- Choosing a materialized view when the requirement is always-current row-level data with no refresh lag.
- Presenting a temporary view as a governed enterprise gold object.
- Writing gold as an unmanaged path when Unity Catalog governance is required.
- Forgetting that gold should represent business-friendly grain and naming, not raw source naming.
- Exposing bronze/raw tables directly to BI users when silver/gold curated objects are expected.

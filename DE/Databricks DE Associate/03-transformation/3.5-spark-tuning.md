# 3.5 Spark tuning and re-measure

## Exam objective

> Understand the basic tuning parameters (spark.sql.shuffle.partitions, spark.default.parallelism, spark.executor/driver.memory, spark.sql.autoBroadcastJoinThreshold) and re-measure the performance.

## Must know

- Tuning is an experiment: **measure baseline -> change one thing -> rerun -> compare**.
- **`spark.sql.shuffle.partitions`** controls the default number of partitions created after Spark SQL/DataFrame shuffle operations such as joins and aggregations.
- **`spark.default.parallelism`** controls default parallelism for RDD operations and some lower-level Spark operations. DataFrame workloads are more often affected by SQL configs.
- **`spark.executor.memory`** is memory per executor JVM. Too little can cause executor OOM, spills, or task failures.
- **`spark.driver.memory`** is memory for the driver. Driver OOM often comes from `collect()`, huge query plans, or large metadata/actions returning to the driver.
- **`spark.sql.autoBroadcastJoinThreshold`** controls the maximum table size Spark considers for automatic broadcast joins.
- The default `spark.sql.autoBroadcastJoinThreshold` is **10 MB**, specifically **10,485,760 bytes**. When Spark estimates one side of a join is below this threshold, it can automatically use a **Broadcast Hash Join**, sending that small table to each worker to avoid an expensive shuffle.
- Setting `spark.sql.autoBroadcastJoinThreshold` to **`-1`** disables automatic broadcast joins. This can relieve executor memory pressure from the broadcast phase and force Spark toward a standard shuffle join strategy such as `SortMergeJoin`.
- **Adaptive Query Execution (AQE)** may coalesce shuffle partitions, change join strategy, and handle some skew at runtime.
- Databricks performance should also consider Photon, Delta table statistics, optimized file layout, and predictive optimization, but the objective here focuses on the basic Spark parameters.

## Performance measurement loop

1. Run the original query or job.
2. Record duration, shuffle read/write, spill, skewed tasks, input rows, and output rows.
3. Identify the likely bottleneck.
4. Change one setting or query pattern.
5. Rerun with comparable input data and cluster size.
6. Compare metrics, not just intuition.

## Tuning parameter table

| Parameter | What it affects | Increase when | Decrease when |
|-----------|-----------------|---------------|---------------|
| `spark.sql.shuffle.partitions` | Number of partitions after SQL/DataFrame shuffles | Tasks are huge, spills are high, cores underused | Many tiny tasks, scheduling overhead dominates |
| `spark.default.parallelism` | Default RDD parallelism | RDD tasks are too few | Too many tiny RDD tasks |
| `spark.executor.memory` | Per-executor heap | Executors OOM or spill heavily | Cluster is over-provisioned or memory unused |
| `spark.driver.memory` | Driver heap | Driver OOM, large metadata/query planning | Rarely lowered for tuning |
| `spark.sql.autoBroadcastJoinThreshold` | Auto broadcast join eligibility | Small dimensions are not being broadcast | Broadcast causes memory pressure |

## Symptom to action

| Symptom | Likely cause | First checks/actions |
|---------|--------------|----------------------|
| Long stage with heavy shuffle | Large join or aggregation | Check shuffle partitions, join keys, data skew |
| One task much slower than others | Data skew | Inspect key distribution, use salting or skew-aware techniques |
| High disk spill | Not enough memory or partitions too large | Increase partitions, executor memory, or reduce data before shuffle |
| Thousands of tiny tasks | Too many partitions or tiny files | Lower shuffle partitions, compact data, use optimized writes |
| Large-small join still shuffles | Broadcast not selected | Check table size/stats, use broadcast hint, threshold |
| Driver OOM | Too much data returned to driver | Avoid `collect`, `toPandas`, large `display`, or huge lists |
| Executor OOM | Task data too large, cache too large, bad partitioning | Increase executor memory, repartition, unpersist |

## PySpark / API essentials

```python
# Inspect configs.
print(spark.conf.get("spark.sql.shuffle.partitions"))
print(spark.conf.get("spark.sql.autoBroadcastJoinThreshold"))

# Set notebook/session-level configs before the query you want to test.
spark.conf.set("spark.sql.shuffle.partitions", "96")
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 50 * 1024 * 1024)  # 50 MB
```

```python
from pyspark.sql.functions import broadcast

fact = spark.table("main.silver.orders")
dim = spark.table("main.silver.customers")

# Use an explicit broadcast hint only when dim is small enough.
result = fact.join(broadcast(dim), on="customer_id", how="left")

# Repartition by a common downstream key before a large write if useful.
(
    result
    .repartition(96, "order_date")
    .write
    .format("delta")
    .mode("overwrite")
    .saveAsTable("main.gold.orders_enriched")
)
```

## What to inspect in Spark UI / Query Profile

| Metric or plan node | Meaning |
|---------------------|---------|
| `Exchange` | Shuffle boundary |
| `BroadcastExchange` | Small side is being broadcast |
| `SortMergeJoin` | Common large-large join strategy |
| `BroadcastHashJoin` | Broadcast join strategy |
| Shuffle read/write | Data moved across executors |
| Spill memory/disk | Task did not have enough memory for in-memory processing |
| Task duration spread | Skew if one/few tasks are much slower |
| Input/output rows | Confirms row explosion or filtering worked |

## Practical tuning examples

Small dataset with 200 shuffle partitions:

```python
spark.conf.set("spark.sql.shuffle.partitions", "16")
```

Large aggregation with huge tasks:

```python
spark.conf.set("spark.sql.shuffle.partitions", "400")
```

Small dimension not broadcasting:

```python
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 100 * 1024 * 1024)
```

Broadcast phase causing executor memory pressure:

```python
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)
```

Only keep a setting if the rerun improves measured runtime, shuffle, or spill without creating new memory risk.

## Exam traps

- Tuning blindly without re-measuring in Spark UI or Query Profile.
- Assuming more partitions is always faster. Too many partitions create scheduling overhead.
- Assuming fewer partitions is always faster. Too few partitions create huge tasks and spills.
- Increasing executor memory to hide a skewed join instead of fixing skew.
- Confusing driver memory with executor memory.
- Using `collect()` or `toPandas()` on a large DataFrame and then blaming executor memory.
- Raising broadcast threshold so high that Spark tries to broadcast a table that does not fit in executor memory.
- Forgetting that `spark.sql.autoBroadcastJoinThreshold = -1` disables automatic broadcast joins; use it only when broadcast memory pressure is the actual bottleneck.

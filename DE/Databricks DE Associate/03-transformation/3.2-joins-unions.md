# 3.2 Joins and unions

## Exam objective

> Combine DataFrames with operations such as Inner join, left join, broadcast join, multiple keys, cross join, union, and union all.

## Must know

- **Inner join** returns only rows with matching keys on both sides.
- **Left join** returns all rows from the left side and matching rows from the right side. Missing right-side matches become null.
- **Broadcast join** sends a small table to all executors so Spark can avoid shuffling the large table.
- **Multiple-key joins** use all listed key columns; every key must match.
- **Cross join** creates a Cartesian product. It is valid but usually expensive and dangerous.
- **PySpark DataFrame `union()` / `unionAll()`** keeps duplicate rows. In modern Spark, `unionAll()` is an alias for `union()`.
- **SQL `UNION`** removes duplicates. **SQL `UNION ALL`** keeps duplicates.
- Prefer **`unionByName()`** for DataFrames when column order might differ.
- A duplicate key on either side of a join can multiply rows. This is normal join behavior, not a Spark bug.

## Join mental model

| Join type | Rows returned | Common use |
|-----------|---------------|------------|
| `inner` | Only matching rows | Keep valid facts that have required dimension matches |
| `left` | All left rows plus right matches | Preserve facts while enriching with optional dimensions |
| `left_semi` | Left rows that have a match, left columns only | Existence filter |
| `left_anti` | Left rows with no match | Find missing dimension keys or invalid records |
| `cross` | Every left row paired with every right row | Intentional combinations, small lookup expansion |

For the certification, inner and left joins are the core targets, but semi and anti joins are useful for data quality examples.

## When to use what

| Situation | Approach |
|-----------|----------|
| Large fact plus small dimension | Broadcast the small dimension |
| Both sides large | Let Spark use shuffle/sort-merge joins; tune only after measuring |
| Need all orders even without customer details | Left join orders to customers |
| Need only orders with valid customer details | Inner join |
| Need orders whose customer_id is missing from dimension | Left anti join |
| Stack same-schema daily datasets | `unionByName` |
| Stack datasets with added columns over time | `unionByName(..., allowMissingColumns=True)` |
| Need SQL distinct stack | SQL `UNION` |
| Need SQL append-like stack | SQL `UNION ALL` |

## PySpark / API essentials

```python
from pyspark.sql.functions import broadcast, col

orders = spark.table("main.silver.orders")
customers = spark.table("main.silver.customers")

# Inner join on one key.
orders_with_customers = orders.join(customers, on="customer_id", how="inner")

# Left join on multiple keys.
orders_enriched = orders.join(customers, on=["customer_id", "region"], how="left")

# Explicit join condition when names differ.
joined = (
    orders.alias("o")
    .join(
        customers.alias("c"),
        col("o.customer_id") == col("c.id"),
        "left"
    )
    .select("o.*", col("c.segment").alias("customer_segment"))
)

# Broadcast the small side.
orders_fast = orders.join(broadcast(customers), on="customer_id", how="left")

# Find records that do not match a dimension table.
missing_customers = orders.join(customers, on="customer_id", how="left_anti")

# Cartesian product only when intentional.
small_calendar = spark.table("main.silver.calendar").filter("date >= current_date()")
combinations = customers.select("customer_id").crossJoin(small_calendar.select("date"))
```

## Union examples

```python
events_2025 = spark.table("main.silver.events_2025")
events_2026 = spark.table("main.silver.events_2026")

# Keeps duplicates and matches by position. Risky if column order differs.
stacked_position = events_2025.union(events_2026)

# Keeps duplicates and matches by column name. Safer exam answer.
stacked_by_name = events_2025.unionByName(events_2026)

# Allows schema drift by filling missing columns with null.
stacked_evolved = events_2025.unionByName(events_2026, allowMissingColumns=True)

# Remove duplicates after a DataFrame union if the business rule requires it.
deduped = stacked_by_name.dropDuplicates(["event_id"])
```

## SQL (exam)

```sql
SELECT
  o.order_id,
  o.customer_id,
  o.amount,
  c.segment
FROM main.silver.orders AS o
INNER JOIN main.silver.customers AS c
  ON o.customer_id = c.customer_id
 AND o.region = c.region;
```

```sql
-- Broadcast hint for a small dimension.
SELECT /*+ BROADCAST(c) */
  o.order_id,
  c.segment
FROM main.silver.orders AS o
LEFT JOIN main.silver.customers AS c
  ON o.customer_id = c.customer_id;
```

```sql
-- SQL UNION removes duplicates.
SELECT event_id, event_ts FROM main.silver.events_2025
UNION
SELECT event_id, event_ts FROM main.silver.events_2026;

-- SQL UNION ALL keeps duplicates.
SELECT event_id, event_ts FROM main.silver.events_2025
UNION ALL
SELECT event_id, event_ts FROM main.silver.events_2026;
```

## Performance cues

- Spark may automatically broadcast small tables using `spark.sql.autoBroadcastJoinThreshold`.
- Adaptive Query Execution can change join strategy at runtime based on observed sizes.
- Broadcast only small tables. Broadcasting a large table can cause memory pressure.
- Cross joins are expensive because row count is `left_count * right_count`.
- Fresh table statistics help the optimizer choose better join plans. SQL: `ANALYZE TABLE table_name COMPUTE STATISTICS`.
- For multiple joins, joining smaller dimensions first can reduce intermediate data size.

## UI / product surfaces

- **Spark UI / SQL tab**: inspect physical plans for `BroadcastHashJoin`, `BroadcastExchange`, `SortMergeJoin`, and `Exchange`.
- **Query Profile** in Databricks SQL: identify shuffle-heavy joins, skewed tasks, and expensive stages.
- **Catalog Explorer lineage**: confirm gold or silver tables depend on the expected inputs.

## Exam traps

- Saying PySpark `union()` removes duplicates. It does not.
- Confusing SQL `UNION` with SQL `UNION ALL`.
- Using `union()` on DataFrames with the same column names but different column order.
- Forgetting that left join can introduce nulls in right-side columns.
- Missing a join condition in SQL and accidentally creating a cross join.
- Joining before deduplicating a dimension table and causing row multiplication.
- Broadcasting the large fact table instead of the small dimension.

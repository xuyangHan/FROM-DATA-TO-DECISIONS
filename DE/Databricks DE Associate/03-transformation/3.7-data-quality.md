# 3.7 Data quality

## Exam objective

> Apply data quality checks and validation rules to ensure reliable Silver and Gold datasets.

## Must know

- Data quality should be enforced before users consume silver and gold, not only inside dashboards.
- Common checks: required fields, valid ranges, accepted values, type validity, uniqueness, referential integrity, freshness, row count changes, and duplicate rates.
- Data quality actions are business choices: **drop invalid rows**, **quarantine invalid rows**, **fail the pipeline**, or **warn and continue**.
- Delta table constraints can enforce **`NOT NULL`** and **`CHECK`** rules at write time.
- Primary key and foreign key constraints in Unity Catalog are informational, not enforced.
- Lakeflow Spark Declarative Pipelines expectations can track, drop, fail, or allow records depending on expectation configuration.
- Quality metrics should be observable: row counts, invalid counts, null rates, duplicate counts, expectation results, and pipeline failures.

## Quality check categories

| Category | Example rule | Common implementation |
|----------|--------------|-----------------------|
| Completeness | `order_id IS NOT NULL` | Filter, expectation, `NOT NULL` |
| Validity | `amount >= 0` | `CHECK` constraint, filter, expectation |
| Conformity | `status IN ('new','paid','shipped')` | Standardize and validate |
| Type validity | timestamp string parses successfully | Cast then check for null |
| Uniqueness | one row per `order_id` | `dropDuplicates`, duplicate report |
| Referential integrity | every `customer_id` exists in customers | Inner/anti join check |
| Freshness | max event time within expected SLA | Aggregate check |
| Volume | row count not unexpectedly zero or 10x higher | Job assertion/monitor |

## Action decision table

| Requirement clue | Best action |
|------------------|-------------|
| "Bad data must never reach silver" | Drop or quarantine before silver write |
| "Pipeline must stop if invalid records exist" | Fail on quality check |
| "Need audit trail of rejected records" | Write quarantine table |
| "Dashboard should continue but quality issue visible" | Warn, log metrics, notify |
| "Enforce rule for all future writes" | Delta `CHECK` or `NOT NULL` constraint |
| "Managed pipeline with DQ metrics" | Lakeflow expectations |

## PySpark / API essentials

```python
from pyspark.sql.functions import col, count, current_timestamp, lit, to_timestamp

bronze = spark.table("main.bronze.orders_raw")

staged = (
    bronze
    .withColumn("order_ts", to_timestamp(col("order_ts_raw"), "yyyy-MM-dd HH:mm:ss"))
    .withColumn("amount", col("amount_raw").cast("decimal(12,2)"))
)

valid_condition = (
    col("order_id").isNotNull()
    & col("order_ts").isNotNull()
    & col("amount").isNotNull()
    & (col("amount") >= 0)
)

valid = staged.filter(valid_condition)

invalid = (
    staged
    .filter(~valid_condition)
    .withColumn("_quarantine_reason", lit("required field missing, invalid timestamp, or invalid amount"))
    .withColumn("_quarantined_at", current_timestamp())
)

invalid.write.format("delta").mode("append").saveAsTable("main.quarantine.orders_invalid")
valid.dropDuplicates(["order_id"]).write.format("delta").mode("overwrite").saveAsTable("main.silver.orders")
```

## Fail a job on hard quality gates

```python
invalid_count = invalid.count()

if invalid_count > 0:
    raise ValueError(f"Data quality failed: {invalid_count} invalid order rows")
```

Use hard failures when downstream correctness matters more than availability. Use quarantine plus alerting when business users need the pipeline to continue with valid rows.

## Referential quality with joins

```python
orders = spark.table("main.silver.orders")
customers = spark.table("main.silver.customers")

missing_customer_keys = orders.join(customers, on="customer_id", how="left_anti")

if missing_customer_keys.count() > 0:
    missing_customer_keys.write.mode("append").saveAsTable("main.quarantine.orders_missing_customer")
```

## SQL (exam)

```sql
-- Add a write-time constraint. Existing rows must already satisfy it.
ALTER TABLE main.silver.orders
ADD CONSTRAINT valid_amount CHECK (amount >= 0);
```

```sql
-- Create a quarantine table from invalid rows.
CREATE OR REPLACE TABLE main.quarantine.orders_invalid AS
SELECT
  *,
  'invalid required field or amount' AS quarantine_reason,
  CURRENT_TIMESTAMP() AS quarantined_at
FROM main.bronze.orders_raw
WHERE order_id IS NULL
   OR CAST(amount_raw AS DECIMAL(12,2)) < 0
   OR CAST(amount_raw AS DECIMAL(12,2)) IS NULL;
```

```sql
-- Duplicate key report.
SELECT order_id, COUNT(*) AS row_count
FROM main.silver.orders
GROUP BY order_id
HAVING COUNT(*) > 1;
```

## Lakeflow expectations

Lakeflow Spark Declarative Pipelines expectations let you attach data quality rules to pipeline datasets and track results in the pipeline event log and UI.

Current Lakeflow Python examples use the `dp` module:

```python
from pyspark import pipelines as dp

@dp.table
@dp.expect("valid_amount", "amount >= 0")
@dp.expect_or_drop("valid_order_id", "order_id IS NOT NULL")
def silver_orders():
    return spark.readStream.table("main.bronze.orders_raw")
```

Common expectation behaviors:

| Behavior | Meaning |
|----------|---------|
| Expect / warn | Record metric but keep rows |
| Expect or drop | Drop rows that violate the rule |
| Expect or fail | Fail the pipeline update on violation |

## Delta constraints

```sql
CREATE TABLE main.silver.products (
  product_id STRING NOT NULL,
  price DECIMAL(10,2),
  CONSTRAINT positive_price CHECK (price >= 0)
);
```

Important details:

- Check constraints fail writes that violate the condition.
- Adding a check constraint validates existing data first.
- Constraints require Delta tables.
- `PRIMARY KEY` and `FOREIGN KEY` constraints are useful metadata but are not enforced as uniqueness/referential checks.

## UI / product surfaces

- **Lakeflow Spark Declarative Pipelines UI**: expectation results, dropped records, failed updates, event log, lineage graph.
- **Lakeflow Jobs**: task failure, retry behavior, alerts, and run output for custom assertions.
- **Catalog Explorer**: table constraints, schema, lineage, comments, and permissions.
- **Spark UI / Query Profile**: cost of validation joins and duplicate checks.

## Exam traps

- Validating only in Power BI or dashboards after bad data has already reached gold.
- Silently dropping invalid rows with no metric, quarantine, or alert.
- Assuming Delta primary keys enforce uniqueness. They are informational.
- Adding a `CHECK` constraint to a table that already has violating rows and expecting it to succeed.
- Using `dropDuplicates` as a substitute for a real rule about which duplicate row should survive.
- Failing an entire pipeline for minor non-critical warnings when quarantine would satisfy the requirement better.
- Forgetting that quality checks should be re-run after transformations, joins, and aggregations, not only at ingestion.

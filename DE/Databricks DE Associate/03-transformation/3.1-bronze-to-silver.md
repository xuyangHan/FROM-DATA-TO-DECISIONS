# 3.1 Bronze to Silver cleaning

## Exam objective

> Implement data cleaning by reading bronze tables with PySpark/SQL, cleaning nulls, standardizing data types, and writing to new silver tables.

## Must know

- **Bronze** tables preserve raw or lightly validated source data. They are the audit-friendly landing layer.
- **Silver** tables are cleaned, validated, deduplicated, type-standardized, and easier for downstream users to consume.
- Do **not** overwrite bronze in place during cleaning. Read bronze and write a new silver table.
- Databricks recommends building silver from bronze rather than writing directly to silver from ingestion, because bronze isolates source drift and corrupt records from curated tables.
- Silver should usually include at least one non-aggregated, validated representation of each important source record.
- Common silver operations: null handling, type casting, timestamp parsing, string standardization, deduplication, schema normalization, joins, and quality checks.
- Use **Unity Catalog three-part names** such as `main.silver.orders` so governance, lineage, permissions, and discovery work.
- Use **Delta tables** for silver targets unless a question clearly states otherwise.

## Cleaning mental model

```
main.bronze.orders_raw
  -> keep raw fields available for audit
  -> filter or quarantine invalid records
  -> standardize strings, dates, timestamps, numbers, booleans
  -> deduplicate by business key
  -> add useful metadata columns
  -> write main.silver.orders
```

Bronze answers "what arrived?" Silver answers "what is usable and conformed?"

## Null handling

| Situation | Preferred action | Example |
|-----------|------------------|---------|
| Required key is null | Drop or quarantine row | `filter(col("order_id").isNotNull())` |
| Optional attribute is null | Keep null | `ship_date` for unshipped orders |
| Missing categorical value | Fill with known placeholder if accepted by business | `fillna({"country": "unknown"})` |
| Missing numeric measure | Usually keep null unless zero is truly meaningful | Do not turn unknown revenue into `0` casually |
| Null nested collection but row should remain | Use `explode_outer` later | Keeps parent row |

Exam shortcut: if the column is part of the primary/business key, a null usually makes the row invalid for silver.

## Type standardization

| Source issue | Cleaning operation |
|--------------|--------------------|
| String timestamp | `to_timestamp(col, pattern)` |
| String date | `to_date(col, pattern)` |
| Number stored as string | `cast("decimal(10,2)")`, `cast("int")`, `cast("double")` |
| Mixed casing | `lower(trim(col))` or `upper(trim(col))` |
| Extra whitespace | `trim`, `regexp_replace` |
| Boolean as `Y/N` or `1/0` | `when(...).otherwise(...)` |
| Invalid cast result | Produces null in many cases; filter, quarantine, or validate |

## PySpark / API essentials

```python
from pyspark.sql.functions import col, current_timestamp, lower, regexp_replace, to_date, to_timestamp, trim, when

bronze = spark.table("main.bronze.orders_raw")

cleaned = (
    bronze
    .withColumn("order_id", trim(col("order_id")))
    .withColumn("email", lower(trim(col("email"))))
    .withColumn("order_ts", to_timestamp(col("order_ts_raw"), "yyyy-MM-dd HH:mm:ss"))
    .withColumn("order_date", to_date(col("order_date_raw"), "yyyy-MM-dd"))
    .withColumn("amount", regexp_replace(col("amount_raw"), "[$,]", "").cast("decimal(12,2)"))
    .withColumn(
        "is_priority",
        when(col("priority_flag").isin("Y", "1", "true"), True)
        .when(col("priority_flag").isin("N", "0", "false"), False)
    )
    .filter(col("order_id").isNotNull())
    .filter(col("order_ts").isNotNull())
    .filter(col("amount").isNotNull())
    .drop("order_ts_raw", "order_date_raw", "amount_raw", "priority_flag")
    .dropDuplicates(["order_id"])
    .withColumn("_silver_loaded_at", current_timestamp())
)

cleaned.write.format("delta").mode("overwrite").saveAsTable("main.silver.orders")
```

Use `mode("append")` for incremental silver loads. Use `mode("overwrite")` carefully for rebuilds, backfills, or exam examples where a full replacement is clearly intended.

## SQL (exam)

```sql
CREATE OR REPLACE TABLE main.silver.orders AS
SELECT
  TRIM(order_id) AS order_id,
  LOWER(TRIM(email)) AS email,
  TO_TIMESTAMP(order_ts_raw, 'yyyy-MM-dd HH:mm:ss') AS order_ts,
  TO_DATE(order_date_raw, 'yyyy-MM-dd') AS order_date,
  CAST(REGEXP_REPLACE(amount_raw, '[$,]', '') AS DECIMAL(12,2)) AS amount,
  CURRENT_TIMESTAMP() AS _silver_loaded_at
FROM main.bronze.orders_raw
WHERE order_id IS NOT NULL
  AND TO_TIMESTAMP(order_ts_raw, 'yyyy-MM-dd HH:mm:ss') IS NOT NULL
  AND CAST(REGEXP_REPLACE(amount_raw, '[$,]', '') AS DECIMAL(12,2)) IS NOT NULL;
```

## Valid vs invalid split

For production-style pipelines, do not silently drop data that may need investigation.

```python
staged = (
    bronze
    .withColumn("order_ts", to_timestamp(col("order_ts_raw"), "yyyy-MM-dd HH:mm:ss"))
    .withColumn("amount", col("amount_raw").cast("decimal(12,2)"))
)

valid = staged.filter("order_id IS NOT NULL AND order_ts IS NOT NULL AND amount IS NOT NULL")
invalid = staged.filter("order_id IS NULL OR order_ts IS NULL OR amount IS NULL")

valid.write.mode("overwrite").saveAsTable("main.silver.orders")
invalid.write.mode("append").saveAsTable("main.quarantine.orders_invalid")
```

## UI / product surfaces

- **Catalog Explorer**: compare bronze and silver schemas, owners, comments, lineage, permissions, and table history.
- **Notebook display / data profile**: inspect null rates, distributions, and invalid values before deciding cleanup rules.
- **Lakeflow Jobs**: schedule and monitor recurring bronze-to-silver notebooks or SQL tasks.
- **Lakeflow Spark Declarative Pipelines**: define streaming tables, materialized views, and expectations for managed pipelines.

## Exam traps

- Overwriting bronze and losing the original source record.
- Writing silver only to a file path instead of a Unity Catalog table.
- Filling every null with `0` or `"unknown"` without checking whether that changes meaning.
- Casting strings to dates/numbers and ignoring the new nulls created by bad values.
- Using `inferSchema` repeatedly for curated silver tables instead of standardizing an explicit schema.
- Deduplicating on all columns when the business key is just `order_id`.
- Treating silver as gold: heavy BI aggregates usually belong in gold, not the core cleaned silver table.

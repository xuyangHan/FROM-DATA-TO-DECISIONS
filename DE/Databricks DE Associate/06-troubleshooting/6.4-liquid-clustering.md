# 6.4 Liquid clustering & predictive optimization

## Exam objective

> Understand the features of Liquid Clustering and predictive optimization.

## Must know

- **Liquid clustering** is a Delta table layout optimization that organizes data by clustering keys to improve data skipping and query performance.
- Databricks recommends liquid clustering for new Delta tables in many workloads instead of older layout strategies such as heavy partitioning or Z-ordering.
- Liquid clustering is defined with `CLUSTER BY`.
- Liquid clustering is **not the same as partitioning**. It does not create a rigid directory structure by partition column.
- Liquid clustering is **incremental**. `OPTIMIZE` reclusters only data that needs clustering unless you force full reclustering.
- You can change clustering keys over time more flexibly than traditional partitioning.
- You can specify up to **four** clustering keys when choosing keys manually.
- Good clustering keys are columns frequently used in filters, especially on large or fast-growing tables.
- **Predictive optimization** automatically runs maintenance operations for Unity Catalog managed tables.
- Predictive optimization can run `OPTIMIZE`, `VACUUM`, and `ANALYZE`.
- Predictive optimization uses serverless jobs compute and may create billable usage.
- Predictive optimization supports **Unity Catalog managed tables**, not external tables.

## Why liquid clustering matters

Data layout affects how much data a query must scan.

Without useful layout:

```sql
SELECT *
FROM main.silver.events
WHERE event_date = DATE '2026-05-21'
  AND country = 'CA';
```

Spark may need to scan many files to find matching rows.

With useful clustering:

- Rows with similar clustering key values are colocated more effectively.
- Delta data skipping can avoid more irrelevant files.
- Queries filtering on common keys can read less data.

## Liquid clustering vs partitioning vs Z-order

| Feature | What it does | Best for | Exam note |
|---------|--------------|----------|-----------|
| Partitioning | Creates physical directory partitions by column values | Low-cardinality, stable partition columns on very large tables | Can create small files if high cardinality |
| Z-order | Rewrites files to colocate related values across columns | Older Delta layout pattern | Liquid clustering is preferred for new Databricks table layout scenarios |
| Liquid clustering | Adaptive clustering by keys without rigid partition directories | Large, growing, skewed, or changing-access-pattern tables | Use `CLUSTER BY`; run `OPTIMIZE` or predictive optimization |
| Predictive optimization | Automatically runs table maintenance | UC managed tables | Runs `OPTIMIZE`, `VACUUM`, `ANALYZE` |

Do not combine every layout feature. Liquid clustering is not compatible with partitioning or Z-ordering on the same table.

## When to use liquid clustering

Good candidates:

- Large tables queried repeatedly by the same filter columns.
- Fast-growing Delta tables.
- Tables with high-cardinality filter columns where partitioning would create too many directories.
- Tables with data skew.
- Tables with changing access patterns.
- Tables where traditional partitioning would produce partitions that are too small or too large.
- Streaming tables and materialized views where liquid clustering is supported.

Less useful candidates:

- Tiny tables where layout does not matter.
- Tables rarely queried.
- Tables where queries always scan all rows.
- Tables already well-served by simple partitioning and not changing.

## Choosing clustering keys

Choose keys based on query filters, not based on what "looks like a primary key."

Good choices:

- `event_date` for time-window queries.
- `customer_id` if many queries filter by customer and the table is large.
- `country` plus `event_date` if dashboards commonly filter by both.
- `store_id` for store-level operational queries.

Risky choices:

- A column never used in filters.
- Too many keys on smaller tables.
- Columns with poor statistics.
- Highly correlated columns where one key would be enough.
- Unique identifiers if queries almost never filter by them.

Exam shortcut: **cluster by columns used in common filters**, especially when partitioning would be too rigid or high-cardinality.

## SQL (exam)

Create a clustered table:

```sql
CREATE TABLE main.silver.events (
  event_id STRING,
  event_ts TIMESTAMP,
  event_date DATE,
  country STRING,
  customer_id STRING,
  amount DECIMAL(12, 2)
)
USING DELTA
CLUSTER BY (event_date, country);
```

Enable clustering on an existing unpartitioned table:

```sql
ALTER TABLE main.silver.events
CLUSTER BY (event_date, country);
```

Use automatic liquid clustering where supported:

```sql
CREATE TABLE main.silver.events_auto (
  event_id STRING,
  event_ts TIMESTAMP,
  event_date DATE,
  country STRING,
  customer_id STRING
)
USING DELTA
CLUSTER BY AUTO;
```

Trigger clustering/compaction manually:

```sql
OPTIMIZE main.silver.events;
```

Force full reclustering when needed and supported:

```sql
OPTIMIZE main.silver.events FULL;
```

Check table details:

```sql
DESCRIBE DETAIL main.silver.events;
```

## Predictive optimization

Predictive optimization automatically chooses tables that benefit from maintenance and queues maintenance operations.

It can run:

| Operation | Purpose |
|-----------|---------|
| `OPTIMIZE` | Improve file size/layout and trigger incremental clustering |
| `VACUUM` | Remove old unreferenced data files after the retention window |
| `ANALYZE` | Collect/update statistics for the optimizer |

Important constraints:

- Supports Unity Catalog managed tables.
- Does not support external tables.
- Can be enabled at account, catalog, or schema level.
- Catalog/schema settings can inherit from parent settings.
- Requires appropriate privileges: account admin for account, catalog owner for catalog, schema owner for schema.
- Uses serverless jobs compute for maintenance.

## Predictive optimization SQL

Enable at catalog level:

```sql
ALTER CATALOG main ENABLE PREDICTIVE OPTIMIZATION;
```

Disable at schema level:

```sql
ALTER SCHEMA main.silver DISABLE PREDICTIVE OPTIMIZATION;
```

Return to inherited setting:

```sql
ALTER SCHEMA main.silver INHERIT PREDICTIVE OPTIMIZATION;
```

Check status:

```sql
DESCRIBE CATALOG EXTENDED main;
DESCRIBE SCHEMA EXTENDED main.silver;
DESCRIBE TABLE EXTENDED main.silver.events;
```

Track operations if the system table is available:

```sql
SELECT
  start_time,
  end_time,
  catalog_name,
  schema_name,
  table_name,
  operation_type,
  status
FROM system.storage.predictive_optimization_operations_history
ORDER BY start_time DESC;
```

## Manual maintenance vs predictive optimization

| Situation | Better answer |
|-----------|---------------|
| UC managed table and platform should handle maintenance | Enable predictive optimization |
| External table | Manual maintenance; predictive optimization does not apply |
| Need immediate reclustering after a large load | Run `OPTIMIZE` manually |
| Already using predictive optimization | Avoid duplicate scheduled `OPTIMIZE` jobs unless there is a clear special case |
| Need longer time travel retention | Set retention property before relying on `VACUUM` |

## PySpark / API essentials

SQL is usually clearer for the exam, but PySpark can create and maintain clustered tables through SQL calls:

```python
spark.sql("""
CREATE TABLE IF NOT EXISTS main.silver.events (
  event_id STRING,
  event_date DATE,
  country STRING,
  amount DECIMAL(12, 2)
)
USING DELTA
CLUSTER BY (event_date, country)
""")

spark.sql("OPTIMIZE main.silver.events")
```

## UI / product surfaces

- **Catalog Explorer**: inspect table metadata, ownership, and table properties.
- **Catalog/schema settings**: predictive optimization may be enabled/disabled or inherited.
- **SQL editor**: run `ALTER CATALOG`, `ALTER SCHEMA`, `DESCRIBE`, and `OPTIMIZE`.
- **System tables**: track predictive optimization operation history where available.

## Exam traps

- Partitioning by a high-cardinality column like `user_id` when liquid clustering is the better layout answer.
- Using `ZORDER` as the preferred new-table answer when the scenario points to liquid clustering.
- Thinking liquid clustering removes the need for `OPTIMIZE`; clustering is commonly triggered through `OPTIMIZE` or predictive optimization.
- Enabling predictive optimization for an external table.
- Confusing `OPTIMIZE` with `VACUUM`: `OPTIMIZE` compacts/reclusters; `VACUUM` deletes old unreferenced files.
- Forgetting predictive optimization can run `ANALYZE`, not only `OPTIMIZE`.
- Scheduling manual `OPTIMIZE` everywhere while predictive optimization is already managing the table.
- Choosing clustering keys because they are unique identifiers instead of because queries filter on them.

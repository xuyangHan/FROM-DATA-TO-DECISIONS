# 6.3 Spark UI bottlenecks

## Exam objective

> Identify common performance bottlenecks such as data skew, shuffling, and disk spilling by interpreting stage-level metrics in the Spark UI.

## Must know

- The Spark UI is where you investigate Spark task and stage behavior after the Jobs UI has identified a slow task or run.
- A Spark **job** is made of **stages**. A stage is a group of tasks that can run without a shuffle boundary.
- A **task** is one unit of work for one partition of data inside a stage.
- A long-running Spark job is usually diagnosed by finding the **longest stage** first.
- Stage-level metrics such as input, output, shuffle read, shuffle write, spill, task count, and task duration spread tell you what kind of bottleneck you have.
- **Shuffle** means data is redistributed across executors, usually after wide operations such as joins, aggregations, distinct, window functions, and repartition.
- **Skew** means one or a few partitions/tasks are much larger or slower than the rest.
- **Spill** means Spark could not keep intermediate data in memory and moved data to disk, which is much slower.
- Adding more workers can help when parallelism or total capacity is the issue, but it often does not fix skew.

## Spark UI workflow

1. Open the slow task/run from Lakeflow Jobs.
2. Open **Spark UI** or **query profile**, depending on task type and compute.
3. Find the Spark job or SQL query that took the most time.
4. Open the longest stage.
5. Compare stage I/O: **Input**, **Output**, **Shuffle Read**, and **Shuffle Write**.
6. Check number of tasks.
7. Check task duration distribution: median/75th percentile vs max.
8. Check memory and disk spill.
9. Connect the symptom to a likely cause before changing code or cluster size.

## Stage-level metrics

| Metric | What it means | What to suspect |
|--------|---------------|-----------------|
| Input | Data read from storage | Large source scan, missing filter, poor file pruning |
| Output | Data written to storage | Large write, row explosion, too many output files |
| Shuffle read | Data read from other executors after shuffle | Join, aggregation, repartition, distinct, window |
| Shuffle write | Data written for later shuffle read | Upstream wide transformation |
| Spill memory | Data spilled from memory during processing | Memory pressure or huge partitions |
| Spill disk | Data actually written to disk | Expensive memory pressure, often shuffle-heavy |
| Task count | Number of partitions/tasks in a stage | Too few tasks, too many tasks, or poor partitioning |
| Max task duration | Slowest task in stage | Skew if much higher than p75/median |
| GC time | Time spent in garbage collection | JVM memory pressure |

## Shuffle

Shuffle is required when Spark needs rows with the same key on the same executor.

Common shuffle operations:

- `groupBy`, `agg`, `countDistinct`, `dropDuplicates`
- Joins that are not broadcast joins
- `repartition`
- `distinct`
- Window functions with `PARTITION BY`
- Set operations that require deduplication

In a physical plan, shuffle often appears as an `Exchange`.

High shuffle is not automatically wrong, but it is a performance cost. The exam answer is usually to identify shuffle-heavy operations and then reduce shuffled data, broadcast a small side, tune partitions, or fix skew.

## Skew

Skew happens when data is unevenly distributed across partitions.

Spark UI evidence:

- One or a few tasks run much longer than the others in the same stage.
- Max task duration is much higher than median or 75th percentile.
- Some tasks read far more shuffle data than other tasks.
- Cluster utilization drops while one straggler task remains.

Common causes:

- A join key has a dominant value such as `unknown`, `null`, `US`, or one huge customer.
- Aggregating by a low-cardinality or heavily imbalanced key.
- Window function partitions are uneven.
- One file or partition is much larger than others.

Typical fixes:

| Situation | Fix |
|-----------|-----|
| Small dimension joined to large fact | Broadcast the small table |
| One hot key dominates | Filter separately, salt the key, or split hot-key processing |
| Null/default keys create huge group | Clean/filter/null-handle before join or aggregation |
| Window partition too large | Add a better partition key if business logic allows |
| AQE can help | Ensure adaptive query execution is enabled |

## Spill

Spill happens when Spark runs out of memory for intermediate data and moves data to disk.

Spark UI evidence:

- Stage page shows memory spill or disk spill.
- Long shuffle stage with high spill.
- Executors tab shows memory pressure or failed executors.
- GC time may be high.

Common causes:

- Too few shuffle partitions, causing huge per-task data.
- Large join/aggregation state.
- Bad skew causing one partition to be huge.
- Over-caching.
- Large broadcast that does not fit comfortably.
- UDFs or Python workers using too much memory.

Typical fixes:

| Situation | Fix |
|-----------|-----|
| Huge partitions spill | Increase `spark.sql.shuffle.partitions` or repartition appropriately |
| Broadcast causes memory pressure | Lower broadcast threshold or remove broadcast hint |
| Cache consumes memory | Unpersist unused DataFrames/tables |
| Skew creates spill | Fix skew, not just executor memory |
| Data is larger than expected | Filter/project earlier and avoid row explosion |
| Memory truly insufficient | Increase executor memory or choose memory-optimized workers |

## One task vs many tasks

Task count is a simple but important clue.

| Spark UI clue | Meaning |
|---------------|---------|
| One task in a long stage | Not enough parallelism, single file/partition, `coalesce(1)`, or non-parallel operation |
| Few very large tasks | Too few partitions; each task has too much data |
| Thousands of tiny tasks | Too many partitions or many tiny files; scheduling overhead |
| Many tasks but one straggler | Skew |
| Many tasks fail repeatedly | Data-dependent error, executor OOM, bad UDF, or flaky infrastructure |

## Bottleneck decision table

| Spark UI observation | Likely bottleneck | First response |
|----------------------|-------------------|----------------|
| Huge shuffle read/write | Wide join/aggregation/repartition | Inspect query plan and join/agg keys |
| One task much slower than others | Skew | Check key distribution; broadcast/salt/filter hot key |
| High disk spill | Memory pressure | Increase partitions, reduce data before shuffle, fix skew, or increase memory |
| High input, low shuffle | Storage scan / I/O bound | Check filters, partition pruning, file layout, table statistics |
| High output | Large write or row explosion | Check joins/explodes and output file count |
| One task only | Poor parallelism | Remove `coalesce(1)`, repartition, check source file layout |
| Many tiny tasks | Scheduling overhead | Reduce partitions or compact small files |
| Executor lost | OOM, spot loss, autoscaling, infrastructure | Check event log and Executors tab |

## PySpark / API essentials

Broadcast a small table:

```python
from pyspark.sql.functions import broadcast

orders = spark.table("main.silver.orders")
customers = spark.table("main.silver.customers")

result = orders.join(broadcast(customers), on="customer_id", how="left")
```

Tune shuffle partitions for a test run:

```python
spark.conf.set("spark.sql.shuffle.partitions", "400")
```

Remove unused cached data:

```python
cached_df.unpersist()
spark.catalog.clearCache()
```

Investigate key skew:

```python
from pyspark.sql.functions import desc

(
    orders
    .groupBy("customer_id")
    .count()
    .orderBy(desc("count"))
    .show(20, truncate=False)
)
```

Salt a skewed key only when needed:

```python
from pyspark.sql.functions import col, floor, rand

salted_orders = orders.withColumn("salt", floor(rand() * 16))

# Real salting needs both sides of the join adjusted correctly.
# Use it for known hot keys, not as a default pattern.
```

## SQL (exam)

Broadcast hint:

```sql
SELECT /*+ BROADCAST(c) */
  o.order_id,
  c.segment
FROM main.silver.orders AS o
LEFT JOIN main.silver.customers AS c
  ON o.customer_id = c.customer_id;
```

Repartition hint:

```sql
SELECT /*+ REPARTITION(200, order_date) */
  order_date,
  SUM(amount) AS revenue
FROM main.silver.orders
GROUP BY order_date;
```

Update table statistics when optimizer decisions seem poor:

```sql
ANALYZE TABLE main.silver.customers COMPUTE STATISTICS;
```

## Relation to 3.5 tuning

Section 3.5 covers basic knobs. In troubleshooting, the important skill is matching the knob to the metric.

| Metric tells you | Related tuning idea |
|------------------|---------------------|
| Huge shuffle partitions | Adjust `spark.sql.shuffle.partitions` |
| Small table not broadcast | Check stats or `spark.sql.autoBroadcastJoinThreshold` |
| Driver OOM | Avoid `collect()` / `toPandas()`; adjust `spark.driver.memory` only if needed |
| Executor OOM/spill | Fix partitioning/skew/cache; consider `spark.executor.memory` |
| Few tasks | Increase parallelism/repartition |
| Tiny tasks | Decrease partitions or compact files |

## Exam traps

- Adding workers for skew and expecting the straggler task to disappear.
- Increasing executor memory before checking whether one skewed key is the real cause.
- Lowering shuffle partitions because "fewer tasks sounds faster" and causing huge spills.
- Raising shuffle partitions blindly and creating thousands of tiny tasks.
- Confusing driver OOM with executor OOM.
- Using `collect()` or `toPandas()` and then blaming Spark shuffle.
- Caching every DataFrame and causing spill/OOM.
- Looking at job-level duration only and never opening the longest stage.
- Assuming all shuffles are bad; some are required, but expensive shuffles should be understood.

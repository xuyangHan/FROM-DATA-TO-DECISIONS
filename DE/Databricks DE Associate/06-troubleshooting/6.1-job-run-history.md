# 6.1 Job run history trends

## Exam objective

> Identify trends in job performance using the Lakeflow Jobs run history view to compare current execution times against historical baselines.

## Must know

- **Run history** is the first place to answer: "Is this job slower than normal, or was this one run unusual?"
- The Lakeflow Jobs **Runs** tab shows current and completed runs for a job, with matrix and list views.
- The run history matrix includes each task in the job, so you can compare total job duration and individual task duration across runs.
- Databricks keeps job run history in the Jobs UI for up to **60 days**. Preserve important run output or logs before that window expires.
- A good baseline compares successful runs over a meaningful window, such as the last 7 days, last 10 runs, or the same weekday/time window.
- Compare both **total run duration** and **task duration**. A job can be slow because one task regressed, because tasks waited for upstream dependencies, or because compute was slow to start.
- A single slow run is not automatically a Spark tuning problem. First decide whether the pattern is isolated, gradual, sudden, or tied to failures.
- A run details page can link from the job-level view into task output, logs, query profiles, or Spark UI, depending on task type and compute.

## What to compare

| Metric | Why it matters |
|--------|----------------|
| Run status | Separate successful slow runs from failed or repaired runs |
| Total duration | Measures user-facing job completion time |
| Task duration | Finds the task that changed inside a multi-task job |
| Queue / setup time | Points to concurrency, compute acquisition, policy, quota, or cluster startup issues |
| Retry attempts | A "successful" run can be slow because failed attempts were retried |
| Trigger type | Scheduled, file-arrival, table-update, manual, or API-triggered runs may have different data volumes |
| Run as / code version | Helps connect regression to deployment, permissions, or identity changes |
| Input volume | More data can explain longer runtime without a code regression |

## Baseline patterns

| Pattern in run history | Likely interpretation | Next step |
|------------------------|-----------------------|-----------|
| One isolated slow run | Transient infra issue, temporary source slowness, spot loss, or one-time data spike | Open that run and compare task details/logs |
| Sudden step-change after a date | New code, config, library, cluster, trigger, or table layout change | Check recent deploys and job edits |
| Gradual increase over weeks | Data growth, accumulating small files, stale layout, or source growth | Check input size, table maintenance, and task metrics |
| Failures clustered at same time | Upstream system, quota, permissions, or schedule overlap | Inspect first failed task and external dependencies |
| Same task always dominates | Critical path or task-level performance bottleneck | Deep dive into that task with Spark UI or query profile |
| Slow only when triggered by data | Larger file batches or table updates | Compare file counts, record counts, and trigger backlog |

## Job-level vs task-level trend

For a multi-task job, do not stop at total duration.

Example:

```text
Run A total duration: 40 min
Run B total duration: 65 min
```

The total got worse, but the reason might be:

| Possibility | Evidence |
|-------------|----------|
| One task got slower | Same task duration increased across runs |
| A task waited longer | Task start time shifted because upstream task finished late |
| More branches ran | Conditional branch or For each iterations changed |
| Repair/retry included | Run duration includes failed attempt plus repair time |
| Compute startup delayed | Large gap before the first task actually executed |

The exam often tests this distinction with a downstream task marked **Skipped** or a total run that is slow because an upstream task is the actual blocker.

## Practical run-history workflow

1. Open **Jobs & Pipelines > Jobs**.
2. Select the job.
3. Open the **Runs** tab.
4. Use the matrix or list view to compare recent successful and failed runs.
5. Identify whether the abnormal duration is in the full run or in a specific task.
6. Open the abnormal run.
7. Check graph/timeline/list view for task start time, end time, duration, status, and retries.
8. Open the slow or failed task's output, logs, query profile, or Spark UI.
9. Compare with a known-good run, preferably with similar input volume.

## What "historical baseline" means

A baseline is not one random previous run. It should be comparable.

Better baselines:

- Last 10 successful production runs with the same trigger.
- Same weekday/time, if source volume has weekly seasonality.
- Last run before and after a deployment.
- A run with similar input row count or file count.
- Median or 75th percentile duration, not only the fastest run.

Weak baselines:

- Comparing dev and prod runs with different compute sizes.
- Comparing a tiny manual test run to a full scheduled production run.
- Comparing a repaired run to a clean run without noticing the repair.
- Comparing total duration without checking task duration.

## Trend calculations

You do not need advanced math for the exam, but you should be comfortable reasoning like this:

```text
Baseline: 12 min average over last 10 successful scheduled runs
Current: 22 min successful run
Change: 83% slower
```

That is a performance regression worth investigating.

```text
Baseline: 12 min
Current: 14 min
Input records: doubled
```

That might be acceptable scaling, depending on SLA.

## UI / product surfaces

- **Jobs & Pipelines > Jobs**: job list and job details.
- **Runs tab**: matrix/list view of active and completed runs.
- **Run total duration row**: total duration and state for each run in matrix view.
- **Run details**: graph, timeline, and list views for multi-task jobs.
- **Task run details pane**: task status, start/end time, duration, output, logs, metrics, cluster, and source.
- **Go to latest successful run**: useful reference point for baseline comparison.
- **Spark UI / query profile**: deeper task-level diagnosis after the Jobs UI identifies the suspicious task.

## PySpark / API essentials

The objective is UI-first, but instrumentation helps make run-history comparisons useful.

```python
import time

start = time.time()

# ETL logic here
rows_written = silver_df.count()

duration_sec = round(time.time() - start, 2)

dbutils.jobs.taskValues.set(key="rows_written", value=rows_written)
dbutils.jobs.taskValues.set(key="duration_sec", value=duration_sec)

print(f"rows_written={rows_written}, duration_sec={duration_sec}")
```

Do not overuse `count()` in production just for logging if it adds expensive extra actions. Prefer metrics already produced by your pipeline, table history, or task output when available.

## SQL (exam)

Jobs UI is the main exam surface. SQL/system tables are useful awareness, especially for account-wide monitoring:

```sql
SELECT
  workspace_id,
  job_id,
  run_id,
  period_start_time,
  period_end_time,
  result_state
FROM system.lakeflow.job_run_timeline
WHERE job_id = 123456789
ORDER BY period_start_time DESC;
```

Exact system-table columns can vary by feature availability and cloud rollout. For the associate exam, know that system tables can support monitoring, but choose the Jobs UI when the question says "run history view."

## Exam traps

- Treating a single slow run as a trend without a baseline.
- Comparing a failed or repaired run to clean successful runs without noticing retries/repair.
- Blaming a downstream task when the upstream task failed or finished late.
- Ignoring compute acquisition/startup time and only tuning Spark code.
- Comparing runs with different trigger types, input sizes, or compute configurations.
- Looking only at total job duration and missing the one task on the critical path.
- Assuming a run marked successful was healthy; it might be slower because of retries.

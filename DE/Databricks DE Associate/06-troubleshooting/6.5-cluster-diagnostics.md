# 6.5 Cluster failures & OOM

## Exam objective

> Diagnose cluster startup failures, library conflicts, and out-of-memory issues.

## Must know

- Cluster and task failures can be caused by infrastructure, configuration, libraries, permissions, code, data volume, or memory pressure.
- Start diagnosis from the highest-level symptom, then move inward:
  1. Lakeflow Jobs run status.
  2. Task run details.
  3. Compute event log.
  4. Driver logs / worker logs.
  5. Spark UI stage and executor details.
- The **compute event log** captures lifecycle events such as cluster creation, start, termination, resize, and configuration changes.
- **Init script** start/finish events appear in cluster event logs; detailed init script logs are in cluster logs when log delivery is configured, or under `/databricks/init_scripts`.
- **Driver logs** contain notebook/job output, Python/Scala exceptions, and driver-side errors.
- **Worker/executor logs** help diagnose failed tasks, executor exits, and executor OOM.
- **Driver OOM** is different from **executor OOM**. Use the symptom and logs to decide which side is out of memory.
- Library conflicts often come from incompatible package versions, runtime mismatch, duplicate JARs, or cluster-scoped libraries that require restart/session refresh.

## Diagnostic map

| Symptom | First place to check | Likely causes |
|---------|----------------------|---------------|
| Cluster will not start | Compute event log | Policy violation, quota/capacity, cloud permissions, init script failure, bad Spark config |
| Job task fails before user code | Task run details and compute logs | Cluster startup, library installation, repo checkout, permissions |
| `LIBRARY_INSTALLATION_ERROR` | Libraries tab and task logs | Bad package, missing repository, incompatible runtime, network/repo access |
| Python `ImportError` / `ModuleNotFoundError` | Notebook output and cluster Libraries tab | Package not installed in active session, wrong environment, library install failed |
| Java `NoSuchMethodError` / `ClassNotFoundException` | Driver/executor logs | JAR version conflict or missing connector |
| Driver lost / driver OOM | Driver logs and Spark UI | `collect()`, `toPandas()`, huge broadcast metadata, too much data returned to driver |
| Executor lost / executor OOM | Spark UI Executors tab, failed task logs, event log | Skew, spill, cache pressure, large partition, UDF memory, spot loss |
| Tasks fail after several retries | Failed stage details | Data-dependent error, OOM, bad file, flaky external service |

## Cluster startup failures

Cluster startup failures happen before or during compute initialization.

Common causes:

- Invalid cluster policy setting.
- Cloud instance quota/capacity issue.
- Spot/preemptible capacity unavailable or reclaimed.
- Instance profile / service credential / storage permission problem.
- Invalid Spark config.
- Init script failed.
- Library install failed during startup.
- Unsupported runtime for a configured feature or library.
- Network restriction prevents access to package repository or storage.

## Startup diagnosis workflow

1. Open **Compute** and select the failed cluster, or open the failed job task's compute link.
2. Open the **Event log** tab.
3. Find the first error or warning around the failed startup time.
4. If an init script is involved, find `INIT_SCRIPTS_STARTED` and `INIT_SCRIPTS_FINISHED`.
5. Open cluster logs for init script stdout/stderr if available.
6. Check the **Libraries** tab if startup failed during library installation.
7. Check the job/task run details for termination codes such as cluster error, invalid cluster request, library installation error, or unauthorized error.

Exam shortcut: when a cluster fails after an init script change, check the **compute event log** first, then init script logs.

## Init script issues

Init scripts are powerful but easy to break.

Common problems:

- Script path no longer exists or is inaccessible.
- Script references files not available on the cluster.
- Syntax error or non-zero exit code.
- Installing system packages not compatible with runtime image.
- Long-running install that times out.
- Script works on driver but fails on workers.
- Permissions on workspace files, UC volumes, or cloud storage are missing.

Good practice:

- Store init scripts in recommended locations, such as Unity Catalog volumes or workspace files depending on runtime and workspace setup.
- Keep init scripts small and deterministic.
- Prefer cluster libraries or policy-managed libraries when possible.
- Test changes on dev compute before production jobs use them.

## Library conflicts

Library conflicts can break startup or fail at runtime.

Common symptoms:

| Symptom | Meaning |
|---------|---------|
| `ModuleNotFoundError` | Python package is not installed in the active environment/session |
| `ImportError` | Package installed but incompatible or missing native dependency |
| `NoSuchMethodError` | JVM/JAR version mismatch |
| `ClassNotFoundException` | Missing JAR/class or connector not installed |
| `LIBRARY_INSTALLATION_ERROR` | Install failed before task code ran |
| Works in one cluster, fails in another | Different runtime, access mode, or library set |

## Library diagnosis workflow

1. Check whether the library is cluster-scoped, notebook-scoped, job-scoped, bundle-defined, or policy-enforced.
2. Open the cluster **Libraries** tab.
3. Check install status and version.
4. Confirm the Databricks Runtime version is compatible with the package.
5. Restart the cluster or start a new notebook session when needed.
6. Remove duplicate or conflicting versions.
7. For JARs/connectors, align Spark/Scala/connector versions.
8. For production, pin versions in bundle/job configuration rather than relying on manual cluster state.

Important details:

- A notebook already attached to a cluster may not see a newly installed cluster library until a new session starts.
- Uninstalling a cluster library is completed on cluster restart; until then it can show as pending restart.
- Policies can enforce libraries and prevent user-managed install/uninstall.
- DBFS root library storage is deprecated/disabled in newer runtimes for many cases; prefer workspace files, Unity Catalog volumes, package repositories, or cloud object storage.

## Driver OOM

The driver coordinates the Spark application. It should not receive all data.

Common driver OOM causes:

- `df.collect()` on a large DataFrame.
- `df.toPandas()` on data larger than driver memory.
- `display()` or `show()` on too much data.
- Creating a huge Python list on the driver.
- Collecting distinct keys to loop in Python.
- Very large query plan or metadata operations.
- Broadcasting a table that is too large.

Fix patterns:

| Bad pattern | Better pattern |
|-------------|----------------|
| `df.collect()` on millions of rows | Write to Delta or aggregate in Spark |
| `toPandas()` on large data | Sample, limit, aggregate, or use pandas API on Spark where appropriate |
| Loop over collected keys | Use distributed joins/grouping |
| Build one huge Python object | Keep computation in Spark DataFrames |
| Increase workers for driver OOM | Reduce driver-side data or increase driver memory if justified |

## Executor OOM

Executors run distributed tasks. Executor OOM is usually about per-task data size or memory-heavy operations.

Common executor OOM causes:

- Skewed partitions.
- Too few shuffle partitions.
- Heavy aggregation or join state.
- Large broadcast variable.
- Over-caching.
- Memory-heavy UDFs.
- Window function with huge partitions.
- Streaming state too large.
- Explode or join accidentally multiplies rows.

Fix patterns:

| Cause | Better response |
|-------|-----------------|
| Skew | Fix hot keys, salting, broadcast small side, AQE |
| Huge partitions | Increase shuffle partitions or repartition by better key |
| Too much cache | Unpersist and cache only reused data |
| Large broadcast | Disable/reduce broadcast or update table stats |
| UDF memory | Replace with built-in Spark functions when possible |
| Window OOM | Add/adjust `PARTITION BY` if business logic allows |
| True memory shortage | Use memory-optimized workers or more executor memory |

## Driver vs executor OOM

| Clue | Driver OOM | Executor OOM |
|------|------------|--------------|
| Error mentions driver lost | Common | Possible but less direct |
| Error mentions executor lost | Less likely | Common |
| Code uses `collect()` / `toPandas()` | Strong signal | Not primary |
| Spark UI shows failed tasks in one stage | Possible | Strong signal |
| One task has huge input/spill | Less likely | Strong signal |
| Increasing workers helps | Usually no | Sometimes |
| Increasing driver memory helps | Sometimes | Usually no |
| Increasing executor memory helps | Usually no | Sometimes |

Exam shortcut: `collect()` on a large DataFrame causes **driver OOM**. Skew/spill/large partitions usually cause **executor OOM**.

## PySpark / API essentials

Avoid driver collection:

```python
# Bad: returns all rows to the driver.
rows = df.collect()

# Better: keep computation distributed.
(
    df.groupBy("country")
      .count()
      .write
      .mode("overwrite")
      .saveAsTable("main.gold.country_counts")
)
```

Use limits only for inspection, not full extraction:

```python
df.select("order_id", "amount").limit(20).show(truncate=False)
```

Check for skew before assuming the cluster is too small:

```python
from pyspark.sql.functions import desc

(
    df.groupBy("customer_id")
      .count()
      .orderBy(desc("count"))
      .show(20, truncate=False)
)
```

Unpersist cached data:

```python
df_cached.unpersist()
spark.catalog.clearCache()
```

Temporarily test whether broadcast is causing memory pressure:

```python
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)
```

Do not leave emergency settings in production without measuring the result.

## SQL (exam)

Avoid returning huge results to the client/driver:

```sql
-- Better than SELECT * into a notebook driver.
CREATE OR REPLACE TABLE main.gold.daily_country_sales AS
SELECT
  order_date,
  country,
  SUM(amount) AS revenue
FROM main.silver.orders
GROUP BY order_date, country;
```

Check skew with SQL:

```sql
SELECT customer_id, COUNT(*) AS row_count
FROM main.silver.orders
GROUP BY customer_id
ORDER BY row_count DESC
LIMIT 20;
```

## UI / product surfaces

- **Jobs run details**: high-level task failure, output, error message, and compute link.
- **Compute > Event log**: cluster lifecycle events and startup/termination clues.
- **Compute > Driver logs**: stdout, stderr, and log4j driver logs.
- **Spark UI > Executors**: executor failures, memory, logs, task failures.
- **Spark UI > Stages**: failed stages, spill, skew, task duration spread.
- **Compute > Libraries**: installed packages, versions, status, pending restart.
- **Compute > Metrics**: CPU, memory, JVM heap, shuffle read/write, failed/completed tasks.

## Exam traps

- Increasing workers for driver OOM caused by `collect()`.
- Increasing driver memory for executor OOM caused by skew.
- Treating a skipped downstream task as the root failure.
- Ignoring the compute event log for startup failures.
- Debugging application code before checking library installation errors.
- Forgetting a library uninstall may require restart.
- Assuming a package installed on one cluster exists on another cluster.
- Fixing OOM only by adding memory when the real issue is row explosion, skew, or bad partitioning.
- Confusing init script event log entries with full init script stdout/stderr.
- Ignoring cloud quota, capacity, and policy restrictions when clusters cannot start.

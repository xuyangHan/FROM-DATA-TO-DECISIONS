# 1.1 Platform core components

## Exam objective

> Understand the core components of the Databricks Data Intelligence Platform, such as its architecture, Delta Lake, and Unity Catalog.

## Must know

- **Data Intelligence Platform**: unified platform for ETL, analytics, BI, ML, and AI on a lakehouse foundation.
- **Lakehouse**: cloud object storage + open table formats + compute engines + governance. It keeps data in low-cost object storage while adding warehouse-style reliability and management.
- **Control plane**: Databricks-managed services such as the workspace UI, notebooks, jobs, cluster management, query planning services, metadata services, and account/workspace administration.
- **Data plane**: compute resources and data storage in the customer's cloud account, where Spark jobs and SQL queries process data.
- **Workspace**: the working environment for notebooks, Lakeflow Jobs, repos/Git folders, clusters, SQL warehouses, dashboards, and collaboration.
- **Account**: higher-level administration for identity, billing, workspaces, metastores, and account-level settings.
- **Delta Lake**: default storage layer for Databricks tables. It stores Parquet data files plus a transaction log (`_delta_log`) that enables ACID transactions, scalable metadata, time travel, schema enforcement, batch, and streaming.
- **Unity Catalog (UC)**: centralized governance layer for data and AI assets. It manages permissions, discovery, lineage, auditability, sharing, and access to tables, views, volumes, functions, and models.
- **Three-level namespace**: `catalog.schema.object`, such as `main.silver.orders`. For tables, the object is usually a table or view.
- **Volumes**: Unity Catalog-governed storage for non-tabular files. Use volumes for files such as JSON drops, images, PDFs, model artifacts, and other file-based data.
- **Delta Sharing**: open protocol for secure live sharing of data and AI assets across organizations, often governed through Unity Catalog.

## Architecture mental model

```
Databricks account
  -> workspaces
  -> Unity Catalog metastore
       -> catalogs
       -> schemas
       -> tables, views, volumes, functions, models

Cloud object storage
  -> Delta data files: Parquet
  -> Delta transaction log: _delta_log

Compute
  -> all-purpose clusters, job clusters, serverless compute, SQL warehouses
  -> reads and writes governed assets through Unity Catalog
```

For the exam, think in layers:

| Layer | Purpose | Examples |
|-------|---------|----------|
| Storage | Durable files in cloud object storage | S3, ADLS, GCS |
| Table format | Reliability on top of files | Delta Lake |
| Governance | Permissions, lineage, discovery, audit | Unity Catalog |
| Compute | Runs SQL, PySpark, jobs, pipelines | Clusters, SQL warehouses |
| Orchestration | Schedules and coordinates work | Lakeflow Jobs |

## Delta Lake essentials

Delta Lake is what turns raw files in object storage into reliable lakehouse tables.

| Feature | Why it matters |
|---------|----------------|
| ACID transactions | Writes succeed completely or fail without corrupting table state |
| Transaction log | Records table versions, metadata, files added/removed, and commit history |
| Time travel | Query older table versions for debugging or reproducibility |
| Schema enforcement | Rejects writes that do not match the table schema, unless schema evolution is enabled |
| `MERGE` | Supports upserts, CDC, and SCD-style patterns |
| Batch + streaming | Same Delta table can be used by batch jobs and Structured Streaming |
| `OPTIMIZE` | Compacts and lays out data files to improve query performance |
| `VACUUM` | Removes old files no longer needed by the current table state or retention period |

## Unity Catalog essentials

Unity Catalog is the default answer when a question mentions centralized governance, cross-workspace permissions, lineage, audit, discovery, secure file access, or the `catalog.schema.table` naming pattern.

| UC object | What it is |
|-----------|------------|
| Metastore | Top-level regional container for metadata and governance |
| Catalog | Top-level data organization unit inside a metastore |
| Schema | Database-like container inside a catalog |
| Table | Structured data object, usually Delta for Databricks-managed tables |
| View | Saved query with permissions applied through UC |
| Volume | Governed location for non-tabular files |
| Function | Governed reusable SQL/Python function |
| External location | Governed path in cloud storage, backed by a storage credential |

## Managed vs external assets

| Asset type | Storage lifecycle | Choose when |
|------------|-------------------|-------------|
| Managed table | Unity Catalog manages metadata and underlying data files | New Databricks-first tables; simplest governance and maintenance |
| External table | UC manages access, but data files live in a cloud location you manage | Existing cloud data, shared access with non-Databricks tools, custom lifecycle needs |
| Managed volume | UC manages the storage location for files | New governed file storage inside Databricks |
| External volume | UC governs an existing cloud storage path | Existing files that need Databricks access control and audit |

Exam shortcut: **managed** means Databricks/UC controls the data lifecycle; **external** means UC governs access, but you still manage the underlying cloud storage layout and lifecycle.

## When to use what

| Need | Choose |
|------|--------|
| Governed table access with SQL grants | Unity Catalog |
| Reliable ETL table storage | Delta table |
| Upsert changed records | Delta `MERGE` |
| Reproduce or debug older results | Delta time travel |
| Govern files instead of tables | Unity Catalog volume |
| Share live data without copying | Delta Sharing |
| Browse assets, grants, and lineage | Catalog Explorer |
| Improve small-file performance | `OPTIMIZE`, or predictive optimization for UC managed tables |

## PySpark / API essentials

```python
# Read a governed table
orders = spark.table("main.bronze.orders")

# Write a managed Delta table in Unity Catalog
orders.write.mode("overwrite").saveAsTable("main.silver.orders")

# Query a prior Delta version
old_orders = spark.read.option("versionAsOf", 10).table("main.silver.orders")

# Use SQL from PySpark for table maintenance
spark.sql("OPTIMIZE main.silver.orders")
```

## SQL (exam)

```sql
-- Fully qualified UC name
SELECT COUNT(*) FROM main.silver.orders;

-- Inspect table metadata and history
DESCRIBE DETAIL main.silver.orders;
DESCRIBE HISTORY main.silver.orders;

-- Time travel
SELECT * FROM main.silver.orders VERSION AS OF 10;
SELECT * FROM main.silver.orders TIMESTAMP AS OF '2026-01-01T00:00:00Z';

-- Upsert with Delta Lake
MERGE INTO main.silver.customers AS target
USING main.bronze.customer_updates AS source
ON target.customer_id = source.customer_id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

-- Basic UC permissions pattern
GRANT SELECT ON TABLE main.silver.orders TO `data_analysts`;
```

## UI / product surfaces

- **Catalog Explorer**: browse catalogs, schemas, tables, views, volumes, lineage, owners, tags, and permissions.
- **Workspace**: notebooks, files, repos/Git folders, jobs, dashboards, and user collaboration.
- **Compute**: clusters, SQL warehouses, policies, libraries, logs, and Spark UI.
- **Workflows / Lakeflow Jobs**: scheduled jobs, task DAGs, retries, triggers, and job clusters.
- **SQL Editor / Databricks SQL**: SQL queries, dashboards, alerts, and SQL warehouses.

## Exam traps

- A **data lake** is mostly files in object storage; a **lakehouse** adds table reliability, governance, and compute access patterns.
- Delta Lake is not a separate database server. It is a table format/storage layer on cloud object storage.
- Do not edit Delta table data files or `_delta_log` files directly.
- `VACUUM` can remove files needed for old time-travel queries if they are outside the retention window.
- Unity Catalog permissions apply to securable objects; cloud IAM alone is not the Databricks governance model for UC tables.
- Use the full `catalog.schema.table` name when working with governed objects.
- DBFS root is not the preferred production storage pattern for new governed workloads; use Unity Catalog tables and volumes.
- Managed and external tables can both be governed by UC, but only managed tables have the data lifecycle managed by UC.

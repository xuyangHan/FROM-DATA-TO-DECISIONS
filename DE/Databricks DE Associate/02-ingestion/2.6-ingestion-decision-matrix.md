# 2.6 Ingestion method prioritization

## Exam objective

> Prioritize between Auto Loader, Lakeflow Connect (standard and managed connectors), partner connectors, and other ingestion methods based on technical requirements such as data volume, ingestion frequency, data types, and governance needs with Unity Catalog.

## Must know

- Start with the **source type**: cloud files, SaaS app, database, message bus, local file, or custom API.
- Then check the **arrival pattern**: one-time, scheduled batch, incremental batch, near-real-time, or continuous streaming.
- Then apply the **most managed option that fits**: managed Lakeflow Connect > Lakeflow Spark Declarative Pipelines / standard connector > Structured Streaming / Auto Loader > custom notebook.
- **Unity Catalog-governed Delta tables** are the default landing target for exam scenarios that mention lineage, ACLs, ownership, audit, or governance.
- **Auto Loader** is the default answer for high-scale incremental file ingestion from S3/ADLS/GCS/Volumes, especially with frequent arrivals or schema drift.
- **COPY INTO** is the SQL-first answer for simpler scheduled file ingestion with thousands of files, known file drops, and moderate scale.
- **Lakeflow Connect managed connectors** are the default answer for supported SaaS apps and databases when minimal custom code is desired.
- **Lakeflow Connect standard connectors** are for sources that need more customization through Structured Streaming, Lakeflow Spark Declarative Pipelines, or SQL streaming tables.
- **JDBC/REST notebooks** are fallback patterns for custom sources or unsupported connectors, and should be scheduled with Lakeflow Jobs.
- **Partner/community connectors** are acceptable when first-party Databricks connectors do not support the source or the scenario explicitly requires a third-party tool.
- **Local file upload** is for one-time/small file loads or experimentation, not recurring production ingestion.

## Decision Algorithm

Use this in exam questions:

1. Is it a supported SaaS app or database connector?
   Choose **Lakeflow Connect managed connector**.

2. Is it cloud object storage files?
   Choose **Auto Loader** for high volume/frequent/schema drift; choose **COPY INTO** for simple SQL scheduled batch.

3. Is it a streaming/message source such as Kafka, Kinesis, Pub/Sub, or Pulsar?
   Choose **standard connector** with Structured Streaming or Lakeflow Spark Declarative Pipelines.

4. Is it a one-off local file?
   Use **Add data UI** or upload to a **Unity Catalog volume**.

5. Is it unsupported JDBC/REST/custom API?
   Use **notebook/Python task + Lakeflow Job**, landing to UC Delta or raw files plus Auto Loader.

6. Does the question mention governance, lineage, ACLs, or `catalog.schema.table`?
   Ensure the method writes to a **Unity Catalog table**, not only DBFS or unregistered cloud paths.

## Fast Decision Matrix

| Requirement or clue | Best fit | Why |
|---------------------|----------|-----|
| Salesforce, Workday, HubSpot, Jira, NetSuite, ServiceNow, Google Analytics | Lakeflow Connect managed SaaS connector | Databricks-managed ingestion, auth, retries, monitoring |
| MySQL/PostgreSQL/SQL Server with CDC available | Lakeflow Connect managed database CDC connector | Captures database changes incrementally |
| Database has cursor column but CDC is unavailable | Lakeflow Connect query-based connector | Uses high-water mark/cursor on schedule |
| Kafka/Kinesis/Pub/Sub/Pulsar stream | Standard connector with Structured Streaming or Lakeflow Spark Declarative Pipelines | Message-source streaming pattern |
| Millions of files or frequent file arrival | Auto Loader | Scalable file discovery and checkpointed incremental processing |
| JSON/CSV files with schema drift | Auto Loader | Strong schema inference/evolution and rescued data support |
| Moderate scheduled cloud-file drops owned by SQL team | COPY INTO | SQL-first, idempotent, simpler ops |
| Exact subset of files to reload | COPY INTO with `FILES` or `PATTERN` | Easier targeted file reload |
| Reference file in a UC volume, overwritten monthly | Batch read in a Lakeflow pipeline table or materialized view | Current-state reference data, not a stream of new files |
| Small one-time local CSV | Add data UI or upload to UC volume | Simple, not a production pipeline |
| Non-tabular files like PDFs/images | Unity Catalog volume | Governed file storage, not table ingestion |
| Custom REST API, no connector | Notebook/Python task + Lakeflow Job | Custom code is needed |
| Large REST extracts | Land pages/files to volume/cloud path, then Auto Loader | Avoid driver OOM and preserve raw files |
| JDBC table, no managed connector | JDBC notebook/Python task + Lakeflow Job | Custom database extract |
| Large JDBC table | JDBC with partitioning and source-side predicates | Parallelize safely and reduce extract size |
| Need replayable raw landing | Stage raw files in volume/cloud path, then Auto Loader | Raw audit + scalable ingestion |
| Need only latest state from source | SCD type 1 or dedup/merge downstream | Current-state table |
| Need source history of changes | SCD type 2 when connector supports it, or append raw + downstream history | Historical tracking |

## Tool Comparison

| Tool | Source type | Strengths | Weaknesses |
|------|-------------|-----------|------------|
| Add data UI / file upload | Local files, samples | Fast, simple | Not for recurring production pipelines |
| COPY INTO | Cloud files or UC volumes | SQL-first, idempotent, good for scheduled batch | Weaker for millions of files, frequent arrivals, complex schema drift |
| Auto Loader | Cloud files or UC volumes | Scalable incremental files, schema evolution, checkpoints, file notifications | More streaming concepts to manage |
| Lakeflow Connect managed | Supported SaaS apps/databases | Least custom code, managed retries, incremental ingestion, monitoring | Connector/source capability dependent |
| Lakeflow Connect standard | Streaming/message/file sources needing customization | Flexible and production-ready with pipelines | More configuration/code than managed connectors |
| JDBC notebook/job | Unsupported relational databases | Flexible SQL extracts, partitioned reads | You manage credentials, state, retries, source load |
| REST notebook/job | Unsupported APIs | Flexible, works for bespoke APIs | Driver memory, pagination, rate limits, custom state |
| Partner connector | Third-party source/tooling | Useful when first-party path unavailable | Adds external vendor dependency |

## COPY INTO vs Auto Loader

| Clue | Choose COPY INTO | Choose Auto Loader |
|------|------------------|-------------------|
| Team wants SQL-only simple batch | Yes | Maybe, with streaming tables |
| File count is thousands over time | Yes | Also possible |
| File count is millions or more | No | Yes |
| Files arrive continuously | No | Yes |
| Schema evolves often | Maybe | Yes |
| Need file notifications / file events | No | Yes |
| Need explicit file subset reload | Yes | Harder |
| Need checkpointed streaming semantics | No | Yes |

Exam shortcut: `COPY INTO` is simpler; **Auto Loader scales better**.

## Connect vs Custom Code

| Clue | Choose Lakeflow Connect | Choose JDBC/REST notebook |
|------|-------------------------|---------------------------|
| Supported connector exists | Yes | Usually no |
| Need minimal custom code | Yes | No |
| Need source-specific managed retries/schema evolution | Yes | No |
| API is custom/internal and unsupported | No | Yes |
| Need highly custom request logic | No | Yes |
| Need database CDC with supported DB | Yes | No |
| Need simple ad hoc extract from unsupported DB | No | Yes |

Exam shortcut: if the scenario names a supported managed connector, choose it before custom code.

## Governance Checklist

| Requirement | Exam-safe answer |
|-------------|------------------|
| Lineage | Write to UC tables and read/write through Databricks |
| Access control | Use Unity Catalog tables, volumes, external locations, and connections |
| Cloud-file access | Prefer UC volumes or external locations |
| SaaS/database credentials | Use UC connections or secrets |
| Production scheduling | Use Lakeflow Jobs or Lakeflow Spark Declarative Pipelines |
| Raw landing | Bronze Delta table or governed volume path |
| Long-term production storage | Avoid DBFS root; use UC tables/volumes |

## Source-First Decision Tree

```
What is the source?

Local file
  -> One-time/small: Add data UI or UC volume
  -> Recurring/large: land in cloud/volume, then Auto Loader or COPY INTO
  -> Reference file overwritten in place: batch read in pipeline table/materialized view

Cloud object storage files
  -> Millions/frequent/schema drift: Auto Loader
  -> Simple scheduled SQL batch: COPY INTO
  -> Production SQL streaming table: Auto Loader through SQL/Lakeflow pipeline

SaaS application
  -> Managed connector exists: Lakeflow Connect managed
  -> No connector: REST notebook/Python task + Lakeflow Job
  -> Partner required: partner connector

Relational database
  -> Managed CDC connector exists and CDC is available: Lakeflow Connect database connector
  -> Cursor column, no CDC: query-based connector
  -> Unsupported/custom: JDBC notebook/Python task + Lakeflow Job

Message bus
  -> Kafka/Kinesis/Pub/Sub/Pulsar: standard connector with Structured Streaming or Lakeflow pipeline

Need governance
  -> Target must be Unity Catalog Delta table or governed volume
```

## PySpark / API Essentials

```python
# Exam pattern: always land governed data to a UC table.
target = "main.bronze.events_raw"

# Auto Loader for high-volume/frequent files.
(spark.readStream.format("cloudFiles")
  .option("cloudFiles.format", "json")
  .option("cloudFiles.schemaLocation", "/Volumes/main/landing/_schemas/events")
  .load("/Volumes/main/landing/events")
  .writeStream
  .option("checkpointLocation", "/Volumes/main/landing/_checkpoints/events")
  .trigger(availableNow=True)
  .toTable(target))
```

```python
# JDBC fallback for unsupported database sources.
jdbc_df = (spark.read.format("jdbc")
  .option("url", jdbc_url)
  .option("dbtable", "public.orders")
  .option("user", dbutils.secrets.get("prod", "db_user"))
  .option("password", dbutils.secrets.get("prod", "db_password"))
  .option("partitionColumn", "order_id")
  .option("lowerBound", 1)
  .option("upperBound", 10000000)
  .option("numPartitions", 8)
  .load())

jdbc_df.write.mode("append").saveAsTable("main.bronze.orders_raw")
```

## SQL (exam)

```sql
-- SQL-first scheduled file ingestion.
COPY INTO main.bronze.events_raw
FROM '/Volumes/main/landing/events'
FILEFORMAT = JSON
COPY_OPTIONS ('force' = 'false');
```

```sql
-- SQL streaming-table style ingestion for scalable file ingestion.
CREATE OR REFRESH STREAMING TABLE main.bronze.events_raw
AS SELECT *
FROM STREAM read_files(
  '/Volumes/main/landing/events',
  format => 'json'
);
```

```sql
-- Lakeflow pipeline reference data: batch read the current overwritten file.
-- Do not use STREAM read_files for a monthly overwritten reference file.
CREATE OR REFRESH MATERIALIZED VIEW main.bronze.product_reference
AS SELECT *
FROM read_files(
  '/Volumes/main/reference/products.csv',
  format => 'csv',
  header => true
);
```

```sql
-- Always verify the target is governed and inspectable.
DESCRIBE DETAIL main.bronze.events_raw;
DESCRIBE HISTORY main.bronze.events_raw;
```

## UI / Product Surfaces

- **Add data / Data ingestion**: local files and Lakeflow Connect managed connector entry points.
- **Catalog Explorer**: UC tables, volumes, external locations, connections, permissions, and lineage.
- **SQL editor**: `COPY INTO`, SQL streaming tables, and validation queries.
- **Notebooks**: Auto Loader, JDBC, REST, and custom PySpark ingestion.
- **Lakeflow Spark Declarative Pipelines**: production Auto Loader and standard connector pipelines.
- **Lakeflow Jobs**: scheduling, dependencies, retries, parameters, alerts, and job compute.
- **System tables / run history / pipeline event logs**: monitoring, costs, and troubleshooting.

## Exam Traps

- Choosing `COPY INTO` for millions of small files when Auto Loader is the scaling answer.
- Choosing Auto Loader for a supported SaaS connector when Lakeflow Connect is available.
- Choosing custom JDBC/REST code before checking for Lakeflow Connect managed connectors.
- Choosing Lakeflow Connect for generic S3/ADLS/GCS file drops; use Auto Loader or `COPY INTO`.
- Landing data to DBFS or an unregistered path when the question requires Unity Catalog governance.
- Forgetting that SQL-only does not automatically mean `COPY INTO`; SQL streaming tables can use Auto Loader for scalable file ingestion.
- Using `STREAM read_files` for a reference file that is overwritten in place once a month. Use a batch pipeline table or materialized view that rereads the current file on refresh.
- Treating every incremental problem as streaming. Scheduled incremental batch is often enough.
- Treating every source as append-only. Updates/deletes may require CDC, SCD mode, dedup, or `MERGE`.
- Ignoring source-system load. JDBC partitioning can overload external databases.
- Using all-purpose interactive compute as the production ingestion compute instead of Lakeflow Jobs, serverless jobs, jobs compute, or pipelines.
- Using partner connectors as the default when a first-party Databricks connector is stated.

# 2.5 JDBC / REST & Jobs

## Exam objective

> Use JDBC/ODBC or REST clients in notebooks to land data into cloud storage or directly into Unity-Catalog-governed tables, usually orchestrated and scheduled with Lakeflow Jobs.

## Must know

- Use custom **JDBC**, **ODBC**, or **REST** ingestion when a first-party Lakeflow Connect connector is unavailable or does not meet the requirement.
- The exam-safe target is still a **Unity Catalog Delta table**, usually a bronze table such as `main.bronze.pg_orders_raw`.
- **JDBC** ingestion uses `spark.read.format("jdbc")` or `spark.read.jdbc(...)` to pull from external relational databases.
- **REST** ingestion usually runs Python code in a notebook or Python task, calls an API, converts the response into a DataFrame, and writes to Delta.
- Use **secrets** or Unity Catalog connections for credentials. Never hardcode database passwords, API keys, or bearer tokens in notebook code.
- Use **Lakeflow Jobs** to schedule, parameterize, retry, monitor, and productionize custom ingestion notebooks.
- Prefer **serverless jobs compute** when supported, or **classic jobs compute** when you need custom libraries, drivers, network settings, or unsupported serverless features.
- Use **job parameters** or notebook widgets for incremental windows such as `start_ts`, `end_ts`, `run_date`, or `last_id`.
- For large JDBC tables, use **parallel reads** with `partitionColumn`, `lowerBound`, `upperBound`, and `numPartitions`.
- Too many JDBC partitions can overload the source database. Parallelism is a source-system trade-off, not just a Spark tuning knob.
- Avoid loading huge REST responses into driver memory. Use pagination, incremental API filters, streaming downloads, or land raw files first.
- For large extracts, consider a two-step pattern: JDBC/REST to cloud files or UC volume, then Auto Loader or `COPY INTO` into a bronze table.

## Custom ingestion mental model

```
External database or API
  -> notebook / Python task
  -> secrets or UC connection for credentials
  -> incremental query, pagination, or cursor
  -> bronze UC Delta table or staged files
  -> Lakeflow Job schedule, retries, alerts, parameters
```

Use this pattern when managed ingestion does not fit. If the source has a supported managed connector, Lakeflow Connect is usually the better exam answer.

## When to use what

| Scenario | Choose |
|----------|--------|
| Supported SaaS/database connector exists and meets needs | Lakeflow Connect |
| External relational DB, no managed connector, simple extract | JDBC notebook/Python task |
| External relational DB, very large table | JDBC with partitioning and incremental predicates |
| API returns small paginated JSON | REST notebook/Python task |
| API returns large files or massive payloads | Download to cloud storage/volume, then Auto Loader or `COPY INTO` |
| Need repeatable scheduled ingestion | Lakeflow Job with notebook or Python task |
| Need near-real-time file ingestion | Auto Loader, not REST polling unless required |
| Need ODBC specifically from a client outside Databricks | ODBC client pattern; inside notebooks, JDBC/Python libraries are more common |

## Direct table vs staged files

| Pattern | Use when | Trade-off |
|---------|----------|-----------|
| JDBC/REST directly to UC Delta | Small/medium data, simple schema, straightforward append/overwrite | Simple, fewer moving parts |
| JDBC/REST to UC volume/cloud files, then Auto Loader | Large extracts, raw file audit, replay needs, schema drift | More robust, but more components |
| JDBC/REST to temp view, then SQL transform | Small intermediate cleanup before write | Keep temp data small |
| API pagination to many small JSON files | Need restartability and raw capture | Pair with Auto Loader for scalable ingestion |

## JDBC essentials

| Option | Purpose |
|--------|---------|
| `url` | JDBC URL, such as `jdbc:postgresql://host:5432/db` |
| `dbtable` | Source table or subquery alias |
| `query` | SQL query to run at the source, useful for filters or joins |
| `user` / `password` | Credentials, usually pulled from secrets |
| `driver` | JDBC driver class when needed |
| `partitionColumn` | Numeric/date/timestamp column used to split reads |
| `lowerBound` / `upperBound` | Bounds for partitioning the source reads |
| `numPartitions` | Number of parallel JDBC connections/tasks |
| `fetchsize` | Rows fetched per round trip; useful for performance tuning |

Partitioning note: `lowerBound` and `upperBound` define partition stride, not a filter by themselves. Use a `WHERE` clause if you need to limit the extracted rows.

## Incremental patterns

| Source pattern | State to track | Example parameter |
|----------------|----------------|-------------------|
| Updated timestamp | Last successful `updated_at` | `start_ts`, `end_ts` |
| Monotonic ID | Last processed ID | `last_id` |
| API cursor | Next page/token from API | `next_cursor` |
| Date partition | Job run date | `run_date` |
| Source CDC table | Last CDC version or timestamp | `last_lsn`, `last_version` |

Store incremental state in a small Delta audit/control table, job parameters, or a trusted source cursor mechanism. Do not rely only on notebook variables for production state.

## PySpark / API essentials

```python
# Notebook parameters passed from Lakeflow Jobs.
dbutils.widgets.text("start_ts", "")
dbutils.widgets.text("end_ts", "")

start_ts = dbutils.widgets.get("start_ts")
end_ts = dbutils.widgets.get("end_ts")
```

```python
# JDBC read with secrets and a source-side incremental predicate.
user = dbutils.secrets.get("prod", "pg_user")
password = dbutils.secrets.get("prod", "pg_password")

query = f"""
(
  SELECT
    order_id,
    customer_id,
    updated_at,
    amount
  FROM public.orders
  WHERE updated_at >= TIMESTAMP '{start_ts}'
    AND updated_at < TIMESTAMP '{end_ts}'
) AS orders_window
"""

orders = (spark.read.format("jdbc")
  .option("url", "jdbc:postgresql://host.example.com:5432/sales")
  .option("dbtable", query)
  .option("user", user)
  .option("password", password)
  .option("driver", "org.postgresql.Driver")
  .option("fetchsize", 10000)
  .load())

orders.write.mode("append").saveAsTable("main.bronze.pg_orders_raw")
```

```python
# JDBC parallel read for a large table.
orders = (spark.read.format("jdbc")
  .option("url", "jdbc:postgresql://host.example.com:5432/sales")
  .option("dbtable", "public.orders")
  .option("user", user)
  .option("password", password)
  .option("partitionColumn", "order_id")
  .option("lowerBound", 1)
  .option("upperBound", 10000000)
  .option("numPartitions", 8)
  .load())
```

```python
# REST pattern with pagination. Keep pages small and append to Delta.
import requests

token = dbutils.secrets.get("prod", "api_token")
url = "https://api.example.com/events"
headers = {"Authorization": f"Bearer {token}"}

all_rows = []
next_cursor = None

while True:
    params = {"limit": 1000, "cursor": next_cursor, "start": start_ts, "end": end_ts}
    response = requests.get(url, headers=headers, params=params, timeout=60)
    response.raise_for_status()
    payload = response.json()

    all_rows.extend(payload["items"])
    next_cursor = payload.get("next_cursor")
    if not next_cursor:
        break

events = spark.createDataFrame(all_rows)
events.write.mode("append").saveAsTable("main.bronze.api_events_raw")
```

For large REST extracts, write each page to files in a volume/cloud path and use Auto Loader. Avoid building one giant Python list on the driver.

## SQL (exam)

```sql
-- After custom ingestion, downstream SQL should read governed UC tables.
SELECT COUNT(*) AS rows
FROM main.bronze.pg_orders_raw;
```

```sql
-- Track incremental state in a small Delta control table.
CREATE TABLE IF NOT EXISTS main.ops.ingestion_watermarks (
  pipeline_name STRING,
  last_success_ts TIMESTAMP
);

SELECT last_success_ts
FROM main.ops.ingestion_watermarks
WHERE pipeline_name = 'pg_orders';
```

```sql
-- Bronze to silver dedup after append-style ingestion.
CREATE OR REPLACE TABLE main.silver.orders AS
SELECT *
FROM (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY order_id
      ORDER BY updated_at DESC
    ) AS rn
  FROM main.bronze.pg_orders_raw
)
WHERE rn = 1;
```

## Lakeflow Jobs setup

| Setting | Exam notes |
|---------|------------|
| Task type | Notebook task or Python script/wheel task |
| Compute | Serverless jobs compute when supported; classic jobs compute for custom drivers/network needs |
| Parameters | Pass `run_date`, `start_ts`, `end_ts`, source name, or table name |
| Schedule | Time-based trigger for recurring extracts |
| Retries | Use retries for transient network/API/database failures |
| Notifications | Alert on failure, duration threshold, or skipped runs |
| Dependencies | Ingest task before downstream silver/gold tasks |
| Secrets | Scope permissions should allow only the job/run identity to read needed credentials |

## UI / product surfaces

- **Secrets**: create secret scopes and store database passwords or API tokens.
- **Catalog Explorer**: inspect UC target tables and permissions; create/manage UC connections where appropriate.
- **Jobs & Pipelines / Lakeflow Jobs**: schedule notebook/Python tasks, configure parameters, retries, compute, alerts, and dependencies.
- **Compute / Libraries**: install JDBC drivers or Python packages when needed.
- **Run history**: monitor duration, failures, retries, logs, and outputs.
- **Spark UI**: inspect JDBC read parallelism, stages, task skew, and spill if the extract is heavy.

## Exam traps

- Wrong: hardcode passwords or API tokens in notebook code. Right: read credentials from Databricks secrets or Unity Catalog connections.
- Wrong: choose custom JDBC/REST code when Lakeflow Connect has a managed connector that meets the requirement. Right: prefer the managed connector first.
- Wrong: read a huge JDBC table without partitioning, causing a slow single-threaded extract. Right: use `partitionColumn`, `lowerBound`, `upperBound`, and `numPartitions` when the source can handle parallel reads.
- Wrong: set very high `numPartitions` just to make Spark read faster. Right: balance JDBC parallelism against source database capacity.
- Wrong: use `lowerBound` and `upperBound` as row filters. Right: they only define JDBC partition stride; use a source-side `WHERE` predicate to filter rows.
- Wrong: pull a huge REST response into driver memory with `requests.get(...).json()` and one big list. Right: use pagination, incremental API filters, streaming downloads, or land pages as raw files.
- Wrong: run production ingestion manually from a notebook. Right: schedule and monitor it with Lakeflow Jobs.
- Wrong: write output to ungoverned DBFS paths when the scenario asks for Unity Catalog governance and lineage. Right: write to Unity Catalog tables or governed volumes.
- Wrong: overwrite bronze every run and lose source history. Right: append or merge incrementally so bronze preserves raw/source change history when required.
- Wrong: ignore retry/backoff for REST rate limits, HTTP 429, or transient database/network errors. Right: add retries, backoff, timeouts, and failure alerts.
- Wrong: assume job parameters automatically appear inside notebooks. Right: pass parameters through Lakeflow Jobs and read them with widgets or task parameter syntax.

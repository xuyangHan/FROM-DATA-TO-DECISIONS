# 2.2 COPY INTO

## Exam objective

> Use the COPY INTO command to incrementally load files from cloud object storage (ADLS/S3/GCS) into Unity-Catalog-governed tables.

## Must know

- **`COPY INTO`** is a SQL command for loading files into a Delta table.
- It is **incremental and idempotent by default**: files already loaded from the source location are skipped on later runs.
- It is best for **scheduled batch file ingestion** from cloud object storage or Unity Catalog volumes.
- It works well when a SQL team owns the pipeline and the source is a predictable file drop.
- It is usually **not** the best answer for very high-volume or continuously arriving files; use Auto Loader or streaming tables for that.
- The target should be a **Unity Catalog Delta table** such as `main.bronze.orders_raw`.
- The target table must exist before `COPY INTO` runs. It can be a normal table with a defined schema or an empty schemaless Delta table used with schema inference/evolution.
- Source access should usually go through a **Unity Catalog volume** or a cloud path covered by a **Unity Catalog external location**.
- For UC volumes, users need source file access such as `READ VOLUME`. For external locations, users need `READ FILES`.
- For target tables, users need normal UC privileges such as `USE CATALOG`, `USE SCHEMA`, and permission to write to the table.
- Supported file formats include **CSV, JSON, XML, Avro, ORC, Parquet, text, and binary files**.
- `FORMAT_OPTIONS` are reader options for the file format, such as `header`, `inferSchema`, `delimiter`, `multiLine`, or `ignoreCorruptFiles`.
- `COPY_OPTIONS` control the copy operation, especially `force` and target table `mergeSchema`.
- `force = false` is the safe default. `force = true` disables idempotency and reloads files even if they were loaded before.
- Append **`VALIDATE`** or **`VALIDATE ALL`** to check that source files can be parsed, match the target table schema, and satisfy constraints without inserting rows.

## COPY INTO mental model

```
Cloud files or UC volume path
  -> COPY INTO command
  -> existing Unity Catalog Delta table
  -> Delta transaction log records the load
  -> next run skips files already loaded
```

COPY INTO tracks loaded files as part of the Delta table's transaction history. You do not manage a streaming checkpoint for `COPY INTO`.

## When to use what

| Scenario | Choose |
|----------|--------|
| SQL-first team needs scheduled file ingestion | `COPY INTO` |
| Moderate recurring CSV/JSON/Parquet drops | `COPY INTO` |
| One-time or occasional backfill from known files | `COPY INTO` |
| Need to filter exact files by name | `COPY INTO` with `FILES` |
| Need to filter files by glob | `COPY INTO` with `PATTERN` |
| Need to preview parsing/schema issues without writing | `COPY INTO ... VALIDATE` |
| Millions of tiny files or continuous file arrival | Auto Loader |
| Schema drift in high-volume JSON files | Auto Loader with schema evolution |
| SaaS or database connector exists | Lakeflow Connect |
| Need upserts/CDC into existing rows | Land with `COPY INTO`, then transform with `MERGE`; `COPY INTO` itself is not an upsert tool |

## COPY INTO vs Auto Loader

| Requirement | COPY INTO | Auto Loader |
|-------------|-----------|-------------|
| SQL-only simple batch load | Strong fit | Possible, but more streaming-oriented |
| Incrementally skip old files | Yes | Yes |
| Checkpoint required | No | Yes |
| Very large file counts | Weaker | Strong fit |
| Continuous file arrival | Weaker | Strong fit |
| Schema drift handling | Some support through options | Stronger support |
| File notifications | No | Yes |
| Runs as repeated SQL task | Strong fit | Usually notebook/pipeline |

## Key syntax pieces

| Clause | Purpose |
|--------|---------|
| `COPY INTO target_table` | Target must be an existing Delta table |
| `FROM '/Volumes/...'` | Recommended source pattern for UC-governed file access |
| `FROM 's3://...'` / `abfss://...` / `gs://...` | Source cloud path, usually governed by external location |
| `FILEFORMAT = CSV` | Declares the file format |
| `FORMAT_OPTIONS (...)` | File reader options |
| `COPY_OPTIONS (...)` | Copy behavior options |
| `FILES = (...)` | Load only a named list of files; limit is 1000 files |
| `PATTERN = '...'` | Load files matching a glob pattern |
| `VALIDATE` | Validate files and schema without writing rows |
| `VALIDATE ALL` | Validate all data that would be loaded without writing rows |
| `FROM (SELECT ... FROM source)` | Transform, cast, rename, or select columns before loading |

`FILES` and `PATTERN` are alternatives; do not use both in the same command.

Bad-row parse modes are file reader options, so set them in `FORMAT_OPTIONS` when supported by the file format:

| Mode | Bad rows |
|------|----------|
| `PERMISSIVE` | Keep the row and partially parse what can be parsed |
| `FAILFAST` | Fail the command/pipeline when a malformed row is found |
| `DROPMALFORMED` | Discard malformed rows |

## SQL (exam)

```sql
-- Recommended pattern: load from a Unity Catalog volume.
COPY INTO main.bronze.orders_raw
FROM '/Volumes/main/landing/orders'
FILEFORMAT = PARQUET
COPY_OPTIONS ('force' = 'false');
```

```sql
-- Load from a cloud path governed by a Unity Catalog external location.
COPY INTO main.bronze.orders_raw
FROM 'abfss://landing@account.dfs.core.windows.net/orders/'
FILEFORMAT = CSV
FORMAT_OPTIONS (
  'header' = 'true',
  'inferSchema' = 'true'
)
COPY_OPTIONS ('force' = 'false');
```

```sql
-- Load only selected files.
COPY INTO main.bronze.orders_raw
FROM '/Volumes/main/landing/orders'
FILEFORMAT = JSON
FILES = ('orders_2026_05_01.json', 'orders_2026_05_02.json');
```

```sql
-- Load files matching a glob pattern.
COPY INTO main.bronze.orders_raw
FROM '/Volumes/main/landing/orders'
FILEFORMAT = CSV
PATTERN = 'daily/orders_2026_05_*.csv'
FORMAT_OPTIONS ('header' = 'true');
```

```sql
-- Validate parsing/schema/constraints without writing rows.
COPY INTO main.bronze.orders_raw
FROM '/Volumes/main/landing/orders'
FILEFORMAT = JSON
VALIDATE ALL;
```

Exam shortcut: before inserting data with `COPY INTO`, append **`VALIDATE`** or **`VALIDATE ALL`**. Use `VALIDATE n ROWS` when only a sample should be checked.

```sql
-- Cast or rename source columns before writing.
COPY INTO main.bronze.orders_raw
FROM (
  SELECT
    _c0::STRING AS order_id,
    _c1::STRING AS customer_id,
    _c2::TIMESTAMP AS order_ts,
    _c3::DOUBLE AS amount
  FROM '/Volumes/main/landing/orders_csv'
)
FILEFORMAT = CSV
FORMAT_OPTIONS ('header' = 'false');
```

## Result and monitoring

- The command result reports load information such as loaded files, skipped files, and skipped corrupt files depending on the operation.
- Use `DESCRIBE HISTORY catalog.schema.table` to inspect Delta table operation history and metrics.
- Use `DESCRIBE DETAIL catalog.schema.table` to confirm the target is a Delta table and inspect metadata.
- In Databricks Runtime 15.2 and above, `VACUUM` can clean up unreferenced metadata files created by `COPY INTO`.

## UI / product surfaces

- **SQL editor**: run `COPY INTO` on a SQL warehouse.
- **Notebook `%sql` or `spark.sql`**: run `COPY INTO` from notebook compute.
- **Lakeflow Jobs**: schedule a SQL task or notebook task that runs `COPY INTO`.
- **Catalog Explorer**: verify target table, permissions, lineage, and table history.
- **Volumes / External locations**: manage governed access to source files.

## Exam traps

- `COPY INTO` is **not** a continuous streaming tool. For continuous or high-scale file ingestion, choose Auto Loader or streaming tables.
- `COPY INTO` is **not** an upsert command. Use it to land files, then use transformations or `MERGE` if you need upserts.
- The target must be an existing Delta table. If no target exists, create it first.
- Already-loaded files are skipped even if the file contents later change. Use `force = true` only when you intentionally want to reload.
- `force = true` can create duplicate rows if the pipeline is append-only and not deduplicated downstream.
- A source cloud path must be accessible through UC permissions, such as a volume or external location. Otherwise the command fails with access errors.
- Do not confuse `FORMAT_OPTIONS ('inferSchema' = 'true')` with `COPY_OPTIONS ('mergeSchema' = 'true')`: the first infers the source file schema; the second evolves the target Delta table schema.
- `FILES` has a file-count limit and is for explicit file names; use `PATTERN` for glob-style filtering.
- Running many concurrent `COPY INTO` commands is not a performance trick. One command over many files is usually better; concurrent copies are mainly for independent file sets.
- `COPY INTO` is idempotent by file path/load tracking, not by primary key. Duplicate records in different files can still be loaded.

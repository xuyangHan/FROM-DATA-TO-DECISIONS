# 2.3 Auto Loader

## Exam objective

> Use Auto Loader with schema enforcement and schema evolution in batch modes (for example, directory listing or file notification) to land data into Unity-Catalog-governed tables.

## Must know

- **Auto Loader** incrementally and efficiently ingests new files from cloud object storage into Delta tables.
- The PySpark source format is **`cloudFiles`**, used with `spark.readStream.format("cloudFiles")`.
- Auto Loader is built on **Structured Streaming**, but it can run as **incremental batch** with `Trigger.AvailableNow`.
- **`Trigger.AvailableNow`** processes all files that arrived before the query starts, then stops. Files that arrive after the run starts are picked up on the next run.
- The target should be a **Unity Catalog-governed Delta table**, usually a bronze table such as `main.bronze.orders_raw`.
- A unique **checkpoint location** is required for each Auto Loader stream. It tracks discovered files and processing progress.
- A **schema location** is required when Auto Loader infers or evolves schema. It stores inferred schemas under a `_schemas` directory.
- **Directory listing mode** is the default and easiest to start with.
- **File notification mode / file events** is recommended for better scalability and performance on larger workloads.
- Auto Loader can process existing files in a directory and then continue with newly arriving files.
- Auto Loader does **not guarantee file processing order**, regardless of detection mode. Design downstream logic to handle out-of-order arrivals.
- Compared with `COPY INTO`, Auto Loader is usually better for high file volume, frequent arrival, schema drift, and production incremental ingestion.

## Auto Loader mental model

```
Cloud files or Unity Catalog volume
  -> spark.readStream.format("cloudFiles")
  -> file discovery state in checkpoint
  -> schema tracking in schemaLocation
  -> writeStream to Unity Catalog Delta table
  -> rerun with same checkpoint to process only new files
```

Important distinction:

| Location | Purpose |
|----------|---------|
| `checkpointLocation` | Streaming state: offsets, discovered files, committed batches, exactly-once progress |
| `cloudFiles.schemaLocation` | Schema inference/evolution metadata |
| Source path | Files to ingest |
| Target table | Delta table where rows are written |

## When to use what

| Scenario | Choose |
|----------|--------|
| Thousands or millions of files | Auto Loader |
| Files arrive frequently or continuously | Auto Loader |
| JSON/CSV files may add new columns over time | Auto Loader with schema evolution |
| Need SQL-only simple scheduled file load | `COPY INTO` may be simpler |
| Need file notifications or highly scalable discovery | Auto Loader file notification mode / file events |
| Need to process all current files and stop | Auto Loader with `Trigger.AvailableNow` |
| Need near-real-time micro-batches | Auto Loader with `processingTime` trigger |
| Need SaaS/database managed ingestion | Lakeflow Connect, not Auto Loader |

## File detection modes

| Mode | How it works | Use when | Watch out |
|------|--------------|----------|-----------|
| Directory listing | Lists the input directory to find new files | Quick start, moderate workloads, fewer cloud setup requirements | Can be less efficient at very large scale |
| File notification / file events | Uses cloud file events and queues to discover files | High volume, production workloads, frequent arrival | Requires supported cloud/storage setup; file events on external location are preferred |

Setting `cloudFiles.useNotifications = true` enables file notification mode. Auto Loader can then use cloud-native event services such as AWS SQS/SNS, Azure Event Grid, or GCP Pub/Sub to identify new files instead of repeatedly listing the source directory. This is usually more efficient and cost-effective than directory listing for deeply nested storage structures, large directories, and infrequent data arrivals because it avoids traversing the whole file tree just to find new files.

You can switch detection modes across restarts and still keep exactly-once processing guarantees when you keep the same checkpoint.

## Schema inference and evolution

Auto Loader can infer schemas, track schemas, and respond to changes in source files.

| Option | Purpose |
|--------|---------|
| `cloudFiles.schemaLocation` | Stores inferred schema and schema changes |
| `cloudFiles.inferColumnTypes` | Infer JSON/CSV/XML column types instead of reading them as strings |
| `cloudFiles.schemaHints` | Tell Auto Loader expected column types or add expected future columns |
| `cloudFiles.schemaEvolutionMode` | Controls behavior when new columns appear |
| `rescuedDataColumn` | Stores unexpected or mismatched fields instead of dropping them |
| `mergeSchema` on write | Allows target Delta table schema to evolve during the write |

Default inference behavior to remember:

| Format | Default inferred types |
|--------|------------------------|
| JSON, CSV, XML | Strings by default |
| Parquet, Avro | Uses encoded file schema |
| Text, binaryFile | Fixed schema |

Use `cloudFiles.inferColumnTypes = true` when you want Auto Loader to infer JSON/CSV/XML types like Spark's regular DataFrame reader.

## Schema evolution modes

| Mode | Behavior when new columns appear |
|------|----------------------------------|
| `addNewColumns` | Default when no schema is provided. Stream fails, schema is updated with new columns, then a restart can continue |
| `rescue` | Stream does not fail; new or mismatched fields go into the rescued data column |
| `failOnNewColumns` | Stream fails and will not continue until schema or data is fixed |
| `none` | Does not evolve schema; new columns are ignored unless `rescuedDataColumn` is configured |
| `addNewColumnsWithTypeWidening` | Adds new columns and widens supported type changes; unsupported type changes are rescued |

Exam shortcut: `addNewColumns` may intentionally fail the stream on a new column so the schema can be updated. In production, configure Lakeflow Jobs retries/restarts so the stream can continue after schema evolution.

In a standard Auto Loader setup outside Lakeflow Spark Declarative Pipelines, `cloudFiles.schemaEvolutionMode = addNewColumns` follows a specific sequence when a new column appears:

1. Auto Loader detects the new column.
2. It updates the schema metadata in `cloudFiles.schemaLocation`.
3. The stream or `AvailableNow` batch fails with an exception.
4. The job must restart to continue processing with the new schema.

In production, this restart is usually handled by the Databricks Workflow / Lakeflow Jobs retry mechanism.

## PySpark / API essentials

```python
# Incremental batch Auto Loader: process current files, then stop.
(spark.readStream.format("cloudFiles")
  .option("cloudFiles.format", "json")
  .option("cloudFiles.schemaLocation", "/Volumes/main/landing/_schemas/orders")
  .option("cloudFiles.inferColumnTypes", "true")
  .option("cloudFiles.schemaEvolutionMode", "addNewColumns")
  .option("rescuedDataColumn", "_rescued_data")
  .load("/Volumes/main/landing/orders_json")
  .writeStream
  .option("checkpointLocation", "/Volumes/main/landing/_checkpoints/orders_raw")
  .option("mergeSchema", "true")
  .trigger(availableNow=True)
  .toTable("main.bronze.orders_raw"))
```

```python
# Continuous micro-batch pattern for lower latency.
(spark.readStream.format("cloudFiles")
  .option("cloudFiles.format", "csv")
  .option("cloudFiles.schemaLocation", "/Volumes/main/landing/_schemas/clicks")
  .option("header", "true")
  .load("/Volumes/main/landing/clicks")
  .writeStream
  .option("checkpointLocation", "/Volumes/main/landing/_checkpoints/clicks_raw")
  .trigger(processingTime="1 minute")
  .toTable("main.bronze.clicks_raw"))
```

```python
# Use schema hints when inference picks the wrong type or a column may appear later.
(spark.readStream.format("cloudFiles")
  .option("cloudFiles.format", "json")
  .option("cloudFiles.schemaLocation", "/Volumes/main/landing/_schemas/events")
  .option("cloudFiles.schemaHints", "event_id STRING, amount DOUBLE, tags ARRAY<STRING>")
  .load("/Volumes/main/landing/events")
  .writeStream
  .option("checkpointLocation", "/Volumes/main/landing/_checkpoints/events_raw")
  .trigger(availableNow=True)
  .toTable("main.bronze.events_raw"))
```

```python
# Rate limit an AvailableNow run so huge backfills are split into micro-batches.
(spark.readStream.format("cloudFiles")
  .option("cloudFiles.format", "parquet")
  .option("cloudFiles.schemaLocation", "/Volumes/main/landing/_schemas/transactions")
  .option("cloudFiles.maxFilesPerTrigger", 1000)
  .option("cloudFiles.maxBytesPerTrigger", "10g")
  .load("/Volumes/main/landing/transactions")
  .writeStream
  .option("checkpointLocation", "/Volumes/main/landing/_checkpoints/transactions_raw")
  .trigger(availableNow=True)
  .toTable("main.bronze.transactions_raw"))
```

## Auto Loader vs COPY INTO

| Requirement | Auto Loader | COPY INTO |
|-------------|-------------|-----------|
| High file volume | Best fit | Weaker |
| Frequent new files | Best fit | Weaker |
| SQL-only simple batch | Possible through streaming tables | Strong fit |
| Schema evolution/rescue | Strong fit | Limited |
| File notifications/events | Yes | No |
| Needs checkpoint | Yes | No |
| Idempotent file ingestion | Yes, through checkpoint | Yes, through Delta load tracking |

## UI / product surfaces

- **Notebook**: run Auto Loader with PySpark `readStream`.
- **Lakeflow Spark Declarative Pipelines**: recommended managed pattern for Auto Loader ingestion and streaming tables.
- **Lakeflow Jobs**: schedule `AvailableNow` Auto Loader notebooks or pipelines.
- **Catalog Explorer**: verify target UC tables, lineage, and permissions.
- **Structured Streaming UI / query profile**: monitor input rows, micro-batches, latency, and failures.
- **External locations / Volumes**: govern source file access.

## Exam traps

- Using `spark.read` instead of `spark.readStream.format("cloudFiles")`. That is not Auto Loader.
- Forgetting `checkpointLocation` on the write side.
- Reusing one checkpoint path for different streams or different source/target combinations.
- Confusing `checkpointLocation` with `cloudFiles.schemaLocation`; they track different metadata.
- Providing a fixed schema and expecting default `addNewColumns` behavior. When a schema is provided, the default evolution mode is `none`.
- Assuming Auto Loader processes files in arrival order. It does not guarantee order.
- Choosing `COPY INTO` for millions of small files or frequent arrivals when Auto Loader is the better fit.
- Enabling aggressive `cloudFiles.maxFileAge` and accidentally causing duplicate or missed ingestion.
- Applying lifecycle cleanup policies to the checkpoint directory. Losing checkpoint files can corrupt the stream state.
- Treating `AvailableNow` as continuous streaming. It processes currently available files and stops.

# 2.4 Lakeflow Connect

## Exam objective

> Configure Lakeflow Connect to reliably ingest data from diverse enterprise sources into Unity-Catalog-governed tables.

## Must know

- **Lakeflow Connect** is Databricks' ingestion framework for connecting enterprise sources to Databricks.
- Use it when the source is a supported **SaaS application**, **database**, or **streaming/message source**, and you want less custom ingestion code.
- The exam-safe target is a **Unity Catalog-governed Delta table**, usually a bronze table such as `main.bronze.salesforce_accounts`.
- Lakeflow Connect target tables in Unity Catalog use **Delta** format by default, and **Delta is required** for the resulting target tables.
- **Managed connectors** are the most automated option. They create managed ingestion pipelines, handle source-specific authentication, retries, incremental ingestion, schema evolution, and operational monitoring.
- **Standard connectors** offer more customization. They are commonly used through Structured Streaming, Lakeflow Spark Declarative Pipelines, or SQL streaming table patterns.
- **Community connectors** are open-source extensions for sources that do not have first-party managed connector support.
- **Partner connectors** are third-party tools. On the exam, prefer first-party Lakeflow Connect when the scenario says a supported connector exists.
- A **connection** is a Unity Catalog securable object that stores endpoint, credentials, owner, and configuration for an external system.
- An **ingestion pipeline** reads from the source and writes to destination tables. Managed connector pipelines run on serverless compute by default.
- For supported relational databases, managed database connectors can use **change data capture (CDC)** to capture changes incrementally.
- Some database ingestion can be **query-based**, using a cursor or high-water mark column instead of CDC logs.
- Lakeflow Connect can create schedules and Lakeflow Jobs around ingestion pipelines.
- Governance, lineage, ownership, and access control come from **Unity Catalog**, not from putting credentials in notebook code.

## Lakeflow Connect mental model

```
External source
  -> Unity Catalog connection
  -> Lakeflow Connect connector
  -> ingestion pipeline
  -> Unity Catalog streaming/Delta destination tables
  -> downstream silver and gold transformations
```

For database CDC connectors:

```
Source database
  -> UC connection
  -> ingestion gateway
  -> staging volume
  -> ingestion pipeline
  -> destination streaming tables / Delta tables
```

## Connector types

| Connector type | Best for | How it behaves |
|----------------|----------|----------------|
| Managed SaaS connector | Salesforce, Workday, HubSpot, Jira, NetSuite, ServiceNow, Google Analytics, and other supported apps | Uses source APIs, managed pipeline, incremental reads when supported |
| Managed database CDC connector | MySQL, PostgreSQL, SQL Server when CDC/binlog/change tracking is available | Captures database changes and applies them incrementally to Delta tables |
| Query-based connector | Databases with a suitable cursor column but no CDC setup | Runs scheduled queries and tracks a high-water mark |
| Standard connector | Kafka, Kinesis, Pub/Sub, Pulsar, SFTP, cloud files, and other sources needing customization | Used through Structured Streaming, Lakeflow Spark Declarative Pipelines, or SQL streaming tables |
| Community connector | Source not covered by managed connectors | Open-source connector maintained outside the core managed set |
| Partner connector | Third-party ingestion tool | Fallback when first-party Lakeflow Connect does not cover the requirement |

## Managed vs standard connectors

| Requirement | Managed connector | Standard connector |
|-------------|-------------------|--------------------|
| Lowest operational overhead | Best fit | More work |
| Source-specific auth and API maintenance | Built in | You configure more |
| Automated retries and schema evolution | Built in where supported | Depends on implementation |
| Fine-grained transformation/custom logic during ingestion | Less flexible | More flexible |
| Need Structured Streaming code control | Not the main path | Best fit |
| Supported SaaS/database source | Usually preferred | Use only if managed path lacks needed features |
| Kafka/Kinesis/Pub/Sub streaming | Usually not managed SaaS/database | Standard connector pattern |

Exam shortcut: start with the **most managed layer** that supports the source and requirement. Drop down to standard connectors, Structured Streaming, or custom code only when the managed connector does not fit.

## When to use what

| Scenario | Choose |
|----------|--------|
| Salesforce data should land in bronze with minimal custom code | Lakeflow Connect managed SaaS connector |
| PostgreSQL/MySQL/SQL Server changes must be captured continuously | Lakeflow Connect managed database CDC connector |
| Database table has `updated_at` cursor but CDC is unavailable | Query-based connector |
| Kafka events require custom parsing and streaming transformations | Standard connector through Structured Streaming or Lakeflow Spark Declarative Pipelines |
| S3/ADLS/GCS file drops | Auto Loader or `COPY INTO`, not managed Lakeflow Connect |
| One-off small REST API not supported by a connector | Notebook/Python task + Lakeflow Job |
| Need third-party source not covered by Databricks connectors | Community or partner connector |
| Need UC lineage and access control on landed data | Write connector output to UC tables |

## Key objects and privileges

| Object | What it stores or does | Exam notes |
|--------|-------------------------|------------|
| Unity Catalog connection | Endpoint, credentials, config, owner, metadata | Users need `CREATE CONNECTION` to create or `USE CONNECTION` to consume |
| Ingestion pipeline | Defines source objects, destination tables, schedule, and ingestion behavior | Managed connectors create Lakeflow pipelines |
| Destination table | UC table where source data lands | Usually bronze; often implemented as streaming tables |
| Ingestion gateway | Extracts snapshots/change logs for managed database CDC connectors | Runs on classic compute and must reach the source database |
| Staging volume | Temporary UC volume used by database CDC connectors | Buffers extracted changes before pipeline applies them |
| Lakeflow Job | Orchestrates scheduled ingestion pipeline runs | Created automatically for schedules in some managed connector flows |

Do not confuse a **connection** with a **storage credential**. A connection stores credentials for external systems such as SaaS apps and databases; a storage credential governs cloud object storage access.

## Incremental ingestion patterns

| Pattern | How changes are tracked | Good fit |
|---------|-------------------------|----------|
| SaaS managed connector | Source API cursor, updated timestamp, or connector-specific change tracking | Apps such as Salesforce or ServiceNow |
| Database CDC connector | Database change logs, binlogs, CDC tables, or change tracking | Relational databases with CDC enabled |
| Query-based connector | Cursor column / high-water mark | Databases where CDC is not available |
| Standard streaming connector | Streaming offsets/checkpoints | Kafka, Kinesis, Pub/Sub, Pulsar |

First run usually performs an initial load or snapshot. Later runs ingest only new or changed records when the connector/source supports incremental behavior.

## SCD and history tracking

| Mode | Meaning | Use when |
|------|---------|----------|
| `SCD_TYPE_1` | Latest value overwrites older value | You only need current state |
| `SCD_TYPE_2` | Keeps history of row changes with version metadata | You need historical changes for auditing or point-in-time analysis |
| `APPEND_ONLY` | Appends each ingested row without merging | Source is append-only or you want raw event history |

Not every connector supports every history tracking mode, so exam answers should respect connector capability if the scenario states a limitation.

## PySpark / API essentials

Lakeflow Connect ingestion is mostly UI/API/bundle-driven. In PySpark, you usually consume the synced bronze tables.

```python
# Read data landed by a managed connector.
accounts = spark.table("main.bronze.salesforce_accounts")

# Example downstream silver cleanup after ingestion.
clean_accounts = (
    accounts
    .dropDuplicates(["Id"])
    .withColumnRenamed("Id", "account_id")
)

clean_accounts.write.mode("overwrite").saveAsTable("main.silver.accounts")
```

```python
# Pattern for validating a connector landed data.
landed = spark.table("main.bronze.salesforce_accounts")

landed.printSchema()
print(landed.count())
```

Column names for connector metadata vary by connector and destination mode, so check the created table schema instead of memorizing one universal metadata column.

## UI / product surfaces

- **Add data / Data ingestion**: start managed connector flows from the Databricks UI.
- **Jobs & Pipelines**: create or monitor ingestion pipelines and schedules.
- **Catalog Explorer**: create/manage Unity Catalog connections, inspect destination tables, owners, permissions, and lineage.
- **Lakeflow Jobs**: view scheduled pipeline runs and task status.
- **Pipeline event logs / health metrics**: troubleshoot ingestion failures, retries, schema changes, and data quality.
- **System tables**: monitor usage and cost where enabled, such as `system.billing.usage`.

## Exam traps

- Choosing Lakeflow Connect for generic cloud file drops. Use Auto Loader or `COPY INTO` for S3/ADLS/GCS file ingestion.
- Writing custom JDBC/REST notebook code when a managed connector exists and meets the requirement.
- Storing source credentials in plaintext notebook variables. Use Unity Catalog connections, secrets, or connector-managed auth.
- Confusing **query federation** with **managed ingestion**. Federation queries external data in place; Lakeflow Connect ingests data into Delta tables.
- Assuming every connector supports incremental ingestion or SCD type 2. Capability is connector-dependent.
- Forgetting that database CDC connectors need network connectivity from the ingestion gateway to the source database.
- Forgetting serverless compute requirements for managed ingestion pipelines.
- Landing data outside Unity Catalog when the scenario requires lineage, ACLs, and governed tables.
- Using partner connectors as the default answer when a first-party Lakeflow Connect connector is listed.
- Assuming deletes from a source always physically delete destination data. Delete/history behavior depends on connector and SCD mode.

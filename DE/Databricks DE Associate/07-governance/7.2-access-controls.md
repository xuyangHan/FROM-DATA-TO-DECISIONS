# 7.2 GRANT / REVOKE / DENY

## Exam objective

> Configure access controls using the UI and SQL by applying GRANT, REVOKE, and DENY privileges to principals (users, groups, and service principals) at appropriate levels of the security hierarchy.

## Must know

- Unity Catalog uses privileges on securable objects.
- Common principals are **users**, **groups**, and **service principals**.
- Grant to **groups** for humans and **service principals** for production jobs.
- `GRANT` adds privileges.
- `REVOKE` removes privileges.
- `SHOW GRANTS` inspects privileges.
- In Unity Catalog, users have no data access by default unless access is granted directly, inherited, or implied by ownership.
- **Important current-docs correction**: `DENY` is not supported for Unity Catalog objects. `DENY` applies to legacy `hive_metastore` objects. If an exam scenario says UC and `DENY`, treat it as a trap and use least privilege, `REVOKE`, groups, table grants, row filters, or masks instead.
- Access to a table usually requires both object privilege and usage privileges on parents.

## Security hierarchy

Unity Catalog object hierarchy:

```text
metastore
  catalog
    schema
      table / view / materialized view / streaming table
      volume
      function
```

Three-level table names:

```text
catalog.schema.table
```

Privilege inheritance:

- Grants on a **catalog** inherit to current and future schemas and child objects where applicable.
- Grants on a **schema** inherit to current and future tables, views, volumes, and functions in that schema.
- Metastore-level grants do **not** grant data access to all child data objects.
- Parent usage privileges are still required to reach child objects.

## Common privileges

| Privilege | Meaning |
|-----------|---------|
| `USE CATALOG` | Can access/catalog-name-resolve objects in a catalog |
| `USE SCHEMA` | Can access/name-resolve objects in a schema |
| `SELECT` | Can read tables and views |
| `MODIFY` | Can write/modify table data and many table properties |
| `CREATE SCHEMA` | Can create schemas in a catalog |
| `CREATE TABLE` | Can create tables in a schema or inherited scope |
| `CREATE VOLUME` | Can create volumes |
| `CREATE FUNCTION` | Can create functions |
| `EXECUTE` | Can execute functions/models |
| `READ VOLUME` | Can read files in a volume |
| `WRITE VOLUME` | Can write files in a volume |
| `BROWSE` | Can discover metadata without data access |
| `APPLY TAG` | Can assign tags to objects |
| `MANAGE` | Can manage privileges, transfer ownership, and drop without being owner |
| `ALL PRIVILEGES` | All applicable privileges for the object type, but not every special privilege |

Important: `ALL PRIVILEGES` does not include `MANAGE`, `EXTERNAL USE SCHEMA`, or `EXTERNAL USE LOCATION`.

## Required privilege patterns

| Operation | Typical required privileges |
|-----------|-----------------------------|
| Read table/view | `USE CATALOG`, `USE SCHEMA`, `SELECT` |
| Write existing table | `USE CATALOG`, `USE SCHEMA`, `MODIFY` |
| Create table | `USE CATALOG`, `USE SCHEMA`, `CREATE TABLE` |
| Execute function | `USE CATALOG`, `USE SCHEMA`, `EXECUTE` |
| Read volume files | `USE CATALOG`, `USE SCHEMA`, `READ VOLUME` |
| Create external table | Parent catalog/schema privileges plus `CREATE EXTERNAL TABLE` on external location |
| Manage grants on object | Ownership, `MANAGE`, parent owner, or metastore admin depending on object |

## Least privilege choices

| Scenario | Better grant |
|----------|--------------|
| Analysts read one table | `SELECT` on that table plus parent `USE` privileges |
| Analysts read every current/future table in one schema | `SELECT` on schema plus `USE CATALOG`, `USE SCHEMA` |
| BI users should discover objects but not read data | `BROWSE` on catalog |
| Job writes one silver table | `MODIFY` on that table plus parent `USE` privileges |
| Job creates many silver tables | `CREATE TABLE` and `USE SCHEMA` on `main.silver` |
| Team manages permissions for a schema | `MANAGE` on schema plus parent `USE` privileges |

## SQL essentials

### Grant read access to one table

```sql
GRANT USE CATALOG ON CATALOG main TO `analysts`;
GRANT USE SCHEMA ON SCHEMA main.gold TO `analysts`;
GRANT SELECT ON TABLE main.gold.revenue TO `analysts`;
```

### Revoke read access

```sql
REVOKE SELECT ON TABLE main.gold.revenue FROM `analysts`;
```

`REVOKE` succeeds even if the privilege was not present. Think of it as making sure the grant is absent.

### Show grants

```sql
SHOW GRANTS ON TABLE main.gold.revenue;
SHOW GRANTS `analysts` ON TABLE main.gold.revenue;
SHOW GRANTS ON SCHEMA main.gold;
```

### Grant schema-wide read access

```sql
GRANT USE CATALOG ON CATALOG main TO `finance_analysts`;
GRANT USE SCHEMA ON SCHEMA main.gold TO `finance_analysts`;
GRANT SELECT ON SCHEMA main.gold TO `finance_analysts`;
```

This covers current and future tables/views in the schema.

### Grant job service principal access

```sql
GRANT USE CATALOG ON CATALOG main TO `spn_lakeflow_jobs`;
GRANT USE SCHEMA ON SCHEMA main.silver TO `spn_lakeflow_jobs`;
GRANT CREATE TABLE ON SCHEMA main.silver TO `spn_lakeflow_jobs`;
GRANT MODIFY ON SCHEMA main.silver TO `spn_lakeflow_jobs`;
```

For a job that only writes an existing table, grant `MODIFY` on that table instead of the whole schema.

### Legacy DENY example - not for Unity Catalog

```sql
-- Legacy hive_metastore pattern only. Not supported for UC objects.
DENY SELECT ON TABLE hive_metastore.default.some_table TO `contractors`;

-- Remove a legacy DENY with REVOKE on the same privilege/object/principal.
REVOKE SELECT ON TABLE hive_metastore.default.some_table FROM `contractors`;
```

For Unity Catalog, restrict access by not granting access, revoking inherited/direct grants, splitting groups, or applying row filters/column masks.

## Ownership vs MANAGE

| Concept | Key point |
|---------|-----------|
| Owner | Has all capabilities on the object and can grant/revoke, transfer ownership, and drop |
| `MANAGE` | Can manage privileges and ownership/drop behavior, but does not automatically grant data access |
| `ALL PRIVILEGES` | Does not include `MANAGE` |
| Ownership inheritance | Ownership does not inherit, but owners of containers can manage child objects |

Exam shortcut: a user with `MANAGE` can manage permissions, but may still need `USE CATALOG` / `USE SCHEMA` to exercise that management.

## PySpark / API essentials

Access control commands are SQL commands. In notebooks, use `spark.sql` if your current principal has the needed administrative privilege.

```python
spark.sql("GRANT SELECT ON TABLE main.gold.revenue TO `analysts`")
spark.sql("SHOW GRANTS ON TABLE main.gold.revenue").show(truncate=False)
```

Consumers do not need storage credentials to read UC tables by name. They need UC privileges.

```python
df = spark.table("main.gold.revenue")
```

## UI / product surfaces

- **Catalog Explorer > object > Permissions**: grant and revoke privileges.
- **Catalog Explorer > object > Overview**: owner and metadata.
- **Catalog Explorer > Metastore**: metastore-level privileges.
- **SQL editor / notebook**: `GRANT`, `REVOKE`, `SHOW GRANTS`.
- **Account console / identity provider**: manage users, groups, and service principals.

## Exam traps

- Granting `SELECT` on a table without `USE CATALOG` and `USE SCHEMA`.
- Granting broad catalog/schema privileges when table-level access is enough.
- Using a personal user/token as the production job identity instead of a service principal.
- Assuming `DENY` works on Unity Catalog objects.
- Assuming `REVOKE` blocks access if the user still has inherited access through another group.
- Forgetting group membership: a user may regain access through another group.
- Thinking `MANAGE` automatically includes `SELECT`.
- Thinking `ALL PRIVILEGES` includes `MANAGE`.
- Granting access to external storage directly when the user only needs UC table access.

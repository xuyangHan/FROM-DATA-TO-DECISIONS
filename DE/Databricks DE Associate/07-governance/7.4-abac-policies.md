# 7.4 Unity Catalog ABAC policies

## Exam objective

> Understand Unity Catalog ABAC policies to centrally control row-level filtering and column masking for sensitive data.

## Must know

- ABAC means **attribute-based access control**.
- In Unity Catalog, the attributes are **governed tags** on securable objects.
- ABAC policies can apply **row filters** or **column masks** centrally.
- Policies attach at the catalog, schema, or table level.
- Policies match target tables/columns dynamically using tag conditions.
- ABAC is the scalable choice when the same masking/filtering rule must apply across many tables.
- ABAC does not replace `GRANT`. Users still need base object privileges such as `USE CATALOG`, `USE SCHEMA`, and `SELECT`.
- ABAC policies use governed tags, not ordinary ungoverned tags.

## Why ABAC exists

Table-level row filters and column masks work well for a few tables:

```sql
ALTER TABLE main.gold.customers
ALTER COLUMN email SET MASK main.security.mask_email;
```

But this becomes hard to maintain if hundreds of columns across many schemas have the same policy requirement.

ABAC lets governance teams define one policy:

```text
For all columns tagged pii=email in this catalog,
mask values for everyone except pii_readers.
```

When new tagged columns appear, the policy applies automatically.

## ABAC vs table-level masks/filters

| Feature | ABAC policy | Table-level row filter / mask |
|---------|-------------|-------------------------------|
| Scope | Catalog, schema, or table | Individual table |
| Matching | Governed tags and conditions | Explicit `ALTER TABLE` |
| Scales to many tables | Yes | Harder |
| Controlled centrally | Yes | Usually table owner |
| Applies to new tagged assets | Yes | No, must configure each table |
| Requires base table grants | Yes | Yes |
| Best for | Enterprise-wide sensitive data policy | Small number of table-specific rules |

Exam shortcut: **many tables + same sensitive classification + central control = ABAC**.

## Core components

| Component | Purpose |
|-----------|---------|
| Governed tag | Account-level controlled tag key/value such as `pii = email` |
| Tag assignment | Apply governed tags to catalogs, schemas, tables, or columns |
| Policy scope | Where the policy is attached: catalog, schema, or table |
| Table condition | Which tables in scope match the policy |
| Column condition | Which columns in matching tables are targeted |
| Policy UDF | SQL function that implements mask/filter logic |
| Principals | Users/groups/service principals the policy applies to |
| Exceptions | Principals exempt from the policy |

## Governed tags

Governed tags are centrally defined tags with controlled values and permissions.

Key points:

- Governed tag definitions are account-level.
- Assigning a governed tag requires permission on the tag, plus object/tag privileges.
- Tags applied to catalogs or schemas can be inherited by child objects for ABAC evaluation.
- Column tags must be applied directly to columns; table tags do not automatically become column tags.
- Tag changes can take a few minutes to take effect.
- Do not put sensitive values in tag names or values because tag metadata is stored as plain text.

Example tag assignment:

```sql
ALTER TABLE main.gold.customers
ALTER COLUMN email
SET TAGS ('pii' = 'email');

ALTER TABLE main.gold.customers
SET TAGS ('sensitivity' = 'high');
```

Newer runtimes also support `SET TAG` syntax:

```sql
SET TAG ON COLUMN main.gold.customers.email pii = email;
SET TAG ON TABLE main.gold.customers sensitivity = high;
```

## Policy types

| Policy type | What it does | UDF return |
|-------------|--------------|------------|
| Row filter policy | Excludes rows from query results | `BOOLEAN` |
| Column mask policy | Replaces column values | Same/castable type as target column |

## SQL: column mask ABAC policy

Example idea: mask all columns tagged `pii = ssn` for all users except HR admins.

```sql
CREATE OR REPLACE FUNCTION main.security.ssn_last4(ssn STRING)
RETURNS STRING
RETURN concat('***-**-', right(ssn, 4));

CREATE POLICY mask_ssn
ON CATALOG main
COLUMN MASK main.security.ssn_last4
TO 'All Users'
EXCEPT 'hr_admins'
FOR TABLES
MATCH COLUMNS has_tag_value('pii', 'ssn') AS ssn
ON COLUMN ssn;
```

Meaning:

- Policy scope is catalog `main`.
- It applies to the listed principals.
- HR admins are exempt.
- It searches for columns tagged `pii = ssn`.
- Matching columns are masked by the UDF.

## SQL: row filter ABAC policy

Example idea: hide EU rows from a group on high-sensitivity tables that have a column tagged as geographic region.

```sql
CREATE OR REPLACE FUNCTION main.security.non_eu_region(region STRING)
RETURNS BOOLEAN
RETURN region <> 'EU';

CREATE POLICY hide_eu_customers
ON SCHEMA main.gold
COMMENT 'Hide EU rows from analysts on high sensitivity tables'
ROW FILTER main.security.non_eu_region
TO `non_eu_analysts`
FOR TABLES
WHEN has_tag_value('sensitivity', 'high')
MATCH COLUMNS has_tag('geo_region') AS region
USING COLUMNS (region);
```

Meaning:

- Policy scope is schema `main.gold`.
- It applies only to tables tagged `sensitivity = high`.
- It finds a column tagged `geo_region`.
- It passes that column into the filter UDF.
- Rows where the UDF returns false are excluded.

Unity Catalog supports at most one effective row filter function for a table for a given user. If a table needs multiple row-level security conditions, such as both region and department restrictions, combine the logic into one SQL UDF:

```sql
CREATE OR REPLACE FUNCTION main.security.allowed_transaction(
  region STRING,
  department STRING
)
RETURNS BOOLEAN
RETURN region = current_user_region()
   AND department = current_user_department();

ALTER TABLE main.gold.transactions
SET ROW FILTER main.security.allowed_transaction
ON (region, department);
```

Exam shortcut: **multiple row restrictions on one table = one row filter UDF with combined `AND` logic**.

## Permissions and compute requirements

To create/manage ABAC policies:

- Own the policy scope object or have `MANAGE` on it.
- Have `EXECUTE` on the policy UDF, unless defining inline logic where supported.
- Use governed tags on target objects.
- Use serverless compute or a supported Databricks Runtime for ABAC policy operations.

Current docs list ABAC compute support as:

- Serverless compute.
- Standard compute on Databricks Runtime 16.4+.
- Dedicated compute on Databricks Runtime 16.4+ with fine-grained access control filtering enabled.

## UI / product surfaces

- **Catalog Explorer > catalog/schema/table > Policies**: create, edit, view, and delete ABAC policies.
- **Catalog Explorer > object > Tags**: apply governed tags.
- **Catalog Explorer > column metadata**: apply column tags.
- **SQL editor / notebook**: `CREATE POLICY`, `SHOW POLICIES`, `DESCRIBE POLICY`, `DROP POLICY`.
- **Account/admin tag management**: define governed tags and allowed values.

## Inspect applied masks and filters

Use `DESCRIBE EXTENDED` or its synonym `DESCRIBE TABLE EXTENDED` to inspect detailed Unity Catalog table metadata:

```sql
DESCRIBE EXTENDED main.gold.customers;

DESCRIBE TABLE EXTENDED main.gold.customers;
```

This output can show table properties, including applied row filters and column masking functions. Row filters typically appear in the table properties section, while masking functions are listed alongside the specific column definitions.

## Design workflow

1. Define the governed tag taxonomy, such as `pii`, `sensitivity`, `geo_region`.
2. Grant the right teams permission to assign governed tags.
3. Create simple SQL UDFs for masking or filtering.
4. Create ABAC policies at the highest appropriate scope.
5. Tag tables and columns.
6. Test access with privileged and restricted groups.
7. Monitor for untagged sensitive columns and policy conflicts.

## What ABAC does not do

- It does not grant `SELECT`.
- It does not replace catalog/schema/table privileges.
- It does not secure direct access to raw cloud files outside Databricks.
- It does not work if assets are not tagged or are tagged with the wrong key/value.
- It does not remove the need to test query behavior and performance.

## Exam traps

- Choosing per-table masks when the scenario says hundreds of tables or enterprise-wide policy.
- Forgetting base grants: ABAC restricts users who can already query, but does not grant access.
- Using ungoverned tags instead of governed tags.
- Tagging the table but forgetting to tag the sensitive column for a column mask policy.
- Assuming column tags inherit from table tags.
- Creating a policy at too narrow a scope, then wondering why new schemas are not covered.
- Forgetting exception groups for administrators, compliance, or policy function owners.
- Trying to apply separate row filters for region and department on the same table instead of combining both rules in one filter UDF.
- Using complex UDFs when a simple SQL function would be easier to optimize and maintain.
- Ignoring runtime/serverless requirements for ABAC-secured tables.
